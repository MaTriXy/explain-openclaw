#### Configurable Context Visibility ([`35e160514`](https://github.com/openclaw/openclaw/commit/35e1605147))

Adds a `contextVisibility` configuration field across all messaging channels, controlling how supplemental context — quoted messages, thread history, and forwarded content — is filtered before reaching the AI model. Three modes: `"all"` (include all supplemental context regardless of sender), `"allowlist"` (only include context from senders in the channel allowlist), and `"allowlist_quote"` (allowlist filtering plus always preserve explicit quote/reply context). The evaluation logic lives in `src/security/context-visibility.ts` via `evaluateSupplementalContextVisibility()`, `shouldIncludeSupplementalContext()`, and `filterSupplementalContextItems()`. Per-channel and per-account override resolution is handled by `resolveChannelContextVisibilityMode()` in `src/config/context-visibility.ts`. All functions are exposed to channel plugins through `src/plugin-sdk/security-runtime.ts` and `src/plugin-sdk/config-runtime.ts`. The companion commit `694d12a90b` applies the new field across all 12 channel provider configs. Thanks @steipete.

**New security control.** Operators can now prevent supplemental context from non-allowlisted senders from reaching the AI model. Configurable at the global `channels.defaults.contextVisibility`, per-channel, and per-account level.

#### Sanitize Plugin Activation Reasons in CLI Output ([`f911bbc35`](https://github.com/openclaw/openclaw/commit/f911bbc353))

Renames the old resolveEffectiveEnableState → `resolveEffectivePluginActivationState` (defined in `src/plugins/config-state.ts`) across 10 production files (`.enabled` → `.activated`) as a semantic rename with no behavior change. The security-relevant addition is in `src/cli/plugins-cli.ts`: the `activationReason` field from plugin records is now passed through `sanitizeTerminalText()` (from `src/terminal/safe-text.ts`) before being displayed in `openclaw plugins list --verbose` output. This strips ANSI escape codes and all C0/C1 control characters, and converts embedded newlines and tabs to visible `\n`/`\t` literals. A dedicated test case (`"sanitizes activation reasons in verbose output"`) verifies that `\u001B[31m` escape codes and embedded `\n`/`\t` in `activationReason` are correctly sanitized. Thanks @vincentkoc.

**New security control.** Plugin metadata from untrusted sources (third-party plugins) can no longer inject ANSI escape sequences or control characters into terminal output via the verbose plugins list.

#### Preserve Restrictive Tool Allowlists ([`193fdd6e3`](https://github.com/openclaw/openclaw/commit/193fdd6e3b))

Fixes a bug in `analyzeAllowlistByToolType()` in `src/agents/tool-policy.ts` where a tool allowlist containing only plugin tools was silently stripped — returning `{ ...policy, allow: undefined }` — which effectively replaced a restrictive allowlist with allow-all behavior. The function was renamed from the old stripPluginOnlyAllowlist. The root cause was a logic inversion: the old implementation tracked a hasCoreEntry flag and stripped when false; it would incorrectly strip allowlists that contained unknown or plugin-only tools even when the operator intended them to be restrictive. The fix changes the policy return to always preserve the allowlist as-is (`policy` instead of `{ ...policy, allow: undefined }`); `pluginOnlyAllowlist` is now an informational flag only (renamed from the old strippedAllowlist variable). Operators who set `tools.allow: ["plugin:mytool"]` to restrict agent tool access to specific plugins would have had those restrictions silently discarded prior to this fix. Thanks @eleqtrizit, @david.silva, @drobison.

**Addresses CWE-269 (Improper Privilege Management).** Restrictive tool allowlists are now correctly enforced rather than silently stripped.

#### Restrict iOS A2UI Action Dispatch to Trusted Canvas URLs ([`49d08382a`](https://github.com/openclaw/openclaw/commit/49d08382a9))

Adds trust-boundary enforcement for the A2UI bridge on iOS. A new `trustedRemoteA2UIURL: URL?` property on `ScreenController` is set only when `navigate(to:trustA2UIActions:true)` is explicitly called. The `normalizeTrustedRemoteA2UIURL()` helper normalizes the trusted URL by lowercasing scheme and host and stripping URL fragments, preventing trivial bypass via case variation or fragment injection. The `isTrustedCanvasUIURL()` method is updated to return `true` for remote URLs only when the normalized request URL matches the stored trusted origin; without a stored trusted origin, all remote A2UI bridge actions are rejected. Local file-based canvas URLs (`isFileURL`) continue to use the existing `canvasScaffoldURL` path. Thanks @eleqtrizit, @drobison.

**Cross-reference: [GHSA-4p4f-fc8q-84m3](https://github.com/openclaw/openclaw/security/advisories/GHSA-4p4f-fc8q-84m3).** Closes the cross-origin A2UI action injection path on the iOS canvas bridge.

#### Gate Gateway Snapshot Metadata to Admin-Scoped Clients ([`676b74805`](https://github.com/openclaw/openclaw/commit/676b748056))

The `buildGatewaySnapshot()` call in `src/gateway/server/ws-connection/message-handler.ts:1049` now passes `{ includeSensitive: scopes.includes(ADMIN_SCOPE) }`. Non-admin clients (operator, operator.approvals, etc.) receive a redacted snapshot with sensitive operational metadata filtered out. The matching `includeSensitive` gate in `src/gateway/server/health-state.ts` controls what data is included for the two privilege tiers. New tests in `src/gateway/server.auth.browser-hardening.test.ts` cover the admin vs. non-admin snapshot content boundary. Thanks @eleqtrizit, @drobison.

**Cross-reference: [GHSA-2f7j-rp58-mr42](https://github.com/openclaw/openclaw/security/advisories/GHSA-2f7j-rp58-mr42).** Sensitive gateway operational state is now gated to admin-scope clients only.
