#### Validate webhook secret before reading request body ([`7e49e98f7`](https://github.com/openclaw/openclaw/commit/7e49e98f79))

The Telegram webhook handler now validates the `x-telegram-bot-api-secret-token` header *before* reading or buffering the request body. New helper `hasValidTelegramWebhookSecret()` performs a constant-time comparison using `timingSafeEqual` to prevent timing-based secret guessing. When the header is absent or does not match the configured token, the request is rejected with a 403 response before any body I/O occurs. Previously, the body was buffered first and the secret checked afterward, allowing unauthenticated callers to exhaust server resources by sending oversized request bodies. The `secret` parameter remains typed as optional but is enforced as mandatory at runtime — a missing or empty configuration causes the server to throw at startup rather than accepting requests. Addresses GHSA-jq3f-vjww-8rq7 (HIGH: Telegram webhook unauthenticated resource exhaustion via pre-body validation bypass).

#### Make device bootstrap tokens single-use to prevent scope escalation ([`1803d16d5`](https://github.com/openclaw/openclaw/commit/1803d16d5c))

Device bootstrap setup codes are now single-use. `verifyDeviceBootstrapToken()` deletes the token record from persisted state before returning a success result: after locating a matching entry, it removes the entry with `delete state[entry.token]` and calls `persistState()` to flush the deletion to disk, then returns `{ ok: true }`. Previously the record remained in place after a successful verification, allowing the same code to be replayed — an attacker who obtained a bootstrap setup code could re-submit it to mutate the scope of an existing pending pairing request, potentially escalating privileges on an already-paired device. Addresses GHSA-63f5-hhc7-cx6p (HIGH: Bootstrap setup code replay enables scope escalation on pending pairing requests).
