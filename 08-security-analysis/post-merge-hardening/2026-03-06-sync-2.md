#### Enforce Prompt Hook Policy with Runtime Validation ([`688b72e15`](https://github.com/openclaw/openclaw/commit/688b72e15))

Introduces per-plugin hook policy controls via the `allowPromptInjection` flag in `plugins.entries.*.hooks` configuration. The `constrainLegacyPromptInjectionHook` wrapper intercepts `before_agent_start` handlers and calls `stripPromptMutationFieldsFromLegacyHookResult` to block prompt mutation fields while preserving legacy `modelOverride`/`providerOverride` behavior. `isPromptInjectionHookName` classifies hooks into prompt-injection and model-override categories, enabling fine-grained access control at plugin load time.

**Audit 2 Claim 5 impact:** This directly implements the RBAC control layer for plugin hooks. Plugin authors can now selectively enable prompt injection capabilities per-hook via the `allowPromptInjection` config flag; without it, `before_agent_start` hook results have prompt mutation fields stripped before they reach the agent runner.

#### Decouple Discord Inbound Worker Timeout from Listener Timeout ([`063e493d3`](https://github.com/openclaw/openclaw/commit/063e493d3))

Separates the Carbon gateway listener timeout (`listenerTimeout`, covering normalization and enqueue work, 120 s default) from a new inbound worker timeout (`inboundWorker.runTimeoutMs`, covering queued Discord agent turns, 30 min default). New `timeouts.ts` exports `normalizeDiscordInboundWorkerTimeoutMs` and `runDiscordTaskWithTimeout` to enforce per-worker budget independently. The refactored `DiscordInboundJobPayload` and `DiscordInboundWorker` isolate runtime references from serializable job state, preventing the previous misconfiguration where the gateway listener timeout capped long-running agent work.

**Resilience impact:** Prevents agent turns from being incorrectly terminated by the gateway's short-lived listener timeout. There is no direct audit claim linkage; this is defense-in-depth for availability and predictable execution budget enforcement.

#### Compaction/Safeguard: Add Summary Quality Audit Retries ([`036c32971`](https://github.com/openclaw/openclaw/commit/036c32971))

Adds an optional quality-audit retry loop for safeguard compaction summaries via `qualityGuard.enabled` and `qualityGuard.maxRetries` config. New `extractOpaqueIdentifiers` and `auditSummaryQuality` functions validate summary structure and detect identifier loss between original context and compacted output. New `wrapUntrustedPromptDataBlock` sanitizes and wraps untrusted compaction results in `<untrusted-text>` markers with HTML entity encoding, hardening the session context against prompt injection via manipulated compaction summaries.

**Audit 2 Claim 1 impact:** Directly strengthens the compaction injection hardening referenced in Audit 2 Claim 1. The `wrapUntrustedPromptDataBlock` wrapper is the primary enforcement point — any compaction summary that cannot be verified as structurally sound is wrapped and neutralized before reaching the agent's context window.

#### Grant senderIsOwner for Internal Channels with operator.admin Scope ([`60d33637d`](https://github.com/openclaw/openclaw/commit/60d33637d))

Adds `senderIsOwnerByScope` logic in `resolveCommandAuthorization` that grants owner-level authorization when `isInternalMessageChannel(ctx.Provider)` is true AND `ctx.GatewayClientScopes` includes `'operator.admin'`. The resulting `senderIsOwner` is now `senderIsOwnerByIdentity || senderIsOwnerByScope`, ensuring internal channel operators with the admin scope receive owner privileges. External channels (Telegram, Discord, etc.) are unaffected — the scope-based grant applies only to internal/webchat providers.

**Audit 2 Claim 5 impact (escalated):** Intentional extension of authorization logic. The change is narrowly scoped: scope-based elevation requires both an internal channel AND the `operator.admin` client scope, preventing external channel actors from gaining elevated access via scope alone. The escalated classification reflects that this touches the ownership determination path in `resolveCommandAuthorization`.

#### Bump dompurify to 3.3.2 ([`999b7e4ed`](https://github.com/openclaw/openclaw/commit/999b7e4ed))

Security patch bumping dompurify from 3.3.1 to 3.3.2 in the web UI layer (`ui/package.json` + lockfile). dompurify is the primary HTML sanitization library protecting against XSS in the control UI. No application code changes — dependency version bump only.

**Escalated:** XSS dependency hardening. The control UI renders user-generated content through dompurify; keeping this library current closes any vulnerabilities patched between minor versions.

#### Propagate mediaLocalRoots Through Slack Send Path ([`7830366f3`](https://github.com/openclaw/openclaw/commit/7830366f3))

Restores local media allowlist enforcement in the Slack send path by threading `mediaLocalRoots` through the call chain: `slack-actions.ts` → `slack.actions.ts` → `slack/actions.ts`. Without this fix, local file upload allowlist validation was bypassed in the Slack path — `sendMessageSlack` did not receive the allowlist needed for upload path verification.

**Escalated:** Allowlist enforcement restoration. This is a correctness fix for a CVE-era security control that was silently not applied to the Slack code path. The fix is purely additive (propagating an existing parameter) with no behavior change on non-Slack paths.
