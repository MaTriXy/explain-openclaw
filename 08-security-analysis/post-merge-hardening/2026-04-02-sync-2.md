#### Reject mixed trusted-proxy token config ([`6c679e5f0`](https://github.com/openclaw/openclaw/commit/6c679e5f049a289233716f0ffe3ea314497ff671))

`src/gateway/auth.ts` now rejects configurations where `trusted-proxy` mode is mixed with a bearer token setup, failing closed rather than allowing the ambiguous auth configuration to proceed. Also adds fail-closed behavior for loopback trusted-proxy auth, so a gateway listening on loopback with trusted-proxy mode but without a valid token config fails authentication rather than granting implicit access. Co-authored with @boy-hack (#58371).

#### Filter fetched Matrix room context by sender allowlist ([`8a563d603`](https://github.com/openclaw/openclaw/commit/8a563d603b70ef6338915f0527bee87282c3bad5))

Matrix handler in `extensions/matrix/src/matrix/monitor/handler.ts` now applies sender allowlist checks (`allowFrom`, per-room `allowFrom`) to messages fetched when building thread and reply context for AI processing. Previously, messages from senders blocked by the allowlist could appear in fetched context even though their direct messages would be rejected. An attacker in a shared Matrix room could send messages that bypass allowlist enforcement by appearing only in context, not as a direct message. Also drops raw sender IDs from allowlist context logs to reduce data exposure. **Corresponds to GHSA-rg8m-3943-vm6q.**

#### Fail closed on dangerous skill installs ([`0d7f1e2c8`](https://github.com/openclaw/openclaw/commit/0d7f1e2c84eca65df7dee890d9c30e2a841c030a))

When `src/plugins/install-security-scan.runtime.ts` encounters a scan error or returns a `security_scan_blocked` / `security_scan_failed` verdict, the install now fails closed rather than proceeding (fail-open). The `InstallSecurityScanResult.blocked` field (at `src/plugins/install-security-scan.ts:9-14`) captures both the block code and reason. The macOS app (`GatewayConnection.swift`) adds a user-facing confirmation prompt for dangerous installs. Companion `44b9936136` adds an explicit `--unsafe-install` opt-in for operators who need to override. **Corresponds to GHSA-cwq8-6f96-g3q4.**

#### Rebind hook agent session keys to the target agent ([`1ca12ec8b`](https://github.com/openclaw/openclaw/commit/1ca12ec8bf16f07c6fde6f40e0a4ecab45eaccf8))

Hook agents in `src/gateway/hooks.ts` now rebind their session keys to the target agent's key rather than inheriting the hook dispatcher's session key. Additionally, `src/gateway/server-http.ts` validates normalized dispatch keys on the HTTP path. Without this fix, a hook agent could operate with session credentials scoped to the dispatcher agent, potentially accessing session state from a different agent's context or impersonating the wrong principal in tool policy evaluation.

#### Reject shell init-file script matches in exec approvals ([`0c8375424`](https://github.com/openclaw/openclaw/commit/0c83754246ddcd8fde36df2e36b60a6803e2b4e7))

Previously, an approved rule for `bash` could be bypassed by appending `--init-file` to load arbitrary script code at shell startup, executing outside the allow-always policy.

#### Reject oversized image inputs before decode ([`0ed4f8a72`](https://github.com/openclaw/openclaw/commit/0ed4f8a72b0cde22d84e2be6a5c61b6b2e0be05c))

#### Verify inbound Nostr DM signatures before pairing ([`4ee742174`](https://github.com/openclaw/openclaw/commit/4ee742174f3f0b2f71f5b6fdf6f21099f0cfeaa8))

`extensions/nostr/src/nostr-bus.ts` now verifies NIP-01 event signatures on inbound direct messages before using them to identify pairing reply targets. Previously, an attacker could craft an unsigned or wrongly-signed DM with a spoofed pubkey to hijack a pairing session — the gateway would accept it as a valid pairing signal and bind the session to the attacker's key instead of the legitimate sender's.

#### Prevent session death loop on overloaded fallback ([`1fcd179d8`](https://github.com/openclaw/openclaw/commit/1fcd179d8c5e1d9cbb895f25c7b8044c18b06e15))

Returns a `kind:final` response with a user-visible error on retry exhaustion rather than looping indefinitely, preventing a denial-of-service condition via resource exhaustion on heavily loaded gateways.
