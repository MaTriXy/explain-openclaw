#### Add compatible bundle support ([`dd40741e18`](https://github.com/openclaw/openclaw/commit/dd40741e18))

Plugin bundles (Codex App, Claude, Cursor format) now run a security scan before installation completes. The new `installBundleFromSourceDir()` function in `src/plugins/install.ts` calls `skillScanner.scanDirectoryWithSummary()` on the bundle directory. If any `critical`-severity findings are detected (e.g., dangerous code patterns that could be used in a malicious setupCommand), a warning is logged naming the file and line. If only `warn`-severity patterns are found, users are directed to `openclaw security audit --deep`. The scan runs before the bundle is activated, giving operators a chance to reject suspicious bundles.

New types `BundlePluginManifest` and loader `loadBundleManifest()` in `src/plugins/bundle-manifest.ts` validate bundle structure and capabilities before the install proceeds. The manifest loader also enforces `rejectHardlinks: true` to prevent hardlink-based escape attempts.

#### Fix bundled plugin roots and skill assets ([`47fd8558cd`](https://github.com/openclaw/openclaw/commit/47fd8558cd))

`ACPX_PLUGIN_ROOT` in `extensions/acpx/src/config.ts` was previously computed as `path.resolve(path.dirname(fileURLToPath(import.meta.url)), "..")`, which is correct for the source layout (`src/config.ts` → `extensions/acpx/`) but wrong when bundled (the entry file lives at the plugin root, making `..` point to the parent directory).

The fix introduces `resolveAcpxPluginRoot()`, which walks up the directory tree up to 3 levels looking for both `openclaw.plugin.json` and `package.json` before settling on a root. If no valid root is found, it falls back to the old behaviour. This ensures plugin-local npm installs (e.g., `npm install acpx@version`) run inside `dist/extensions/acpx/` rather than the parent directory, keeping plugin install scope correctly contained.

#### Harden context engine ownership ([`7931f06c00`](https://github.com/openclaw/openclaw/commit/7931f06c00))

Plugin context engine registration in `src/plugins/registry.ts` now uses `registerContextEngineForOwner()` instead of the open `registerContextEngine()`. Each plugin's engine registration is tagged with `plugin:${record.id}` as the owner, and `allowSameOwnerRefresh: true` permits a plugin to re-register its own engine (e.g., on hot-reload) without allowing a different owner to claim the same slot.

Previously, a plugin could import `registerContextEngine` directly from the context-engine module SDK and register an engine under any ID, including IDs already claimed by privileged system components. The ownership enforcement closes this escalation path — an attempt by plugin B to claim plugin A's engine slot is now rejected at registration time.
