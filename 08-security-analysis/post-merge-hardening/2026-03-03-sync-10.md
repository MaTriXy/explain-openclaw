#### Suppress NO_REPLY token in chat transcript rendering ([`0743463b8`](https://github.com/openclaw/openclaw/commit/0743463b8))

The `NO_REPLY` sentinel token is used internally to suppress delivery when the agent decides not to respond (e.g., in cron heartbeat mode). Before this fix, a `NO_REPLY`-only assistant message could survive into the webchat transcript history passed to the model, causing the token to appear verbatim in rendered chat transcripts.

`sanitizeChatHistoryMessages()` now iterates the transcript and calls `extractAssistantTextForSilentCheck()` on each assistant entry to obtain its visible text — handling both flat `text`/`content` fields and structured content-block arrays. If `isSilentReplyText()` confirms the extracted text is entirely the `SILENT_REPLY_TOKEN` sentinel (`NO_REPLY`), the message is dropped from the transcript before it reaches the client. The function returns the original slice unchanged when no messages are dropped, avoiding a copy on the common path.

#### Wire telegram disableAudioPreflight config validation and precedence tests ([`1fa2488db`](https://github.com/openclaw/openclaw/commit/1fa2488db))

The `disableAudioPreflight` option lets operators turn off the audio attachment preflight check (a round-trip that verifies the voice file is valid before committing to processing it). This commit wires the config field into `TelegramTopicSchema` and `TelegramGroupSchema` so the option can be set at the topic, group, or global level.

Precedence is resolved by `firstDefined()`: topic-level config overrides group-level, which overrides the global default. In `bot-message-context.ts`, the `disableAudioPreflight` value is derived from `firstDefined(topicConfig?.disableAudioPreflight, groupConfig?.disableAudioPreflight, ...)`. This ensures that a per-topic opt-out takes effect without requiring a group-wide or global change. Validation tests confirm the precedence chain and flag invalid combinations at config load time.

#### Prevent Windows PATH from poisoning docker exec ([`f257818ea`](https://github.com/openclaw/openclaw/commit/f257818ea))

Audit 2 Claim 8 identified that env variable injection — particularly `LD_PRELOAD` — could influence Docker exec behavior. This commit addresses a related vector: host `PATH` leaking into `docker exec` calls.

`buildDockerExecArgs()` now explicitly skips the `PATH` key when iterating the env map passed via `-e` flags. The code comment explains the rationale: a Windows-style `PATH` (with drive letters and backslash separators) passed directly to `docker exec -e PATH=...` poisons Docker's executable lookup, causing `/bin/sh` to be treated as not found. Instead, any custom `PATH` value is forwarded as `OPENCLAW_PREPEND_PATH` and prepended to the container's system PATH after `/etc/profile` sources — preserving system paths while still honoring operator-configured tool paths. The exec command also uses the absolute path `/bin/sh` rather than relying on the container's `PATH` to resolve the shell, removing a dependency on correct PATH state at exec time.
