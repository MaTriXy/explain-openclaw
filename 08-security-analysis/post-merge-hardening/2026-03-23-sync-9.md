#### Browser Proxy Profile Allowlist Enforcement ([`eac93507c`](https://github.com/openclaw/openclaw/commit/eac93507c3))

When a node proxy configures the allowProfiles config key, the gateway now enforces access control via three new helpers extracted into `src/browser/request-policy.ts`: `isPersistentBrowserProfileMutation()` detects profile create (`/profiles/create` POST) and delete (`/profiles/*` DELETE) operations, `resolveRequestedBrowserProfile()` extracts the requested profile name from query string or request body, and `normalizeBrowserRequestPath()` canonicalizes paths for consistent matching. Profile mutations are blocked outright when allowProfiles is configured; all other profile requests are validated against the allowlist before forwarding. This refactor centralizes least-privilege browser proxy policy and improves testability.

#### Centralize Discovery Target Handling ([`fe5819887`](https://github.com/openclaw/openclaw/commit/fe5819887b))

New module `src/infra/gateway-discovery-targets.ts` centralizes Bonjour discovery endpoint resolution. `buildGatewayDiscoveryTarget()` constructs gateway targets from resolved Bonjour beacons using `resolveGatewayDiscoveryEndpoint()`, which validates host and port from DNS SRV/A/AAAA records and fails closed if either is missing or invalid. This prefers resolved DNS records over unauthenticated TXT metadata, reducing SSRF surface from manipulated discovery announcements. Three command modules — `gateway-status`, `onboard-remote`, and `discover` — were refactored to use the centralized helpers, eliminating duplicated endpoint-picking logic.

#### Audit Synology Dangerous Name Matching ([`c42cb1ca6`](https://github.com/openclaw/openclaw/commit/c42cb1ca66))

Extended the security audit framework (`src/security/audit-channel.ts`) to detect when Synology Chat has `dangerouslyAllowNameMatching` enabled, at both the global channel level and per-account level. New helper `formatChannelAccountNote()` centralizes account-labeling logic for multi-account audit findings. This audit control surfaces risky configurations that allow username-matched messages to bypass sender verification in Synology Chat.

#### Gate Android Canvas Bridge to Trusted Pages ([`8b02ef133`](https://github.com/openclaw/openclaw/commit/8b02ef1332))

New Kotlin object `CanvasActionTrust` (`apps/android/.../node/CanvasActionTrust.kt`) implements `isTrustedCanvasActionUrl()` to validate canvas action URLs before the A2UI bridge processes them. `MainViewModel.isTrustedCanvasActionUrl()` and `NodeRuntime.isTrustedCanvasActionUrl()` delegate canvas URL trust checks to this class. Only URLs matching the trusted origin list may trigger canvas bridge actions, preventing untrusted web content from invoking node-runtime bridge operations on Android.

#### Share Local File Access Guards ([`dc90d3b1d`](https://github.com/openclaw/openclaw/commit/dc90d3b1d3))

New module `src/media/local-media-access.ts` provides `assertLocalMediaAllowed()`, which validates media paths against configured `localRoots`, rejects Windows network paths via `assertNoWindowsNetworkPath()`, and raises typed `LocalMediaAccessError` with codes `path-not-allowed`, `network-path-not-allowed`, `invalid-root`, `invalid-file-url`, and `unsafe-bypass`. Shared guards from `src/infra/local-file-access.ts` replace duplicated validation logic previously scattered across Bluebubbles, WhatsApp media, pi-tools.read, and message-action-params, eliminating five separate ad-hoc path checks and adding structured error codes.

#### Harden Secondary Local Path Seams ([`93880717f`](https://github.com/openclaw/openclaw/commit/93880717f1))

Three secondary access paths — embedded Pi image loading (`src/agents/pi-embedded-runner/run/images.ts`), sandbox path validation (`src/agents/sandbox-paths.ts`), and media-understanding attachment normalization (`src/media-understanding/attachments.normalize.ts`) — now call `assertNoWindowsNetworkPath()` and `safeFileURLToPath()` from `src/infra/local-file-access.ts`. These seams were not covered by the assertLocalMediaAllowed() refactor in `dc90d3b1d` and could have allowed Windows UNC paths or remote-host `file://` URLs to reach file reads.

#### Block Remote-Host File URLs in Loaders ([`4fd7feb0f`](https://github.com/openclaw/openclaw/commit/4fd7feb0fd))

`safeFileURLToPath()` in `src/infra/local-file-access.ts` throws "file:// URLs with remote hosts are not allowed" when a `file://` URL contains a non-local hostname (anything other than empty string or `localhost`). WhatsApp media and the web-media loader now route all `file://` URL resolution through `safeFileURLToPath()`, blocking SSRF vectors where an attacker could supply `file://attacker-host/etc/passwd`-style URLs to reach remote SMB shares or trigger SSRF via the file scheme.

#### Fail Closed on Unresolved Discovery Endpoints ([`deecf68b5`](https://github.com/openclaw/openclaw/commit/deecf68b59))

`resolveGatewayDiscoveryEndpoint()` in `src/infra/bonjour-discovery.ts` now returns `null` when host or port resolution produces an empty or zero value, instead of allowing unresolved components to propagate as invalid WebSocket URLs. Gateway status, onboard-remote, and discover commands treat a `null` endpoint as a hard failure. This complements the `fe5819887` centralization and closes a gap where partially-resolved Bonjour beacons could produce malformed connection targets.

#### Fail Closed on Shared Synology Webhook Paths ([`980940aa5`](https://github.com/openclaw/openclaw/commit/980940aa58))

In multi-account Synology Chat setups, `extensions/synology-chat/src/gateway-runtime.ts` now rejects two classes of ambiguous webhook configurations at gateway startup: (1) `"inherited-shared-webhook-path"` — an account without an explicit `webhookPath` that would inherit the base channel path in a multi-account setup; (2) `"duplicate-webhook-path"` — two accounts resolving to the same path. Both cases raise an error and refuse to register the route. Operators may opt in to inherited paths via `dangerouslyAllowInheritedWebhookPath: true` per-account.

#### Harden Voice-Call Webhook Pre-Auth Guards ([`651dc7450`](https://github.com/openclaw/openclaw/commit/651dc7450b))

`VoiceCallWebhookServer` in `extensions/voice-call/src/webhook.ts` now validates the webhook signature header before reading the request body — missing header triggers an immediate 401 response. Body reads are capped at `MAX_WEBHOOK_BODY_BYTES` (from `WEBHOOK_BODY_READ_DEFAULTS.preAuth.maxBytes`) with a `WEBHOOK_BODY_TIMEOUT_MS` deadline, and concurrent pre-auth requests per source IP are rate-limited. This closes an unauthenticated resource-exhaustion path where attackers could stream arbitrarily large bodies to the voice-call webhook endpoint before any credential check.
