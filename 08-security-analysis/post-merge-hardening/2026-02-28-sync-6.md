#### Detect groupPolicy=allowlist with empty groupAllowFrom ([`f1bf55868`](https://github.com/openclaw/openclaw/commit/f1bf55868))

The `detectEmptyAllowlistPolicy` function in `src/commands/doctor-config-flow.ts` now also checks `groupPolicy` configurations — not just `dmPolicy` — for empty allowlists. Previously, a user who set `groupPolicy="allowlist"` without populating `groupAllowFrom` (or `allowFrom`) would silently drop all group and channel messages with only a verbose-level log, as the runtime group access decision resolver fails closed on empty allowlists. The fix adds two inner helpers: `usesSenderBasedGroupAllowlist()` excludes channels that enforce group access via channel/space config rather than sender lists (discord, slack, googlechat), and `allowsGroupAllowFromFallback()` identifies channels that do not fall back from `groupAllowFrom` to `allowFrom` (googlechat, imessage, matrix, msteams, irc). The doctor command now surfaces distinct warnings for each case, prompting users to populate `groupAllowFrom` or switch to `groupPolicy="open"`.

#### Azure Custom Provider Credential Routing ([`978d9ae19`](https://github.com/openclaw/openclaw/commit/978d9ae19), [`4ed12c18a`](https://github.com/openclaw/openclaw/commit/4ed12c18a))

The `requestOpenAiVerification` function in `src/commands/onboard-custom.ts` was refactored to route Azure OpenAI verification requests correctly. A new `isAzureUrl()` helper detects Azure endpoints by checking for `.services.ai.azure.com` and `.openai.azure.com` hostnames. Azure requests now use `buildAzureOpenAiHeaders()` which sets the `api-key` header (per the Azure OpenAI API spec), while standard OpenAI requests use `buildOpenAiHeaders()` which sets `Authorization: Bearer`. The `978d9ae19` commit established the initial Azure validation fix (swapping `Authorization: Bearer` for `api-key`, removing the `model` field, and changing `max_tokens` to `max_completion_tokens`); the `4ed12c18a` commit completed the refactor by adding `isAzureUrl()` and splitting the request path into explicit Azure and non-Azure branches. This prevents custom Azure provider credentials from being sent with the wrong header format, which would cause authentication failures and could expose the API key in error responses.

#### Harden Azure Verification Test Coverage ([`89e158fc9`](https://github.com/openclaw/openclaw/commit/89e158fc9))

This commit adds targeted test coverage in `src/commands/onboard-custom.test.ts` that validates the Azure-specific request shape produced by the credential routing changes above: the `api-key` header is present (not `Authorization: Bearer`), the `api-version` query parameter is included, the `model` field is absent (Azure uses deployment name via URL path), and `max_completion_tokens` is used instead of `max_tokens`.
