#### Gateway: tighten forwarded client and pairing guards ([`fc2d29ea9`](https://github.com/openclaw/openclaw/commit/fc2d29ea9))

The gateway's X-Forwarded-For processing now filters loopback entries when walking the forwarded IP chain. New helper `isLoopbackAddress()` is used inside `resolveForwardedClientIp()` to skip forged loopback hops that could be inserted by an attacker to make a remote request appear to come from `127.0.0.1`. Device pairing approval is also tightened: `approveDevicePairing()` now accepts a `callerScopes` parameter, and the return type becomes the union `ApproveDevicePairingResult` which carries an explicit `forbidden` variant so the approval path fails auditably rather than silently.

#### Integrations: tighten inbound callback and allowlist checks ([`a47722de7`](https://github.com/openclaw/openclaw/commit/a47722de7))

Webhook and inbound callback authorization is hardened across four integrations. Google Chat's `verifyGoogleChatRequest()` adds `expectedAddOnPrincipal` validation, binding incoming callbacks to a specific registered add-on principal. Mattermost's interaction handler adds an `authorizeButtonClick` step that gates slash-command callbacks by scope before dispatching. The `NextcloudTalkRoomMatch` logic is corrected to match only on `roomToken` (not `roomName`), preventing room collision attacks where two rooms share the same human-readable name. Twitch's allowlist check now explicitly blocks empty allowlists rather than treating them as open (allow-all).

#### Plugins: preserve scoped ids and reserve bundled duplicates ([`8d44b16b7`](https://github.com/openclaw/openclaw/commit/8d44b16b7))

Scoped plugin IDs (`@org/plugin`) are now encoded via `encodePluginInstallDirName()` as `@[hashed-id]` on disk, preventing name collisions between scoped and unscoped plugins that could be exploited by crafting a plugin ID matching an existing unscoped entry. Plugin resolution precedence is also adjusted: bundled plugins now rank higher than workspace-local overrides unless explicitly configured, closing a vector where an untrusted plugin placed in the workspace directory could shadow a bundled, trusted plugin.

#### Harden device token rotation denial paths ([`7c0a849ed`](https://github.com/openclaw/openclaw/commit/7c0a849ed))

`rotateDeviceToken()` now returns a structured `RotateDeviceTokenResult` union type instead of a bare nullable, making denial reasons explicit: `unknown-device-or-role`, `missing-approved-scope-baseline`, and `scope-outside-approved-baseline` are all distinct variants. A new helper `logDeviceTokenRotationDenied()` emits an audit log entry on each denial. Crucially, both the "unknown device" and "missing scope" denial paths return the same public error message, preventing information leakage about whether a device ID is registered at all.

#### Nodes: recheck queued actions before delivery ([`ec2c6d83b`](https://github.com/openclaw/openclaw/commit/ec2c6d83b))

A new function `resolveAllowedPendingNodeActions()` re-validates queued node commands at pull time (before delivery to the client), not just at enqueue time. It calls `isNodeCommandAllowed()` against the current allowlist and client capability state, dropping commands that have become unauthorized due to policy changes between enqueue and delivery. This closes a policy timing window: previously, a command could be queued under permissive policy and executed after policy tightening.

#### Harden mention pattern regex compilation ([`ff61343d7`](https://github.com/openclaw/openclaw/commit/ff61343d7))

User-supplied mention patterns are now compiled through `compileConfigRegex()`, which wraps `compileSafeRegex()` to reject patterns with nested repetition constructs (e.g., `(a+)+$`) that cause ReDoS on large input strings. Empty patterns are also rejected. Provider-specific mention stripping for Discord, Slack, and WhatsApp uses new `stripRegexes()` helpers that compile regexes at load time rather than per-match, and cache the compiled results. Rejected patterns are logged with rate limiting to avoid log flooding.

#### Add ACP and subagent session binding ([`89e3969d6`](https://github.com/openclaw/openclaw/commit/89e3969d6))

The Feishu integration gains ACP (agent control protocol) session binding, enabling persistent multi-agent routing. Session binding is managed via `ensureConfiguredAcpRouteReady()` and `resolveConfiguredAcpRoute()` from the shared ACP persistent-bindings module, with session lifecycle coordinated through `getSessionBindingService()`. A new `buildFeishuConversationId()` function standardizes conversation IDs across group, topic, and DM scopes, replacing ad-hoc string concatenation that had caused scope collision bugs. While primarily a feature addition, this establishes ACP session isolation boundaries for Feishu that prevent cross-session message routing.

#### Harden remote cdp probes ([`a472f988d`](https://github.com/openclaw/openclaw/commit/a472f988d))

Remote Chrome DevTools Protocol (CDP) endpoints are now validated through `assertCdpEndpointAllowed()` before any WebSocket or HTTP connection is made. The check blocks CDP targets in private/loopback address space (RFC 1918, link-local) unless the `dangerouslyAllowPrivateNetwork` policy flag is set. A companion helper `redactCdpUrl()` strips credentials and session tokens from CDP URLs before they appear in logs or error messages. Together these prevent SSRF attacks that could probe internal services by disguising a private-network CDP target as a legitimate remote browser.

#### Gateway: scrub credentials from endpoint snapshots ([`f0202264d`](https://github.com/openclaw/openclaw/commit/f0202264d))

Config snapshot serialization now redacts credentials embedded in endpoint URLs. A new helper `isUserInfoUrlPath()` identifies config paths that hold base or HTTP URLs (those ending in `.baseUrl` or `.httpUrl`), and for each such path the snapshot logic calls `stripUrlUserInfo()` to extract and replace the userinfo component with a `REDACTED_SENTINEL` before the snapshot is stored or exported. This prevents raw API keys and passwords embedded as `https://key:secret@api.example.com` from leaking into audit logs, support exports, or diagnostics.

#### Subagents: restrict follow-up messaging scope ([`7679eb375`](https://github.com/openclaw/openclaw/commit/7679eb375))

Follow-up message sends from subagent controllers are now gated by `ensureControllerOwnsRun()` before dispatch. Additionally, leaf subagents (those with `controlScope !== "children"`) are now explicitly blocked from sending messages to other sessions, returning a `forbidden` status rather than attempting the operation. This closes a scenario where a subagent could send follow-up messages to sessions it did not originate, enabling unauthorized cross-session message delivery.

#### Tools: revalidate workspace-only patch targets ([`8e97b752d`](https://github.com/openclaw/openclaw/commit/8e97b752d))

The `apply-patch` tool's `remove` and `mkdirp` operations now call `assertSandboxPath()` when `workspaceOnly` mode is active, enforcing that file removal and directory creation targets remain within the workspace root (`options.cwd`). Previously only file writes were sandboxed; delete and mkdir operations could target paths outside the workspace by crafting a patch with an `..` traversal in the target path. The `allowFinalSymlinkForUnlink` and `allowFinalHardlinkForUnlink` flags permit final-hop link resolution to match expected unlink behavior.

#### Plugins: reserve context engine ownership ([`85dd0ab2f`](https://github.com/openclaw/openclaw/commit/85dd0ab2f))

The context engine registry now tracks ownership per engine ID. `registerContextEngine()` returns a `ContextEngineRegistrationResult` union type: `{ ok: true }` on success or `{ ok: false; existingOwner: string }` when another plugin has already claimed that engine ID. This prevents a second plugin from silently overwriting a registered context engine, which could be exploited by a malicious plugin to hijack another plugin's context processing pipeline.

#### CLI/completion: fix generator OOM and harden plugin registries (#45537) ([`bbb0c3e5d`](https://github.com/openclaw/openclaw/commit/bbb0c3e5d))

Plugin hook registration now checks for duplicate hook names before accepting a new entry. If a hook with the same `name` is already in the registry, the registration is rejected with a diagnostic error noting which plugin holds the existing registration. Similarly, empty command arrays are rejected at registration time. These guards prevent plugins from accidentally or maliciously registering duplicate hooks that could interfere with each other's execution or bypass hook dispatch ordering.

#### ACP: require admin scope for mutating internal actions ([`229426a25`](https://github.com/openclaw/openclaw/commit/229426a25))

ACP actions that mutate agent state — `spawn`, `cancel`, `steer`, `close`, `status`, `set-mode`, `set`, `cwd`, `permissions`, `timeout`, `model`, and `reset-options` — are now collected in the `ACP_MUTATING_ACTIONS` set and gated through `requireGatewayClientScopeForInternalChannel()` before execution. A client without admin scope receives a `forbidden` response when attempting any of these actions via the internal ACP channel. This enforces the RBAC boundary for ACP operations, preventing unprivileged clients from manipulating agent sessions.

#### Webhooks: tighten pre-auth body handling ([`5e78c8bc9`](https://github.com/openclaw/openclaw/commit/5e78c8bc9))

Pre-authentication webhook body reads are now bounded by explicit limits. For the Google Chat add-on webhook profile, constants `ADD_ON_PREAUTH_MAX_BYTES` (16 KB) and `ADD_ON_PREAUTH_TIMEOUT_MS` (3 s) are applied when reading the request body before the JWT signature is verified. Equivalent limits are applied in the Mattermost and MS Teams webhook handlers. Without these limits, an attacker could send an oversized or slowly-dripped body to exhaust server memory or hold connections open before authentication is checked.

#### ACP: fail closed on conflicting tool identity hints ([`e4c61723c`](https://github.com/openclaw/openclaw/commit/e4c61723c))

When resolving a tool's name from multiple identity sources (`fromMeta`, `fromRawInput`, `fromTitle`), the ACP client now returns `undefined` (fail closed) if any two non-null sources disagree. Previously, a conflict between `fromMeta` and `fromTitle` could allow a tool invocation to proceed under an unintended identity. The updated logic normalizes each source via `normalizeToolName()` and rejects the call if any pair of present sources produces different normalized names, preventing identity confusion attacks where a tool could masquerade as a different registered tool.
