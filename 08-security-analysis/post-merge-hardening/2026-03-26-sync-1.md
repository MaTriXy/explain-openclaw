#### Prefer freshest duplicate session rows in reads ([`40f820ff7`](https://github.com/openclaw/openclaw/commit/40f820ff7))

Gateway session read paths now use `resolveFreshestSessionEntryFromStoreKeys` instead of the non-deterministic `.find(Boolean)` pattern when duplicate session store keys exist. This affects session preview and history endpoints. The old approach could non-deterministically return stale session data when duplicate store keys were present, creating a race window where consumers could observe outdated session state. The new helper selects the freshest entry by timestamp, closing the race window and ensuring consistent reads.

#### Prefer deterministic session usage targets ([`7a7e4cd4c`](https://github.com/openclaw/openclaw/commit/7a7e4cd4c))

`buildStoreBySessionId` now delegates to `resolvePreferredSessionKeyForSessionIdMatches` when multiple session store keys resolve to the same session ID. Previously, non-deterministic key selection during usage tracking could mask unauthorized session activity or produce inconsistent usage attribution when a session appeared under more than one store key. The change ensures usage records point to a stable, predictable key regardless of store ordering.

#### Enforce spawned session visibility in key resolve ([`57fd0a9b2`](https://github.com/openclaw/openclaw/commit/57fd0a9b2))

The session key resolver now enforces sandbox-aware visibility for spawned sessions. Before this commit, key resolution could traverse beyond a spawned session's sandbox boundary and resolve keys belonging to peer or parent sessions. The fix ensures that spawned sessions are restricted to their own visible session set when resolving keys, preventing cross-session key leakage via the key resolution path.

#### Require operator.admin for mutating internal /allowlist commands ([`56eeec409`](https://github.com/openclaw/openclaw/commit/56eeec409))

Internal `/allowlist` mutation commands now require `operator.admin` scope via `requireGatewayClientScopeForInternalChannel`. Previously, callers with lower privilege scopes could invoke allowlist-mutating commands on internal channels. The new scope gate ensures only fully-privileged operators can modify the internal allowlist, hardening the RBAC boundary around a privileged administrative operation. This is the same class of fix as the earlier `/reset` RBAC hardening — privileged internal commands should require the highest-tier scope.

#### Close sandbox media root bypass for mediaUrl/fileUrl aliases ([`1d7cb6fc0`](https://github.com/openclaw/openclaw/commit/1d7cb6fc0))

`SANDBOX_MEDIA_PARAM_KEYS` was extended from three entries (`media`, `path`, `filePath`) to five by adding `mediaUrl` and `fileUrl`. The `normalizeSandboxMediaParams` function iterates these keys to sandbox-confine media parameters before they reach the message transport layer. The two missing aliases could previously be used to reference files outside the sandbox media root by supplying a `mediaUrl` or `fileUrl` parameter instead of the canonical `filePath`. Tests confirm that both sandbox-relative paths and remote HTTP URLs are handled correctly through the aliases after the fix.

#### Block non-owner authorized senders from changing /send policy ([`47dc7fe81`](https://github.com/openclaw/openclaw/commit/47dc7fe81))

`rejectNonOwnerCommand` is now applied to the `/send` command handler. Authorized senders who are not the session owner could previously modify the `/send` policy (the outbound-send enable/disable toggle) even though the policy controls what the session owner receives. The fix gates `/send` policy changes to the session owner only, preventing authorized but non-owner participants from silently modifying delivery behavior.

#### Enforce sandbox visibility for session_status ids ([`fb04801ed`](https://github.com/openclaw/openclaw/commit/fb04801ed))

The `session_status` tool now uses `resolveVisibleSessionReference` (sandbox-scoped) instead of `loadCombinedSessionStoreForGateway` + `resolvePreferredSessionKeyForSessionIdMatches` (gateway-wide). The old path would look up session IDs across the entire combined store, meaning a sandboxed agent could query the status of sessions it had no visibility into. The new path restricts lookups to the caller's visible session set, preventing cross-sandbox session ID enumeration via the `session_status` tool.
