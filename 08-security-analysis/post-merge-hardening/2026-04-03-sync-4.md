#### Make x_search auth plugin-owned ([`3872a866a`](https://github.com/openclaw/openclaw/commit/3872a866a1))

Moves xAI x_search API key authentication from the core config path (`tools.web.x_search.apiKey`) to a plugin-owned path under `plugins.entries.xai.config.*` via `migrateLegacyXSearchConfig()` in `src/config/legacy-x-search.ts:49`. The `X_SEARCH_RULE` auto-migration rule at `src/config/legacy.migrations.runtime.ts:263` transparently migrates existing credentials on load and drops the legacy path from `src/secrets/target-registry-data.ts`. The old `readLegacyXSearchApiKey()` core function is removed. This enforces the credential boundary established for Firecrawl in `38d2faee2` (Apr 3 sync 2): plugin-controlled tools own their auth in plugin config rather than the core schema, reducing the attack surface for config injection vectors that target core tool config paths.

**Addresses Audit Claim:** Partial Audit 2 Claim 1 (Config injection RCE via setupCommand) — x_search credentials are no longer reachable through core config injection paths.

#### Allow trusted explicit proxy media fetches ([`4251ad663`](https://github.com/openclaw/openclaw/commit/4251ad6638))

Adds `allowPrivateProxy?: boolean` to the `PinnedDispatcherPolicy` union variant (mode: `"explicit-proxy"`) at `src/infra/net/ssrf.ts:289`. In `src/infra/net/fetch-guard.ts:116`, when this flag is present, `assertExplicitProxyAllowed()` conditionally sets `allowPrivateNetwork: true` to permit the proxy to resolve to a private address. This implements a narrow, controlled SSRF exception: Telegram's extension can opt into private network access only for pre-configured, trusted explicit proxy URLs, while all other callers continue enforcing strict SSRF baseline policy. Without this exception, a Telegram bot configured to route media through a legitimate local proxy (e.g., `socks5://127.0.0.1:1080`) would be falsely rejected.

#### Make Windows exec hints accurate and dynamic ([`36d953aab`](https://github.com/openclaw/openclaw/commit/36d953aab6))

Updates the exec denial hint messages in `src/agents/bash-tools.exec.ts` to accurately reflect Windows-specific allowlist behavior. Previously, hints shown when exec was blocked on Windows used POSIX-centric language that did not match Windows command patterns or the argPattern allowlist semantics. The new hints are generated dynamically based on the platform detected at runtime, ensuring users see actionable instructions that correspond to the actual Windows allowlist configuration. This is the UX companion to `fff633377` (below) — accurate hints are load-bearing for operators configuring Windows exec policies.

#### Implement Windows argPattern allowlist flow ([`fff633377`](https://github.com/openclaw/openclaw/commit/fff6333773))

Completes the Windows exec allowlist by implementing `argPattern`-based command matching in `src/infra/exec-command-resolution.ts`. The key addition is `matchArgPattern()` at `src/infra/exec-command-resolution.ts:266`, which matches allowlist entries against command arguments using null-byte or space-delimited regex patterns. `matchAllowlist()` at `src/infra/exec-command-resolution.ts:323` is extended to apply `argPattern` matching on Windows platforms (deliberately skipped on non-Windows to avoid cross-platform policy divergence).

In `src/infra/exec-approvals-allowlist.ts`, `evaluateSegments()` at line 369 gains `isDirectShellPositionalCarrierInvocation()` detection at line 708: shell wrapper invocations that are positional carriers (e.g., PowerShell `-File` style) are handled separately from inline-eval invocations, preventing allowlist bypass via carrier argument reordering. The `POWERSHELL_WRAPPERS` set and `POSIX_INLINE_COMMAND_FLAGS` constants gate the PowerShell-specific code path. This commit (1,247 insertions) is the largest exec approval change in this sync cycle and completes the Windows exec hardening track begun in prior syncs.

**New security control.** Windows argPattern allowlist closes the gap where Windows nodes could not enforce argument-level exec restrictions.

#### Normalize transport policy wiring ([`b0f94a227`](https://github.com/openclaw/openclaw/commit/b0f94a227b))

Centralizes provider transport policy resolution in `src/agents/provider-request-config.ts` and adds `assertExplicitProxyAllowed()` at `src/infra/net/fetch-guard.ts:96`. This function validates that explicit proxy URLs (1) use `http:` or `https:` protocol, and (2) have a hostname that passes `resolvePinnedHostnameWithPolicy()` SSRF checks before the connection is dispatched. Without this, a provider configured with a malformed or internal-network proxy URL could bypass SSRF controls by routing through the explicit proxy path. The commit also ensures `allowPrivateNetwork` and `capabilities` fields flow through a unified `ResolvedProviderRequestPolicyConfig` type across OpenAI WebSocket, stream wrappers, and media understanding providers (14 production files), eliminating per-callsite policy divergence that had created inconsistent SSRF enforcement across provider backends.

**New security control.** Explicit proxy URL validation enforces SSRF policy on the proxy-routing path.
