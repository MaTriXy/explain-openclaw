#### Harden shared auth resolution across systemd, discord, and node host ([`25252ab5a`](https://github.com/openclaw/openclaw/commit/25252ab5a))

Extracts gateway credential resolution into a dedicated `connection-auth.ts` module with explicit auth precedence controls. The new `resolveGatewayConnectionAuth()` and `resolveGatewayConnectionAuthFromConfig()` functions accept `remoteTokenPrecedence` and `remotePasswordPrecedence` parameters to prevent credential leakage when multiple auth sources are present across systemd, Discord, and node-host deployment modes. Comprehensive test coverage was added for auth fallback chains.

#### Unify spawned metadata and extract attachments service ([`eed403dc7`](https://github.com/openclaw/openclaw/commit/eed403dc7))

Extracts attachment materialization into dedicated `subagent-attachments.ts` with strict validation via `decodeStrictBase64()` (returns `null` on malformed input, invalid base64, or oversized data) and `materializeSubagentAttachments()` which enforces per-file and total-bytes quotas (defaults: 5 MB total, 1 MB per file, 50 files max). Also introduces `SpawnedToolContext` and `SpawnedRunMetadata` types to standardize subagent context propagation across `subagent-spawn.ts`. This partially mitigates Legitimate Gap 3 (attachment size enforcement added; outPath path traversal validation remains open).

#### Narrow dm shared group policy typing ([`b6318d4df`](https://github.com/openclaw/openclaw/commit/b6318d4df))

Narrows the `groupPolicy` parameter in `resolveDmGroupAccessDecision()` from an implicit string to the explicit `GroupPolicy` enum type, ensuring only valid values (`"open"`, `"disabled"`, `"allowlist"`) are accepted. This prevents undefined group policy states from silently defaulting to a permissive or restrictive mode.

#### Share matched group policy evaluation ([`b0d924676`](https://github.com/openclaw/openclaw/commit/b0d924676))

Exports `evaluateMatchedGroupAccessForPolicy()` via the plugin SDK for consistent group access decisions across channel implementations. The function returns an explicit `MatchedGroupAccessDecision` type with reasons (`"allowed"`, `"disabled"`, `"empty_allowlist"`, `"not_allowlisted"`), replacing duplicated allowlist decision logic in DM security checks and the Nextcloud plugin.

#### Land #39247 from @jasonQin6 (subagent workspace inheritance) ([`ab54532c8`](https://github.com/openclaw/openclaw/commit/ab54532c8))

Restricts workspace directory overrides in spawned subagent runs to spawned contexts only via `resolveIngressWorkspaceOverrideForSpawnedRun()`, which checks for `spawnedBy` presence before applying the override. Adds a regression test verifying that tool-arg workspace overrides are ignored in non-spawned contexts, keeping workspace override internal-only and preventing external API exposure.

#### Honor exec-approvals ask=off for gateway/node runs ([`173132165`](https://github.com/openclaw/openclaw/commit/173132165))

Fixes exec-approvals to properly honor the `ask=off` setting for gateway/node execution contexts. Previously, the `ask=off` policy could be bypassed for certain gateway and node host execution paths; this ensures the approval gate is consistently skipped when explicitly disabled, reducing unnecessary friction for operators who have configured autonomous execution. Relates to Audit 2 Claim 5 (approval controls).

#### Support negative IDs in groupAllowFrom (#36753) ([`a7f6e0a92`](https://github.com/openclaw/openclaw/commit/a7f6e0a92))

Fixes `normalizeAllowFrom()` in `src/telegram/bot-access.ts` to accept negative Telegram group/channel IDs by changing the validation regex from `/^\d+$/` to `/^-?\d+$/`. Telegram group and channel IDs are negative integers; the previous regex rejected all group IDs, forcing operators to use `groupPolicy: "open"` as a workaround — which disabled allowlist-based access control entirely.

#### Block sandboxed slash spawns ([`61000b8e4`](https://github.com/openclaw/openclaw/commit/61000b8e4))

Prevents slash command spawning from within sandboxed ACP agent sessions in `src/agents/acp-spawn.ts`. Sandboxed sessions now block ACP slash command initiation to prevent potential sandbox escape via slash command chains. This addresses the ACP sandbox inheritance gap reported in GHSA-474h-prjg-mmw3.

#### Restore isolated delivery defaults ([`4e07bdbdf`](https://github.com/openclaw/openclaw/commit/4e07bdbdf))

Restores isolated delivery defaults for cron job execution in `src/cron/isolated-agent/run.ts`, ensuring cron hook jobs use the isolated sandbox configuration by default. A previous refactor had inadvertently changed the default delivery mode; this fix re-establishes the security-preferred isolated execution path.

#### Share missing-sender matched allowlist evaluation ([`566a821e5`](https://github.com/openclaw/openclaw/commit/566a821e5))

Extracts missing-sender allowlist evaluation logic into a shared function used consistently across LINE and Telegram channel implementations. Consolidating this logic ensures that missing-sender access control decisions are made uniformly rather than through per-channel ad-hoc implementations that could diverge.

#### Share allowlist provider warning resolution ([`2b5407052`](https://github.com/openclaw/openclaw/commit/2b5407052))

Consolidates allowlist provider warning resolution across all channel extensions (8 production files) into shared plugin SDK logic. This eliminates per-channel divergence in how allowlist warnings are surfaced, ensuring consistent user-facing behavior and reducing the risk of silent allowlist misconfiguration in any single channel.

#### Share onboarding allowlist entry parsing ([`f319ec2da`](https://github.com/openclaw/openclaw/commit/f319ec2da))

Consolidates onboarding allowlist entry parsing across LINE, Telegram, and Zalo channel onboarding modules into shared plugin SDK functions. Unified parsing ensures that allowlist entry validation during onboarding is consistent across channels.

#### Daemon CLI: resolve token drift from gateway credentials ([`c5fb66174`](https://github.com/openclaw/openclaw/commit/c5fb66174))

Resolves token drift between the daemon CLI and gateway credentials, ensuring the daemon CLI operates with the current gateway credential state rather than a stale cached value. Credential drift could cause authentication inconsistencies in daemon-managed deployments.
