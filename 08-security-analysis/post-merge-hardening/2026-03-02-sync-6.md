#### Improve DM topics support (#30579) (thanks @kesor) ([`c13b35b83`](https://github.com/openclaw/openclaw/commit/c13b35b83))

Adds per-DM topic configuration via the new `TelegramDirectConfig` type. When a Telegram DM conversation uses the platform's topic/thread feature, operators can now configure topic-specific `allowFrom` sender overrides and a `requireTopic` flag. When `requireTopic: true` is set, the gateway rejects DMs that arrive without a thread ID — preventing senders from bypassing topic-scoped access controls by messaging outside any topic. The per-DM `allowFrom` overrides take precedence over account-level allowlists, enabling fine-grained sender restrictions per DM topic. Access gating is enforced in `bot-message-context.ts`, with the `groupAllowOverride` / `dmAllowFrom` resolution chain giving per-topic overrides highest priority. Session key generation is also updated to incorporate the DM topic ID, ensuring topic-isolated sessions are properly segregated.

#### Normalize inline SecretRef token/key to tokenRef/keyRef in runtime snapshot ([`8a4d8c889`](https://github.com/openclaw/openclaw/commit/8a4d8c889))

Closes a credential write-back path in the runtime secrets snapshot. Prior to this fix, `auth-profiles.json` entries that used an inline SecretRef object directly in the `key` or `token` field (e.g., `"key": { "source": "env", "id": "MY_KEY" }`) were never promoted to their canonical `keyRef`/`tokenRef` form during profile load. This meant the inline form — and any subsequently resolved plaintext — could be written back to disk on every `updateAuthProfileStoreWithLock` call. The fix adds promotion logic inside `collectApiKeyProfileAssignment` and `collectTokenProfileAssignment`: when an inline SecretRef is detected in `key`/`token`, it is moved to `keyRef`/`tokenRef` and the original field is deleted from the in-memory profile object. The normalized profile is then written to disk in canonical (ref-only) form, preventing plaintext credential exposure.
