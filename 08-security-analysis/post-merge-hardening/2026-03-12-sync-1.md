#### Propagate real gateway client into plugin subagent runtime ([`a1520d70f`](https://github.com/openclaw/openclaw/commit/a1520d70f))

Fixes GHSA-xw77-45gv-p728. Plugin HTTP routes carrying the `auth:plugin` designation (public webhooks without gateway auth) previously dispatched runtime subagent calls using a hardcoded synthetic client with admin, approvals, and pairing scopes. This allowed any unauthenticated external request reaching a plugin route to invoke privileged gateway methods — including `sessions.delete` and `agent.run` — through the plugin runtime layer.

The fix introduces `createPluginRouteRuntimeClient` in `plugins-http.ts` to scope the runtime client per route: unauthenticated plugin routes receive a non-admin synthetic client limited to `operator.write`, while gateway-authenticated connections retain admin-capable scopes. The boundary is enforced per-request via `withPluginRuntimeGatewayRequestScope`, preventing privilege inheritance from the plugin runtime into gateway method dispatch.

#### Enforce caller-scope subsetting in device.token.rotate ([`dafd61b5c`](https://github.com/openclaw/openclaw/commit/dafd61b5c))

Fixes GHSA-4jpw-hj22-2xmc. The `device.token.rotate` gateway method accepted arbitrary caller-supplied scopes and forwarded them directly to `rotateDeviceToken` without verifying the caller actually held those scopes. An attacker with a pairing-scoped token on an already-approved device could request rotation to `operator.admin`, escalating to full admin permissions on any device whose `approvedScopes` included that scope.

The fix introduces a caller-scope subsetting check via `resolveMissingRequestedScope` in `devices.ts`: requested scopes are validated against the caller's `client.connect.scopes` using `roleScopesAllow` before `rotateDeviceToken` is invoked. If the caller lacks any requested scope, the method rejects with `missing scope: <scope>`, preventing privilege escalation through token rotation.

#### Add nodes to owner-only tool policy fallbacks ([`62d5df28d`](https://github.com/openclaw/openclaw/commit/62d5df28d))

Fixes GHSA-r26r-9hxr-r792. The `OWNER_ONLY_TOOL_NAME_FALLBACKS` set in `tool-policy.ts` — consulted by `applyOwnerOnlyToolPolicy()` when no explicit tool policy is configured — excluded the `nodes` tool. While `applyOwnerOnlyToolPolicy()` correctly stripped `gateway` and `cron` from non-owner senders, `nodes` remained accessible. The `nodes` tool internally issues `node.pair.approve` (requiring `operator.pairing` scope) and `node.invoke` (requiring `operator.write` scope), allowing non-owner users to approve pending node pairings and invoke arbitrary commands on paired nodes — potentially reaching `system.run` on nodes that expose it.

The fix adds `'nodes'` to `OWNER_ONLY_TOOL_NAME_FALLBACKS`. The `isOwnerOnlyToolName` helper (used by the fallback path) now returns `true` for the `nodes` tool, so non-owner senders no longer receive it when no explicit policy grants access.

#### Enforce browser origin check regardless of proxy headers ([`ebed3bbde`](https://github.com/openclaw/openclaw/commit/ebed3bbde))

Fixes GHSA-5wcw-8jjv-m286 (private advisory). In trusted-proxy mode, `resolveHandshakeBrowserSecurityContext` in `message-handler.ts` set `enforceOriginCheckForAnyClient` to `false` when `hasProxyHeaders` was true — skipping browser origin validation entirely for connections arriving through a trusted reverse proxy. An attacker serving content from an untrusted origin could connect to the gateway through the proxy, have the proxy inject trusted identity headers, and obtain `operator.admin` access via the `sharedAuthOk`/`roleCanSkipDeviceIdentity` path — without the origin check that normally catches cross-origin browser WebSocket connections.

The fix removes the `hasProxyHeaders` exemption from `resolveHandshakeBrowserSecurityContext`: origin validation now runs for all browser-originated WebSocket connections regardless of proxy header presence, closing the bypass.
