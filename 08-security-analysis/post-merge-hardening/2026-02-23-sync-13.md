#### SSRF IP Validation Consolidated to ipaddr.js ([`333fbb863`](https://github.com/openclaw/openclaw/commit/333fbb863))

Replaces manual hex-based IPv4/IPv6 address parsing with the `ipaddr.js` library in a new shared module `src/shared/net/ip.ts`. Key functions extracted into the shared module: `isBlockedSpecialUseIpv4Address()` (checks RFC special-use ranges via `BLOCKED_IPV4_SPECIAL_USE_RANGES`), `isPrivateOrLoopbackIpAddress()` (IPv4 + IPv6), `extractEmbeddedIpv4FromIpv6()` (detects IPv4-in-IPv6 transition addresses using `EMBEDDED_IPV4_SENTINEL_RULES`). The consolidation closes a long-standing gap where IP checks were duplicated across `ssrf.ts` and `gateway/net.ts` using inconsistent custom logic. Addresses Audit 2 Claim 4.

#### SSRF Host/IP Block Check Centralization ([`44dfbd23d`](https://github.com/openclaw/openclaw/commit/44dfbd23d))

Centralizes all host and IP blocking into `isBlockedHostname()` and `isBlockedHostnameOrIp()` in `ssrf.ts`, removing previously scattered duplicate checks. `assertAllowedHostOrIpOrThrow()` and `assertAllowedResolvedAddressesOrThrow()` now call the same shared classifier for both pre-DNS and post-DNS phases, eliminating drift between the two check points. Addresses Audit 2 Claim 4.

#### SSRF Dispatcher IPv6 Family Fallback ([`e9ed688c2`](https://github.com/openclaw/openclaw/commit/e9ed688c2))

Adds `autoSelectFamily: true` to `createPinnedDispatcher()` so that when a pinned SSRF-safe host resolves to both IPv4 and IPv6 addresses, the dispatcher falls back to IPv4 if IPv6 connectivity fails (Node.js "Happy Eyeballs" algorithm). Without this, IPv6-first stacks could stall on connection to hosts that have both record types. Addresses Audit 2 Claim 4.

#### Local-Host and Tailnet CIDR Unification ([`f14ebd743`](https://github.com/openclaw/openclaw/commit/f14ebd743))

Unifies private/loopback IP detection across `ssrf.ts` and `gateway/net.ts` into `isPrivateOrLoopbackAddress()` delegating to `isPrivateOrLoopbackIpAddress()` from `shared/net/ip.ts`. Removes redundant CIDR check functions previously spread across multiple files. Defense-in-depth: a consistent single classifier prevents divergence between inbound (gateway) and outbound (SSRF) network checks.

#### File Log Size Cap Added ([`8cc744ef1`](https://github.com/openclaw/openclaw/commit/8cc744ef1))

Adds `maxFileBytes` field to `LoggerSettings` with `DEFAULT_MAX_LOG_FILE_BYTES = 500 MB`. `buildLogger()` now tracks `currentFileBytes` and checks `nextBytes > settings.maxFileBytes` before each log line write. When the cap is hit, a warning line is appended once (`warnedAboutSizeCap` flag), then all further writes are suppressed. `resolveMaxLogFileBytes()` validates the config value (positive finite number). Configurable via `logging.maxFileBytes` in the config schema. Prevents runaway disk usage from verbose logging (DoS prevention for persistent-process deployments).

#### Core Auto-Updater with Dry-Run Preview ([`f442a3539`](https://github.com/openclaw/openclaw/commit/f442a3539))

Introduces a core auto-updater mechanism (`feat(update)`). Adds `--dry-run` preview mode. Supply chain security consideration: the update path should be verified for integrity checks on downloaded packages. The implementation establishes a foundation for coordinated update delivery; the security posture of the update channel will depend on how package signing/verification is implemented in downstream tooling.

#### Telegram webhookPort Config Wired ([`d0e676326`](https://github.com/openclaw/openclaw/commit/d0e676326))

Wires `webhookPort` through Telegram config and startup sequence, fixing a previously silent misconfiguration where the port was ignored. Config validation improvement: Telegram webhook deployments now correctly respect the configured port, preventing silent fallback to incorrect ports.

#### Node 22+ DNS IPv4 Default for Telegram ([`53adae9ce`](https://github.com/openclaw/openclaw/commit/53adae9ce))

Applies `dnsResultOrder=ipv4first` as the default for Node 22+ where DNS resolution defaults changed to prefer IPv6, causing Telegram fetch failures in IPv4-only environments. Applied at startup to normalize DNS resolution behavior across Node versions.
