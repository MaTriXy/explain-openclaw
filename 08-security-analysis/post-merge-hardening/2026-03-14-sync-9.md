#### Harden external content marker sanitization ([`b7afc7bf4`](https://github.com/openclaw/openclaw/commit/b7afc7bf40))

The external content boundary marker detection in `src/security/external-content.ts` now strips six invisible Unicode characters before processing marker text, preventing attackers from splitting trusted/untrusted content markers with visually identical but encoding-distinct characters. New constant `MARKER_IGNORABLE_CHAR_RE` covers zero-width space (U+200B), zero-width non-joiner (U+200C), zero-width joiner (U+200D), word joiner (U+2060), zero-width no-break space (U+FEFF), and soft hyphen (U+00AD). The `foldMarkerText()` function applies this regex to strip ignorable characters before matching, so a marker like `<<<EXTERNAL_UNTRUSTED_CONTENT>>>` cannot be defeated by embedding U+200B between characters. Parametrized tests cover each ignorable character type on both start and end markers.

#### Harden powershell wrapper detection ([`28a49aaa3`](https://github.com/openclaw/openclaw/commit/28a49aaa34))

The `POWERSHELL_INLINE_COMMAND_FLAGS` set in `src/infra/shell-inline-command.ts` gains two new entries: `-f` and `-file`. PowerShell's `-File` parameter executes a script file as an inline command — previously, only `-c`, `-command`, `-encodedcommand`, `-enc`, and `-e` were blocked. Without this fix, passing `-file path/to/script.ps1` to the PowerShell wrapper could bypass inline command detection and execute arbitrary script files. The new entries close this gap in the flag-based blocklist used by the shell injection detection layer.

#### Harden macos shell continuation parsing ([`6b49a604b`](https://github.com/openclaw/openclaw/commit/6b49a604b4))

The macOS exec command resolution parser gains `nextShellSignificantCharacter()` in `ExecCommandResolution.swift`, which skips backslash-newline line-continuation sequences when scanning shell command tokens. Previously, a command written as `dangerous-cmd\<newline>arg` would confuse the significant-character scanner, causing the exec approval engine to misidentify the start of the command token. An attacker could exploit this to slip a disallowed command past the macOS exec approval check by injecting a backslash-newline continuation at a strategic position.

#### Harden macos env wrapper resolution ([`e1fedd438`](https://github.com/openclaw/openclaw/commit/e1fedd4388))

The macOS exec approval system replaces `parseFirstToken()` with `resolveShellSegmentExecutable()` in `ExecCommandResolution.swift`. The new function uses `tokenizeShellWords()` to split the shell segment into tokens and then calls `ExecEnvInvocationUnwrapper.unwrapDispatchWrappersForResolution()` to strip `env`/dispatch wrappers before resolving the true executable. Previously, a command prefixed with `env VAR=val malicious-cmd` would resolve as `env` (a universally trusted binary) rather than `malicious-cmd`, allowing the actual executable to escape approval scrutiny. The unwrapper step ensures the real executable is identified regardless of how many wrapper layers surround it.

#### Bind macOS skill trust to resolved paths ([`4d686b47f`](https://github.com/openclaw/openclaw/commit/4d686b47f0))

The macOS exec approval trust mechanism in `ExecApprovalEvaluation.swift` upgrades from name-only lookup to name-plus-path validation. `SkillBinsCache` now exposes `currentTrust()` returning a `[String: Set<String>]` map (executable name → set of trusted resolved paths) instead of the previous `currentBins()` returning a flat `Set<String>` of names. New helper `isSkillAutoAllowed()` walks the resolved-path set for each approval resolution and returns `true` only when both the executable name and resolved path match a trusted entry. This closes a skill binary name hijacking vector: placing a binary with the same name as a trusted skill in a different directory previously earned it auto-allow status because only the name was checked.
