#### Security: split audit runtime surfaces ([`c7137270d`](https://github.com/openclaw/openclaw/commit/c7137270d1))

Splits `src/security/audit.ts` into three discrete runtime boundary modules — `audit.nondeep.runtime.ts`, `audit.deep.runtime.ts`, and `audit-channel.collect.runtime.ts` — and introduces three async load helpers: `loadAuditNonDeepModule()`, `loadAuditDeepModule()`, and `loadAuditChannelModule()`. These are used throughout `audit.ts` to defer evaluation of the corresponding security-audit sub-systems until they are actually needed. The change reduces the set of modules that must be evaluated at startup, improving isolation from config-loading side effects and shrinking the early-import surface area for the security audit subsystem.

#### Status: lazy-load security audit commands ([`d5b12f505`](https://github.com/openclaw/openclaw/commit/d5b12f505c))

Creates `src/security/audit.runtime.ts` as a thin re-export (`export { runSecurityAudit } from "./audit.js"`) and updates `src/commands/status-json.ts` and `src/commands/status.command.ts` to import it lazily via a module promise (`securityAuditModulePromise`). Previously the status commands imported `runSecurityAudit` statically, forcing full audit module evaluation on every status command invocation. The lazy pattern defers this until a security audit is actually requested.

#### Security: lazy-load audit config snapshot IO ([`5f42389d8`](https://github.com/openclaw/openclaw/commit/5f42389d8d))

Converts `src/security/audit-extra.async.ts` to lazy-load `createConfigIO` via a new `loadConfigModule()` helper. The call to `createConfigIO` now occurs inside `readConfigSnapshotForAudit()` only when a config snapshot is actually required, deferring module evaluation and reducing initialization overhead for audit paths that don't access the filesystem config snapshot.

#### Security: lazy-load deep skill audit helpers ([`a2119efe1`](https://github.com/openclaw/openclaw/commit/a2119efe1c))

Converts `src/security/audit-extra.async.ts` to lazy-load `loadWorkspaceSkillEntries` via a new `loadSkillsModule()` helper. The skills module is now imported on demand inside `collectInstalledSkillsCodeSafetyFindings()` rather than at module load time, deferring its evaluation until a deep skill scan (`--deep` audit) is actually requested.

#### Security: trim audit policy import surfaces ([`4cb46f223`](https://github.com/openclaw/openclaw/commit/4cb46f223c))

Extracts two policy evaluation functions into dedicated boundary modules: `isToolAllowedByPolicies` moves to `src/agents/tool-policy-match.ts`, and `resolveAllowedAgentIds` moves to `src/gateway/hooks-policy.ts`. The security audit files (`audit-extra.async.ts`, `audit-extra.sync.ts`) import from these new focused modules rather than from the larger `pi-tools.policy.ts` and `hooks.ts`. This reduces circular dependencies and makes the policy evaluation surface easier to audit in isolation.

#### Add structured card actions and interactive approval flows ([`fa62231af`](https://github.com/openclaw/openclaw/commit/fa62231afc))

Implements a new security layer for Feishu card interactions. Introduces `FEISHU_CARD_INTERACTION_VERSION = "ocf1"` (in `extensions/feishu/src/card-interaction.ts`) as a schema version stamp embedded in every card action payload. `decodeFeishuCardAction()` validates incoming card actions against: (1) version mismatch rejection, (2) stale payload detection (configured via `FEISHU_CARD_ACTION_TOKEN_TTL_MS` = 15 minutes in `extensions/feishu/src/card-action.ts`), and (3) wrong-user/conversation rejection. Processed action tokens are deduplicated to prevent replay. This makes Feishu interactive approval flows (used for agent task approvals) resistant to stale replay and context-swapping attacks.
