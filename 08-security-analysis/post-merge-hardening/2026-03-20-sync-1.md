#### Matrix Credential Write Isolation ([`f4f0b171d`](https://github.com/openclaw/openclaw/commit/f4f0b171d))

Matrix credential handling was split into separate read and write boundary modules. A new `credentials-read.ts` file consolidates all credential read functions using static imports, while a new `credentials-write.runtime.ts` file exports `saveMatrixCredentials` and `touchMatrixCredentials` through a dynamic import wrapper (`await import("./credentials.js")`). This runtime boundary means the underlying credentials module is only loaded on demand — write operations no longer pull in the full credential implementation at startup.

The calling code in `client/config.ts` uses a local `loadCredentialsWriter()` helper that dynamically imports from `credentials-write.runtime.ts`, keeping write path initialization deferred until the first actual credential write. This directly follows the dynamic-import guardrail documented in CLAUDE.md: mixing static and dynamic imports for the same module in production code is forbidden, so the write side gets its own dedicated `.runtime.ts` boundary. The isolation reduces the blast radius of any credential-writing code path by ensuring it is never eagerly loaded.

This is a defense-in-depth improvement for Audit 1 Claim 1 (credential storage architecture). No credential storage format changed; only the loading boundary was tightened.

#### macOS Exec Allowlist Parity ([`c4a4050ce`](https://github.com/openclaw/openclaw/commit/c4a4050ce))

A parity gap in the macOS exec command allowlist evaluation was fixed. Previously, the allow-always list was checked only against the inferred/formatted argument vector form of a command; on the gateway side, both the raw and formatted forms are evaluated. The macOS implementation now checks both forms.

`resolveAllowAlwaysPatterns()` was added to `ExecCommandResolution` to collect allow-always patterns from the exec approval configuration, with depth-limited recursion (depth < 3) and deduplication to prevent runaway traversal. `allowlistEvaluationRawCommand()` was added to `ExecSystemRunCommandValidator` as the canonical form resolver — given a command and optional raw string, it determines which representation to use for allowlist matching (the `rawCommand` value if available, otherwise the formatted argv). `ExecApprovalEvaluation` gains an `allowAlwaysPatterns: [String]` field that carries the collected patterns through the evaluation pipeline, used in `ExecApprovalsSocket` when iterating allowed patterns.

Together these changes close the behavioral gap between macOS and the gateway's exec RBAC evaluation (Audit 2 Claim 5). The fix relates to GHSA-f8r2-vg7x-gh8m (allowlist overmatch on POSIX paths) and GHSA-rw39-5899-8mxp (misleading shell payloads) — both of which rely on inconsistent form evaluation for their attack surface.
