#### Configurable failure alerts for repeated job errors ([`4637b90c0`](https://github.com/openclaw/openclaw/commit/4637b90c0))

Adds operator-configurable failure notifications when a cron job errors repeatedly. The new `CronFailureAlert` type defines per-job alert routing (channel + destination) with a cooldown threshold controlled by `lastFailureAlertAtMs` state. At alert time, `normalizeCronMessageChannel()` and `normalizeTo()` in `src/cron/service/timer.ts` normalize the target before handing off to `sendCronFailureAlert()` — which routes through the existing outbound delivery pipeline, inheriting all channel validation and delivery controls already in place.

The alert configuration uses the same Zod schema (with `.strict()`) that governs all other cron config, preventing injection of unexpected fields. No new delivery surface is created; the control sits entirely within the existing job-monitoring path.

#### Add retry policy for one-shot jobs on transient errors ([`ea3955cd7`](https://github.com/openclaw/openclaw/commit/ea3955cd7))

Implements configurable retry for one-shot cron jobs that fail on transient errors (`rate_limit`, `network`, `timeout`, `server_error`). The `CronRetryConfig` type (in `src/config/types.cron.ts`) accepts `maxAttempts` (0–10), `backoffMs` (1–10 entries), and `retryOn` (an enum-constrained list of transient error types). Retry gating is enforced by `isTransientCronError()` and `errorBackoffMs()` in `src/cron/service/timer.ts` — only known transient error strings produce a retry; permanent errors pass through unchanged.

This replaces a hardcoded `DEFAULT_BACKOFF_SCHEDULE_MS` constant with a per-job configurable schedule, improving operator control without opening any new execution path or bypassing authorization checks.

#### Add audit logging for job create/update/remove ([`62a7683ce`](https://github.com/openclaw/openclaw/commit/62a7683ce))

Adds gateway-side audit log entries for cron job lifecycle events (create, update, remove) via `context.logGateway.info()` in `src/gateway/server-methods/cron.ts`. Removal is only logged when `result.removed` is true, avoiding noise from no-op calls. This creates an auditable event trail for operators tracking job manipulation — relevant to Audit 1 Claim 2 (audit trail for privileged operations) and Audit 2 Claim 1 (config injection visibility). No authorization or execution logic changes.

#### Slack: harden slash and interactions ingress checks ([`0f36ee5a2`](https://github.com/openclaw/openclaw/commit/0f36ee5a2))

Adds `shouldDropMismatchedSlackEvent()` checks at the entry points for slash commands (`src/slack/monitor/slash.ts`) and interaction events (`src/slack/monitor/events/interactions.ts`). The guard compares the incoming event's api_app_id and team_id fields against the configured account, now also handling the nested team.id field variant that some Slack event types use. Events with a mismatched app or team ID are silently dropped (after `ack()`) with a logged warning, preventing unauthorized workspaces from triggering gateway commands.

This closes a gap where Slack interaction payloads carrying `team.id` in the nested format (rather than top-level `team_id`) could bypass the existing team identity check, potentially allowing events from unrecognized workspaces to proceed.

#### Android: return NOT_AUTHORIZED when notify permission is lost ([`6f63fc288`](https://github.com/openclaw/openclaw/commit/6f63fc288))

Hardens `SystemHandler.kt` to explicitly enforce Android's `POST_NOTIFICATIONS` permission at runtime. Previously, posting a notification when the permission was revoked would fail silently or throw an uncaught `SecurityException`. Now the handler catches the `SecurityException`, logs it, and returns a `NOT_AUTHORIZED` error code to the caller — making permission revocation visible and consistently handled across the notification path.

#### Android: guard notification post permission ([`348a7dd5b`](https://github.com/openclaw/openclaw/commit/348a7dd5b))

Adds a proactive permission check in `SystemHandler.kt` before attempting to post any notification. If `POST_NOTIFICATIONS` is not granted, the post is skipped and an appropriate error is returned without triggering a `SecurityException`. Combined with `6f63fc288`, this establishes a two-layer defence: proactive check (this commit) and catch-on-revocation (prior commit), ensuring notification delivery never proceeds without the required Android permission.

#### Enforce model allowlist for cron subagents ([`d0ca02e96`](https://github.com/openclaw/openclaw/commit/d0ca02e96))

Isolated cron sessions are subagents. Previously, `subagents.model` set in agent config was not applied to cron sessions, leaving them to always use the default model. This commit wires `normalizeModelSelection()` to read `subagents.model` from agent config (or global defaults) and then validates the result through `resolveAllowedModelRef()` before applying it. If the model fails the allowlist check, the default model is used — enforcing that cron subagents cannot use an unauthorized model even if `subagents.model` is set to an unrecognized or disallowed string.
