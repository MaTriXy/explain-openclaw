#### Security audit: suggest valid gateway.nodes.denyCommands entries ([`d95cf256e`](https://github.com/openclaw/openclaw/commit/d95cf256e))

The security audit routine that validates `gateway.nodes.denyCommands` now provides actionable suggestions when an entry doesn't match any known node command pattern. Two new internal helpers power this: `editDistance()` at `src/security/audit-extra.sync.ts:243` implements a space-optimized Levenshtein DP algorithm, and `suggestKnownNodeCommands()` at `:270` performs a two-phase lookup — a fast prefix scan followed by fuzzy matching with a score-normalized threshold.

`collectNodeDenyCommandPatternFindings()` at `:974` calls these helpers (line 1004) when a config entry doesn't match any registered node command name. Previously, operators received only a generic "not a known node command" warning. Now they receive a `did you mean: [alternatives]` list drawn from the actual command registry, reducing the chance that misconfigured `denyCommands` entries silently fail to block anything. The check covers both exact-name misses (e.g., `system.Run` instead of `system.run`) and pattern-style entries that the exact-match system can never fire on (e.g., `system.*`). Relates to Audit 2 Claim 5 (RBAC/permission boundary controls).

#### Add Permissions-Policy header to default security headers ([`4b17d6d88`](https://github.com/openclaw/openclaw/commit/4b17d6d88))

`setDefaultSecurityHeaders()` at `src/gateway/http-common.ts:11` now emits a `Permissions-Policy` header with value `camera=(), microphone=(), geolocation=()` (line 17). This header restricts browser feature access for all responses served through the gateway HTTP layer, preventing embedded pages from accessing camera, microphone, or geolocation APIs — even if a page were compromised by a script injection or XSS. The existing security header set (`X-Content-Type-Options: nosniff`, `Referrer-Policy: no-referrer`, `Strict-Transport-Security`) is unchanged; this adds a new browser-level capability restriction policy. Test coverage added to `src/gateway/http-common.test.ts`.

#### Add X-Content-Type-Options nosniff header to media route ([`a8dd9ffea`](https://github.com/openclaw/openclaw/commit/a8dd9ffea))

The media file serving route at `src/media/server.ts:36` now explicitly sets `X-Content-Type-Options: nosniff` before serving each response. While `setDefaultSecurityHeaders()` in `src/gateway/http-common.ts` applies this header for the main gateway HTTP handler, the media server runs as a separate Express application instance (created at `server.ts:108`, routes attached via `attachMediaRoutes()` at `:28`) that does not share gateway middleware. This was a gap: browsers could MIME-sniff the content type of gateway-served media files, potentially treating a crafted payload as executable HTML or script. The `nosniff` directive closes this gap for media responses.

#### Strip partial API token from status labels ([`e8cb0484c`](https://github.com/openclaw/openclaw/commit/e8cb0484c))

`resolveModelAuthLabel()` at `src/agents/model-auth-label.ts:11` previously embedded partial API key snippets into model auth status strings using the now-removed `formatApiKeySnippet()` and `formatCredentialSnippet()` helpers. Status labels such as `api-key sk-...a1b2 (ANTHROPIC_API_KEY)` or `token sk-...4321 (profile)` appeared in gateway status commands and plugin status replies visible to channel members.

The fix removes both helpers and simplifies the output: `token (label)` and `api-key (source)` — no credential material in status labels. The change reduces the exposure surface for API key leakage via status messages that may be logged, forwarded, or visible in group chat channels.

#### Align DiscordAccountConfig.token type with SecretInput ([`ff96e41c3`](https://github.com/openclaw/openclaw/commit/ff96e41c3))

`DiscordAccountConfig.token` at `src/config/types.discord.ts:222` was typed as `string | undefined`. This commit changes it to `SecretInput | undefined`, aligning the field with the `SecretInput` union type used for all other credential fields across the config schema.

`SecretInput` supports both inline string values and `SecretRef` objects (e.g., `{ source: "env", id: "DISCORD_BOT_TOKEN" }`), enabling environment-variable-backed or vault-backed token resolution. Previously, Discord bot tokens could only be stored as plaintext in config files. With this change, operators can use `{ source: "env", id: "DISCORD_BOT_TOKEN" }` for secret reference resolution, consistent with how other channel credentials are configured. Four documentation files updated to reflect the new configuration format. Relates to Audit 1 Claim 1 (plaintext credential handling).

#### Support SecretRef-style env credentials in account resolver ([`b7589b32a`](https://github.com/openclaw/openclaw/commit/b7589b32a))

`resolveFeishuCredentials()` at `extensions/feishu/src/accounts.ts:122` (one of three overloads) previously only handled string values for `appId`, `appSecret`, `verificationToken`, and `encryptKey`. When these fields contained `SecretRef` objects (`{ source: "env", id: "VAR_NAME" }`), the resolver returned `null`, causing Feishu channels to fail silently when configured with env-backed secrets.

The fix introduces two internal helpers: `normalizeString()` at `:132` (type-safe string extraction from unknown values) and `resolveSecretLike()` at `:140`, which first attempts direct string resolution and then, on the `allowUnresolvedSecretRef` onboarding path, performs an inline `process.env` lookup for `source: "env"` references. This brings Feishu credential resolution to feature parity with other channels that already supported `SecretRef`-style configuration. Relates to Audit 1 Claim 1 (plaintext credential handling — Feishu credentials can now be stored as env refs).
