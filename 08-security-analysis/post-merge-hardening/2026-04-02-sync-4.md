#### Fix: move bootstrap session grammar into plugin-owned session-key surfaces ([`bea53d7a3`](https://github.com/openclaw/openclaw/commit/bea53d7a3))

Refactors session key parsing for channel-specific conversation grammars (Telegram topics, Feishu sender scopes) out of the core `session-key-utils.ts` and into plugin-owned hooks via a new `src/channels/plugins/session-conversation.ts` facade. The four routing functions are:

- `resolveSessionConversation()` at `src/channels/plugins/session-conversation.ts:218` — central dispatch for parsing session keys via plugin hooks
- `resolveSessionConversationRef()` at line 230 — parses raw session keys and delegates to plugin-registered grammar
- `resolveSessionThreadInfo()` at line 255 — extracts thread/topic identifiers via plugin grammar
- `resolveSessionParentSessionKey()` at line 269 — resolves parent session keys through plugin boundaries

Case-insensitive regex matching (`/i` flag) is added to Feishu conversation-id parsing and Telegram topic parsing to prevent case-sensitive access bypasses. All readers of session keys in agents, auto-reply, model overrides, and delivery info are migrated to use plugin-delegated parsing. Moving grammar into plugin boundaries reduces the risk that core code accidentally replicates or differs from plugin-specific parsing logic, which could lead to session key misrouting across trust boundaries.

#### Make overload failover configurable ([`418fa12df`](https://github.com/openclaw/openclaw/commit/418fa12df))

Makes provider overload rate-limit handling configurable rather than hardcoded. Two new fields in `src/config/types.auth.ts` govern behavior:

- `auth.cooldowns.overloadedProfileRotations` (line 33, default: 1) — caps how many same-provider profile rotations occur before fallback
- `auth.cooldowns.overloadedBackoffMs` (line 38, default: 0ms) — fixed delay between overload retries

Replaces hardcoded constants with config-driven helpers `resolveOverloadFailoverBackoffMs()` at `src/agents/pi-embedded-runner/run/helpers.ts:31` and `resolveOverloadProfileRotationLimit()` at line 35. Default settings preserve existing behavior (1 rotation, 0ms backoff) while allowing operators to tune per deployment. This prevents tight profile rotation cascades when multiple providers are simultaneously rate-limited. Addresses profile rotation cascade risk from [#58348](https://github.com/openclaw/openclaw/issues/58348).

#### Add owner-key task access boundaries ([`7cd0ff2d8`](https://github.com/openclaw/openclaw/commit/7cd0ff2d8))

Introduces mandatory owner-key based access control for all task records. New `src/tasks/task-owner-access.ts` seam provides:

- `canOwnerAccessTask()` at line 14 — enforces ownership: session-scoped tasks require matching `ownerKey`; system-scoped tasks (cron, background ops) bypass session ownership
- `getTaskByIdForOwner()` at line 21 — returns task only if caller's `ownerKey` matches
- `findTaskByRunIdForOwner()` at line 29 — owner-scoped lookup by run ID
- `listTasksForRelatedSessionKeyForOwner()` at line 37 — lists tasks visible to a specific owner

Replaces `requesterSessionKey` with an explicit `ownerKey` + `scopeKind: "session" | "system"` tuple. `assertTaskOwner()` at `src/tasks/task-registry.ts:63` validates ownership at record creation using the `taskIdsByOwnerKey` index at line 49. A database migration handles legacy cron rows. All existing task readers are migrated to owner-scoped access functions, preventing cross-session task visibility and unauthorized task inspection across user sessions.

#### Restore owner-key task scope ([`80ed55332`](https://github.com/openclaw/openclaw/commit/80ed55332))

Re-establishes `assertTaskOwner()` at `src/tasks/task-registry.ts:63` and `taskIdsByOwnerKey` index at line 49 after a partial revert broke the access control boundary introduced in `7cd0ff2d8`. Repairs ACP control plane (`src/acp/control-plane/manager.core.ts`) and cron service (`src/cron/service/ops.ts`, `src/cron/service/timer.ts`) to use correct owner-scoped task operations. #### Make task-store writes atomic ([`66413487c`](https://github.com/openclaw/openclaw/commit/66413487c))

Wraps all task-store mutations in `withWriteTransaction()` at `src/tasks/task-registry.store.sqlite.ts:419`. Without this, concurrent task creation or modification could produce partial state writes — leaving `ownerKey`, `scopeKind`, or indexed maps in an inconsistent state. Adds `src/tasks/task-registry.store.ts` abstraction layer and updates `src/tasks/task-registry.ts` to use the transactional write path. Inconsistent ownership state could allow `assertTaskOwner()` checks to pass or fail incorrectly depending on interleaving.

#### Normalize task create compatibility ([`1816d6a4e`](https://github.com/openclaw/openclaw/commit/1816d6a4e))

Normalizes the task creation path in `src/tasks/task-registry.ts` to ensure `ownerKey` and `scopeKind` are always populated when a new task record is created. Prevents null-owner tasks arising from legacy creation paths that predate the owner-key refactor. Without this fix, tasks created via legacy codepaths would have a null `ownerKey` and could bypass `assertTaskOwner()` access control checks on all subsequent reads.

#### Show agent-local task counts when session tasks are empty ([`fdad8ea3b`](https://github.com/openclaw/openclaw/commit/fdad8ea3b))

Adds an agent-local fallback in `src/auto-reply/reply/commands-status.ts` (line 85): when the session task list is empty, the status command returns an agent-local count string instead of silently rendering zero. This prevents the edge case where a status command might inadvertently query cross-session task counts when no session tasks are found, defaulting instead to the narrower agent-local scope.

#### Restore QR bootstrap onboarding handoff (#58382) (thanks @ngutman) ([`69fe99937`](https://github.com/openclaw/openclaw/commit/69fe99937))

Restores two security controls that were regressed in a prior refactor:

1. **Device binding check** — `verifyDeviceBootstrapToken()` at `src/infra/device-bootstrap.ts:172` now enforces the device binding assertion before accepting a bootstrap token. Without this, a bootstrap token issued for device A could be redeemed on device B.

2. **TLS fingerprint cleanup** — `clearFingerprint()` at `apps/shared/OpenClawKit/Sources/OpenClawKit/GatewayTLSPinning.swift:39` and a new `clearAllFingerprints()` function are called on onboarding reset. Without clearing stale pins, a device that previously paired with a gateway would reject a new TLS certificate (from a new gateway or regenerated cert), blocking legitimate re-enrollment. The gateway authentication control UI surfaces are also updated to reflect reset state.
