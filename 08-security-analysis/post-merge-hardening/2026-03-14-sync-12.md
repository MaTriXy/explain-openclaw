#### Sanitize SCP remote path to prevent shell metacharacter injection ([`a54bf71b4`](https://github.com/openclaw/openclaw/commit/a54bf71b4))

The iMessage SCP staging path (`scpFile()` in `stage-sandbox-media.ts`) previously used the caller-supplied remote attachment path verbatim. An iMessage sender could inject shell metacharacters (`;`, `|`, `&`, `$`, backtick, etc.) into the attachment filename, which were then passed unquoted to `/usr/bin/scp`, enabling command execution on the configured remote host.

New helper `normalizeScpRemotePath()` in `scp-host.ts` validates the remote path before use. The function rejects: paths not starting with `/`, control characters (code points ≤ 0x1f and 0x7f), and any character in `SCP_REMOTE_PATH_UNSAFE_CHARS` (`\`, `'`, `"`, backtick, `$`, `;`, `|`, `&`, `<`, `>`). `scpFile()` now calls `normalizeScpRemotePath()` and throws on rejection before constructing the scp argument, ensuring the sanitized path (`safeRemotePath`) is always used. Companion `isSafeScpRemotePath()` provides a boolean predicate for callers that prefer it.

This resolves GHSA-g2f6-pwvx-r275 (medium severity; credited to @lintsinghua).

#### Tighten safe bin runtime policy coverage ([`71a3dd80e`](https://github.com/openclaw/openclaw/commit/71a3dd80e))

Safe binary checks in `exec-safe-bin-runtime-policy.ts` compare an executable name against the allowlist after normalization. Previously, `normalizeSafeBinName()` stripped directory components but left Windows executable suffixes intact. A binary named `curl.exe` or `npm.cmd` would not match an allowlist entry of `curl` or `npm`, but more importantly, a crafted path with an appended `.exe` suffix could bypass the suffix-based allowlist on Windows.

`normalizeSafeBinName()` now strips `.exe`, `.cmd`, `.bat`, and `.com` extensions (case-insensitive) after extracting the trailing path component, normalizing Windows executables to their base name before allowlist comparison. This prevents extension-appended bypass attempts on Windows hosts.

#### Tighten path guard coverage ([`ff6636ed5`](https://github.com/openclaw/openclaw/commit/ff6636ed5))

The path containment check `isPathInside()` in `path-guards.ts` determines whether a resolved target path is inside a resolved root path. On POSIX the function used `path.resolve()` correctly. On Windows it used the same POSIX `path.resolve()`, which may produce inconsistent results when paths contain mixed forward/backward slashes or UNC prefixes.

The fix switches the Windows branch to `path.win32.resolve()` for both root and target resolution, ensuring platform-correct normalization. This closes a potential path traversal bypass on Windows where mixed-slash inputs could cause the containment check to return true for a path outside the intended root.

#### Tighten device identity helper coverage ([`e8c300c35`](https://github.com/openclaw/openclaw/commit/e8c300c35))

Two functions in `device-identity.ts` lacked a guard against zero-length decoded key material. `normalizeDevicePublicKeyBase64Url()` decodes a base64url-encoded public key and re-encodes it; `deriveDeviceIdFromPublicKey()` hashes raw key bytes to produce a device identifier. Without a length check, an empty byte array would pass through both paths silently: `normalizeDevicePublicKeyBase64Url()` would return an empty encoded string, and `deriveDeviceIdFromPublicKey()` would hash the empty buffer, producing the fixed SHA-256 value `e3b0c44298fc1c149...` as the device ID — causing any two devices with empty keys to collide.

Both functions now return `null` immediately when `raw.length === 0`, treating zero-length decoded key material as invalid input.
