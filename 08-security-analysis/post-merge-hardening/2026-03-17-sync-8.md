#### Restore control-ui query token compatibility ([`6101c023b`](https://github.com/openclaw/openclaw/commit/6101c023b))

The control-UI's `applySettingsFromUrl()` in `ui/src/ui/app-settings.ts` previously read the gateway authentication token exclusively from the URL hash fragment (`hashParams.get("token")`). After this fix, it reads from the hash first and falls back to the query string (`hashParams.get("token") ?? params.get("token")`). This restores compatibility with token-passing flows that use query params instead of hash fragments — for example, generated share links or gateway URLs constructed by automation. The token is still only used for control-UI pairing and is not exposed beyond this call site.

#### Harden zizmor workflow diffing ([`916db21fe`](https://github.com/openclaw/openclaw/commit/916db21fe))

The zizmor security scanner step in `.github/workflows/ci.yml` previously determined the diff base SHA by executing a Python subprocess that parsed `$GITHUB_EVENT_PATH` JSON at runtime. This was replaced with a git-backed approach: `BASE_SHA` is set via a GitHub Actions expression (`github.event.before` for pushes, `github.event.pull_request.base.sha` for PRs) and then validated with `git cat-file -e "${BASE_SHA}^{commit}"` before use. If the SHA is absent, zero-valued, or unavailable in the shallow clone, the job now exits cleanly with a skip message rather than failing. This removes a Python subprocess from the security scanning path and eliminates the risk of malformed event JSON affecting SHA derivation.

#### Add image_generate tool ([`3a456678e`](https://github.com/openclaw/openclaw/commit/3a456678e))

A new `image_generate` agent tool (`src/agents/tools/image-generate-tool.ts`) is added to the coding profile. It supports "generate" and "list" actions, takes a `prompt` and optional `model`, `image`/`images` (for edit mode), `size`, `resolution`, and `count` fields. The tool is only registered when `cfg.agents.defaults.imageGenerationModel` is configured (`hasConfiguredImageGenerationModel()`) — it silently absent-loads in unconfigured installs. Image generation calls route through `generateImage()` and `listRuntimeImageGenerationProviders()` from `src/image-generation/runtime.js`, which use the configured provider credentials. User-supplied prompts are passed directly to external image APIs (Google, OpenAI), making this a new external API call surface where prompt injection from message content could reach an image API. The tool entry is added to the core tool definitions in `src/agents/tool-catalog.ts` (at lines 237–243) and a description is added to the system prompt's tool description map.

#### Restore full gate ([`449127b47`](https://github.com/openclaw/openclaw/commit/449127b47))

The plugin SDK barrel file (`src/plugin-sdk/`) had two HTTP registration modules — `http-path.js` and `http-registry.js` — missing from its exports after a prior refactor. This commit re-exports both (`export * from "../plugins/http-path.js"` and `export * from "../plugins/http-registry.js"`) and also adds `registerContextEngine` and `ContextEngineFactory` type exports. These re-exports restore the plugin HTTP route registration API surface, which includes auth enforcement hooks used by plugins that register HTTP endpoints. Without these exports, plugins relying on the route registration APIs at SDK install time would silently lose auth enforcement on their HTTP endpoints.

#### Telegram: fix named-account DM topic session keys ([`1eb810a5e`](https://github.com/openclaw/openclaw/commit/1eb810a5e))

When a named (non-default) Telegram account was matched via the "default" binding — i.e., `route.matchedBy === "default"` and `route.accountId !== DEFAULT_ACCOUNT_ID` — DM topic session keys incorrectly reused the generic route session key rather than building a per-account key. The new `resolveTelegramConversationBaseSessionKey()` function in `extensions/telegram/src/conversation-route.ts` detects this condition and constructs a session key with `dmScope: "per-account-channel-peer"` that incorporates the specific `accountId` and direct peer ID. For group chats the original session key is returned unchanged. This prevents DMs received through different named Telegram accounts from sharing a single session context, closing a potential cross-account session leakage path in multi-account Telegram setups.
