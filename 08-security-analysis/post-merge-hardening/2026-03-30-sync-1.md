#### Gateway: track background task lifecycle ([`17c36b509`](https://github.com/openclaw/openclaw/commit/17c36b509))

Adds durable tracking for background ACP task sessions, with `suppressUserDelivery` gating in `dispatch-acp.ts` and `dispatch-acp-delivery.ts` to prevent internal background task output from reaching user-facing channels. `suppressAcpChildUserDelivery` is set in `dispatch-from-config.ts` when the session is identified as a parent-owned background ACP child (`isParentOwnedBackgroundAcpSession`). This is defense-in-depth output containment — background tasks should not leak their intermediate state to users through the main channel delivery path.

#### Local trusted-proxy fallback to require token auth ([`1738d540f`](https://github.com/openclaw/openclaw/commit/1738d540f))

**Closes issue [#50635](https://github.com/openclaw/openclaw/issues/50635).** Rewrites `isLocalDirectRequest()` in `src/gateway/auth.ts` to use only `isLoopbackAddress(req.socket?.remoteAddress)` — the old implementation also accepted requests where the isLocalishHost() helper (defined in `src/gateway/net.ts`) matched the Host header, which allowed `*.ts.net` Tailscale hostnames in the Host header to satisfy the local-direct check and bypass gateway token authentication. The new implementation additionally returns `false` whenever any forwarding headers (`x-forwarded-for`, `x-forwarded-proto`, `x-real-ip`, `x-forwarded-host`) are present, failing closed for all proxied requests regardless of the socket address. This is a security fix: the previous fallback allowed header-spoofing to reach gateway endpoints that skip token auth for local-direct requests.

#### Fail closed when sandbox is unavailable and harden deny followups ([`5d81b6434`](https://github.com/openclaw/openclaw/commit/5d81b6434))

Two related hardening changes in `src/agents/bash-tools.exec.ts` and `src/agents/bash-tools.exec-approval-followup.ts`. First: the exec sandbox unavailability check in `createExecTool` now throws unconditionally whenever `host === "sandbox" && !sandbox` — the old condition had a sandboxHostConfigured guard that left a path for sandbox mode to silently fall through to gateway execution when the sandbox was unavailable. The fix removes that guard entirely. Second: `buildExecApprovalFollowupPrompt()` now detects `"Exec denied ("` prefixed result text and routes it through a new `buildExecDeniedFollowupPrompt()` function that instructs the agent not to run the command again, not to reuse earlier command output, and not to claim new output exists. This prevents agents from replaying denied commands or leaking prior execution context through the followup prompt.

#### Fail closed on silent turns ([`af694def5`](https://github.com/openclaw/openclaw/commit/af694def5))

Defense-in-depth hardening for agent output containment. Adds `silentExpected` gating in `pi-embedded-subscribe.ts` and `agent-runner-execution.ts` — when a followup run is marked `silentExpected`, text chunks are suppressed unless they match `isSilentReplyText(text, SILENT_REPLY_TOKEN)`, and the agent runner skips the followup delivery entirely. Previously, a silent turn could leak partial output through the subscribe handler if the `silentExpected` flag was not properly threaded. This prevents agents from emitting user-visible output on turns that were intended to be silent.

#### Allow memory_search and memory_get in sub-agent sessions ([`6c85c82ba`](https://github.com/openclaw/openclaw/commit/6c85c82ba))

**Security-notable policy change.** Removes `"memory_search"` and `"memory_get"` from `SUBAGENT_TOOL_DENY_ALWAYS` in `src/agents/pi-tools.policy.ts`. These two read-only tools were previously blocked for all sub-agents unconditionally; they are now available to sub-agents by default (same as read, web_search, and web_fetch, which were already allowed). The rationale: both are read-only with no state-modification capability, and multi-agent deployments relying on shared memory for context retrieval required users to manually configure `tools.subagents.tools.alsoAllow`. This is a deliberate policy boundary relaxation: the sub-agent attack surface is modestly expanded (a compromised sub-agent can now query shared memory), but the tools cannot write to memory, send messages, or affect external systems.

Simplification only; authorization logic unchanged. Preceded by `aee799262` below which introduced `isTelegramExecApprovalTargetRecipient`.

#### Accept approval callbacks from forwarding target recipients ([`aee799262`](https://github.com/openclaw/openclaw/commit/aee799262))

**Security-notable approval surface expansion.** When `approvals.exec.targets` routes an exec approval request to a Telegram DM, the recipient now qualifies as an authorized approver via `isTelegramExecApprovalTargetRecipient()` even without `channels.telegram.execApprovals` explicitly configured. The check verifies that `senderId` matches a configured `approvals.exec.targets` entry with `channel: "telegram"`, with `accountId` scoping for multi-bot deployments. This is config-gated: only users explicitly listed in `approvals.exec.targets` qualify. The expansion is intentional — users who receive approval buttons via forwarding were previously unable to act on them without a separate `execApprovals` config entry.
