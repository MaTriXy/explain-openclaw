#### Resolve pinned fs helper python without PATH ([`b86f5d5ea`](https://github.com/openclaw/openclaw/commit/b86f5d5ea))

The sandbox fs-bridge mutation helper previously resolved `python3` by invoking it bare from the shell, relying on the runtime `PATH`. In constrained sandbox environments (empty `PATH`, Docker with minimal env, restricted shells), this caused the mutation helper to fail at runtime. The fix introduces `SANDBOX_PINNED_MUTATION_PYTHON_CANDIDATES` (`src/agents/sandbox/fs-bridge-mutation-helper.ts:9`) — an ordered list of absolute paths (`/usr/bin/python3`, `/usr/local/bin/python3`, `/opt/homebrew/bin/python3`, `/bin/python3`). `buildPinnedMutationPlan()` (`:288`) now iterates through these candidates first using `[ -x 'path' ]` shell guards, only falling back to `command -v python3` if no absolute candidate is found. A new `SANDBOX_PINNED_MUTATION_PYTHON_SHELL_LITERAL` constant (`:286`) safely escapes the inline Python source for shell injection safety. The commit adds a test case covering the empty-PATH scenario to prevent regression. This is defense-in-depth hardening — it removes a PATH dependency from a security-critical sandbox primitive without changing the mutation helper's logic or sandbox isolation boundary.
