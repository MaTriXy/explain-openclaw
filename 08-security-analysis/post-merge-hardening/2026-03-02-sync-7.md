#### Sessions_spawn: inline attachments with redaction, lifecycle cleanup, and docs ([`a9f118878`](https://github.com/openclaw/openclaw/commit/a9f118878))

Adds inline file attachment support to `sessions_spawn` with multiple layered security controls. `sanitizeToolCallInputs` (in `src/agents/session-transcript-repair.ts`) redacts attachment file contents from transcript persistence so base64 payloads never reach stored logs. `safeRemoveAttachmentsDir` (in `src/agents/subagent-registry.ts`) performs symlink-safe cleanup of the attachment staging directory on session teardown. `decodeStrictBase64` (in `src/agents/subagent-spawn.ts`) validates incoming base64 data — returns null on padding errors, non-base64 chars, or oversized payloads. Files materialize to `.openclaw/attachments/<uuid>/` with configurable limits via `tools.sessions_spawn.attachments` (`maxFiles`, `maxFileBytes`, `maxTotalBytes`, `retainOnSessionKeep`).

#### Centralize boundary traversal and root path checks ([`1de320097`](https://github.com/openclaw/openclaw/commit/1de320097))

Centralizes path-traversal containment into a single validated abstraction. `createLexicalTraversalState` (in `src/infra/boundary-path.ts`) encapsulates traversal state as the explicit `LexicalTraversalState` type tracking `segments`, `canonicalCursor`, `lexicalCursor`, and `preserveFinalSymlink`. `resolvePathWithinRoot` is extracted as a shared helper in `src/infra/fs-safe.ts`, removing duplicated containment logic from `openFileWithinRoot`. Strengthens `isPathInside` containment checks for path traversal defense (CWE-22, Audit 1 Claim 6: Path traversal in agent dirs).

#### Allow ws:// to private network addresses ([`449511484`](https://github.com/openclaw/openclaw/commit/449511484))

`isSecureWebSocketUrl` now rejects plaintext `ws://` connections to non-loopback addresses by default. `isPrivateOrLoopbackHost` classifies RFC 1918, link-local, CGNAT, and IPv6 ULA/loopback ranges. A break-glass override is available via `OPENCLAW_ALLOW_INSECURE_PRIVATE_WS=1` for trusted private networks. The guard is applied in `buildGatewayConnectionDetails`, the `GatewayClient` constructor, and the onboard remote flow (CWE-319, relates to Audit 2 Claim 4: DNS rebinding SSRF via web-fetch).

#### Harden recycled PID lock recovery follow-up ([`5a2200b28`](https://github.com/openclaw/openclaw/commit/5a2200b28))

Addresses PID recycling in session write lock staleness checks — a container-specific issue where a different process can inherit the same PID as a dead lock holder. The lock file now records `starttime` (field 22 of `/proc/pid/stat`) alongside the PID. `isValidLockNumber` (in `src/agents/session-write-lock.ts`) validates that both `pid` and `starttime` payloads are non-negative integers. `getProcessStartTime` (in `src/shared/pid-alive.ts`) reads the start time on Linux; returns null on non-Linux platforms (check is skipped). During staleness evaluation, if the PID is alive but its start time differs from the recorded value, the lock is treated as stale and reclaimed immediately. Backward compatible: existing lock files without `starttime` fall back to the prior PID-alive + age-based logic.

#### Use 0o644 for inbound media files to allow sandbox read access ([`376a52a5b`](https://github.com/openclaw/openclaw/commit/376a52a5b))

Changes `MEDIA_FILE_MODE` in `src/media/store.ts` from `0o600` (owner-only read/write) to `0o644` (world-readable). The prior 0o600 mode prevented Docker sandbox containers running as a different UID from reading inbound media attachments. The 0o644 mode preserves owner-write exclusivity while allowing cross-UID reads within the sandbox boundary. Tests assert that URL-sourced inbound files use 0o644 and include platform-aware assertions for redirect handling.

#### Harden viewer security and docs ([`4a1be9825`](https://github.com/openclaw/openclaw/commit/4a1be9825))

Hardens the diffs extension HTTP handler against unauthenticated remote access and brute-force attempts. `normalizeRemoteClientKey` and `isLoopbackClientIp` (in `extensions/diffs/src/http.ts`) classify requests as local vs. remote. Non-loopback requests are rejected with 404 unless `allowRemoteViewer` is explicitly set to `true`. A `ViewerFailureLimiter` enforces per-client failure rate limits (40 failures per 60-second window; lockout for 60 seconds; max 2,048 tracked keys) with 429 responses and `Retry-After` headers. Invalid artifact ID or token failures from remote clients increment the limiter, protecting against token enumeration.

#### Exec/ACP: inject OPENCLAW_SHELL into child shell env ([`b7615e0ce`](https://github.com/openclaw/openclaw/commit/b7615e0ce))

Injects `OPENCLAW_SHELL` into the environment of all spawned child processes so they can detect they are running inside OpenClaw's execution context. Values: `"tui-local"` for TUI local shell runs (`src/tui/tui-local-shell.ts`), `"acp-client"` for ACP client spawns (`src/acp/client.ts`), and `"acp"` for ACPX runtime process spawns (`extensions/acpx/src/runtime-internals/process.ts`). The marker enables child processes and scripts to distinguish between constrained OpenClaw execution and unrestricted shell invocations — supporting policy-aware behavior and detection of execution context.

#### Guard against year-rollback in croner nextRun ([`4691aab01`](https://github.com/openclaw/openclaw/commit/4691aab01))

Two changes bundled: (1) Cron staleness guard — `src/cron/service/timer.ts` detects when croner's nextRun() returns a timestamp at or before now (a known edge case with certain timezone/date combinations such as Asia/Shanghai), retrying from the next whole second and then from midnight-tomorrow UTC before giving up; prevents runaway cron job scheduling. (2) Google Chat SSRF protection — `extensions/googlechat/src/api.ts` now imports `fetchWithSsrFGuard` from `openclaw/plugin-sdk` and routes all Google Chat API calls through the SSRF guard, preventing server-side request forgery via the Google Chat extension (Audit 2 Claim 4: SSRF via web-fetch).
