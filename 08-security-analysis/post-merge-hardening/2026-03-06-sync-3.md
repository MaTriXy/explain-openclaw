#### Topic-Aware ACP/Subagent Spawning in Telegram ([`d58dafae8`](https://github.com/openclaw/openclaw/commit/d58dafae8))

Introduces configurable thread binding support for Telegram accounts via a new `threadBindings` schema in `TelegramAccountSchemaBase` (`zod-schema.providers-core.ts`). The new `SessionThreadBindingsConfigShape` type adds fields: `enabled`, `idleHours`, `maxAgeHours`, `spawnSubagentSessions`, and `spawnAcpSessions`. A new `thread-bindings.ts` module implements the thread binding lifecycle and a `telegram-context.ts` module handles thread ID extraction. ACP and subagent spawns in Telegram now respect topic/thread boundaries with configurable session persistence, preventing cross-thread context bleed between concurrent sessions.

**Security impact:** Session isolation enhancement. ACP spawns are scoped to their originating Telegram topic/thread, preventing one thread's session context from leaking into another's agent context. No direct audit claim match; defense-in-depth for multi-session deployments.

#### Hostname-Aware Private WebSocket Override ([`d86a12eb6`](https://github.com/openclaw/openclaw/commit/d86a12eb6))

Hardens `isSecureWebSocketUrl()` (in `src/gateway/net.ts`) to distinguish private VPN/Tailnet hostnames from public IP literals when the OPENCLAW_ALLOW_INSECURE_PRIVATE_WS override is set. Uses `net.isIP()` to detect IP literals (returns 4 or 6) vs. hostname strings (returns 0): ws:// is permitted only for non-IP hostnames (e.g., Tailnet DNS names), while ws:// to IP literals is still blocked. Also adds `canBindToHost()` for gateway bind-address validation.

**Security impact:** Private network overlay support without widening public exposure. Enables VPN/Tailnet use cases (ws:// to Tailscale DNS) while maintaining protection against direct ws:// connections to public IP addresses.

#### Session-Scoped Browser Tab Cleanup ([`06a229f98`](https://github.com/openclaw/openclaw/commit/06a229f98))

Introduces the new `src/browser/session-tab-registry.ts` module with `trackSessionBrowserTab()`, `untrackSessionBrowserTab()`, and `closeTrackedBrowserTabsForSessions()`. The browser tool now registers each opened tab (URL, tabId, page reference) keyed by session/windowId when a tab opens, and deregisters it on close. `ensureSessionRuntimeCleanup()` in `src/gateway/server-methods/sessions.ts` calls `closeTrackedBrowserTabsForSessions()` to forcibly close all tabs belonging to cleaned-up sessions.

**Security impact:** Prevents browser session leakage across user sessions. Without this, tabs opened during one session could accumulate and remain accessible in future sessions. This is a new session-isolation control; no direct audit claim match.

#### Dual-Permission Poll Action Gating ([`6dfd39c32`](https://github.com/openclaw/openclaw/commit/6dfd39c32))

Adds a `poll: z.boolean().optional()` gating field to `TelegramActionConfig` schema in `zod-schema.providers-core.ts`. The new `resolveTelegramPollActionGateState()` function (in `src/telegram/accounts.ts`) enforces that BOTH the `sendMessage` AND `poll` permissions are enabled before allowing Telegram poll sending. The poll action handler in `src/channels/plugins/actions/telegram.ts` checks the gate state and throws an error if either permission is disabled, closing a path where partial permission access could enable unauthorized poll actions.

**Audit 2 Claim 5 impact:** Strengthens the action-level RBAC layer for Telegram. Previously, the `sendMessage` permission alone could be leveraged to send polls. The dual-permission gate ensures poll access requires explicit configuration of both permissions, reducing the blast radius of a misconfigured or partially-compromised account configuration.

#### API Key ByteString Safety for HTTP Headers ([`1ab939321`](https://github.com/openclaw/openclaw/commit/1ab939321))

Extends `normalizeSecretInput()` (in `src/utils/normalize-secret-input.ts`) to filter code points above `0xFF` (non-Latin1 characters) from API key strings before HTTP header construction. This prevents the ByteString crash that occurs when pasted API keys contain Unicode artifacts such as smart quotes, zero-width characters, or emoji. The function now returns an auth error on the fail-closed path rather than throwing a runtime crash.

**Security impact:** Hardens the API key normalization pipeline against malformed inputs. Prevents a DoS path where a pasted key with Unicode characters causes a ByteString header construction crash. New control; no direct audit claim match.

#### Required OAuth Scope Enforcement for OpenAI Codex ([`8088218f4`](https://github.com/openclaw/openclaw/commit/8088218f4))

Adds `augmentOpenAIOAuthScopes()` (in `src/commands/openai-codex-oauth.ts`) to enforce required OAuth API scopes before browser handoff during the OpenAI Codex OAuth flow. The function appends the required scopes — `api.responses.write`, `model.request`, and `api.model.read` — to the authorization URL, deduplicating case-insensitively. Tokens obtained via this flow now include the runtime permissions required by OpenAI API calls.

**Audit 1 OAuth claim adjacency:** The Audit 1 findings cover OAuth credential security (Claims 1–4). This commit addresses scope completeness — ensuring tokens have the minimum required permissions, preventing both over-privileged and under-privileged token issuance.

#### Built-In OpenAI Codex OAuth Login Path ([`92b489212`](https://github.com/openclaw/openclaw/commit/92b489212))

Adds `runBuiltInOpenAICodexLogin()` (in `src/commands/models/auth.ts`) to provide an embedded OAuth flow for OpenAI Codex authentication, replacing the previous plugin-delegated path. The new function handles credential storage via `writeOAuthCredentials()`, profile setup, and model defaults, enabling secure Codex auth without requiring external plugin installation.

**Audit 1 OAuth claim adjacency:** Improves OAuth login path security by consolidating credential management within the built-in credential store rather than plugin-delegated flows. Ensures Codex credentials follow the same storage and lifecycle patterns as other providers.

#### SecretRef-Backed Keys in Memory Embeddings ([`fb289b7a7`](https://github.com/openclaw/openclaw/commit/fb289b7a7))

Introduces the new `src/memory/secret-input.ts` module with helpers for secret normalization in memory embedding code paths. Updates `memory-search.ts`, `embeddings.ts`, and related provider files (`embeddings-gemini.ts`, `embeddings-ollama.ts`, `embeddings-remote-client.ts`) to handle SecretRef-backed API keys. Unresolved SecretRef values are now treated as explicit configuration errors (with informative error messages) instead of passing `undefined` to HTTP header construction, which previously caused silent failures or crashes. `types.tools.ts` and `extension-api.d.ts` are updated to expose the `SecretInput` type.

**Security impact:** Proper secret lifecycle management in the embeddings pipeline. Fail-closed behavior for unresolved secrets prevents silent auth failures or undefined values reaching HTTP headers. New control; no direct audit claim match.
