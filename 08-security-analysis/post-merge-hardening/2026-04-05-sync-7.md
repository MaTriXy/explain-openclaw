#### Block UV_PYTHON in workspace dotenv and harden uv installer env [AI] ([`b48b528b0`](https://github.com/openclaw/openclaw/commit/b48b528b0))

`UV_PYTHON` was not previously included in the workspace dotenv blocklist, allowing a malicious `.env` file in a user's workspace to override the Python interpreter used by `uv` during skill installation — potentially redirecting installs to an attacker-controlled binary. This commit adds `UV_PYTHON` to `BLOCKED_WORKSPACE_DOTENV_KEYS` (`src/infra/dotenv.ts:41`) and to the central `src/infra/host-env-security-policy.json` allow-list (line 106). A dedicated proof-of-concept test (`skills-install-uv-python-env-override.poc.test.ts`) was added to prevent regression. Extends the env-var blocklist first established in PR #12 (Gap #1 — already closed).

#### Defer bootstrap token revocation ([`a42f000b5`](https://github.com/openclaw/openclaw/commit/a42f000b5))

Previously, bootstrap tokens could be revoked before the pairing handshake commit had fully succeeded, leaving nodes unable to retry if an error occurred mid-handshake. This commit introduces `getDeviceBootstrapTokenProfile()` at `src/infra/device-bootstrap.ts:226`, which captures the token's profile (scopes, constraints) before revocation so that revocation can be safely deferred until after the handshake commit succeeds. The approach decouples token capture from token destruction, enabling the revoke-on-success pattern implemented in subsequent commits.

#### Track bootstrap profile redemption ([`b08d58c91`](https://github.com/openclaw/openclaw/commit/b08d58c91))

Adds explicit redemption tracking to the bootstrap token lifecycle. `redeemDeviceBootstrapTokenProfile()` at `src/infra/device-bootstrap.ts:243` marks a token profile as consumed, while two companion helpers — `resolveBootstrapProfileScopes()` and `bootstrapProfileSatisfiesProfile()` — enforce that the redeemed profile matches the expected scope set before the handshake is allowed to proceed. Without this, a replay attack could present a previously-issued bootstrap token to satisfy a pairing request it was not intended for.

#### Revoke bootstrap tokens after handshake commit ([`5e3a3c42c`](https://github.com/openclaw/openclaw/commit/5e3a3c42c))

Closes the revocation window opened by `a42f000b5`. After the pairing handshake commit succeeds in `src/gateway/server/ws-connection/message-handler.ts`, the bootstrap token is now revoked immediately. Previously, a successfully-paired node retained a live bootstrap token for the duration of the connection, creating a window during which a replayed or stolen token could be used to initiate a second pairing. With this fix, the token is single-use: it is captured (profile saved), presented, and then destroyed upon successful commit — regardless of whether the connection remains active.

#### Restore bootstrap tokens after send failure ([`39361d13b`](https://github.com/openclaw/openclaw/commit/39361d13b))

A side-effect of the deferred-revocation pattern (`a42f000b5`/`5e3a3c42c`) is that a send failure after profile capture but before successful commit could leave the token permanently revoked, forcing the user to re-pair. `restoreDeviceBootstrapToken()` at `src/infra/device-bootstrap.ts:215` handles this recovery path: if the handshake send fails, the token is restored to its pre-revocation state so that the pairing attempt can be retried without manual intervention. The restore path is only reachable on failure — success still results in permanent revocation.

#### Preserve mixed-role node scopes ([`089125301`](https://github.com/openclaw/openclaw/commit/089125301))

In multi-role gateway deployments (nodes that simultaneously act as both operator and non-operator), the pairing scope resolution was merging or truncating scope sets incorrectly, risking either privilege escalation (non-operator scopes elevated) or capability loss (operator scopes dropped). `resolveRoleScopedDeviceTokenScopes()` at `src/infra/device-pairing.ts:347` now partitions scopes by role before applying them, ensuring each role receives exactly the scopes it is entitled to. The fix is exercised in the accompanying test and has no user-visible impact on single-role deployments.
