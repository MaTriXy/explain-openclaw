#### Use session origin thread metadata in chat routing ([`59cd79d37`](https://github.com/openclaw/openclaw/commit/59cd79d37))

`resolveChatSendOriginatingRoute` in `src/gateway/server-methods/chat.ts` now includes `origin?.threadId` as a third fallback candidate alongside `deliveryContext.threadId` and `lastThreadId` when resolving the originating route for chat messages. The `ChatSendDeliveryEntry` type gains a `threadId` field in its `origin` sub-object. This prevents cross-session thread misattribution when the originating session provides thread metadata but the delivery context does not.

#### Use session origin delivery metadata in outbound targets ([`a9e9c7cbf`](https://github.com/openclaw/openclaw/commit/a9e9c7cbf))

`deliveryContextFromSession` in `src/utils/delivery-context.ts` now falls back to `origin.provider`, `origin.accountId`, and `origin.threadId` when top-level session fields (`channel`, `lastAccountId`) are absent. The `DeliveryContextSessionSource` type gains an `origin` sub-object mirroring the chat routing origin structure. This ensures outbound message targets use correct session attribution even when the delivery context is sparse, preventing messages from being routed to the wrong channel or account.

#### Resolve Telegram slash command bindings from sender peer ([`1c412b1ac`](https://github.com/openclaw/openclaw/commit/1c412b1ac))

`resolveBindingConversationFromCommand` in `src/plugins/commands.ts` now detects synthetic `slash:` prefixed `To` values used by native Telegram slash commands and prefers the `from` peer for binding resolution instead of the synthetic target. Without this fix, native Telegram slash commands could resolve to an incorrect conversation binding, potentially allowing command execution in the wrong context.

#### Plugins: clean up channel config on uninstall ([`8c079a804`](https://github.com/openclaw/openclaw/commit/8c079a804))

`removePluginFromConfig` in `src/plugins/uninstall.ts` now removes the matching `channels.<pluginId>` entry (exact key match) during plugin uninstall, in addition to the existing `plugins.*` cleanup. The `UninstallActions` type gains a `channelConfig` boolean field to track this action. Shared config keys (`defaults`, `modelByChannel`) are explicitly guarded from accidental removal. Without this fix, orphaned channel config entries caused gateway errors when referencing a channel whose backing plugin no longer existed.
