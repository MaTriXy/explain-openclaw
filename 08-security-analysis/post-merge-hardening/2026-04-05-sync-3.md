#### Support default subagent allowlists (#59944) (thanks @hclsys) ([`d92178471`](https://github.com/openclaw/openclaw/commit/d92178471))

Adds `allowAgents` (optional `string[]`) to the `agents.defaults.subagents` config object in `AgentDefaultsSchema` (`zod-schema.agent-defaults.ts`), with corresponding type and generated-schema updates in `types.agent-defaults.ts` and `schema.base.generated.ts`. Operators can now set `agents.defaults.subagents.allowAgents` to define a global fallback allowlist of permitted target agent IDs for `sessions_spawn`, without requiring each agent entry to repeat the same policy. The `"*"` wildcard is supported to allow any configured agent.

#### Fall back to defaults for subagents.allowAgents ([`a57766bad`](https://github.com/openclaw/openclaw/commit/a57766bad))

Implements the runtime fallback at both call sites where `allowAgents` is read: `agents-list-tool.ts` (the `agents_list` tool) and `subagent-spawn.ts` (the `sessions_spawn` handler). Both now resolve `allowAgents` via a two-level chain: `resolveAgentConfig(cfg, requesterAgentId)?.subagents?.allowAgents ?? cfg?.agents?.defaults?.subagents?.allowAgents`, defaulting to `[]` (deny all) when neither is set. This matches the existing pattern used by `runTimeoutSeconds` and other subagent defaults. Prior to this fix, per-agent entries that omitted `allowAgents` would silently deny all spawns rather than inheriting the operator's default policy.

#### Fail closed on missing mode ([`2c7eea8f1`](https://github.com/openclaw/openclaw/commit/2c7eea8f1))

Removes the `?? "local"` fallback from gateway mode resolution in `runGatewayCommand` (`src/cli/gateway-cli/run.ts`). Previously, if `gateway.mode` was absent from the config file — for example after config corruption or a minimal `openclaw onboard` write — the gateway would silently start in local mode. After this fix, `mode` is `undefined` when the field is missing, causing the startup guard to block with an error unless `--allow-unconfigured` is passed. This is a fail-closed change: an absent `gateway.mode` is now treated as an error rather than a safe default.

#### Clarify local mode guardrails ([`e2e1197fa`](https://github.com/openclaw/openclaw/commit/e2e1197fa))

Extracts the gateway startup guard into `getGatewayStartGuardErrors()` in `run.ts`, adding a distinct error branch for `mode === undefined`: "Gateway start blocked: existing config is missing gateway.mode. Treat this as suspicious or clobbered config." This directs operators to the config audit log (`configAuditPath`) and advises re-running `openclaw onboard --mode local` or setting the field manually. The `--allow-unconfigured` flag description is also tightened to clarify it bypasses the guard without repairing the config. Together with [`2c7eea8f1`](https://github.com/openclaw/openclaw/commit/2c7eea8f1), this makes the security posture explicit: a missing mode is suspicious, not benign.
