#### Use icacls /sid for locale-independent Windows ACL audit ([`7735a0b85`](https://github.com/openclaw/openclaw/commit/7735a0b85))

The Windows ACL audit was producing false-positives on non-English locales where `icacls` outputs localized SID display names (e.g., `NT AUTHORITY\???????` for Cyrillic). `classifyPrincipal()` couldn't match these against known trusted principals, causing SYSTEM and Administrators to be flagged as untrusted. The fix introduces `WORLD_SIDS` (a Set of trusted world-equivalent SID strings in canonical `s-1-x-y` form), `normalizeSid()` (strips the leading `*` from `/sid` mode output), and an updated `SID_RE` regex that accepts the optional `*` prefix — ensuring identity classification is locale-independent regardless of the Windows language setting.

#### Secrets: harden SecretRef-safe models.json persistence ([`8e20dd22d`](https://github.com/openclaw/openclaw/commit/8e20dd22d))

API keys and auth headers resolved during `normalizeProviders()` were previously included in the serialized models.json snapshot as plaintext, leaking credentials whenever the config was written to disk. The new `model-auth-markers.ts` module introduces `NON_ENV_SECRETREF_MARKER` (`"secretref-managed"`) and `SECRETREF_ENV_HEADER_MARKER_PREFIX` (`"secretref-env:"`); `resolveNonEnvSecretRefApiKeyMarker()` substitutes these opaque markers instead of the resolved credential value. Provider normalization in `normalizeProviders()` now calls these marker functions for every API key and auth header, so the on-disk representation never contains plaintext secrets — only resolvable references.

#### Strip custom auth headers on cross-origin redirects ([`46715371b`](https://github.com/openclaw/openclaw/commit/46715371b))

The cross-origin redirect guard in `fetch-guard.ts` previously used a block-list (`CROSS_ORIGIN_REDIRECT_SENSITIVE_HEADERS`) containing four header names (`authorization`, `proxy-authorization`, `cookie`, `cookie2`). Any custom auth headers beyond those four — such as `X-API-Key` or bearer tokens under non-standard names — would pass through on cross-origin redirects. The fix replaces the block-list with an allow-list (`CROSS_ORIGIN_REDIRECT_SAFE_HEADERS`) of standard safe headers; `retainSafeHeadersForCrossOriginRedirect()` now discards every incoming header not explicitly in the allow-list, ensuring no custom credentials are forwarded when a redirect crosses origins.

#### Harden proxied local viewer detection ([`44881b022`](https://github.com/openclaw/openclaw/commit/44881b022))

The diffs viewer previously classified a request as "local" solely by checking if the TCP socket address was loopback (`127.0.0.1` or `::1`). An attacker routing through a local proxy (e.g., a malicious tool on the same host) could bypass rate limiting and access controls by presenting a loopback socket while setting `X-Forwarded-For`. The fix introduces `hasProxyForwardingHints()` that inspects all standard forwarding headers, and `resolveViewerAccess()` now requires the socket to be loopback AND all proxy forwarding headers to be absent before setting `localRequest: true`. Helper functions `recordRemoteFailure()` and `resetRemoteFailures()` centralize the conditional limiter calls.

#### Auto-reply: scope allowlist store writes by account ([`70da80bcb`](https://github.com/openclaw/openclaw/commit/70da80bcb))

Allowlist store writes in the auto-reply command handler were not scoped by account, allowing an add or remove operation to affect allowlist entries across accounts. The new `updatePairingStoreAllowlist()` helper includes `accountId` in every `addChannelAllowFromStoreEntry()` and `removeChannelAllowFromStoreEntry()` call, and handles the special case of `DEFAULT_ACCOUNT_ID` by issuing an additional unscoped removal for backwards-compatibility. This prevents one account's allowlist modifications from inadvertently altering another account's access entries.

#### Harden profile mutation proxy guards ([`3a50e46cb`](https://github.com/openclaw/openclaw/commit/3a50e46cb))

Nostr profile mutation endpoints were guarded by a loopback socket check that could be bypassed when the gateway runs behind a reverse proxy. The fix adds `hasNonLoopbackForwardedClient()`, which inspects `X-Forwarded-For` (comma-separated hop list) and `X-Real-IP` headers for any non-loopback address. `normalizeIpCandidate()` strips IPv6 brackets and port suffixes before loopback testing to avoid format-based bypasses. `firstHeaderValue()` normalizes headers that may arrive as arrays. When any non-loopback forwarded client is detected, `enforceLoopbackMutationGuards()` now rejects the mutation request regardless of socket address.

#### Thread command authorization from webhook gate ([`b393b9e8f`](https://github.com/openclaw/openclaw/commit/b393b9e8f))

The Synology Chat channel handler was passing `commandAuthorized: true` unconditionally when delivering messages from the webhook handler to the agent, bypassing the `auth.allowed` result from `authorizeUserForDm()`. This meant any Synology Chat user could issue control commands even if the webhook gate had denied them. The fix threads `commandAuthorized: auth.allowed` from the webhook handler output so that authorization decisions made at the gate boundary are respected by the downstream message delivery path.
