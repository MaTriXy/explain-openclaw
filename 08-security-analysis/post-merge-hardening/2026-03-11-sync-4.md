#### Sanitize Docker env before marking OPENCLAW_CLI ([`bd33a340f`](https://github.com/openclaw/openclaw/commit/bd33a340f))

Docker sandbox env vars are now sanitized via `sanitizeEnvVars()` (with `EnvSanitizationOptions`) before the `OPENCLAW_CLI` exec marker is injected. Previously, sanitization happened after marking, which meant host env vars could leak into the marked env and be inherited by the Docker container. The fix also disables the Windows shell fallback (`allowShellFallback: false`) for Docker execution. Defense-in-depth on LD_PRELOAD / env injection (Audit 2 Claim 8).

#### Split conversation reset from admin reset ([`c91d1622d`](https://github.com/openclaw/openclaw/commit/c91d1622d))

Session reset logic is extracted into a dedicated `session-reset-service.ts` module with `performGatewaySessionReset()`. This separates the user-initiated `/new` or `/reset` conversation command from the admin-level reset RPC, each with their own access control path. The refactor strengthens RBAC by removing a shared handler that could previously be reached via different privilege levels (Audit 2 Claim 5).

#### Pin staged writes and fs mutations ([`a0d546257`](https://github.com/openclaw/openclaw/commit/a0d546257))

Sandbox filesystem bridge mutations are hardened to use `runPinnedWriteHelper()` for all staged writes. Pinned writes open files via fd rather than path, eliminating a symlink substitution window between path resolution and write. Path safety checks are also tightened in `fs-bridge-path-safety.ts`. Addresses Audit 1 Claim 5 (file permission race) and Claim 6 (path traversal via symlink).

#### Unify sandbox fs bridge mutations ([`72b0e00ea`](https://github.com/openclaw/openclaw/commit/72b0e00ea))

A new `fs-bridge-mutation-helper.ts` module centralizes all sandbox fs bridge mutation types (write, append, move, delete). All mutation paths now flow through the unified helper and share the same fd-pinned write path and path safety validation. This eliminates per-call path divergence that could bypass the symlink race protections added in `a0d546257`. Completes Audit 1 Claim 5 / Claim 6 hardening.

#### Simplify system.run approval model ([`68c674d37`](https://github.com/openclaw/openclaw/commit/68c674d37))

The `system.run` approval model is refactored to a simpler, more auditable structure. Approval request construction is centralized in `bash-tools.exec-approval-request.ts`, removing duplicated approval-building logic scattered across the exec host paths. Fewer code paths means a smaller audit surface for approval bypass. This is a preparatory refactor for the exact argv binding in `7289c19f1`.

#### Unify config write target policy ([`3a39dc4e1`](https://github.com/openclaw/openclaw/commit/3a39dc4e1))

Config write authorization is unified through `authorizeConfigWrite()` from `channels/plugins/config-writes.ts`. All `commands-config.ts` paths now call the shared authorization function, ensuring consistent RBAC enforcement for both channel-initiated and admin config commands. Closes a divergence where different command paths applied slightly different write target validation (Audit 2 Claim 5).

#### Bind system.run approvals to exact argv text ([`7289c19f1`](https://github.com/openclaw/openclaw/commit/7289c19f1))

`system.run` approvals are now bound to the exact argv string presented to the user. An approved command pattern cannot be reused for a different command that wasn't shown at approval time. This closes the approval reuse vector in GHSA-8g75-q649-6pv6, where a pre-approved pattern could be matched against a semantically different command constructed by the agent.

#### Pin fs-bridge staged writes ([`11924a702`](https://github.com/openclaw/openclaw/commit/11924a702))

Fs-bridge staged writes in `fs-bridge-shell-command-plans.ts` and `fs-bridge-write-helper.ts` are updated to use `runPinnedWriteHelper()`. All write paths opened via the fs bridge now use fd-pinned writes, completing the symlink race elimination across the full write surface (Audit 1 Claim 5).

#### Gateway: fail closed unresolved local auth SecretRefs ([`0125ce1f4`](https://github.com/openclaw/openclaw/commit/0125ce1f4))

When a gateway auth config entry contains a `SecretRef` that cannot be resolved at runtime, the gateway now fails closed (authentication fails) rather than failing open (treating the unresolved ref as empty/no credential). `resolveConfiguredGatewayCredentialInput()` propagates the `refPath` and `configured` state so unresolved refs are detected and rejected at auth time. Closes Audit 1 Claim 1 (credential handling fail-closed).

#### Fail closed on unbound interpreter approvals ([`daaf211e2`](https://github.com/openclaw/openclaw/commit/daaf211e2))

Node-host interpreter approvals now fail closed when the interpreter cannot be confirmed safe via `isInterpreterLikeSafeBin()`. A new `collectExistingFileOperandIndexes()` helper maps argv file operands to existing paths before approval, preventing approval substitution where a different file is executed than the one shown at approval time. Extends the system.run approval hardening.

#### Harden subagent control boundaries ([`aad014c7c`](https://github.com/openclaw/openclaw/commit/aad014c7c))

Subagent control scope enforcement is strengthened. `SUBAGENT_CONTROL_SCOPES` now defines the valid scope values (`"children"` and `"none"`), and `resolveSubagentController()` validates that the requesting session has the required control scope before allowing control operations. Prevents lateral control (non-parent sessions controlling sibling subagents). Audit 2 Claim 5 / subagent RBAC.

#### Enforce target account configWrites ([`8eac93941`](https://github.com/openclaw/openclaw/commit/8eac93941))

`authorizeConfigWrite()` now checks that the config write target matches the requesting account's bound channels. A config write from channel A cannot target channel B's config scope. Uses `resolveExplicitConfigWriteTarget()` to determine scope, then validates against the request's channel identity. Closes a cross-account config write vector (Audit 2 Claim 5 RBAC).

#### Fail closed for unresolved local gateway auth refs ([`702f6f330`](https://github.com/openclaw/openclaw/commit/702f6f330))

Gateway credential resolution is updated to detect unresolved local auth `SecretRef` entries via `resolveConfiguredGatewayCredentialInput()`. When `refPath` is present but the ref cannot be resolved, the credential is treated as unresolved and the connection is rejected. Works in tandem with `0125ce1f4` to cover both initial auth SecretRef validation and runtime credential drift (Audit 1 Claim 1).

#### Restrict leaf subagent control scope ([`ecdbd8aa5`](https://github.com/openclaw/openclaw/commit/ecdbd8aa5))

Leaf subagents (maximum spawn depth, no children of their own) now have their control scope set to `"none"` regardless of inherited scope. A leaf subagent can no longer issue control commands (steer, cancel, status) against any session, including its own siblings. `SUBAGENT_CONTROL_SCOPES` enforces this at the policy layer. Prevents privilege escalation through spawn-depth-limit boundary (Audit 2 Claim 5 / subagent scope).
