#### Voice Call: canonicalize Plivo V3 replay key ([`85777e726`](https://github.com/openclaw/openclaw/commit/85777e726))

Hardens Plivo V3 webhook signature verification against parameter-reordering attacks. The `createPlivoV3ReplayKey()` function at `extensions/voice-call/src/webhook-security.ts:731` now canonicalizes the parameter set before computing the replay key, ensuring that requests with reordered query parameters are recognized as replays of the same original request and rejected. Previously, parameter reordering could produce a different replay key, allowing an attacker who had captured a legitimately signed webhook to replay it with shuffled parameters and bypass deduplication. The companion `verifyPlivoWebhook()` function at `:874` calls this to construct the replay key before HMAC validation. This directly strengthens **Audit1-claim7** (webhook signature bypass) in the Plivo V3 path.

#### Restrict node pairing approvals ([`4d7cc6bb4`](https://github.com/openclaw/openclaw/commit/4d7cc6bb4))

Introduces `resolveNodeApprovalRequiredScope()` at `src/infra/node-pairing.ts:62` to determine the minimum scope required to approve a pending node pairing request. Approval requests that involve `system.run`-class capabilities now require the caller to hold `operator.write` or `operator.admin`; callers presenting only `operator.pairing` are rejected with a forbidden error. The function is called at `:191` inside the approval flow, with the caller's `callerScopes` checked at `:196`. This directly addresses **Audit2-claim5** (self-approving agent / no RBAC enforcement in pairing paths) by ensuring that scope-escalating pairing approvals are gated by server-side scope verification rather than trusting the caller's claimed access level.

#### Gateway: disconnect revoked device sessions ([`7a801cc45`](https://github.com/openclaw/openclaw/commit/7a801cc45))

Adds defense-in-depth session teardown for revoked devices. The gateway's device management methods now call `context.disconnectClientsForDevice?.()` at `src/gateway/server-methods/devices.ts:177` and `:301` when a device token is revoked or removed. Previously, revoked devices could maintain live WebSocket connections until they timed out, creating a window for continued unauthorized access. Now revocation immediately tears down associated sessions. While not a direct fix for any of the 16 audit claims, this closes a credential-lifecycle gap that was identified as a defense-in-depth hardening opportunity.

#### Telegram: tighten media SSRF policy ([`20c7cbbf7`](https://github.com/openclaw/openclaw/commit/20c7cbbf7))

Tightens the SSRF policy applied during Telegram media resolution. The media fetch path now passes an explicit `ssrfPolicy` that restricts outbound fetches to `api.telegram.org` (plus any operator-configured custom API root host), disables the RFC 2544 benchmark range exemption (`allowRfc2544BenchmarkRange: false`), and validates the resolved hostname against the allowlist before fetching. This prevents a scenario where a crafted `file_path` in a Telegram `getFile` response could redirect the media fetch to an internal network address. The change is defense-in-depth against server-side request forgery through Telegram's file download channel.

#### CLI: confirm discovered remote gateways before saving config ([`d6affb17d`](https://github.com/openclaw/openclaw/commit/d6affb17d))

Adds explicit trust confirmation before a discovered remote gateway is saved to the CLI config. When the Bonjour discovery flow in `src/commands/onboard-remote.ts` resolves a remote gateway, the user is now prompted to confirm they trust the endpoint — displaying the host, port, and TLS fingerprint (if advertised). #### Net: block missing IPv6 special-use ranges ([`d61f8e567`](https://github.com/openclaw/openclaw/commit/d61f8e567))

Adds the missing IPv6 special-use ranges to `BLOCKED_IPV6_SPECIAL_USE_RANGES` at `src/shared/net/ip.ts:26`. The set is extended with ranges that `ipaddr.js` does not classify as private but that must be blocked to prevent SSRF: site-local addresses (`fec0::/10`, checked at `:246-247` via `(address.parts[0] & 0xffc0) === 0xfec0`) and the full set of IANA-designated IPv6 special-use ranges.
