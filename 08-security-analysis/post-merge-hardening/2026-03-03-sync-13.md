## Summary

Voice-call extension now verifies each persisted call against the provider API on gateway restart before restoring it to the active call map. All three providers (Plivo, Twilio, Telnyx) gain a `getCallStatus()` method backed by `guardedJsonApiRequest()` with provider-specific hostname allowlists. This eliminates phantom calls that could block the concurrent-call limit and closes an SSRF avenue in the voice-call extension's outbound provider API calls.

### Voice-Call Provider Status Verification ([`ffa7c13c9`](https://github.com/openclaw/openclaw/commit/ffa7c13c9))

`getCallStatus()` is added to the `VoiceCallProvider` interface and implemented in all three providers. Each implementation calls `guardedJsonApiRequest()` with an explicit `allowedHostnames` constraint tied to the provider's API hostname — `this.apiHost` for Plivo, `api.twilio.com` for Twilio, and `api.telnyxapis.com` for Telnyx. This applies the same SSRF controls already used elsewhere in the codebase to the voice-call extension's outbound HTTP calls.

`CallManager.initialize()` becomes `async` and calls the new `verifyRestoredCalls()` private method before adding persisted calls to the active map. For each persisted call carrying a `providerCallId`, the method queries the provider and drops the call if: the provider reports a terminal status (completed, failed, busy, no-answer, canceled), the call is older than `maxDurationSeconds`, or the provider returns 404. Calls with transient errors (HTTP 5xx) are kept and start a duration-limit timer as a fallback.

## Audit Cross-Reference

**Audit 2 Claim 4 (DNS rebinding / SSRF via web fetch):** The voice-call extension previously made outbound calls to provider APIs without an `allowedHostnames` constraint. This commit closes that gap: `guardedJsonApiRequest()` now enforces hostname pinning for all three voice-call provider API calls, consistent with the SSRF defenses described in the Medium article audit.

The 3 legitimate gaps are unchanged.
