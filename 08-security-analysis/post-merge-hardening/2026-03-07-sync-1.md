#### Extend plugin system to support custom context management ([`fee91fefc`](https://github.com/openclaw/openclaw/commit/fee91fefc))

`registerContextEngine` is added to the plugin registry API alongside existing hook registration points. `resolveContextEngine` resolves the active context engine at session init, falling back to `LegacyContextEngine` when no plugin overrides the slot, preserving backward compatibility. Plugin-supplied context engines run within the same trust boundary as the gateway — no sandbox isolation change is introduced, so context engine plugins should be held to the same trust level as other gateway plugins.

#### Gateway: add path-scoped config schema lookup ([`ff9719550`](https://github.com/openclaw/openclaw/commit/ff9719550))

`lookupConfigSchema` validates incoming path segments against `FORBIDDEN_LOOKUP_SEGMENTS` (containing traversal-relevant tokens) and enforces a maximum depth limit via `MAX_LOOKUP_PATH_SEGMENTS` before performing the lookup. Slash-delimited schema IDs are now permitted for plugin-namespaced schemas, but each segment is independently validated against the blocklist. This prevents a malformed plugin schema ID from traversing the config namespace hierarchy to enumerate or shadow unintended configuration entries.

#### Gateway: discriminate input sources ([`ff334600d`](https://github.com/openclaw/openclaw/commit/ff334600d))

`extractImageContentFromSource` now operates on a discriminated union type (`InputImageSource` with a `data:` variant and a `url:` variant), making the type system enforce the correct handling path at compile time. Previously both `data` and `url` fields coexisted in the same object shape, allowing a runtime value to be handled as either type depending on which field was checked first. The discriminated union eliminates the type confusion class where a crafted input could supply both fields and cause ambiguous processing — only one variant is possible per value.

#### Gateway: coerce chat deliverable route boolean ([`a939a1560`](https://github.com/openclaw/openclaw/commit/a939a1560))

The chat routing decision now explicitly coerces the `ExplicitDeliverRoute` expression to a boolean before evaluating it as a delivery gate. Previously a truthy non-boolean intermediate value (such as an object or non-empty string) could pass the routing check, producing inconsistent delivery behavior. This makes routing decisions uniform: the gate is either `true` (deliver) or `false` (suppress), with no ambiguity from truthy intermediate expressions.

#### Fix Control UI duplicate iMessage replies for internal webchat turns ([`6c39616ec`](https://github.com/openclaw/openclaw/commit/6c39616ec))

`isInternalWebchatTurn` detects whether a turn originated from the internal web UI session, which should not be echoed to the iMessage channel. `ExplicitDeliverRoute` gates iMessage delivery so only genuine external iMessage turns trigger a reply; webchat-originated turns are suppressed on the iMessage path. Without this guard, a user chatting via the Control UI webchat could silently receive duplicate replies on iMessage when both channels were active, exposing conversation content on an unintended surface.

#### Support image_url in OpenAI chat completions ([`9c86a9fd2`](https://github.com/openclaw/openclaw/commit/9c86a9fd2))

External `image_url` content in OpenAI API requests is now fetched through the gateway's existing SSRF-protected `web-fetch` path rather than being forwarded directly to the upstream model API. The feature is opt-in and enforces per-turn fetch budgets with incremental byte counting, preventing unbounded resource consumption from large images. Payload redaction strips raw image bytes from internal logs before dispatch, addressing a secondary data-exposure concern alongside the SSRF defense (Audit 2 Claim 4).

#### CLI: make read-only SecretRef status flows degrade safely ([`0e4245063`](https://github.com/openclaw/openclaw/commit/0e4245063))

`SecretRef` status inspection commands now detect a read-only secrets backend and degrade gracefully with an informative message rather than throwing an uncaught error. The degradation path preserves the audit trail for the inspection attempt, maintaining coverage for Audit 1 Claim 1 even when the backend is unavailable. This prevents sensitive information from leaking through stack traces or error messages when the credentials store is in a restricted state (e.g., CI environments, ephemeral containers, guest mode).

#### Remove bogus codex oauth responses probe ([`bdd368533`](https://github.com/openclaw/openclaw/commit/bdd368533))

`detectMissingResponsesWriteScope` was a probe function that detected a missing OAuth scope by dispatching a test request using the user's live access token to an OpenAI endpoint. The probe was unnecessary: the scope gap could be inferred locally from token metadata, so the live API call served no diagnostic purpose and unnecessarily transmitted the token to an upstream endpoint. Removing it eliminates an OAuth token exposure vector for Codex OAuth users, particularly in environments where outbound request logging is active.

#### Stop mutating authorize url scopes ([`a4a490bae`](https://github.com/openclaw/openclaw/commit/a4a490bae))

`augmentOpenAIOAuthScopes` silently appended additional OAuth scopes to the authorization URL when a Codex OAuth session was detected, widening the scope set beyond what the user reviewed during the consent screen. This silent mutation meant the final authorized scope set could include capabilities not explicitly requested by the user. Removing the function ensures OAuth authorization requests are immutable after construction — no scopes are added after the user initiates the flow.

#### Preserve credential-like tokens in render sanitization ([`8d4a2f2c5`](https://github.com/openclaw/openclaw/commit/8d4a2f2c5))

`isCopySensitiveToken` exempts strings that resemble credentials (OAuth tokens, API keys) from the TUI render sanitization pass. The guard uses `TOKENISH_MIN_LENGTH=24` as the minimum length threshold: strings at or above this length matching credential-like patterns are passed through unsanitized to preserve their full value for the user to copy. Before this fix, the render sanitization pass could silently truncate or mangle credential-length strings in the TUI display, causing copy-paste failures for users retrieving tokens.

#### Remove config.schema from agent gateway tool ([`8c85ad540`](https://github.com/openclaw/openclaw/commit/8c85ad540))

`config.schema` was included in `GATEWAY_ACTIONS` — the enumeration of gateway operations accessible via agent tool calls. Exposing the config schema to agent queries allowed a potentially injected or compromised agent to enumerate all valid configuration keys, providing reconnaissance that could inform targeted config manipulation attacks. Removing the entry reduces the information surface accessible to agents without affecting legitimate agent workflows that interact with typed config values.

#### Doctor: warn on implicit heartbeat directPolicy ([`e5481ac79`](https://github.com/openclaw/openclaw/commit/e5481ac79))

The new `doctor-security.ts` check warns when the heartbeat configuration uses an implicit `directPolicy` (relying on default behavior) rather than an explicit allow/deny declaration. Implicit policies are brittle across version upgrades: a behavior change in how defaults are resolved could silently alter security posture without any operator action. The warning surfaces in `openclaw doctor` output so operators can audit and explicitly declare their intended heartbeat policy.

#### Avoid synthetic tool-result writes on idle-timeout cleanup ([`4daaea119`](https://github.com/openclaw/openclaw/commit/4daaea119))

`clearPendingToolResults` drains any buffered tool-result entries when a session is idle-timed out, preventing stale results from being replayed into a subsequent session. `installSessionToolResultGuard` is called at session initialization and installs this cleanup hook to ensure draining always fires, even during abnormal session termination. Without draining, an attacker with channel access could craft a tool result, trigger an idle timeout, and have the stale result injected into the next session's tool-result queue.

#### Announce delivery with descendant gating, frozen result refresh, and cron retry ([`81b93b9ce`](https://github.com/openclaw/openclaw/commit/81b93b9ce))

`shouldIgnorePostCompletionAnnounce` gates completion announcement delivery by verifying the announcing agent is a descendant of the session's authorization boundary, preventing sub-agents from broadcasting to channels the parent was not authorized to write. The frozen result refresh captures the result snapshot at completion time, preventing concurrent mutation from influencing announcement decisions (TOCTOU hardening). Cron retry logic for announce delivery is added alongside these gates to ensure guaranteed-at-least-once delivery within the authorization boundary.

#### Harden openai-codex oauth refresh fallback ([`fa3fafdde`](https://github.com/openclaw/openclaw/commit/fa3fafdde))

`shouldUseOpenaiCodexRefreshFallback` validates the shape, expiry, and issuer of the current OAuth credential before allowing the Codex-specific refresh fallback path to activate. Previously the fallback was unconditional: any OAuth credential, including partial or malformed ones, could trigger the alternative refresh mechanism. The validation gate ensures the fallback is used only for credentials that genuinely match the Codex OAuth profile, preventing credential-type confusion from silently activating an unintended refresh path.

#### Enforce 600 perms for cron store and run logs ([`0e2bc588c`](https://github.com/openclaw/openclaw/commit/0e2bc588c))

`setSecureFileMode` is called on the cron store and run log files at creation time to set `0600` (owner read/write only) permissions. Previously these files were created with the process umask in effect, which on permissive-umask systems (e.g., `0022`) results in world-readable files containing cron schedules and run log data. Restricting to `0600` prevents other local users on a shared system from reading cron configuration that may include scheduled command content with embedded arguments.
