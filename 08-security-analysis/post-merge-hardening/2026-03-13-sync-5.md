#### Harden nodes owner-only tool gating ([`9692dc766`](https://github.com/openclaw/openclaw/commit/9692dc766))

The nodes tool definition now carries `ownerOnly: true`, restricting invocation to the gateway owner role. The nodes tool surfaces device operations such as screen recording, camera access, location data, and remote notifications — capabilities that require the highest privilege level. A new test file `openclaw-tools.owner-authorization.test.ts` covers the gating logic, alongside additions to the existing `pi-tools.whatsapp-login-gating.test.ts` and `nodes-tool.test.ts`. SECURITY.md was updated to require evidence from shipped release tags (not just `main`) when reporting vulnerability claims against specific versions.

#### Fail closed on ruby approval preload flags ([`ddcaec89e`](https://github.com/openclaw/openclaw/commit/ddcaec89e))

A new constant `RUBY_UNSAFE_APPROVAL_FLAGS` defines the set `{"-I", "-r", "--require"}` in `invoke-system-run-plan.ts`, and the new function `hasRubyUnsafeApprovalFlag()` scans Ruby command argv for these flags, respecting the `--` end-of-options marker. When Ruby is the executable and `hasRubyUnsafeApprovalFlag()` returns true, `resolveMutableFileOperandIndex` returns `null` (fail closed), preventing the approval prompt from rendering. This blocks `-I` (load-path injection) and `-r`/`--require` (file require) from being used to load attacker-controlled Ruby code through the node-host approval mechanism.

#### Fail closed on unreadable loader paths ([`d4f535b20`](https://github.com/openclaw/openclaw/commit/d4f535b20))

The hook loader now uses `resolveExistingRealpath()` (defined in `loader.ts`) instead of `safeRealpathOrResolve`, returning `null` when a hook's base directory no longer exists or is unreadable; the hook entry is then skipped with a warning rather than thrown. All hook names, handler paths, and export names in log output are now sanitized through `safeLogValue()` (which delegates to `sanitizeForLog()` from `terminal/ansi.ts`), preventing ANSI escape injection in log streams. A new helper `maybeWarnTrustedHookSource()` logs an explicit warning when workspace or managed hook code is loaded into the gateway process, surfacing implicit trust decisions that were previously silent.

#### Dedupe repeated agent deliveries by idempotency key ([`2649c03cd`](https://github.com/openclaw/openclaw/commit/2649c03cd))

Hook agent deliveries now support client-supplied idempotency keys. In `hooks.ts`, `resolveOptionalHookIdempotencyKey()` accepts keys from the `idempotency-key` or `x-openclaw-idempotency-key` HTTP headers or from `payload.idempotencyKey`, trimming and rejecting keys longer than `MAX_HOOK_IDEMPOTENCY_KEY_LENGTH` (256 chars). In `server-http.ts`, the replay-cache key is built from three separate SHA-256 fingerprints — token, scope, and idempotency key — so duplicate deliveries with the same key are suppressed without leaking credential material into the cache index.

#### Trim bootstrap token metadata ([`1c7ca391a`](https://github.com/openclaw/openclaw/commit/1c7ca391a))

The `DeviceBootstrapTokenRecord` type in `device-bootstrap.ts` drops the `channel`, `senderId`, `accountId`, and `threadId` fields, and `issueDeviceBootstrapToken()` no longer accepts those parameters. These fields carried channel-routing context (originating platform, sender identity, thread ID) that was written into the persistent token store even though the bootstrap token's purpose is only to facilitate device pairing. Removing them reduces the PII exposure surface in bootstrap token records and eliminates a potential correlation vector if the token store is accessed by an attacker.
