#### Patch hono transitive audit vulnerabilities ([`809f9513a`](https://github.com/openclaw/openclaw/commit/809f9513a))

Two `pnpm.overrides` entries in `package.json` are updated to pin vulnerable transitive versions of hono. `hono` is bumped from `4.11.10` to `4.12.5`, and a new override adds `@hono/node-server` at `1.19.10`. These overrides prevent older hono/node-server versions — pulled in by transitive dependencies — from being installed. The gateway HTTP layer (built on Hono) processes incoming webhook and API traffic, making this a supply chain patch for the active request-handling surface.

#### Align AGENTS.md template section names with post-compaction extraction (#25029) ([`96021a2b1`](https://github.com/openclaw/openclaw/commit/96021a2b1))

`readWorkspaceContextForSummary()` in `compaction-safeguard.ts` previously extracted only `"Session Startup"` and `"Red Lines"` from AGENTS.md. If a user's AGENTS.md used the older template headings (`"Every Session"` and `"Safety"`), the extraction returned empty — silently dropping Red Lines and session context from every compaction summary. This commit adds a fallback: when `extractSections(content, ["Session Startup", "Red Lines"])` returns empty, the function retries with the legacy names. The fix ensures the compaction safeguard's context-preservation guarantees hold regardless of which template version the user's AGENTS.md follows.

#### Compaction/Safeguard: require structured summary headings ([`df0f2e349`](https://github.com/openclaw/openclaw/commit/df0f2e349))

A significant hardening of the compaction summary pipeline against prompt injection. New constants `REQUIRED_SUMMARY_SECTIONS` (5 required headings: `## Decisions`, `## Open TODOs`, `## Constraints/Rules`, `## Pending user asks`, `## Exact identifiers`) enforce structured output from the summarization model. New helper `sanitizeUntrustedInstructionText()` cleans untrusted content before it enters prompts: it normalizes line endings, runs `sanitizeForPromptLiteral()` to strip control characters, caps at `MAX_UNTRUSTED_INSTRUCTION_CHARS = 4000`, and HTML-escapes `<` and `>`. New helper `wrapUntrustedInstructionBlock()` wraps user-provided workspace context in `<untrusted-text>` tags, signaling to the model that the enclosed content is data, not instructions. Compaction summaries missing any required heading are now rejected. This closes a prompt injection path where a malicious AGENTS.md or workspace file could inject instructions into the compaction summary and survive context rollover.

#### Enforce behavioral sweep validation ([`4cc293d08`](https://github.com/openclaw/openclaw/commit/4cc293d08))

The `review_validate_artifacts()` function in `scripts/pr` now validates a `behavioralSweep` JSON object in `.local/review.json`. Required fields: `performed: true`, `status` ∈ `{pass, needs_work, not_applicable}`, `silentDropRisk` ∈ `{none, present, unknown}`, and `branches[]` (each entry must have string `path`, `decision`, `outcome`). When a PR touches runtime files under `src/`, `extensions/`, or `apps/` (excluding test/doc files), `status: not_applicable` is rejected — at least one branch entry is required. Additionally, `.local/pr-meta.json` is now required alongside `review.md` and `review.json`. This gate prevents silent behavioral changes from landing without documented sweep evidence.

#### Pass provider headers to Ollama stream function ([`7597fc556`](https://github.com/openclaw/openclaw/commit/7597fc556))

`createOllamaStreamFn()` in `ollama-stream.ts` gains an optional `defaultHeaders?: Record<string, string>` parameter. These headers are merged at lowest precedence into every fetch request (before request-level `options.headers`), enabling Ollama endpoints behind reverse proxies that require auth headers (e.g. `X-OLLAMA-KEY` via HAProxy). Previously, provider-configured headers were silently discarded at the Ollama stream layer, causing 403 errors against protected local Ollama deployments.

#### Sync rawCommand with hardened argv after executable path pinning ([`c8ebd48e0`](https://github.com/openclaw/openclaw/commit/c8ebd48e0))

`hardenApprovedExecutionPaths()` in `invoke-system-run-plan.ts` now returns an `argvChanged: boolean` flag alongside the hardened `argv`. `buildSystemRunApprovalPlan()` uses this flag to set `rawCommand` correctly: when path pinning changed `argv[0]` (i.e. `argvChanged = true`), `rawCommand` is derived from `formatExecCommand(hardening.argv)`, reflecting the pinned absolute executable path. Previously, `rawCommand` always preserved the original command text even when pinning had replaced the executable — creating a semantic mismatch where the approval record showed the original command while the actual execution used the pinned path. This provides additional hardening relevant to GHSA-h3f9-mjwj-w476 (Node host system.run rawCommand/command mismatch, HIGH, currently pending).

#### Propagate custom provider headers to model objects ([`3fa43ec22`](https://github.com/openclaw/openclaw/commit/3fa43ec22))

`buildInlineProviderModels()` and `resolveModel()` in `model.ts` now include a `headers` field on model objects, merged from both provider-level and model-level configuration. In `resolveModel()`, when a `providerOverride` exists with `headers`, those headers are deep-merged onto any headers already on the model (provider headers take lower precedence than model-level headers). This ensures provider-configured auth headers propagate all the way to model objects — a prerequisite for `7597fc556`'s `createOllamaStreamFn(baseUrl, model.headers)` call site to actually pass provider headers to the Ollama HTTP layer.
