#### Share sender group policy evaluation ([`8c15b8600`](https://github.com/openclaw/openclaw/commit/8c15b8600))

Prior to this commit, each messaging channel independently implemented group sender access evaluation, which created a risk of divergent policy checks as channel-specific implementations drifted. This refactor extracts the shared logic into `evaluateSenderGroupAccessForPolicy()` in `src/plugin-sdk/group-access.ts`, which is now consumed directly by LINE, Signal, and other channel adapters. By centralizing group policy evaluation into a single location, this commit ensures that allowlist checks, mention-gating decisions, and sender permission logic are applied consistently across channels. Inconsistent permission checks (Audit 1 Claim 5) are a recognized defense-in-depth gap; this consolidation directly reduces that risk.

#### Unify gateway SecretRef auth resolution paths ([`fecca6fd8`](https://github.com/openclaw/openclaw/commit/fecca6fd8))

Gateway authentication previously had multiple independent code paths for resolving secret references — plaintext token values, environment variable references, and SecretRef objects — with each path implementing its own lookup and validation logic. This refactor centralizes all resolution into two shared functions in `src/gateway/resolve-configured-secret-input-string.ts`: `resolveConfiguredSecretInputWithFallback()` for optional secret resolution with a fallback, and `resolveRequiredConfiguredSecretRefInputString()` for required secret resolution that fails loudly on missing input. Callers across `startup-auth.ts`, `qr-cli.ts`, `daemon-cli/status.gather.ts`, and other modules are migrated to use these shared functions. Centralizing secret resolution (Audit 1 Claim 1) reduces the risk of inconsistent plaintext token handling and narrows the number of places where a token expiry or format bypass vector could be introduced (Audit 1 Claim 8).

#### Cover recent landed fix paths ([`7ab49a7fb`](https://github.com/openclaw/openclaw/commit/7ab49a7fb))

This test-only commit adds regression coverage for hardening paths introduced by recently landed fixes. The test suite now verifies that control-character sanitization functions strip disallowed characters correctly, preventing log injection and spoofing attacks where malicious content could manipulate structured log output or confuse downstream parsers. Six test files are added, all covering the sanitization and fix paths that were landed in the preceding syncs. No production code is changed.

#### Explain provider-object password bootstrap errors (#39230, thanks @ademczuk) ([`ab704b7ac`](https://github.com/openclaw/openclaw/commit/ab704b7ac))

A common misconfiguration is supplying a full provider object (containing a `password` field) in a position that expects only a plaintext token or password string. Without clear error messaging, this silently resolves to an unexpected value and leaves the gateway in an ambiguous auth state. This fix improves gateway startup validation to detect this pattern early and emit a clear, actionable error message when a provider-object is found where a simple secret string is expected. The fix applies to both token and password bootstrap paths (Audit 1 Claims 1 and 8), ensuring that unresolved or misconfigured secret inputs are caught at startup rather than at first authentication attempt.

#### Land #38725 from @ademczuk ([`3a74dc00b`](https://github.com/openclaw/openclaw/commit/3a74dc00b))

This fix addresses a token refresh race condition in gateway reconnect logic. Previously, clients that attempted to reconnect before a token was available could trigger rate-limiting as if they had presented wrong credentials, causing legitimate clients to be locked out during transient auth delays. The fix refines rate-limit behavior to distinguish between clients that present an incorrect credential (wrong credentials — should be rate-limited) versus clients that present no credential yet (missing credentials — should not be rate-limited until a token is fully available). This reduces the window in which a reconnecting client could be incorrectly penalized, mitigating the token refresh race condition exposure described in Audit 1 Claim 4.

#### Stop webchat route inheritance on channel sessions (#39175, thanks @widingmarcus-cyber) ([`b4bac484e`](https://github.com/openclaw/openclaw/commit/b4bac484e))

Webchat clients connected via a channel-scoped session previously could inherit the route resolution logic of the parent channel, allowing the webchat path to reach delivery channels it was not authorized to use. This fix adds an explicit check via `isWebchatClient()` in `src/utils/message-channel.ts` to the route resolution function `resolveChatSendOriginatingRoute()` in `src/gateway/server-methods/chat.ts`, causing webchat sessions to short-circuit channel-specific route inheritance and use only their own delivery path. This prevents path traversal and unintended cross-channel delivery that could occur when channel-scoped sessions exposed webchat to inherited routes (Audit 1 Claim 6).
