#### Preserve raw activation source for startup plugin loads ([`cee5f960b`](https://github.com/openclaw/openclaw/commit/cee5f960b5))

This commit is the fourth in the allowlist bypass fix chain that began in sync 9 (commits `f7d24c1ed5`, `cd08facd7a`, `bc137951e9`). It corrects a subtler version of the same issue: in `loadGatewayStartupPlugins` (called from `server.impl.ts`), the gateway previously passed `gatewayPluginConfigAtStart` — the config after auto-enable — as the `activationSourceConfig`. The raw config snapshot (`cfgAtStart`) was not forwarded. This meant that when `resolvePluginActivationState` evaluated whether a channel was explicitly configured by the admin, it evaluated against the already-auto-enabled config, making auto-enabled bundled channels appear "explicitly configured" and allowing them to bypass the allowlist gate they should not pass.

The fix adds `activationSourceConfig?: ReturnType<typeof loadConfig>` to `GatewayPluginBootstrapParams` in `src/gateway/server-plugin-bootstrap.ts`. `prepareGatewayPluginLoad` now passes `params.activationSourceConfig ?? params.cfg` to both `applyPluginAutoEnable` and `loadGatewayPlugins` instead of always using `params.cfg`. At the call site in `src/gateway/server.impl.ts`, `loadGatewayStartupPlugins` now explicitly passes `activationSourceConfig: cfgAtStart`, ensuring the activation decision uses the raw config, not the post-auto-enable effective config.

#### Unify plugin activation source plumbing ([`cd38eba31`](https://github.com/openclaw/openclaw/commit/cd38eba316))

The companion refactor to `cee5f960b5`. It introduces the `PluginActivationConfigSource` type in `src/plugins/config-state.ts` (a struct containing `plugins: NormalizedPluginsConfig` + `rootConfig?: OpenClawConfig`) and the `createPluginActivationSource()` factory function. In `resolvePluginActivationState`, the two separate `sourceConfig?: NormalizedPluginsConfig` + `sourceRootConfig?: OpenClawConfig` parameters are replaced by a single `activationSource?: PluginActivationConfigSource`. The `resolveExplicitPluginSelection` call now receives `activationSource.plugins` and `activationSource.rootConfig` from the consolidated struct. The refactor propagates through `src/plugins/loader.ts`, `src/plugins/channel-plugin-ids.ts`, and `src/plugin-sdk/facade-runtime.ts`.

This unification eliminates the risk of future bypasses from passing mismatched separate params (e.g., passing `sourceConfig` without `sourceRootConfig` or vice versa). With the consolidated source, all allowlist activation decisions are guaranteed to use the same config snapshot for both the normalized plugin config and the raw root config.

#### Migrate legacy sandbox perSession alias ([`35cf7d034`](https://github.com/openclaw/openclaw/commit/35cf7d0340))

Adds doctor migration for the deprecated `agents.defaults.sandbox.perSession` and `agents.list[].sandbox.perSession` boolean fields. The migration converts these to the canonical `sandbox.scope: "session" | "shared"` via `sandboxScopeFromPerSession()` in `src/config/legacy.migrations.runtime.ts`. Users running with stale config are auto-migrated on load and warned via the doctor flow (migration ID `agents.sandbox.perSession->scope`). The sandbox scope controls session isolation; misconfigured legacy config could silently set the wrong isolation mode when `scope` was absent and `perSession` was not understood by the new code path.

#### Surface legacy channel streaming aliases ([`fbd361d33`](https://github.com/openclaw/openclaw/commit/fbd361d338))

Adds a new migration file `src/config/legacy.migrations.channels.ts` that detects and warns on legacy channel streaming fields: `streamMode` and boolean `streaming` for Telegram, Discord, and Slack accounts. The migration detects these via `hasLegacyStreamMode()` and per-channel streaming key helpers (`hasLegacyTelegramStreamingKeys`, `hasLegacyDiscordStreamingKeys`, `hasLegacySlackStreamingKeys`) and surfaces them in the doctor output with instructions to migrate. Prevents silent config mis-handling where legacy streaming keys could resolve inconsistently against the new enum-based streaming config.

#### Migrate legacy talk config via doctor ([`316da43dd`](https://github.com/openclaw/openclaw/commit/316da43dd7))

Adds migration for legacy `talk.*` flat fields (`voiceId`, `modelId`, `apiKey`, `voiceAliases`) to the provider-specific tree (`talk.providers.elevenlabs.*`). The migration in `src/config/legacy.migrations.runtime.ts` preserves existing values, warns users, and removes the flat keys. Legacy flat config could be silently ignored by new code paths that only read provider-specific config, potentially leaving provider credentials or voice IDs unreachable.
