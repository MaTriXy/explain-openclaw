#### Late-binding subagent runtime for non-gateway load paths ([`eeb140b4f`](https://github.com/openclaw/openclaw/commit/eeb140b4f0))

Fixes plugin loading when subagents run outside the gateway context (e.g., CLI, direct node invocation). Previously the subagent runtime was eagerly initialized, which could cause isolation gaps when plugins were loaded via non-gateway entry points. The fix makes the binding lazy — resolved only when first needed — ensuring plugin isolation contracts hold regardless of entry point.

#### Remove public extension-api surface ([`2de28379d`](https://github.com/openclaw/openclaw/commit/2de28379dd))

Removes `src/extensionAPI.ts`, which provided a general-purpose hook for extensions to reach OpenClaw internals. Plugins are now restricted to specific SDK subpaths rather than a broad import target. The plugin loader (`loadOpenClawPlugins()`) no longer registers the extensionAPI binding, reducing the implicit API surface available to third-party extensions and shrinking the blast radius of a compromised or malicious plugin.

#### Normalize Discord commands allowFrom auth ([`7178a0d3c`](https://github.com/openclaw/openclaw/commit/7178a0d3cb))

Fixes a normalization bug in Discord native command authorization. The old code checked `Array.isArray(commandsAllowFrom.discord) || Array.isArray(commandsAllowFrom["*"])` and could proceed with an inadvertently mixed-source allowlist. The new code picks one source explicitly — `commandsAllowFrom.discord` takes precedence, falling back to `commandsAllowFrom["*"]`. Uses new `normalizeDiscordAllowList()` and `resolveDiscordAllowListMatch()` helpers instead of the generic `resolveCommandAuthorization()`, making Discord command gating more precise.

#### Secrets: honor caller env during runtime validation ([`467dae53c`](https://github.com/openclaw/openclaw/commit/467dae53cf))

Fixes `collectCandidateAgentDirs()` to accept a caller-provided `env: NodeJS.ProcessEnv` (defaulting to `process.env`) rather than hardcoding the global process environment. The env parameter now propagates through `resolveUserPath()`, `resolveOpenClawAgentDir()`, and `resolveAgentDir()`. In test and isolated environments where `OPENCLAW_AGENT_DIR` differs from the global env, secrets runtime validation now uses the correct environment for path resolution — ensuring secret lookups operate on the intended agent directories.
