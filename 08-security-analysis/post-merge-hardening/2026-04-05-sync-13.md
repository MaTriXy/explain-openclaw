#### Narrow audit browser enablement check ([`9a88a933c`](https://github.com/openclaw/openclaw/commit/9a88a933c))

Refactors `isBrowserEnabled()` in `src/security/audit-extra.sync.ts` (verified at `:175`) to replace a call to the full browser config resolution pipeline with a direct property check: `cfg.browser?.enabled !== false`. The old implementation called the resolution pipeline in a try-catch that silently returned `true` on any resolution error, meaning an erroring config path would cause the audit to treat the browser as enabled. The new check reads the config flag directly and fails closed — if `cfg.browser?.enabled` is absent (undefined) the expression still evaluates to `true`, which is the correct semantic (browser enabled by default), but errors in unrelated config resolution no longer affect the decision.

`collectSmallModelRiskFindings()` (verified at `src/security/audit-extra.sync.ts:1192-1285`) receives more precise browser enablement detection as a result. Explicit test coverage is added for the default-browser-enabled path.

Splits the monolithic `src/plugin-sdk/browser-config.ts` (~434 lines) into four specialized modules at both the plugin-sdk and extension layers:

- `browser-config.ts` (~11 lines) — `resolveBrowserConfig`, profile resolution, and browser-config constants
- `browser-bridge.ts` — browser bridge server helpers
- `browser-cdp.ts` — CDP-related utilities including `redactCdpUrl`
- `browser-control-auth.ts` — `resolveBrowserControlAuth` for browser control authentication

New extension-layer barrels are added at `extensions/browser/browser-bridge.ts`, `extensions/browser/browser-cdp.ts`, and `extensions/browser/browser-control-auth.ts`. Sixteen files are updated with granular imports pointing to the appropriate sub-module rather than the monolithic `browser-config.ts`. Verified: sandbox browser setup at `src/agents/sandbox/browser.ts:356-367` (`startBrowserBridgeServer` call) and `src/agents/sandbox/browser.ts:69-102` (`buildSandboxBrowserResolvedConfig`).

The refactor improves security-relevant encapsulation: CDP URL handling and control authentication are now in dedicated modules, reducing the chance that unrelated changes affect security-sensitive code paths during future browser-related work.

#### Honor bluebubbles setup dm policy accounts ([`625201bdd`](https://github.com/openclaw/openclaw/commit/625201bdd))

Fixes per-account DM policy resolution in `extensions/bluebubbles/src/setup-surface.ts`. Previously, the Bluebubbles setup wizard resolved the DM access policy from the top-level channel config key (`channels.bluebubbles.dm.policy`) regardless of which account was being configured. This meant that in multi-account setups, all accounts would share the top-level DM policy even if per-account overrides were present.

The fix adds account-aware config key resolution: when `accountId` is present and not the default account, the setup wizard reads and writes `channels.bluebubbles.accounts.${accountId}.dm.policy`. This aligns Bluebubbles with the access control semantics of other channel extensions and ensures that DM restrictions configured per-account are actually honored by the setup flow.

#### Honor legacy setup dm policy accounts ([`3d8a03914`](https://github.com/openclaw/openclaw/commit/3d8a03914))

The function gains a `resolveConfigKeys` callback that is invoked with the current `accountId` and returns the appropriate config key pair:

- For non-default accounts: `channels.${channel}.accounts.${accountId}.dmPolicy` + `channels.${channel}.accounts.${accountId}.allowFrom`

The `setPolicy` path calls the appropriate setter — either `patchChannelConfigForAccount` (multi-account) or `setLegacyChannelDmPolicyWithAllowFrom` (single/default account) — ensuring that `allowFrom` wildcards are written to the correct account scope. Test coverage added for Discord and Slack setup surfaces.

#### Share activation context for provider runtimes ([`8b6c22455`](https://github.com/openclaw/openclaw/commit/8b6c22455))

Creates `src/plugins/activation-context.ts` (95 lines) as a shared helper that consolidates plugin activation context computation for provider runtimes. The new module exports `PluginActivationCompatConfig`, `PluginActivationInputs`, and a `resolvePluginActivationInputs()` factory that calls `applyPluginAutoEnable()` and `createPluginActivationSource()` in the correct order, producing a consistent `PluginActivationConfigSource` for use in `providers.runtime.ts`, `web-fetch-providers.shared.ts`, and `web-search-providers.shared.ts`.

This is a companion to the allowlist bypass fix chain from sync 11 (`cee5f960b5`, `cd38eba316`). Those commits fixed the gateway startup path to pass the raw pre-auto-enable config as the activation source; this commit applies the same pattern to provider runtime activation paths. Previously each provider runtime assembled its own activation source independently, risking a drift in semantics (e.g., passing the post-auto-enable config instead of the raw config). The shared factory eliminates that risk across all provider paths.

#### Align cache-ttl anthropic semantics ([`ed297eb8b`](https://github.com/openclaw/openclaw/commit/ed297eb8b))

Adds a `modelApi` parameter to `isCacheTtlEligibleProvider()` in `src/agents/pi-embedded-runner/run/attempt.thread-helpers.ts` and propagates it through `appendAttemptCacheTtlIfNeeded()`. The updated signature `(provider, modelId, modelApi?)` allows `anthropic-family-cache-semantics.ts` and `cache-ttl.ts` to distinguish between Anthropic API types when determining cache-TTL eligibility.

The change prevents cache TTL markers from being incorrectly attached to Anthropic model invocations that use a non-eligible API variant. Incorrect attachment would result in cache TTL tokens being sent to the provider on non-caching request paths, which could produce unexpected prompt-caching behavior or provider errors. The fix is a precision correction rather than a security gate; it is included because `attempt.ts` is in the agent execution path and the previous behavior could cause subtle misbehavior in multi-provider caching scenarios.
