#### Harden bootstrap and transport ready coverage ([`3c70e50af`](https://github.com/openclaw/openclaw/commit/3c70e50af))

`loadState()` in `src/infra/device-bootstrap.ts` now validates the shape of the parsed JSON before incorporating any bootstrap state entries. Previously the function trusted the file contents directly; now `rawState` is checked to be a non-null, non-array object before iteration. Each entry within the map is independently validated — non-object values are silently skipped. Fields are individually coerced: `token` must be a non-empty string (falls back to the map key), `issuedAtMs` must be a number (defaults to `0`), and `ts` mirrors `issuedAtMs` if absent. This means a crafted or corrupted bootstrap state file cannot inject malformed `DeviceBootstrapTokenRecord` objects into the running process state. The change complements the same-day advisory GHSA-63f5-hhc7-cx6p (bootstrap setup code replay hardening) as part of the v2026.3.13 bootstrap security improvements.

#### Retry Telegram inbound media downloads over IPv4 fallback ([`7a53eb7ea`](https://github.com/openclaw/openclaw/commit/7a53eb7ea))

This commit patches [GHSA-xwcj-hwhf-h378](https://github.com/openclaw/openclaw/security/advisories/GHSA-xwcj-hwhf-h378) — Telegram media fetch errors were logging the raw media download URL, which for Telegram bots includes the bot token as a path component. A new `redactMediaUrl()` helper in `src/media/fetch.ts` wraps `redactSensitiveText()` from `src/logging/redact.ts` and is applied to every error message in `fetchRemoteMedia()`. The `sourceUrl` variable is set once at entry using the redacted form and reused in all error paths: HTTP error messages, max-bytes exceeded errors, and generic fetch failures. The commit also adds `fallbackDispatcherPolicy` and `shouldRetryFetchError` parameters to `FetchMediaOptions` to support the Telegram-specific IPv4 fallback retry logic implemented in `src/telegram/fetch.ts`.
