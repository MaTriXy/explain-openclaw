#### Matrix Native Exec Approvals ([`1efa923ab`](https://github.com/openclaw/openclaw/commit/1efa923ab8))

Adds end-to-end native exec approval support for the Matrix channel, allowing operators to approve or deny agent command execution directly within Matrix DMs rather than requiring a separate approval surface.

The implementation is split across two new modules. `exec-approvals.ts` provides the core profile: `getMatrixExecApprovalApprovers()` resolves the approver list from `execApprovals.approvers` with a fallback to `dm.allowFrom`; `normalizeMatrixExecApproverId()` strips wildcard `"*"` entries (returning `undefined`) to prevent wildcard bypass of exec approvals — a stricter policy than plugin approvals, which permit `"*"`. `isMatrixExecApprovalAuthorizedSender()`, `isMatrixExecApprovalClientEnabled()`, `resolveMatrixExecApprovalTarget()`, and `shouldHandleMatrixExecApprovalRequest()` are all derived from `createChannelExecApprovalProfile()`. `shouldSuppressLocalMatrixExecApprovalPrompt()` suppresses duplicate local prompts when native delivery is active and the request matches the Matrix account.

`approval-native.ts` exports the composite `matrixApprovalCapability` and `matrixNativeApprovalAdapter`. The underlying `matrixNativeApprovalCapability` is created with `createApproverRestrictedNativeApprovalCapability` and sets `requireMatchingTurnSourceChannel: true`, meaning exec approvals from a turn that did not originate from Matrix will not be dispatched to Matrix. Origin target resolution via `resolveTurnSourceMatrixOriginTarget()` verifies `turnSourceChannel === "matrix"` and normalizes the target via `resolveMatrixNativeTarget()`; `matrixTargetsMatch()` compares both the `to` field and `threadId` to ensure approval requests route to the correct room and thread. `normalizeComparableTarget()` lowercases scheme and canonicalizes user vs. room target kinds to prevent trivial bypass via case variation.

The adapter further differentiates exec and plugin approvals: `matrixExecOnlyNativeApprovalAdapter` disables native delivery for `approvalKind === "plugin"` (returning `MATRIX_PLUGIN_NATIVE_DELIVERY_DISABLED` with `preferredSurface: "approver-dm"`), while exec approvals use full origin-surface delivery. Plugin approval authorization still routes through the existing `matrixApprovalAuth` (based on `dm.allowFrom`) rather than the exec approval profile.

Across `src/infra/exec-approval-forwarder.ts` and `src/infra/exec-approval-reply.ts`, the `approvalKind: "exec" | "plugin"` discriminant is added to `ExecApprovalReplyMetadata` and `shouldSkipForwardingFallback()`, enabling downstream handlers to distinguish approval types when deciding whether to suppress forwarding fallback. `agentId` and `sessionKey` are also threaded through `buildExecPendingPayload()` and `buildApprovalPendingReplyPayload()` to support session-aware approval routing. `src/plugin-sdk/approval-delivery-helpers.ts` adds `approvalKind` to the `DeliverySuppressionParams` type used by channel adapters. Thanks @gumadeiras.

**New security control.** Exec approval authorization is now enforced natively in Matrix using the same approver-restriction pattern as other channels. Wildcard entries are explicitly rejected for exec approvals (unlike plugin approvals), preventing a misconfigured `"*"` in `execApprovals.approvers` from granting blanket exec authority to any Matrix user. Cross-channel approval confusion is blocked by `requireMatchingTurnSourceChannel: true`.
