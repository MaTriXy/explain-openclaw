#### Add allowBots mention gating ([`65816657c`](https://github.com/openclaw/openclaw/commit/65816657c))

`allowBotsMode` in `message-handler.preflight.ts` controls whether the Discord integration processes messages from other bots. The setting maps `discordConfig.allowBots` to three modes: `"off"` (default — all bot messages dropped), `"all"` (all bot messages accepted), and `"mentions"` (only bot messages that explicitly mention the configured bot are accepted; others are silently dropped). Without this gate, two bots configured in the same channel could trigger each other in a feedback loop. The new preflight check enforces this policy before any message reaches the agent.

#### Discord auto presence health signal ([`e28ff1215`](https://github.com/openclaw/openclaw/commit/e28ff1215))

New `DiscordAutoPresenceState` type (`"healthy"` | `"degraded"` | `"exhausted"`) in `auto-presence.ts` drives automatic Discord status updates based on gateway authentication profile availability. The presence text displayed in Discord is configurable per state via `healthyText`, `degradedText`, and `exhaustedText` config fields. When all auth profiles are in cooldown, the bot presence transitions to `"exhausted"`, surfacing capacity issues visibly to channel members without requiring log inspection. `isProfileInCooldown` in `usage.ts` was updated to accept an optional `now` timestamp parameter to support deterministic testing of cooldown transitions.

#### Discord mention handling ([`d493861c1`](https://github.com/openclaw/openclaw/commit/d493861c1))

`ignoreOtherMentions` tightens mention-based routing so the bot only responds when it is directly and explicitly mentioned. Previously a message mentioning `@everyone` or another bot alongside a broad trigger could inadvertently route to the agent. The preflight check in `message-handler.preflight.ts` now enforces that when `ignoreOtherMentions` is set, messages whose only mentions are directed at other users or bots are dropped before reaching the agent, reducing accidental trigger surface.

#### Substitute YYYY-MM-DD at session startup and post-compaction ([`53727c72f`](https://github.com/openclaw/openclaw/commit/53727c72f))

`readPostCompactionContext` in `post-compaction-context.ts` now replaces `YYYY-MM-DD` placeholders with the current date when loading AGENTS context at session startup and after compaction. Agents using date-stamped memory files (e.g., `memory/YYYY-MM-DD.md`) previously received unresolved placeholder strings, causing stale or broken memory references. `buildBareSessionResetPrompt` in `session-reset-prompt.ts` appends a current-time line for `/new` and `/reset` resets, ensuring agents always have accurate temporal grounding from the first turn.

#### Synthesize strict LINE auth boundary hardening ([`dbccc73d7`](https://github.com/openclaw/openclaw/commit/dbccc73d7))

`createLineWebhookReplayCache` in `bot-handlers.ts` tracks incoming LINE webhook event IDs to detect and reject replay attacks. `groupAllowFrom` constrains which LINE group IDs a configured account may receive messages from. `groupPolicy` determines disposition for messages from groups outside the allowlist (reject or ignore). Together these controls establish a strict inbound trust boundary for LINE webhooks, closing a gap where any LINE event reaching the webhook endpoint could trigger the agent regardless of source group.

#### Synthesize media/auth/routing webhook regressions ([`9a5bfb1fe`](https://github.com/openclaw/openclaw/commit/9a5bfb1fe))

Follow-on to the LINE auth hardening commit ([`dbccc73d7`](https://github.com/openclaw/openclaw/commit/dbccc73d7)), resolving regressions in media fetching, webhook routing, and auth context threading introduced by the strict group-boundary enforcement. `download.ts` media retrieval, `webhook-node.ts` routing logic, and the `webhook.ts` entry-point were corrected to properly propagate auth context through the new allowlist checks. Test coverage was expanded across `bot-handlers`, `bot-message-context`, `download`, `webhook-node`, and `webhook` to prevent regression of the corrected auth and media flow.

#### Allowlist Discord CDN hostnames for SSRF media ([`a7a9a3d3c`](https://github.com/openclaw/openclaw/commit/a7a9a3d3c))

`DISCORD_CDN_HOSTNAMES` in `message-utils.ts` defines the authoritative list of Discord CDN origin hostnames. `mergeHostnameList` merges this list into the operator-configured SSRF policy, and `resolveDiscordMediaSsrFPolicy` produces the final policy object applied to Discord media-fetch requests. Previously, fetching Discord attachment URLs required operators to manually widen the SSRF policy, risking inadvertent over-permissioning. This commit encodes the Discord CDN allowlist at the source while the wildcard hostname entries are constrained to known Discord CDN patterns, reducing SSRF attack surface for media fetch paths.

#### iOS Security Stack 2/5: Concurrency Locks ([`6df57d963`](https://github.com/openclaw/openclaw/commit/6df57d963))

`GatewayConnectionController.swift` wraps probe state (URLSession, URLSessionWebSocketTask, and `didFinish` flag) in an `OSAllocatedUnfairLock`-guarded `ProbeState` struct. All mutation of shared probe fields now occurs inside `state.withLock` closures, eliminating a race condition that could cause double-cancellation or premature session invalidation when multiple threads complete a probe concurrently. `CameraController.swift` receives analogous lock guarding. This is part of the five-commit iOS Security Stack series hardening concurrent access patterns in the iOS app.

#### iOS Security Stack 1/5: Keychain Migrations + Tests ([`ec0eb9f8c`](https://github.com/openclaw/openclaw/commit/ec0eb9f8c))

`GatewaySettingsStore.swift` migrates last-connected gateway details from `UserDefaults` to `KeychainStore`, backed by `GenericPasswordKeychainStore` as the secure storage primitive. A migration routine runs once on first access, writes the structured `LastGatewayConnectionData` payload as a JSON Keychain entry, and removes the plaintext `UserDefaults` entry after a successful write. `GatewayTLSPinning.swift` in `OpenClawKit` receives updated TLS certificate pinning for gateway connections. Tests in `GatewaySettingsStoreTests.swift` and `GatewayConnectionControllerTests.swift` verify migration correctness and TLS pinning behavior. This partially closes Audit 1 Claim 1 (plaintext credential storage) for the iOS app.

#### Scope exec wake dispatch to session key ([`627813aba`](https://github.com/openclaw/openclaw/commit/627813aba))

`scopedHeartbeatWakeOptions` in `session-key.ts` derives heartbeat wake options scoped to a specific session key, ensuring that an exec heartbeat wake event triggers only the agent session that owns the corresponding session key. Without this scoping, a heartbeat wake in a multi-session gateway deployment could inadvertently resume an unrelated session sharing the same host process, leaking execution context across session boundaries. The function is called during exec wake dispatch in `server-node-events.ts` to enforce per-session isolation.
