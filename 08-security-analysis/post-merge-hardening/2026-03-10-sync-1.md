#### Harden macOS launchagent install permissions ([`ce9e91fdf`](https://github.com/openclaw/openclaw/commit/ce9e91fdf))

The launchd install path now sets explicit directory and plist file permissions instead of relying on umask defaults. New constants `LAUNCH_AGENT_DIR_MODE = 0o755` and `LAUNCH_AGENT_PLIST_MODE = 0o644` define the required permission modes. New helper `ensureSecureDirectory()` creates the target directory and strips group and other write bits via `mode & ~0o022`, ensuring that world-writable or group-writable launchd directories cannot be introduced by a permissive process umask. The plist write path explicitly calls `fs.chmod` after writing to enforce `LAUNCH_AGENT_PLIST_MODE`, providing defense-in-depth against mode inheritance from the parent directory.

#### Preserve control-ui auth across refresh ([`f2f561fab`](https://github.com/openclaw/openclaw/commit/f2f561fab))

Patches GHSA-rchv-x836-w7xp. The control UI gateway token was previously passed in the URL query string (`?token=...`), causing it to appear in browser history, server access logs, and HTTP referrer headers. This commit moves token delivery to the URL hash fragment (`#token=...`), which is never sent to servers. The token is read once via `hashParams.get("token")` and immediately stripped from the URL. A new `pendingGatewayToken` field is added to hold the token when a gateway URL change is also detected simultaneously; in that case the token is deferred until the new gateway connection is established rather than being applied against the stale connection.

#### Stabilize launchd paths and appcast secret scan ([`6c579d784`](https://github.com/openclaw/openclaw/commit/6c579d784))

Follows the `ce9e91fdf` permission-hardening chain. Launchd path construction now uses `toPosixPath()` and `path.posix.join()` consistently across install, uninstall, and log-directory paths, preventing Windows-style path separators from appearing in macOS plist files. The `.detect-secrets` tooling configuration (`.secrets.baseline` and `.pre-commit-config.yaml`) is updated to exclude the auto-generated `src/daemon/launchd.ts` path strings and the appcast XML file from false-positive secret-scan alerts.

#### Avoid persisting talk fallback on fresh setup ([`7217b9765`](https://github.com/openclaw/openclaw/commit/7217b9765))

The `baseConfig` initialization in both the non-interactive onboarding command and the interactive wizard previously used `snapshot.valid ? snapshot.config : {}`. This would populate `baseConfig` with the snapshot's config even when the snapshot is structurally valid but the file no longer exists on disk. The fix adds `snapshot.exists` as a secondary guard: `snapshot.valid ? (snapshot.exists ? snapshot.config : {}) : {}`. This prevents stale configuration data — including talk provider fallback settings and associated credentials — from persisting into a fresh setup flow when the config file has been manually deleted or was never written.
