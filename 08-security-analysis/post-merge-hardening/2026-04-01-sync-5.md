#### Harden SecretRef round-trip handling in Control UI and RPC writes ([`81b777c76`](https://github.com/openclaw/openclaw/commit/81b777c76))

`config.apply`, `config.set`, and `config.patch` in `src/gateway/server-methods/config.ts` now call `ensureResolvableSecretRefsOrRespond()` before applying any config change. This preflight validates that every `SecretRef` id in the incoming config can actually be resolved; if any cannot, the RPC returns an error before the config is written. The fix also strengthens redaction semantics: `REDACTED_SENTINEL` values in restored configs are now rejected rather than passed through, closing a path where a caller could echo a literal sentinel string back to bypass redaction. `redactConfigSnapshot()` and `restoreRedactedValues()` in `src/config/redact-snapshot.ts` enforce these stricter semantics. Addresses Audit 1 Claim 1 (plaintext secret storage vectors) and Audit 1 Claim 8 (token expiry/resolution validation).

#### Relabel managed workspace mounts for SELinux ([`7516b423e`](https://github.com/openclaw/openclaw/commit/7516b423e))

`formatManagedWorkspaceBind()` in `src/agents/sandbox/workspace-mounts.ts` now appends `:z` to every Docker bind mount for managed workspaces, enabling SELinux shared relabeling. This applies to all four bind categories: main workspace read-only, main workspace read-write, agent workspace read-only, and agent workspace read-write. The `:z` label instructs SELinux to relabel the volume content so the container process can access it even when DAC (discretionary access control) alone would permit a bypass. `SANDBOX_MOUNT_FORMAT_VERSION` is bumped to `2` to track this mount-schema change and invalidate any stale container recreate caches that predate the relabeling requirement.

#### Add requireAgentId to block sessions_spawn without explicit agen… ([`d4cccda57`](https://github.com/openclaw/openclaw/commit/d4cccda57))

A new `requireAgentId` boolean flag (per-agent in `src/config/types.agents.ts` and defaults-level in `src/config/types.agent-defaults.ts`) gates `spawnSubagentDirect()` in `src/agents/subagent-spawn.ts`. When enabled, any spawn request that omits or passes a blank `agentId` is rejected with status "forbidden" before the subagent is created. Without this guard, an invocation that passed an empty `agentId` could inherit the caller's identity implicitly, conflating trust boundaries between the parent agent and any spawned subagent. The flag is propagated through ZodSchema validation layers so it can be set at individual agent level or inherited from defaults.

#### Pin remote fs bridge reads ([`121870a08`](https://github.com/openclaw/openclaw/commit/121870a08))

`SANDBOX_PINNED_MUTATION_PYTHON` in `src/agents/sandbox/fs-bridge-mutation-helper.ts` gains a `read_file(parent_fd, basename)` function that opens files using `os.open(basename, READ_FLAGS, dir_fd=parent_fd)` — a directory-fd-relative open that pins the lookup to the already-opened parent directory. After opening, `read_file` calls `os.fstat()` and rejects: (1) files where `st_nlink > 1` (hardlinks, which could alias a trusted path to an attacker-controlled target) and (2) non-regular files (devices, sockets, symlinks). Relative path validation in the caller blocks `.`, `..`, and absolute paths from reaching the mount root. Together these defenses prevent symlink-swap TOCTOU races, hardlink aliasing, and path-escape attacks that were theoretically possible when the FS bridge opened files by name without pinning the parent directory. Closes upstream advisory GHSA-9p3r-hh9g-5cmg (CRITICAL sandbox TOCTOU).

#### Restore strict SSRF pinning ([`9d9ee0f31`](https://github.com/openclaw/openclaw/commit/9d9ee0f31))

`fetchWithSsrFGuard()` in `src/infra/net/fetch-guard.ts` had acquired a helper function that introduced an env-proxy bypass path: when an HTTP proxy was configured and the target hostname was not on the explicit allowlist, the code would route the request through the OS-level `EnvHttpProxyAgent` rather than the pinned DNS dispatcher. This bypass allowed a request that nominally used pinned DNS to fall back to unpinned proxy routing, potentially reaching private network addresses. The commit removes the bypass helper entirely and reverts `fetchWithSsrFGuard()` to always call `createPinnedDispatcher()` directly, eliminating the env-proxy bypass path and restoring the strict SSRF pinning that was the original design intent.

#### Regenerate mac host env policy ([`5d8ca42c7`](https://github.com/openclaw/openclaw/commit/5d8ca42c7))

Follow-up to `7ae1bb0c7` (see below): regenerates `apps/macos/Sources/OpenClaw/HostEnvSecurityPolicy.generated.swift` to include `UV_EXTRA_INDEX_URL` in the macOS host env block list. The Swift file is the compiled representation of `src/infra/host-env-security-policy.json` for the macOS app; without this regeneration, the macOS sandbox would not block `UV_EXTRA_INDEX_URL` even though it was added to the JSON source. Together with `7ae1bb0c7`, this ensures the `uv` package index override env vars are blocked on all supported platforms.

#### Block Python package index redirection env vars ([`7ae1bb0c7`](https://github.com/openclaw/openclaw/commit/7ae1bb0c7))

Adds eight Python and `uv` package-index environment variables to the block list in `src/infra/host-env-security-policy.json`: `PIP_INDEX_URL`, `PIP_PYPI_URL`, `PIP_EXTRA_INDEX_URL`, `UV_INDEX`, `UV_INDEX_URL`, `UV_EXTRA_INDEX_URL`, `UV_DEFAULT_INDEX`. When any of these env vars is set in the host environment, an untrusted model could redirect pip or `uv` package downloads to an attacker-controlled index, enabling supply-chain compromise of any Python dependency the sandbox installs. Blocking them at the exec env sanitization layer ensures the host operator's Python/uv environment cannot influence package resolution inside the sandbox. Addresses upstream advisory GHSA-7ggg-pvrf-458v (HIGH: `PIP_INDEX_URL` and `UV_INDEX_URL` bypass host exec env sanitization).

#### Dedupe Discord approval delivery ([`eba41dae4`](https://github.com/openclaw/openclaw/commit/eba41dae4))

`DiscordExecApprovalHandler` in `extensions/discord/src/monitor/exec-approvals.ts` gains a `deliveredChannelIds` `Set<string>` that tracks every Discord channel id to which an approval prompt has already been sent within a single delivery pass. Both the "origin" path and the "DM" path check `deliveredChannelIds.has()` before sending; if the target channel was already delivered to, the delivery is skipped with a debug log. This prevents a duplicate approval prompt when `target=both` resolves the origin route and the approver DM to the same concrete Discord channel (e.g., when the operator uses a DM-only workspace). Duplicate approval prompts could cause users to see the same request twice and approve or deny it inconsistently; in edge cases they could also cause race conditions in the pending-approval tracking state.
