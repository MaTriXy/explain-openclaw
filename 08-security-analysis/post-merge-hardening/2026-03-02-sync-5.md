#### Bind doc create grants to trusted requester context ([`bbab94c1f`](https://github.com/openclaw/openclaw/commit/bbab94c1f))

Feishu doc-create tool actions now bind document grants to the authenticated runtime requester rather than accepting arbitrary owner IDs from tool arguments. `OpenClawPluginToolContext` in `src/plugins/types.ts` gains `requesterSenderId` and `senderIsOwner` fields that carry verified identity from the message pipeline. The Feishu `createDoc()` function in `extensions/feishu/src/docx.ts` reads `options.grantToRequester` (defaulting to `true`) and uses `ctx.requesterSenderId` to determine the grant target. The `doc-schema.ts` schema exposes this as the boolean field `grant_to_requester`, replacing the previous owner_open_id/owner_perm_type Feishu API parameters that accepted user-supplied values.

#### Warn on wildcard control-ui origins and feishu owner grants ([`17bae9368`](https://github.com/openclaw/openclaw/commit/17bae9368))

Adds two security audit checks in `src/security/audit.ts`. First, `isFeishuDocToolEnabled()` inspects channel and account-level configuration to detect when Feishu doc tools are active, and emits a warning when they are — referencing the new permission binding model. Second, a check with ID `gateway.control_ui.allowed_origins_wildcard` fires when the control UI `allowedOrigins` contains a wildcard entry, with severity scaled by gateway exposure (`critical` for externally reachable bindings, `warn` for loopback). Both checks surface in `openclaw security audit` output to help operators identify permissive gateway and Feishu configurations.

#### Check chat allowlist before sender allowlist in group policy ([`8247c25a3`](https://github.com/openclaw/openclaw/commit/8247c25a3))

Fixes a silent authorization rejection in `evaluateTelegramGroupPolicyAccess()` in `src/telegram/group-access.ts`. Previously when `groupPolicy` was `allowlist`, the empty-sender-allowlist guard ran before the chat-level check, causing Telegram groups explicitly configured in the `groups` config (but with no per-sender entries) to be rejected. The fix moves the `checkChatAllowlist` block before the sender guard and introduces a `chatExplicitlyAllowed` flag: when a group has a dedicated `groupConfig` entry (not just a wildcard match) and no per-sender entries exist, the group ID itself acts as authorization and the sender check is skipped entirely.

#### Harden sessions_spawn delivery params and telegram account routing (#31000, #31110) ([`b0c7f1ebe`](https://github.com/openclaw/openclaw/commit/b0c7f1ebe))

Adds `UNSUPPORTED_SESSIONS_SPAWN_PARAM_KEYS` to `createSessionsSpawnTool()` in `src/agents/tools/sessions-spawn-tool.ts` — a blocklist of channel delivery parameters (`target`, `transport`, `channel`, `to`, `threadId`, `thread_id`, `replyTo`, `reply_to`) that `sessions_spawn` does not support. When any of these keys appear in a tool call, the tool immediately throws `ToolInputError` with guidance to use `message` or `sessions_send` for channel delivery instead. Prevents mis-routing of spawned agent sessions via channel delivery parameter injection.

#### Land #31147 plugin install spawn EINVAL (@codertony) ([`a1a8ec687`](https://github.com/openclaw/openclaw/commit/a1a8ec687))

Adds `resolveNpmArgvForWindows()` to `src/process/exec.ts` to resolve `npm` and `npx` invocations to their underlying Node.js scripts (`npm-cli.js`, `npx-cli.js`) on Windows. This is required on Node.js 18.20.2+ following CVE-2024-27980, where spawning `.cmd` wrappers without a shell raises EINVAL as a security measure. The fix routes through `node.exe` directly, maintaining the no-shell spawn model (no shell injection risk introduced) while restoring plugin install compatibility on Windows.
