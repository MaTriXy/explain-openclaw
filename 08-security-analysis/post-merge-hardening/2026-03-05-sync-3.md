#### Prevent Internal Route Leakage in chat.send ([`c4dab17ca`](https://github.com/openclaw/openclaw/commit/c4dab17ca))

The gateway's `chat.send` handler now prevents internal client connections (webchat, TUI, web UI) from inheriting the configured-main session route for external channel delivery. The fix introduces a `shouldDeliverExternally` check (`p.deliver === true`) and restricts configured-main session inheritance to non-webchat/UI clients. Internal clients originating from `isWebchatClient()` now default to internal-only routing; they must pass an explicit `deliver: true` flag to inherit any external channel route. Regression tests cover UI client route isolation and `deliver` flag semantics.

**Impact:** Prevents a class of cross-channel message leakage where webchat or TUI client replies could inadvertently route to external channels (Telegram, Discord, etc.) via the configured-main session path.

#### Prototype-Chain Account Path Validation ([`da0e245db`](https://github.com/openclaw/openclaw/commit/da0e245db))

The `hasExplicitProviderAccountConfig()` function in `src/security/audit-channel.ts` now uses `Object.hasOwn(accounts, accountId)` instead of the `in` operator for account path lookups. The `in` operator traverses the prototype chain, meaning inherited properties such as "toString", "constructor", or "valueOf" could be misinterpreted as explicitly configured provider account paths. `Object.hasOwn()` only checks own enumerable properties, eliminating the prototype chain walk. A regression test verifies that prototype-inherited method names are correctly rejected as non-explicit account paths.

**Impact:** Hardens provider account path validation against prototype pollution scenarios where a crafted account ID matching a prototype property could bypass the "no explicit account config" branch.
