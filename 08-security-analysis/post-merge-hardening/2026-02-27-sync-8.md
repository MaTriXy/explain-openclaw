### Security Details

#### Device-Auth Migration Diagnostics ([`cb9374a2a`](https://github.com/openclaw/openclaw/commit/cb9374a2a))

Adds six new error detail constants for device authentication failure modes:
`DEVICE_AUTH_DEVICE_ID_MISMATCH`, `DEVICE_AUTH_SIGNATURE_EXPIRED`,
`DEVICE_AUTH_NONCE_REQUIRED`, `DEVICE_AUTH_NONCE_MISMATCH`,
`DEVICE_AUTH_SIGNATURE_INVALID`, and `DEVICE_AUTH_PUBLIC_KEY_INVALID`.
New function `resolveDeviceAuthConnectErrorDetailCode()` maps raw auth failure
reasons to typed error codes, improving diagnostic visibility during v2-to-v3
migration. Clearer error granularity aids debugging without exposing additional
attack surface (fail-closed behaviour unchanged).

#### Camera Input Validation: Reject facing=both with deviceId ([`b8373eadd`](https://github.com/openclaw/openclaw/commit/b8373eadd))

Adds explicit input validation to the `nodes:screen_record` tool: a request
specifying both a specific camera `deviceId` and `facing=both` is now rejected
with an error. Previously this ambiguous combination could result in undefined
behaviour. The fix narrows the accepted input space for camera operations.
Adjacent to Audit 2 Claim 2 (outPath lacks path validation, still open as
Legitimate Gap 3) but addresses a distinct input validation gap.

#### Android Command Allowlist Expansion ([`e99b323a6`](https://github.com/openclaw/openclaw/commit/e99b323a6))

Extends the Android node command allowlist with two new constant arrays:
`ANDROID_NOTIFICATION_COMMANDS` (extends `NOTIFICATION_COMMANDS` with
`"notifications.actions"`) and `ANDROID_DEVICE_COMMANDS` (extends
`DEVICE_COMMANDS` with `"device.permissions"` and `"device.health"`).
Platform defaults updated to use the Android-specific arrays rather than
the generic ones. Allowlist expansion is controlled — new commands are
explicitly enumerated rather than using wildcards.

#### Notification Action Race Condition Fix ([`bbab0b005`](https://github.com/openclaw/openclaw/commit/bbab0b005))

Fixes a race condition in the Android notification listener: the listener
service is now rebound before notification actions execute. Previously,
a stale binding could cause notification actions to fail silently or
execute against the wrong service instance. The fix ensures the listener
is in a consistent state before action dispatch.

#### Non-Clearable Notification Open/Reply Fix ([`8807267bf`](https://github.com/openclaw/openclaw/commit/8807267bf))

`actionRequiresClearableNotification()` previously blocked `open` and `reply`
actions on non-clearable notifications. This was incorrect — those actions do
not require the notification to be clearable. The fix allows `open` and `reply`
on non-clearable notifications while continuing to block actions that genuinely
require clearability (e.g., dismiss).
