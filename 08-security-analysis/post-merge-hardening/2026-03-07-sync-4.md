#### Stop shared-main chat.send from inheriting stale external routes ([`563a125c6`](https://github.com/openclaw/openclaw/commit/563a125c6))

The gateway's `chat.send` handler previously inherited `OriginatingChannel` from the session key's stored external route — a session key like `agent:main:slack:direct:U…` could cause webchat replies to be dispatched to Slack or Telegram instead of back to the originating gateway connection. The fix unconditionally sets `OriginatingChannel` to `INTERNAL_MESSAGE_CHANNEL` (`"webchat"`) for all `chat.send` calls, removing route-inheritance entirely.

Additionally, `CHAT_SEND_SESSION_KEY_MAX_LENGTH = 512` and the `ChatSendSessionKeyString` schema type are introduced in `src/gateway/protocol/schema/primitives.ts`, enforcing a byte-length cap on inbound session key strings to prevent oversized-key request attacks.

#### Fix owner-only auth and overlapping skill env regressions ([`15a5e39da`](https://github.com/openclaw/openclaw/commit/15a5e39da))

Two independent regressions fixed together. First, the owner-only authorization gate in `src/auto-reply/command-auth.ts` was missing the `ownerAllowAll` short-circuit in its `senderIsOwner` boolean, causing some configured-owner setups to incorrectly deny access. The fix adds `ownerAllowAll` to the disjunction.

Second, `src/agents/skills/env-overrides.ts` gains reference-counted tracking of active skill env keys via `activeSkillEnvEntries` (a `Map<string, ActiveSkillEnvEntry>`) and `acquireActiveSkillEnvKey()`. When multiple concurrent skills attempt to inject the same env key, the new logic prevents a later skill from overwriting an earlier skill's value, and correctly restores the baseline on release — eliminating env var shadowing races between overlapping skill activations.

#### Prevent echo loop from leaking internal metadata and amplifying NO_REPLY ([`adb9234d0`](https://github.com/openclaw/openclaw/commit/adb9234d0))

Multi-layer defense added to the iMessage channel monitor against two related failure modes: outbound leakage of assistant-internal metadata and recursive echo amplification.

**Outbound sanitization** (`src/imessage/monitor/sanitize-outbound.ts`): `sanitizeOutboundText()` strips internal scaffolding from text before delivery. It calls `stripAssistantInternalScaffolding()` and then removes `INTERNAL_SEPARATOR_RE` (`/(?:#\+){2,}#?/g`), `ASSISTANT_ROLE_MARKER_RE`, and `ROLE_TURN_MARKER_RE` patterns — ensuring thinking/reasoning tags, role turn markers, and OpenClaw-internal separators never reach the iMessage channel.

**Reflection guard** (`src/imessage/monitor/reflection-guard.ts`): `REFLECTION_PATTERNS` detects when an inbound message contains assistant-internal markers (`thinking-tag`, `relevant-memories-tag`, `final-tag`, `internal-separator`, `assistant-role-marker`) — meaning a previously leaked message bounced back as a new inbound. When detected, the message is suppressed to break the amplification loop.

**Echo cache hardening** (`src/imessage/monitor/echo-cache.ts`): TTL for outbound echo entries is extended to catch delayed reflections.

**Loop rate limiter** (`src/imessage/monitor/loop-rate-limiter.ts`): `createLoopRateLimiter()` enforces a sliding-window limit (`DEFAULT_MAX_HITS = 5` per `DEFAULT_WINDOW_MS = 60,000 ms`) that suppresses rapid-fire conversation cycles characteristic of runaway echo loops.

#### Treat unconfigured-owner sessions as owner for ownerOnly tools ([`48b3c4a04`](https://github.com/openclaw/openclaw/commit/48b3c4a04))

The `senderIsOwner` check in `src/auto-reply/command-auth.ts` is extended with a fourth condition: `!ownerAllowlistConfigured && isDirectChat && Boolean(senderId)`. In the default single-user setup where no explicit owner allowlist has been configured, an identified sender in a direct-chat session is now treated as owner for owner-only tools. This corrects a regression where users with a default (unconfigured) setup were incorrectly denied access to owner-only tools they were entitled to use.

#### Strip skill-injected env vars from ACP harness spawn env ([`ae96a8191`](https://github.com/openclaw/openclaw/commit/ae96a8191))

Skills can inject environment variables (e.g., `OPENAI_API_KEY`) into `process.env` for the duration of their execution. Previously, when an ACP harness subprocess was spawned during a skill activation, those skill-injected keys leaked into the child process environment — creating an unintended privilege escalation path where a skill's API credentials could be read by ACP child processes.

The fix adds `activeSkillEnvKeys = new Set<string>()` and `getActiveSkillEnvKeys()` to `src/agents/skills/env-overrides.ts` to track keys currently injected by active skills. The `resolveAcpClientSpawnEnv()` function in `src/acp/client.ts` gains a `stripKeys` option that deletes tracked skill keys from the spawn environment before launching child processes.
