#### Auto-allow private network for local serverUrl and add allowPrivateNetwork to channel schema ([`e43600c9e`](https://github.com/openclaw/openclaw/commit/e43600c9e))

`resolveBlueBubblesServerAccount()` at `extensions/bluebubbles/src/account-resolve.ts:14` now calls `isBlockedHostnameOrIp()` (line 52) before constructing a connection. If the resolved serverUrl is a private or loopback address, `allowPrivateNetwork: true` is automatically set in the returned account object (line 61). Previously this flag was only set when the user had explicitly enabled it in the channel schema, leaving a window where a DNS rebinding attack could silently direct OpenClaw to an intranet host. The commit also adds `allowPrivateNetwork` to the BlueBubbles channel schema so the auto-set value persists across reconnects.

#### Route fetch calls through fetchWithSsrFGuard ([`f92c92515`](https://github.com/openclaw/openclaw/commit/f92c92515))

This commit routes outbound fetch calls from 28 production files (spanning BlueBubbles, Mattermost, Nextcloud Talk, Thread Ownership, and other extensions) through `fetchWithSsrFGuard` (exported from `src/infra/net/fetch-guard.js`), which blocks requests to RFC 1918 / loopback / link-local destinations unless the caller has explicitly declared `allowPrivateNetwork: true`. Before this change, several extensions composed their own fetch wrappers and did not honour the SSRF guard, making it possible for a prompt-injected URL to reach internal network services. The fix consolidates all outbound HTTP through the existing guard surface.

#### Refactor sendMessageBlueBubbles to use resolveBlueBubblesServerAccount and enhance private network handling in tests ([`2513a8d85`](https://github.com/openclaw/openclaw/commit/2513a8d85))

The send path in `extensions/bluebubbles/src/send.ts` previously constructed the BlueBubbles server URL independently, bypassing the hostname-blocking logic introduced in e43600c9e. This commit refactors `sendMessageBlueBubbles()` to call `resolveBlueBubblesServerAccount()` before dispatching, ensuring the `isBlockedHostnameOrIp()` check and `allowPrivateNetwork` auto-detection apply consistently to both incoming-webhook and outbound-message code paths.

#### Require pairing for backend scope upgrades ([`d3d8e316b`](https://github.com/openclaw/openclaw/commit/d3d8e316b))

`shouldSkipBackendSelfPairing()` in `extensions/gateway/src/handshake-auth-helpers.ts` was a shortcut that allowed a backend-initiated session to skip the pairing handshake when requesting elevated scopes. This commit removes the function entirely (26 lines deleted) and simplifies the `skipPairing` assignment in `src/gateway/message-handler.ts` — it is now solely derived from `shouldSkipControlUiPairing()`, ensuring any scope-upgrade request from an AI-controlled session must go through the normal pairing flow. `shouldAllowSilentLocalPairing` (line 48) remains; it covers a separate, narrower case (loopback control-UI) and was not changed.

#### Gateway: require caller scope for subagent session deletion ([`b5d785f1a`](https://github.com/openclaw/openclaw/commit/b5d785f1a))

`createGatewaySubagentRuntime().deleteSession()` in `src/gateway/server-plugins.ts:369` previously injected `syntheticScopes: [ADMIN_SCOPE]` when dispatching the deletion request, effectively granting itself admin authority regardless of the caller's actual scope. This commit removes the `syntheticScopes` override, requiring the subagent to hold a legitimately granted deletion scope. Without this fix, a compromised subagent could delete any session by constructing a `deleteSession` call.

#### Enforce session_status guard after sessionId resolution ([`d9810811b`](https://github.com/openclaw/openclaw/commit/d9810811b))

The `session_status` tool in `src/agents/session-status-tool.ts` evaluates whether the requesting agent may view another session's status. The `isExplicitAgentKey` flag (line 288) distinguishes requests that arrive with a caller-supplied `agentId` from those using an implicit default. Before this commit, the visibility guard at line 370 was evaluated before `sessionId` was resolved from the explicit key, allowing an agent to bypass the guard by supplying a key that resolved to a session it did not own. The fix ensures the `!isExplicitAgentKey` check at line 370 always fires after full resolution.

#### Gateway: align HTTP session history scopes ([`1c4512323`](https://github.com/openclaw/openclaw/commit/1c4512323))

The WebSocket gateway already enforced scope checks for `chat.history` reads; the HTTP gateway path did not have an equivalent gate. This commit adds `resolveGatewayRequestedOperatorScopes()` (defined in `src/gateway/http-auth-helpers.ts:33`) to the HTTP session-history handler and calls `authorizeOperatorScopesForMethod("chat.history")` at `src/gateway/sessions-history-http.ts:175-176` before returning results. Without this gate, an operator with any valid HTTP session token could read session history regardless of whether the token included the `chat.history` scope.

#### Keep plugin HTTP runtime scopes least-privileged ([`ec2dbcff9`](https://github.com/openclaw/openclaw/commit/ec2dbcff9))

`createGatewayPluginRequestHandler()` in `src/gateway/plugins-http.ts` routes external plugin HTTP requests. Previously, all matched routes were handled with the gateway's full scope set. #### Throttle webhook auth guesses ([`5e08ce36d`](https://github.com/openclaw/openclaw/commit/5e08ce36d))

Repeated incorrect webhook auth tokens were not rate-limited, making it possible to brute-force the BlueBubbles webhook secret. This commit introduces `webhookRateLimiter = createFixedWindowRateLimiter(...)` at `extensions/bluebubbles/src/monitor.ts:31`, locking out a source IP after a configurable number of failed attempts within a fixed time window. A trusted-proxy header is also parsed so that X-Forwarded-For is honoured behind a reverse proxy.

#### Throttle repeated webhook auth guesses ([`c2c136ae9`](https://github.com/openclaw/openclaw/commit/c2c136ae9))

Telegram's webhook verification path lacked rate limiting, making it possible to enumerate valid bot tokens via repeated requests. This commit adds `telegramWebhookRateLimiter = createFixedWindowRateLimiter(...)` per-listener at `extensions/telegram/src/webhook.ts:279`, returning HTTP 429 after the threshold is exceeded. Each Telegram listener (identified by bot token) receives its own limiter instance.

#### Synology-chat: throttle webhook token guesses ([`0b4d07337`](https://github.com/openclaw/openclaw/commit/0b4d07337))

The Synology Chat webhook token is a static shared secret that was previously unguarded against brute-force. This commit adds `this.limiter = createFixedWindowRateLimiter(...)` at `extensions/synology-chat/src/security.ts:98` and a lockout path, ensuring repeated incorrect tokens trigger a cooldown period. The per-source-IP window matches the pattern established by the BlueBubbles and Telegram fixes.

#### Feishu: validate webhook signatures before parsing ([`5e8cb2217`](https://github.com/openclaw/openclaw/commit/5e8cb2217))

`parseFeishuWebhookPayload()` in `extensions/feishu/src/monitor.transport.ts:53` previously parsed the JSON body before validating the Feishu webhook signature, allowing a malicious caller to trigger JSON parsing with attacker-controlled bytes before the signature check ran. This commit moves `isFeishuWebhookSignatureValid()` (line 62) to run against the raw request bytes before any parsing occurs, closing a potential payload-injection window. The existing `feishuWebhookRateLimiter` (imported at line 18) continues to apply before signature validation.

ACP permission prompts display the names of tools requested by the AI agent. #### Feishu: reject legacy raw card command payloads ([`81c45976d`](https://github.com/openclaw/openclaw/commit/81c45976d))

#### Honor reaction mention gating ([`f8c986307`](https://github.com/openclaw/openclaw/commit/f8c986307))

#### Google Chat: require stable group ids ([`11ea1f678`](https://github.com/openclaw/openclaw/commit/11ea1f678))
