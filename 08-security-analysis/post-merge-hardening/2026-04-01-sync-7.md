#### Reject escaping symlinks in ssh sandbox uploads ([`3d5af1498`](https://github.com/openclaw/openclaw/commit/3d5af1498))

SSH sandbox uploads previously performed no symlink validation before copying files into the sandbox. A malicious or compromised agent directory could include a symlink pointing outside the intended upload root, allowing files outside the configured directory to be read or overwritten.

This commit adds `assertSafeUploadSymlinks` at `src/agents/sandbox/ssh.ts:342`, called at `:261` before any SSH upload proceeds. The function resolves every symlink in the local directory tree and rejects any that point outside the upload root, throwing an error with the escaping path. This closes the symlink traversal vector in SSH sandbox upload paths. Addresses Audit 1 Claim 6 (path traversal in agent dirs).

#### Enforce trusted-proxy HTTP origin checks ([`6b3f99a11`](https://github.com/openclaw/openclaw/commit/6b3f99a11))

In trusted-proxy HTTP mode, the gateway previously skipped `Origin` header validation on inbound HTTP requests, relying entirely on the proxy to enforce access. A forged `Origin` header (or a bypass via a cross-site request) could reach the gateway API without origin gating.

This commit adds `authorizeTrustedProxyBrowserOrigin` at `src/gateway/auth.ts:378`, invoked at `:489` during HTTP gateway connection authorization. The function validates the `Origin` request header against the configured allowed origins before passing the request to the handler. Without a matching origin, the request is rejected. This closes the origin-forging vector in trusted-proxy HTTP deployments and strengthens Audit 2 Claim 4 (DNS rebinding via web-fetch).

#### Exec approvals: detect command carriers in strict inline eval ([`7bd2761b9`](https://github.com/openclaw/openclaw/commit/7bd2761b9))

In strict inline-eval mode, an AI agent could pass arbitrary code to commands capable of inline execution even when the base binary was allowlisted — because the inline-eval detector did not cover all such carriers. Prior to this commit, `awk`, `find`, `make`, `sed`, and `xargs` were absent from the carrier detection list.

This commit expands `src/infra/exec-inline-eval.ts` to detect these carriers: `awk`/`gawk`/`mawk`/`nawk` (positional inline program, lines 41 and 77), `find` with `-exec`/`-execdir` (line 51), `make`/`gmake` with `--eval` (line 56), `sed`/`gsed` with inline expressions (lines 68, 138), and `xargs` (line 97). When these commands are used in strict inline-eval mode, explicit approval is now required even when the binary is already in the exec allowlist. Further strengthens Audit 2 Claim 7 (shell injection via incomplete regex).

#### Drop auth headers on cross-origin redirects ([`e704323ff`](https://github.com/openclaw/openclaw/commit/e704323ff))

When the SSRF-guarded HTTP fetch followed a redirect to a different origin, request headers — including `Authorization`, `Cookie`, and `x-openclaw-*` custom headers — were forwarded to the redirect target unchanged. A server-side redirect to an attacker-controlled host could harvest these credentials.

This commit introduces NEW `src/infra/net/redirect-headers.ts` with `CROSS_ORIGIN_REDIRECT_SAFE_HEADERS` at `:1` (an allowlist of innocuous headers: `accept`, `accept-encoding`, `accept-language`, `cache-control`, `content-type`, `user-agent`, etc.) and `retainSafeHeadersForCrossOriginRedirect` at `:17`, which strips all headers not on the allowlist. `src/infra/net/fetch-guard.ts` imports this function and applies it at `:201` before following any redirect that crosses origins, ensuring auth-bearing headers are never forwarded to a different host.

#### Redact Nostr privateKey in config views ([`57700d716`](https://github.com/openclaw/openclaw/commit/57700d716))

`SENSITIVE_PATTERNS` at `src/config/schema.hints.ts:134` is the regex array that `mapSensitivePaths()` uses to mark config fields for redaction in `config.get` output, diagnostic exports, and debug snapshots. The existing patterns (`/password/i`, `/secret/i`, `/api.?key/i`, `/encrypt.?key/i`) did not cover Nostr's privateKey field, which contains a raw ed25519 private key.

Without this pattern, a sandboxed agent with `config.get` access could extract the Nostr private signing key from config output.

#### Secrets: hard-fail unsupported SecretRef policy and fix gateway restart token drift ([`788f56f30`](https://github.com/openclaw/openclaw/commit/788f56f30))
