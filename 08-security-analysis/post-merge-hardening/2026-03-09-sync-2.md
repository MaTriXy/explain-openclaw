#### Gateway: add safer password-file input for gateway run ([`4062aa5e5`](https://github.com/openclaw/openclaw/commit/4062aa5e5))

Introduces `src/acp/secret-file.ts` with `readSecretFromFile()` to harden how the gateway reads secrets from disk. Before reading, the function calls `lstatSync()` to inspect the path's metadata: if the path is a symlink it is immediately rejected, preventing symlink substitution attacks where an attacker replaces a file with a symlink to a sensitive target. The file must be a regular file (`isFile()` check); any other file type (device, socket, FIFO) is rejected. A 16 KB size cap (`MAX_SECRET_FILE_BYTES = 16 * 1024`) prevents unbounded reads. These three checks together close the file-reading attack surface noted in Audit 1 Claim 5. A new `--password-file` option on `openclaw gateway run` lets operators supply credentials via file path instead of exposing them inline on the command line or in environment variables.

#### Reject launchd pid sentinel values ([`7f44bc5e9`](https://github.com/openclaw/openclaw/commit/7f44bc5e9))

Fixes a daemon status reporting edge case in `parseLaunchctlPrint()` (`src/daemon/launchd.ts`). On macOS, launchd can return PID 0 as a sentinel value when a service is registered but not running. Previously the parser accepted any finite number for PID, meaning a PID of 0 could be stored in `LaunchctlPrintInfo.pid` and later treated as a valid process reference. The fix routes the raw PID string through `parseStrictPositiveInteger()`, which returns `undefined` for zero and negative values, so only genuine PIDs (> 0) are recorded. This prevents the daemon from misreporting service state based on a sentinel PID.

#### Reuse shared gateway probe auth ([`380eb1c07`](https://github.com/openclaw/openclaw/commit/380eb1c07))

Centralizes gateway probe authentication into `src/gateway/probe-auth.ts`, which exports three functions: `resolveGatewayProbeAuth()` for synchronous config-based auth resolution, `resolveGatewayProbeAuthWithSecretInputs()` for async resolution with explicit credentials, and `resolveGatewayProbeAuthSafe()` for callers that should not throw on unresolved secret refs. Previously, probe auth logic was duplicated across `src/cli/daemon-cli/status.gather.ts` and other call sites, creating a risk that changes to auth semantics (token priority, env var handling, legacy fallback) would be applied inconsistently. Centralizing removes ~100 lines of duplicated handling and ensures all probe call sites share the same auth resolution path.

#### Gateway: stop and restart unmanaged listeners ([`bf9c36212`](https://github.com/openclaw/openclaw/commit/bf9c36212))

Adds lifecycle management for unmanaged gateway listeners (processes started outside the daemon manager). The new `src/cli/daemon-cli/restart-health.ts` module exports `inspectGatewayRestart()` to snapshot service runtime + port occupancy, `waitForGatewayHealthyRestart()` to poll until a new gateway instance becomes healthy, `waitForGatewayHealthyListener()` for port-only health polling, and `terminateStaleGatewayPids()` to kill leftover listener processes. `DEFAULT_RESTART_HEALTH_TIMEOUT_MS` (60 s) and `DEFAULT_RESTART_HEALTH_DELAY_MS` (500 ms) govern the poll window. The lifecycle changes in `src/cli/daemon-cli/lifecycle.ts` and `lifecycle-core.ts` wire these helpers into the restart path and add explicit rejection of restart attempts when unmanaged restart is disabled, preventing unauthorized gateway restarts through the unmanaged listener path.

#### Restore Telegram webhook-mode health after restarts ([`9d7d961db`](https://github.com/openclaw/openclaw/commit/9d7d961db))

Adds `webhookCertPath?: string` to `startTelegramWebhook()` (`src/telegram/webhook.ts`) so operators can supply a self-signed certificate path for webhook registration. When set, the certificate is wrapped in a grammy `InputFile` and passed to `bot.api.setWebhook()` — the Telegram API requirement for self-signed cert deployments. On the health-check side, `ChannelHealthSnapshot` in `src/gateway/channel-health-policy.ts` gains a `mode?: string` field; `evaluateChannelHealth()` now skips the stale-socket check when `snapshot.mode === "webhook"`, because webhook-mode channels receive events on demand rather than maintaining a persistent outgoing socket. Without this exemption, webhook channels with low traffic would be incorrectly flagged as stale-socket failures after a gateway restart, triggering unnecessary restarts.
