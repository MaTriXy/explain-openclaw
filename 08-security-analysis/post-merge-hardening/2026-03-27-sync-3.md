#### Apply security fix ([`8e285d112`](https://github.com/openclaw/openclaw/commit/8e285d112))

`SENSITIVE_PATTERNS` at `src/config/schema.hints.ts:130` is the array of regular expressions that `mapSensitivePaths()` (line 219) uses to recursively mark config fields for redaction in snapshots, dumps, and debug logs. The existing patterns covered `password`, `secret`, and `api.?key` but missed Feishu-specific encryption key fields. This commit adds `/encrypt.?key/i` at line 135, ensuring that config paths such as `channels.feishu.encryptKey` are treated as sensitive and redacted. A companion change to `src/plugin-sdk/secret-input-schema.ts:55` calls `.register(sensitive)` on `secretInputSchema`, registering the secret-input Zod schema with the sensitive registry so every config field backed by that schema is also swept by `mapSensitivePaths`. Without these changes, Feishu webhook encryption keys could appear unredacted in config export output or diagnostic snapshots. This is a defense-in-depth control for credential leakage in config output; it does not alter runtime behavior.

#### Match guild-level entries in Discord exec allowlist ([`afc649255`](https://github.com/openclaw/openclaw/commit/afc649255))

`resolveDiscordNativeCommandAllowlistAccess()` in `extensions/discord/src/monitor/native-command.ts` previously only checked per-user entries in the exec allow-list and ignored entries prefixed with `guild:`. This commit adds a `guildId?: string | null` parameter (line 147) and a guild-level check at lines 159–164: when a non-empty `guildId` is supplied, the function scans the raw allow-list for entries of the form `guild:<id>` and returns `{ configured: true, allowed: true }` immediately on a match. Previously, a Discord server owner who had configured a guild-level allowlist entry received no enforcement — any guild member could execute exec-allowlisted commands regardless of guild restrictions, because the matcher only consulted user-ID entries. The fix ensures guild-level policies are evaluated before user-level policies, which is the correct precedence for a guild-scoped command authorization.

#### Clamp copilot auth refresh overflow (#55360) (thanks @michael-abdo) ([`85b169c45`](https://github.com/openclaw/openclaw/commit/85b169c45))

`parseCopilotTokenResponse()` in `src/agents/github-copilot-token.ts` distinguishes seconds-epoch from milliseconds-epoch token expiry values by comparing against a threshold. The prior threshold of `10_000_000_000` was too small: certain Copilot tokens carry seconds-epoch values above 10¹⁰, inverting the detection branch and computing an expiry ten thousand times smaller than intended. Lines 42 and 48 raise the threshold to `100_000_000_000` (1e11), matching the inline comment. Separately, `clampRuntimeAuthRefreshDelayMs()` added at `src/agents/runtime-auth-refresh.ts:3` clamps the computed refresh delay to `MAX_SAFE_TIMEOUT_MS` (2,147,483,647 ms) via `Math.min(MAX_SAFE_TIMEOUT_MS, Math.max(minDelayMs, refreshAt - now))`. This prevents an integer overflow in `setTimeout()` that would cause the timer to fire immediately (near-zero delay) or expire incorrectly, creating a token refresh race condition. The clamp is applied at `src/agents/pi-embedded-runner/run/auth-controller.ts:182` where the refresh timer is scheduled. **Addresses Audit 1 Claim 4** (token refresh race condition) and strengthens **Audit 1 Claim 8** (insufficient token expiry validation).

#### Remove unconditional "Tools are disabled" prompt injection in CLI runner ([`99f0ea8d4`](https://github.com/openclaw/openclaw/commit/99f0ea8d4))

`prepareCliRunContext()` at `src/agents/cli-runner/prepare.ts:57` previously constructed `extraSystemPrompt` by joining the caller-supplied prompt with the hardcoded string "Tools are disabled in this session. Do not call tools." The intent was to prevent the LLM from calling OpenClaw's embedded API tools on CLI-native backends. However, CLI agents such as Claude Code interpret this text as a blanket prohibition on all tool execution — including their own native Bash, Read, and Write tools — causing silent tool-call failures across cron jobs, group chats, and DM sessions. The fix replaces the array-join construction with `params.extraSystemPrompt?.trim() ?? ""`, removing the injection entirely. CLI backends manage their own tool availability through their native permission system and do not require a gateway-injected override. This is an incorrect system-prompt injection (unconditional adversarial framing of tool availability) and its removal improves both security posture and reliability.

#### Tighten openai ws reasoning replay ([`d7b61228e`](https://github.com/openclaw/openclaw/commit/d7b61228e))

The OpenAI Realtime WebSocket API uses `previous_response_id` to enable incremental turns — the gateway sends only new tool-result inputs rather than the full context when this ID is present. Before this fix, `src/agents/openai-ws-stream.ts` supplied `previous_response_id` unconditionally from `session.manager.previousResponseId` (line 964), even on the "full context" branch (line 974) that fires when no new tool results are found. Sending a stale `previous_response_id` on full-context turns causes the Realtime API to treat the turn as incremental, potentially replaying stale reasoning context from a prior turn into the new response. This fix encapsulates turn-input planning in `planTurnInput()` (line 459), which returns a `{ mode, inputItems, previousResponseId? }` structure. When new tool results are present the function returns `mode: "incremental_tool_results"` with `previousResponseId` set; the `full_context_restart` branch (no new tool results) and `full_context_initial` path leave `previousResponseId` undefined. The API call at line 1033 conditionally spreads `{ previous_response_id: turnInput.previousResponseId }`, ensuring `previous_response_id` is omitted on full-context turns as the WebSocket guide requires.

#### Default and gate apply_patch like write ([`b9c60fd37`](https://github.com/openclaw/openclaw/commit/b9c60fd37))

Two distinct changes harden `apply_patch` tool access. First, `src/agents/pi-tools.ts:363` changes the default from opt-in (`!!applyPatchConfig?.enabled`, falsy when config is absent) to opt-out (`applyPatchConfig?.enabled !== false`, truthy when config is absent). The `apply_patch` tool is now enabled by default when workspace-contained and no explicit override disables it. Second, and more importantly, `src/agents/tool-policy-match.ts:19,28` changes the `apply_patch` policy axis from `exec` to `write`. Previously, an operator who allowed `exec` but denied `write` inadvertently allowed `apply_patch` — a tool that writes arbitrary file content via unified diff hunks — because policy matching only consulted the `exec` allow/deny lists. The fix at line 19 adds a deny check (`matchesAnyGlobPattern("write", deny)`) and at line 28 changes the allow check to `matchesAnyGlobPattern("write", allow)`. This ensures `apply_patch` is treated symmetrically with the `write` tool for sandbox policy purposes. **Partially addresses Audit 1 Claim 5** (insufficient file permission checks) — closes the exec/write allowlist divergence that permitted ungated file writes via `apply_patch`.

#### Use stream-json output for Claude CLI backend to prevent watchdog timeouts ([`623f4d305`](https://github.com/openclaw/openclaw/commit/623f4d305))

`extensions/anthropic/cli-backend.ts:22–23` switches the Claude CLI backend invocation from `--output-format json` to `--output-format stream-json --verbose`. Under `json` mode, the Claude CLI buffers all output until the request completes, producing no stdout during processing. The gateway's no-output watchdog timer (180 s for resume sessions) fired on long-running sessions — those with large context windows (100K+ tokens) or slow API responses — before the CLI could produce any output, terminating the process with "CLI produced no output for 180s and was terminated." Under `stream-json`, the CLI emits NDJSON events (init, assistant, rate_limit, result) throughout processing; each event resets the watchdog timer, which is the intended behavior. `parseCliJsonl()` at `src/agents/cli-output.ts:130` is updated to handle the NDJSON format, extracting result text, session ID, and usage metadata from streamed events. This is primarily a reliability fix that restores correct availability for long-running CLI backend sessions; preventing false-positive watchdog terminations also removes a surface that could be exploited for DoS via large-context flooding.
