#### Separate exec policy and execution targets ([`7f373823b`](https://github.com/openclaw/openclaw/commit/7f373823b0))

This refactor splits the single command-resolution path in `src/infra/exec-command-resolution.ts` into two distinct resolution lanes: execution targets (what actually runs) and policy targets (what is matched against allowlists and approval rules). New exported functions `resolveExecutionTargetResolution()`, `resolvePolicyTargetResolution()`, `resolveExecutionTargetCandidatePath()`, `resolvePolicyTargetCandidatePath()`, `resolveApprovalAuditCandidatePath()`, and `resolvePolicyAllowlistCandidatePath()` make this distinction explicit. This prevents a class of mismatches where the approval path saw a different executable string than the one that ultimately ran — a prerequisite for the wrapper-trust fixes that follow.

#### Split exec and policy resolution for wrapper trust ([`d8cef14eb`](https://github.com/openclaw/openclaw/commit/d8cef14eb1))

Builds on the above refactor to fix wrapper-trust resolution in `src/infra/exec-command-resolution.ts`, `src/infra/exec-approvals-allowlist.ts`, `src/infra/exec-approvals-analysis.ts`, `src/agents/bash-tools.exec-host-gateway.ts`, and `src/node-host/invoke-system-run.ts`. Wrappers that multiplex into target executables now have their policy checked against the wrapper binary (the approval surface) rather than the resolved inner target. This closes a gap where a wrapper could bypass per-command allowlist enforcement by routing through an already-approved parent.

#### Infra: preserve wrapper executable for multiplexer trust ([`32e89b468`](https://github.com/openclaw/openclaw/commit/32e89b4687))

`resolveExecWrapperTrustPlan()` in `src/infra/exec-wrapper-trust-plan.ts` now preserves the wrapper's executable path through the full multiplexer resolution chain. Previously the wrapper path could be lost during candidate resolution, causing `src/infra/exec-command-resolution.ts` to fall back to the raw command string for trust decisions. The fix ensures the original wrapper binary identity is carried through to approval and allowlist checks.

#### Redact sensitive config values in diff panel ([`9dd0530b9`](https://github.com/openclaw/openclaw/commit/9dd0530b97))

The Control UI's config diff panel now calls `isSensitiveConfigPath()` (defined in `ui/src/ui/views/config-form.shared.ts`) inside `renderDiffValue()` in `ui/src/ui/views/config.ts`. Paths matching token, password, secret, or apiKey patterns return `REDACTED_PLACEHOLDER` instead of the raw config value. This prevents API keys and credentials from appearing in plaintext in the UI's live config diff view.

#### Guard channel-auth against prototype-chain pollution and control-char injection ([`6c44b2ea5`](https://github.com/openclaw/openclaw/commit/6c44b2ea50))

`isConfiguredAuthPlugin()` in `src/cli/channel-auth.ts` previously accessed `cfg.channels[plugin.id]` via bracket notation, which allows JavaScript prototype-chain keys (`__proto__`, `constructor`, `prototype`) to escape the config object and reach arbitrary prototype properties. The fix casts `cfg.channels` as `Record<string, unknown>`, then gates every lookup on `isBlockedObjectKey(key)` (from `src/infra/prototype-keys.ts`) and `Object.prototype.hasOwnProperty.call(channels, key)`. Additionally, plugin IDs in ambiguity error messages are now sanitized with `sanitizeForLog()` (from `src/terminal/ansi.ts`) to strip terminal control characters before they reach the output.

#### Prevent stale auth store reverts ([`03231c063`](https://github.com/openclaw/openclaw/commit/03231c0633))

`updateAuthProfileStoreWithLock()` in `src/agents/auth-profiles/store.ts` previously called `ensureAuthProfileStore()` to obtain the store before applying locked updates. `ensureAuthProfileStore()` returns the cached runtime snapshot, which may lag behind writes made by a concurrent CLI process or config-auth command. The fix switches to `loadAuthProfileStoreForAgent()` inside the lock, forcing a fresh disk read. This prevents a live gateway from overwriting fresher credential state with stale in-memory data during usage or cooldown updates.

#### Redact credentials from cache-trace diagnostic output ([`fb6588cb9`](https://github.com/openclaw/openclaw/commit/fb6588cb99))

`src/agents/cache-trace.ts` now passes diagnostic payloads through `sanitizeDiagnosticPayload()` from `./payload-redaction.ts` before they reach any output surface. In parallel, `src/agents/payload-redaction.ts` adds `REDACTED_IMAGE_DATA` and `redactImageDataForDiagnostics()` to strip base64-encoded image data from cache-trace payloads. This closes an information-disclosure path where raw model request/response bodies — including credentials embedded in requests — could appear in diagnostic traces.

#### Infra: tighten shell-wrapper positional-argv allowlist matching ([`3814f956d`](https://github.com/openclaw/openclaw/commit/3814f956d1))

`isSafeBinUsage()` in `src/infra/exec-approvals-allowlist.ts` tightens how shell-wrapper positional arguments are matched against the exec approval allowlist. The previous matching logic could overmatch on positional arguments that resembled allowed binary names without being the actual command token. The fix narrows the pattern to require canonical positional-argument structure, reducing the chance that a crafted argv could claim approval by matching a safe-bin pattern in an unintended position.

#### Support inline script hashes in Control UI CSP ([`a96eded4a`](https://github.com/openclaw/openclaw/commit/a96eded4a0))

`buildControlUiCspHeader()` in `src/gateway/control-ui-csp.ts` now accepts an `inlineScriptHashes` array and emits each entry as a `'sha256-...'` source in the `script-src` directive. When hashes are provided, the blanket `'unsafe-inline'` allowance is replaced by the specific hash allowlist. The companion function `computeInlineScriptHashes()` scans HTML for inline `<script>` blocks and computes their SHA-256 digests, enabling the gateway to enumerate hashes from the Control UI bundle at startup rather than requiring unsafe-inline for all scripts.
