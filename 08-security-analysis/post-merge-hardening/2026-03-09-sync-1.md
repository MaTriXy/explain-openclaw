#### Split fs bridge path safety ([`40dfba85d`](https://github.com/openclaw/openclaw/commit/40dfba85d))

This refactor established the foundation for all subsequent sandbox path-traversal hardening by extracting path safety logic from the monolithic `fs-bridge.ts` into dedicated modules. The new `SandboxFsPathGuard` class in `fs-bridge-path-safety.ts` centralizes all path validation with `assertPathSafety()`, `openReadableFile()`, and `resolveRequiredMount()` methods. Shell command plans were separated into `fs-bridge-command-plans.ts` (later renamed in `bebde34b9`). Tests were split across three new files — `fs-bridge.anchored-ops.test.ts`, `fs-bridge.boundary.test.ts`, and `fs-bridge.shell.test.ts` — establishing clear test boundaries between path safety and command execution. This architectural separation is what made all subsequent per-operation anchoring feasible.

#### Pin fs bridge readfile handles ([`99cfd271d`](https://github.com/openclaw/openclaw/commit/99cfd271d))

This commit eliminated a TOCTOU (Time-Of-Check-Time-Of-Use) race in sandbox file reads. Previously, `readFile()` executed a shell `cat` command via `buildReadFilePlan()`, creating a window between path validation and actual file access where an attacker controlling symlinks inside the sandbox could redirect the read to a different target. The fix uses `openReadableFile()` to obtain a pinned file descriptor (fd) via `openBoundaryFile()` — which validates symlink boundaries at the boundary-file-read layer before returning the fd — then reads synchronously via `fs.readFileSync(fd)` in a `finally` block that guarantees close. Because the fd is pinned to the validated inode at open time, subsequent symlink manipulation cannot affect the read.

#### Anchor fs-bridge mkdirp ([`09cfcf9dd`](https://github.com/openclaw/openclaw/commit/09cfcf9dd))

Directory creation operations in the sandbox fs bridge were vulnerable to path traversal because the shell script received a full user-supplied path: `mkdir -p -- "$1"`. This commit introduces `resolveAnchoredSandboxEntry()` which canonicalizes the parent directory and separates it from the basename. The shell script now receives two distinct arguments: `cd -- "$1"; mkdir -p -- "$2"` where `$1` is the canonical parent and `$2` is the basename only. An attacker supplying a path with `..` components or symlink segments cannot escape the canonical parent, because `cd` to the canonical path establishes the working directory before the relative `mkdir` runs.

#### Anchor fs-bridge destructive ops ([`f195af0b2`](https://github.com/openclaw/openclaw/commit/f195af0b2))

The same canonical-parent-plus-basename anchoring pattern is applied to `remove()` and `rename()` operations. The new `AnchoredSandboxEntry` type carries `canonicalParentPath` and `basename` as separate typed fields, preventing accidental recombination back into a user-supplied path. For `remove()`, the script becomes `cd -- "$1"; rm ... -- "$2"`. For `rename()`, both source and destination are anchored independently: `mkdir -p -- "$2"; cd -- "$1"; mv -- "$3" "$2/$4"` — where `$1`/`$3` are the source canonical-parent/basename and `$2`/`$4` are the destination canonical-parent/basename. This prevents both delete and move operations from escaping the sandbox via relative symlinks or path manipulation.

#### Clarify fs bridge read and shell plans ([`bebde34b9`](https://github.com/openclaw/openclaw/commit/bebde34b9))

This cleanup refactoring clarifies the security boundary between path validation (via the boundary-file-read layer) and shell command execution. `fs-bridge-command-plans.ts` was renamed to `fs-bridge-shell-command-plans.ts` to make explicit that command plans run in a shell context. Method names in `fs-bridge-path-safety.ts` were updated for precision: `assertPathSafetyWithinMount()` became `assertGuardedPathSafety()` and `resolveMountByContainerPath()` became `openBoundaryWithinRequiredMount()`. A `readPinnedFile()` helper was extracted into `fs-bridge.ts` to encapsulate the fd-based read pattern established in `99cfd271d`. These naming changes make it easier to audit which code runs in the shell vs. which code validates boundaries before shell invocation.

#### Fail closed for config-first secretrefs ([`da88d9209`](https://github.com/openclaw/openclaw/commit/da88d9209))

When gateway auth is configured with `precedence: "config-first"` and a secretref like `${GATEWAY_TOKEN}`, the expectation is that the config-side secret wins over the environment variable. A bug allowed silent fall-through: if the secretref failed to resolve (empty or missing) but the corresponding env var existed, the gateway would silently use the env var instead of failing, breaking the user's expressed precedence intent and potentially using a weaker credential. `resolveGatewayCredentialsFromConfig()` now calls `throwUnresolvedGatewaySecretInput()` when a `config-first` secretref for token or password is unresolvable yet an env var is present. This fail-closed behavior forces the operator to fix the secretref rather than silently degrading to env-var auth.

#### Inherit ask from exec-approvals.json when tools.exec.ask unset ([`c6575891c`](https://github.com/openclaw/openclaw/commit/c6575891c))

Exec approval behavior was inconsistent when `tools.exec.ask` was not set in the agent config: the code fell back to `"on-miss"` without checking `exec-approvals.json`, which may have a different `defaults.ask` policy configured by the operator. The fix changes the resolution to `defaults?.ask ?? loadExecApprovals().defaults?.ask ?? "on-miss"` — respecting the `exec-approvals.json` defaults as an intermediate fallback before the hardcoded `"on-miss"`. This ensures that operator-level exec-approval policies are honored even when individual agent configs do not explicitly set `tools.exec.ask`, closing a gap where exec approval defaults could diverge between agents.

#### Block cached device token override fallback ([`f236742dc`](https://github.com/openclaw/openclaw/commit/f236742dc))

The gateway client had a fallback path where `resolvedDeviceToken` (the cached device token from the authentication handshake) could be used even when explicit credentials were configured. The condition now checks `!(explicitGatewayToken || this.opts.password?.trim())` before using the cached token, ensuring that if an explicit gateway token or password is present, the cached device token is not used as a substitute. This prevents a scenario where a stale or different-identity device token could implicitly override explicit operator-configured credentials.

#### Honor commands.allowFrom in native command auth ([`c22a4450e`](https://github.com/openclaw/openclaw/commit/c22a4450e))

Telegram native command authorization was not checking `cfg.commands.allowFrom` — the operator-configured allowlist for which users may invoke native commands. The fix integrates `resolveCommandAuthorization()` into `resolveTelegramCommandAuth()` when `commandsAllowFrom` is configured, routing command auth through the unified authorization path that respects the allowFrom policy. Without this fix, `commands.allowFrom` had no effect on native command access, meaning commands could be invoked by users outside the intended allow set.

#### Config: fail closed invalid config loads ([`2c7fb5495`](https://github.com/openclaw/openclaw/commit/2c7fb5495))

Previously, `loadConfig()` would return a partial or default config when the config file was invalid (e.g., JSON5 parse errors, schema validation failures), silently falling back rather than failing. This commit makes `loadConfig()` throw on invalid configs, adding a separate `readBestEffortConfig()` function for contexts that genuinely need a non-throwing fallback (CLI diagnostics, status commands). The fail-closed behavior prevents agents and the gateway from operating under invalid configuration that could leave security controls in an unexpected state. The CLI is updated to use `readBestEffortConfig()` where explicit best-effort semantics are required.

#### Use group allowlist for native command auth in groups ([`02eef1d45`](https://github.com/openclaw/openclaw/commit/02eef1d45))

In Telegram groups, native command authorization was not using the group-level allowlist (`effectiveGroupAllow`) as an authorizer. This commit passes `effectiveGroupAllow` as a second authorizer in `resolveTelegramCommandAuth()` for group messages, so that group-configured sender allowlists are enforced for native commands. Without this fix, native commands in group contexts bypassed the group-specific allowFrom policy.

#### Honor allow-always for bash script invocations ([`9d2b29299`](https://github.com/openclaw/openclaw/commit/9d2b29299))

The exec-approvals allowlist matching had a gap for bash script invocations: when an agent invoked bash with a script file path (e.g., `bash /path/to/script.sh`), only the `bash` executable itself was checked against the allowlist, not the script path. An `allow-always` entry for a specific script would never match. This commit adds `resolveShellWrapperScriptCandidatePath()` to extract the candidate script path from shell wrapper invocations, then runs a second `matchAllowlist()` call against the script path. The `extractShellWrapperInlineCommand()` check ensures that inline shell commands (not script files) are not incorrectly matched as file paths.

#### Restore sender-only allowFrom validation ([`13ed6afe6`](https://github.com/openclaw/openclaw/commit/13ed6afe6))

A regression in `normalizeAllowFrom()` allowed negative integers (Telegram group IDs, which use negative numbers) to pass through sender-only allowFrom validation. The regex was changed from `/^-?\d+$/` to `/^\d+$/` to reject negative integers, restoring the invariant that sender-only allowFrom entries must be positive user IDs. Group IDs should not appear in sender-specific allowlists; allowing them could expand sender authorization unintentionally when the allowlist was intended for individual users only.

#### Cover sender-only groupAllowFrom normalization ([`240b143bd`](https://github.com/openclaw/openclaw/commit/240b143bd))

Test coverage for the `normalizeAllowFrom()` regex fix in `13ed6afe6`, specifically verifying that negative integers (group IDs) are rejected from sender-only groupAllowFrom configurations. Ensures the authorization invariant is regression-tested.
