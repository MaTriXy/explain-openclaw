#### `ff607adc69` — Sandbox home credential bind block

`getBlockedBindReason()` (`src/agents/sandbox/validate-sandbox-security.ts:110`) is exported and called from `validateBindMounts()` (`:280`) during Docker sandbox validation. This commit adds `getBlockedHostPaths()` (`:136`) and `getBlockedHomeRoots()` (`:146`) — both private helpers — that enumerate the user home directory and resolve all credential subdirectory paths (`~/.aws`, `~/.ssh`, `~/.gnupg`, `~/.config/gcloud`, etc.) as blocked bind-mount sources. A Docker container that requested a bind of any home credential path now receives a `BlockedBindReason` and is rejected before container start by `validateSandboxSecurity()` (`:377`). This closes a credential exfiltration vector where an AI agent could exfiltrate cloud or SSH credentials by requesting a Docker bind mount targeting credential directories under the host home. No existing audit claims addressed; new sandbox hardening.

#### `e19dce0aed` — fail-closed before_tool_call hook runner

`BEFORE_TOOL_CALL_HOOK_FAILURE_REASON` (`src/agents/pi-tools.before-tool-call.ts:27`) is a constant set to `"Tool call blocked because before_tool_call hook failed"`. Previously, an unexpected exception in the `before_tool_call` hook runner propagated upward uncaught, allowing tool execution to continue if the hook threw. The fix wraps the hook runner in a try/catch at line 373 that returns `{ blocked: true, reason: BEFORE_TOOL_CALL_HOOK_FAILURE_REASON }` on any error, ensuring tool calls are blocked rather than passed through when the hook fails. This is a fail-closed hardening for the hook execution pipeline — a hook that crashes now blocks the tool call instead of silently succeeding. No existing audit claims addressed; new defense-in-depth.

#### `42ffdf882f` — Cross-origin redirect auth header stripping (GHSA-6mgf-v5j7-45cr)

`retainSafeHeadersForCrossOriginRedirectHeaders()` (`src/infra/net/fetch-guard.ts:129`) is exported and used at line 273 when following cross-origin redirects. The companion private `retainSafeHeadersForCrossOriginRedirect()` (`:135`) takes the full `RequestInit` and returns a copy with only safe headers retained. Previously, the redirect handler used a narrow sensitive-header denylist (stripping only known-bad headers like Authorization), which could miss custom auth headers not on the denylist. The fix switches to a safe-header allowlist: only Accept, Accept-Language, Content-Language, Content-Type, and a small set of CORS-safe headers are forwarded across origin boundaries — everything else is dropped. This directly remediates GHSA-6mgf-v5j7-45cr (HIGH severity), which describes the pre-fix behavior as forwarding custom authorization headers on cross-origin redirects.

#### `e8e7d1fab3` — Non-interactive auth choices exclude untrusted workspace plugins

`resolveManifestProviderAuthChoices()` (`src/plugins/provider-auth-choices.ts:32`) now accepts `includeUntrustedWorkspacePlugins?: boolean` (`:36`). The gate at line 47 — `params?.includeUntrustedWorkspacePlugins === false && !resolveEffectiveEnableState(...).enabled` — returns an empty array for workspace-origin plugins that are not explicitly enabled when the flag is `false`. Non-interactive onboarding (which sets this flag to `false`) therefore cannot acquire provider credentials on behalf of untrusted workspace plugins that have not been explicitly enabled by the operator. This narrows the attack surface for a malicious workspace plugin that attempts to harvest provider API keys during automated setup flows. No existing audit claims addressed; new auth isolation.

#### `e4b5027c5e` — Extension seam refactor (security audit channels)

This large refactor (234 files) moves channel-specific security audit logic from `src/security/` into their respective extension packages. `collectDiscordSecurityAuditFindings()` (`extensions/discord/src/security-audit.ts:65`) and `isDiscordMutableAllowEntry()` (`extensions/discord/src/security-audit.ts:24`) are now extension-owned. The core `src/security/audit.ts` retains the generic `runSecurityAudit()` orchestrator and `SecurityAuditFinding` type, while per-channel audit collection is now encapsulated within each extension. No security controls were removed; this is an architectural ownership refactor that reduces the footprint of channel-specific logic in core security paths. No audit claims affected.
