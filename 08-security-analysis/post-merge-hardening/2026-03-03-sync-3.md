#### Fail closed plugin auth path canonicalization ([`93b072402`](https://github.com/openclaw/openclaw/commit/93b072402))

Plugin authentication now fails closed against URL encoding bypass attacks. `buildCanonicalPathCandidates()` iterates up to `MAX_PATH_DECODE_PASSES` (32) rounds of percent-decoding, tracking `decodePassLimitReached` when a path still contains decodable sequences at the decode limit. New `hasSecurityPathCanonicalizationAnomaly()` returns true when the canonical path exhibits `malformedEncoding` or `decodePassLimitReached`, signaling a potential obfuscation attempt. New `shouldEnforceGatewayAuthForPluginPath()` combines anomaly detection with `isProtectedPluginRoutePath()` — if either condition is true, gateway auth is required. This means a path that is not directly in the protected set but exhibits suspicious encoding still requires auth, preventing dot-segment traversal via multi-round percent-encoding. `enforcePluginRouteGatewayAuth` is consolidated into `server/http-auth.ts`. Patches GHSA-mwxv-35wr-4vvj (Gateway plugin auth bypass via encoded dot-segment traversal).

#### Preserve dns pinning for strict web SSRF fetches ([`345abf0b2`](https://github.com/openclaw/openclaw/commit/345abf0b2))

Closes a TOCTOU window in the SSRF guard where env proxy settings could bypass DNS pinning. Previously, `fetchWithSsrFGuard()` switched to the env proxy dispatcher when an env proxy was configured, skipping the pinned DNS resolver that enforces IP-based SSRF checks. This created a window where the pre-check (using pinned DNS) could resolve to a safe IP while the actual connection (via env proxy) could be redirected. Now `fetchWithSsrFGuard()` only enables the env proxy dispatcher when `dangerouslyAllowEnvProxyWithoutPinnedDns: true` is explicitly set; all other calls use DNS-pinned dispatch regardless of env proxy configuration. `fetchWithWebToolsNetworkGuard()` also loses its env-proxy default — callers must opt in via `useEnvProxy: true`, which `withTrustedWebToolsEndpoint()` does for its internal trusted endpoints only. Slack's file upload path explicitly sets the dangerous flag for its known-safe upload endpoint. Addresses Audit 2 Claim 4 (DNS rebinding SSRF via web-fetch).

#### Harden inbound sender id handling ([`ee1b14763`](https://github.com/openclaw/openclaw/commit/ee1b14763))

ZaloUser inbound message routing now rejects null, empty, and whitespace-only sender IDs before authorization. `isSenderAllowed()` accepts `string | undefined` and short-circuits to false when the sender ID is absent or contains only whitespace, preventing evaluation of an empty string against the allowlist. `processMessage()` trims and validates `senderId` early, dropping messages with missing identities before any allowlist or routing logic runs. This closes a path where a crafted message with no sender ID could be routed against a wildcard allowlist entry.

#### Share auth, routing, and stream/account helpers ([`ed21b63bb`](https://github.com/openclaw/openclaw/commit/ed21b63bb))

Centralizes authentication, routing, and account resolution logic into the plugin SDK, reducing duplicated access control code across channel extensions. New shared helpers include `resolveDirectDmAuthorizationOutcome()` for DM policy enforcement, `resolveSenderCommandAuthorizationWithRuntime()` for command authorization with runtime context, `listConfiguredAccountIds()` and `resolveAccountWithDefaultFallback()` for account resolution, and `resolveAllowlistMatchByCandidates()` for allowlist matching. Five extensions and seven agent/gateway modules now use these shared APIs, concentrating authorization logic in centrally-maintained, auditable functions and reducing the risk of inconsistent permission checks across channels (related to Audit 1 Claims 5 and 6).
