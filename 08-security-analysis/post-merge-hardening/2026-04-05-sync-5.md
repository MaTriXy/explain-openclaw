#### Harden ollama tool-call replay (#52253) (thanks @Adam-Researchh) ([`87abcfd6a`](https://github.com/openclaw/openclaw/commit/87abcfd6a))

Extracts a new module `extensions/ollama/src/ollama-json.ts` containing two JSON parsing utilities: `quoteUnsafeIntegerLiterals` walks raw JSON text and quotes any integer literal exceeding `Number.MAX_SAFE_INTEGER` before parsing, and `parseJsonPreservingUnsafeIntegers` wraps `JSON.parse` with that preprocessing so precision-losing integer coercion cannot silently corrupt large IDs or arguments returned by ollama. A companion `normalizeOllamaCompatMessageToolArgs` function (in `extensions/ollama/src/stream.ts`) iterates all `message.tool_calls[].function.arguments` and `message.function_call.arguments` fields in a request payload, coercing raw JSON strings or malformed values into proper objects via `ensureArgsObject`. This normalization is applied in `wrapOllamaCompatNumCtx` before the payload is sent, and `parseJsonPreservingUnsafeIntegers` is also used directly when streaming `OllamaChatResponse` lines. The fix prevents tool-call replay failures caused by ollama returning tool arguments as serialized JSON strings with large integer IDs that would lose precision under standard `JSON.parse`.

#### Move ollama synthetic auth precedence into extension ([`49936f606`](https://github.com/openclaw/openclaw/commit/49936f606))

Generalizes the previously ollama-specific `shouldDeferSyntheticOllamaProfileAuth` in `src/agents/model-auth.ts` into a provider-agnostic `shouldDeferSyntheticProfileAuth` function that delegates to a new plugin SDK hook, `shouldDeferProviderSyntheticProfileAuthWithPlugin`. The hook is typed via `ProviderDeferSyntheticProfileAuthContext` (added to `src/plugins/types.ts` and re-exported from `src/plugin-sdk/plugin-entry.ts`) and declared as an optional method `shouldDeferSyntheticProfileAuth` on the provider plugin entry type. The ollama extension implements this hook in `extensions/ollama/index.ts`, returning `true` when `resolvedApiKey` matches `DEFAULT_API_KEY` â€” preserving the pre-existing behavior where a synthetic "ollama-local" profile key defers to real environment or config credentials rather than locking into the synthetic profile. The refactor removes the hard-coded ollama identity check from core auth resolution while making the deferral mechanism extensible to any provider that registers the hook.
