#### Sanitize workspace deps before plugin install ([`1bd79add8`](https://github.com/openclaw/openclaw/commit/1bd79add8))

The plugin installer now scrubs `workspace:*` version specifiers from
`devDependencies` before running `npm install` in a plugin directory. The new
helper `sanitizeManifestForNpmInstall()` reads `package.json` from the target
directory, strips any dev-dependency entries whose version spec starts with
`workspace:`, and rewrites the file before install proceeds. If all
`devDependencies` entries are workspace-relative, the entire `devDependencies`
key is removed. This prevents monorepo-internal workspace paths from leaking
into npm install invocations outside the monorepo, which would otherwise cause
the install to fail or resolve unintended local packages.

#### Unify credential precedence across entrypoints ([`08431da5d`](https://github.com/openclaw/openclaw/commit/08431da5d))

Gateway credential resolution is now centralised in `src/gateway/credentials.ts`
via two new exported functions: `resolveGatewayCredentialsFromValues()` accepts
raw token/password values with explicit precedence rules, and
`resolveGatewayCredentialsFromConfig()` reads from a full `OpenClawConfig` object
and dispatches to the appropriate local or remote credential path. Both functions
respect a configurable `GatewayCredentialPrecedence` (`env-first` or
`config-first`) so that callers can declare whether environment variables take
precedence over config-file values or vice versa. Previously, credential
selection logic was duplicated across the daemon lifecycle entrypoint, the agent
gateway tool, and the gateway auth module — each with subtly different ordering.
The refactor eliminates those divergences, ensuring that `OPENCLAW_GATEWAY_TOKEN`,
`OPENCLAW_GATEWAY_PASSWORD`, and their legacy counterparts are read and ranked
consistently regardless of which code path initiates the connection.

#### Normalize SSRF IP parsing and monitor typing ([`98427453b`](https://github.com/openclaw/openclaw/commit/98427453b))

The SSRF guard in `src/infra/net/ssrf.ts` now correctly rejects IPv4 addresses
embedded inside IPv6 transition formats. The new helper
`parseIpv6WithEmbeddedIpv4()` in `src/shared/net/ip.ts` parses mixed-notation
addresses (e.g. `::1.2.3.4`, `64:ff9b:1::10.0.0.1`, `2002:7f00:0001::`) by
extracting the embedded IPv4 component and resolving it through a table-driven
`EMBEDDED_IPV4_SENTINEL_RULES` list that covers the IPv4-compatible (`::w.x.y.z`),
NAT64 (`64:ff9b:1::`), 6to4 (`2002::`), Teredo (`2001:0000::`), and ISATAP IID
formats. The RFC 2544 benchmarking range (198.18.0.0/15, represented as
`RFC2544_BENCHMARK_PREFIX`) is added to the blocked IPv4 special-use set. In
addition, `isLegacyIpv4Literal()` now detects non-canonical decimal notation such
as octal (`0177.0.0.1`), hexadecimal, and abbreviated forms (`127.1`) that some
resolvers accept but that the previous guard would pass as non-IP hostnames. These
changes close SSRF bypass vectors where an attacker could supply an IPv6-encoded
private address or a non-standard IPv4 literal to slip past the block list.

#### Reuse signal uuid detection and payload types ([`b3c78e5e0`](https://github.com/openclaw/openclaw/commit/b3c78e5e0))

The outbound Signal session handler now reuses `looksLikeUuid()` from
`src/signal/identity.ts` to distinguish UUID-format recipients from phone-number
recipients when building `sourceUuid` / `sourceNumber` fields. Previously, the
outbound session module duplicated this heuristic inline. On the type side,
`QueuedDelivery extends QueuedDeliveryPayload` in `src/infra/outbound/delivery-queue.ts`
consolidates the queue entry shape by lifting shared payload fields into a base
type, reducing duplication across the delivery queue and outbound session modules.
This is a code-quality refactor with no new security boundary, but it tightens
the Signal routing logic and removes inconsistencies in how UUID vs. phone-number
recipients are identified.
