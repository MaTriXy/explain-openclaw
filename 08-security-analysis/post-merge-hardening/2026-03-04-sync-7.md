#### Narrow legacy route inheritance for custom session keys (openclaw#33932) thanks @Takhoffman ([`b4e4e25e7`](https://github.com/openclaw/openclaw/commit/b4e4e25e7))

The `hasLegacyChannelPeerShape` boolean in `src/gateway/server-methods/chat.ts` previously allowed any legacy channel peer session to inherit prior external route metadata. This commit adds an extra guard: `sessionChannelHint === routeChannelCandidate`. Route inheritance now only occurs if the session's channel hint explicitly matches the incoming route's channel candidate, preventing stale routes from bleeding across sessions where the channel identity diverges.

#### Preserve route inheritance for legacy channel session keys (openclaw#33919) thanks @Takhoffman ([`8a7d1aa97`](https://github.com/openclaw/openclaw/commit/8a7d1aa97))

Introduces `hasLegacyChannelPeerShape` in `src/gateway/server-methods/chat.ts` and changes `canInheritDeliverableRoute` to use `(isChannelScopedSession || hasLegacyChannelPeerShape)`. Before this fix, legacy channel peer sessions (sessions whose peer shape uses a non-canonical scope string) could not inherit route metadata at all, causing delivery failures. The fix restores inheritance for these legacy sessions while respecting the `CHANNEL_SCOPED_SESSION_SHAPES` set that canonically defines `["direct", "dm", "group", "channel"]`.

#### Add per-topic agent routing for forum groups [AI-assisted] ([`58bc9a241`](https://github.com/openclaw/openclaw/commit/58bc9a241))

Adds an optional `agentId` field to `TelegramTopicConfig` in `src/config/types.telegram.ts` with zod validation in `zod-schema.providers-core.ts`. When set, `src/telegram/bot-message-context.ts` re-derives the session key using the topic's designated agent via `rawTopicAgentId = topicConfig?.agentId?.trim()`. Each forum topic gets its own isolated workspace, memory, and session rather than sharing the default agent. The account fail-closed guard (owner verification) is preserved for all topic-routed sessions.

#### Unify session delivery invariants for duplicate suppression ([`7f2708a8c`](https://github.com/openclaw/openclaw/commit/7f2708a8c))

This commit establishes two sentinel sets in `src/gateway/server-methods/chat.ts`: `CHANNEL_AGNOSTIC_SESSION_SCOPES` (scopes that are not tied to a specific channel: `main`, `direct`, `dm`, `group`, `channel`, `cron`, `run`, `subagent`, `acp`, `thread`, `topic`) and `CHANNEL_SCOPED_SESSION_SHAPES` (canonical channel-bound shapes). These replace ad-hoc inline checks previously scattered across `src/auto-reply/reply/reply-payloads.ts`, `src/auto-reply/reply/session-delivery.ts`, and `src/utils/delivery-context.ts`. Duplicate-suppression logic now evaluates topic and session-scope invariants consistently, preventing the same message from being delivered twice when session keys share overlapping shapes.

#### Follow-up: align ingress, atomic paths, and channel tests with credential semantics ([`1c200ca7a`](https://github.com/openclaw/openclaw/commit/1c200ca7a))

**Audit 1 Claim 6 / Audit 2 Claim 2.** The `writeViaSiblingTempPath()` function in `src/browser/output-atomic.ts` now resolves both `rootDir` and the parent directory of `targetPath` through `fs.realpath()` before the boundary check, with a `catch` fallback to `path.resolve()`. This prevents symlink alias escape: an attacker-controlled symlink in the target directory could previously redirect a write to an arbitrary path outside the sandbox root because the pre-symlink-resolved path appeared within bounds. The `src/gateway/server-methods/agents-mutate.test.ts` symlink safety test description was updated from "allows in-workspace symlink targets for get/set" to "allows in-workspace symlink reads but rejects writes through symlink aliases", making the intended invariant explicit. This commit also aligns credential semantics in `src/agents/auth-profiles.ts`, `src/agents/auth-profiles/credential-state.ts`, and `src/agents/auth-profiles/oauth.ts`.

#### Use HTTPS git URL for api-beta ([`5ce53095c`](https://github.com/openclaw/openclaw/commit/5ce53095c))

Changes the `@tloncorp/api` dependency in `extensions/tlon/package.json` from the short `github:tloncorp/api-beta#hash` form to the explicit `git+https://github.com/tloncorp/api-beta.git#hash` form. The short `github:` protocol instructs pnpm/npm to resolve the dependency via SSH, using the system's configured SSH keys. In CI/CD environments this exposes SSH credentials to the package resolution process. Switching to an explicit HTTPS URL removes the SSH key dependency for this third-party package.

#### Plugin-sdk: add channel subpaths and migrate bundled plugins ([`1278ee924`](https://github.com/openclaw/openclaw/commit/1278ee924))

Adds typed channel-specific subpath exports to `openclaw/plugin-sdk`: `discord.ts`, `slack.ts`, `imessage.ts`, `whatsapp.ts`, `line.ts`, and `signal.ts` under `src/plugin-sdk/`. Plugins are migrated from the monolithic `"openclaw/plugin-sdk"` entry to scoped imports (`"openclaw/plugin-sdk/discord"`, `"openclaw/plugin-sdk/slack"`, etc.). A new build-time enforcement script `scripts/check-no-monolithic-plugin-sdk-entry-imports.ts` uses `ROOT_IMPORT_PATTERN` to detect any bundled plugin entry file that still imports from the monolithic root, failing the build if found. This prevents plugins from accidentally accessing internal APIs they were not meant to consume, reducing the attack surface exposed to plugin authors.
