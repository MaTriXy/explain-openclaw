#### Config: newline-join sandbox setupCommand arrays ([`9b8e64247`](https://github.com/openclaw/openclaw/commit/9b8e64247))

The `setupCommand` field in the Docker sandbox schema (`src/config/zod-schema.agent-runtime.ts:105-108`) now accepts both a plain string and an array of strings via a Zod `.union()` + `.transform()`. When the value is an array, elements are joined with newlines before the string type coercion: `value.join("\n")`. This prevents shell command arrays from being space-concatenated by config tooling, ensuring each array element is treated as a separate shell statement when passed to `execDocker()` at `src/agents/sandbox/docker.ts:472` via `sh -lc`. Reduces injection surface when array-form `setupCommand` entries are constructed from user-supplied strings.

#### Split gateway auth suites and share safe write path checks ([`83c8406f0`](https://github.com/openclaw/openclaw/commit/83c8406f0))

Gateway auth test files were refactored from a single large file into three focused suites (`server.auth.control-ui.suite.ts`, `server.auth.default-token.suite.ts`, `server.auth.modes.suite.ts`) sharing helpers via `server.auth.shared.ts`. Crucially, `openWritableFileWithinRoot()` (`src/infra/fs-safe.ts:320`) was introduced to unify symlink-safe file opening across `archive.ts` and `fs-safe.ts` write flows. It enforces `O_NOFOLLOW` flags where available and performs post-open realpath verification to detect race conditions where parent directories are swapped between path resolution and file open. This centralizes symlink-traversal defenses that were previously duplicated.

#### Harden zip write race handling ([`7dac9b05d`](https://github.com/openclaw/openclaw/commit/7dac9b05d))

`openZipOutputFile()` (`src/infra/archive.ts:280`) was hardened against TOCTOU symlink rebind races. After opening the output file, `resolveOpenedFileRealPathForHandle()` (`src/infra/fs-safe.ts:292`) verifies that the realpath of the opened file handle still resolves within the expected destination boundary — detecting cases where a parent directory symlink was swapped between the initial path check and the file open. A new test utility `withRealpathSymlinkRebindRace()` (`src/test-utils/symlink-rebind-race.ts:4`) provides a controlled environment for testing symlink rebind attacks, used in both `fs-safe.test.ts` and `archive.test.ts`.

#### Table-drive wrapper approval pinning tests ([`aee27d0e3`](https://github.com/openclaw/openclaw/commit/aee27d0e3))

Extracted `shouldPinExecutableForApproval()` (`src/node-host/invoke-system-run-plan.ts:56`) to explicitly encode the decision logic for when an approved executable's `argv[0]` should be pinned versus when wrapper argv structure must be preserved. Direct (non-wrapped) approved commands are pinned to prevent argument-shifting attacks where `argv[0]` rewriting could change the effective binary. Wrapper commands retain their argv shape to preserve the semantics under which they were approved. Refactored tests into table-driven format to make the approval boundary logic explicitly testable.

All install and write paths now share `openWritableFileWithinRoot()` and `writeFileFromPathWithinRoot()` from `src/infra/fs-safe.ts`, ensuring consistent symlink-traversal and TOCTOU race detection regardless of the install entry point.

#### Harden webhook auth-before-body handling ([`d3e8b17aa`](https://github.com/openclaw/openclaw/commit/d3e8b17aa))

All webhook handlers now enforce authentication before reading the request body. `beginWebhookRequestPipelineOrReject()` (`src/plugin-sdk/webhook-request-guards.ts:179`) gates the body read behind an auth check and tracks the request via `createWebhookInFlightLimiter()` (`src/plugin-sdk/webhook-request-guards.ts:84`). `WEBHOOK_BODY_READ_DEFAULTS` (`src/plugin-sdk/webhook-request-guards.ts:13`) provides separate pre-auth limits (64 KB, 5 s) and post-auth limits (1 MB, 30 s). LINE, BlueBubbles, and GoogleChat webhook handlers were updated to use these defaults, preventing slowloris-style and unauthenticated body-read DoS attacks. A lint script `scripts/check-webhook-auth-body-order.mjs` guards against future regressions where auth and body-read order is reversed.

#### Unify root-bound write hardening ([`104d32bb6`](https://github.com/openclaw/openclaw/commit/104d32bb6))

`assertCanonicalPathWithinBase()` (`src/infra/install-safe-path.ts:66`) was introduced for unified root-boundary validation: it resolves the canonical path and verifies the result remains within the supplied `baseDir` before permitting the write. Used by skill downloads (`src/agents/skills-install-download.ts:71`), package installs (`src/infra/install-package-dir.ts:57`), and browser output. `writeFileFromPathWithinRoot()` provides atomic file copy with full symlink-traversal and TOCTOU detection via `openWritableFileWithinRoot()` (`src/infra/fs-safe.ts:320`). New tests verify that symlink rebind races cannot escape root boundaries.

#### Split webhook ingress and policy guards ([`1c9deeda9`](https://github.com/openclaw/openclaw/commit/1c9deeda9))

BlueBubbles and GoogleChat `monitor.ts` ingress files were split into focused modules (`monitor-debounce.ts`, `monitor-access.ts`, `monitor-types.ts`, `monitor-webhook.ts`). `respondWebhookBodyReadError()` (`src/plugin-sdk/webhook-request-guards.ts:58`) was extracted as a centralized error dispatcher for `readWebhookBodyOrReject()` and `readJsonWebhookBodyOrReject()`, ensuring consistent HTTP status codes (413 for oversized, 408 for timeout, 400 for malformed) across all body-read failure paths. The `beginWebhookRequestPipelineOrReject()` and `createWebhookInFlightLimiter()` guards from d3e8b17aa are now re-exported through the plugin SDK (`src/plugin-sdk/index.ts:141-142`), enabling third-party webhook extensions to adopt the same auth-before-body guard pattern.

#### Preserve system.run wrapper approval semantics ([`dded56962`](https://github.com/openclaw/openclaw/commit/dded56962))

`hardenApprovedExecutionPaths()` (`src/node-host/invoke-system-run-plan.ts:66`) received a new `shellCommand` parameter (line 69). When a shell wrapper is present (`shellCommand != null`), `argv[0]` rewriting is skipped — rewriting `env sh -c ...` to `/bin/sh sh -c ...` would shift positional arguments and silently change the executed binary from the approved one. A `wrapperChain` check additionally preserves transparent wrapper argv structure to prevent approval bypass via argument position shifts. Addresses GHSA-q399-23r3-hfx4 (HIGH) where post-approval executable rebind was possible via PATH-token identity mismatch.
