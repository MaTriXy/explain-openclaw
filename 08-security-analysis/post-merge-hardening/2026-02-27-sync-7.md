#### macOS LaunchAgent TLS Certificate Fix ([`d33f24c4e`](https://github.com/openclaw/openclaw/commit/d33f24c4e))

`buildServiceEnvironment()` and `buildNodeServiceEnvironment()` in `src/daemon/service-env.ts` now default `NODE_EXTRA_CA_CERTS` to `/etc/ssl/cert.pem` on macOS when the variable is not already set in the environment. launchd services do not inherit the shell environment, so Node's `undici/fetch` could not locate the macOS system CA bundle when the gateway ran as a LaunchAgent, causing silent TLS verification failures for HTTPS requests (Telegram, webhooks). User-supplied `NODE_EXTRA_CA_CERTS` always takes precedence. The fix was also refactored (`9d52dcf1f`) to accept an injectable `platform` parameter, making the CA defaulting logic unit-testable without relying on `process.platform`.

#### Telegram Group Auth Boundary Documented ([`1f68010bd`](https://github.com/openclaw/openclaw/commit/1f68010bd))

This documentation commit clarifies the security boundary introduced in release `2026.2.25+`: group sender authorization does **not** inherit DM pairing-store approvals. The runtime enforces this in `src/telegram/bot/helpers.ts` with an explicit comment ("DM pairing store entries are not a group authorization source"). Operators must configure `groupAllowFrom` explicitly for Telegram groups; the DM pairing store only governs direct messages. This is the Telegram-channel analogue to GHSA-gp3q-wpq4-5c5h (LINE group allowlist scope mismatch, published 2026-02-26).
