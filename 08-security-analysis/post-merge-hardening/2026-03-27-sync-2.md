#### Gateway: require requester ownership for HTTP session kills ([`02cf12371`](https://github.com/openclaw/openclaw/commit/02cf12371))

`handleSessionKillHttpRequest()` in `src/gateway/session-kill-http.ts` previously allowed any bearer token holder to kill any session by calling the helper `canBearerTokenKillSessions()`, which returned `true` for operator-scoped tokens. This commit removes `canBearerTokenKillSessions()` entirely and removes the `allowBearerOperatorKill` code path. Session termination is now gated on one of two conditions: the request originates from a local admin, or the requester's session key matches the target session key. Requests that satisfy neither condition receive a 403 with message "Session kills require a local admin request or requester session ownership." (line 86). This closes the vector where a compromised or exfiltrated operator bearer token could be used to remotely terminate any active session. This commit is the patch for **GHSA-9p93-7j67-5pc2**.

#### Include dashboard children in owner filters ([`cb46b08ef`](https://github.com/openclaw/openclaw/commit/cb46b08ef))

`listSessionsFromStore()` at `src/gateway/session-utils.ts:1259` filters sessions by the `spawnedBy` ownership field when the caller supplies an ownership scope. Before this fix (line 1317), the filter only checked `entry?.spawnedBy === spawnedBy`, which excluded dashboard child sessions that track their parent via `parentSessionKey` rather than `spawnedBy`. Dashboard-spawned child sessions could therefore not be seen by their owner through the filtered list path, creating a visibility gap. The fix changes line 1317 to `entry?.spawnedBy === spawnedBy || entry?.parentSessionKey === spawnedBy`, ensuring both relationship types are honoured when scoping a session list to an owner.

#### Dedupe gateway session resolve visibility ([`a1a9819be`](https://github.com/openclaw/openclaw/commit/a1a9819be))

This refactor extracts three helper functions into `src/gateway/sessions-resolve.ts` that were previously duplicated inline. `resolveSessionVisibilityFilterOptions()` (line 19) centralises the filter-parameter assembly. `noSessionFoundResult()` (line 28) returns a consistent 400-class error. `isResolvedSessionKeyVisible()` (line 35) checks whether a resolved canonical key appears in the ownership-filtered session list; if not, session resolution returns `noSessionFoundResult`. Prior to this change, the visibility check was inlined at multiple call sites with slightly inconsistent parameter construction, risking subtle divergence. Centralising the check ensures any future change to the visibility policy propagates uniformly.

#### Expose spawned session owners in sessions list ([`1062a048e`](https://github.com/openclaw/openclaw/commit/1062a048e))

The `sessions-list-tool` previously omitted `spawnedBy` from the `SessionListRow` output type. Operators and agents using the sessions list to make authorization decisions lacked a field to identify which parent session had spawned a given child. This commit adds `spawnedBy` to `SessionListRow` and surfaces it in the tool output, giving callers the ownership metadata needed to apply correct scope checks. The addition is informational (no change to filter logic), but complements `a1a9819be` and `cb46b08ef` by ensuring that the ownership model is visible in list outputs.

#### Expose parent session keys in sessions list ([`c041fcc04`](https://github.com/openclaw/openclaw/commit/c041fcc04))

Companion to `1062a048e`: adds `parentSessionKey` to the `SessionListRow` type and tool output. Dashboard child sessions relate to their parent via `parentSessionKey` rather than `spawnedBy`; without this field in the list output, callers could not reconstruct parent-child relationships. Together with the `spawnedBy` exposure in the previous commit, this gives consumers of the sessions list a complete picture of session ownership chains, which is the prerequisite for enforcing session-scope authorization decisions at higher layers.

#### Preserve cli sessions across model changes ([`12100719b`](https://github.com/openclaw/openclaw/commit/12100719b))

Introduces `CliSessionBinding` (defined in `src/config/sessions/types.ts:68`) — a type that stores a CLI provider's `sessionId` together with three integrity hashes: `authProfileId`, `extraSystemPromptHash` (SHA-256 of the extra system prompt, computed by `hashCliSessionText()` at `src/agents/cli-session.ts:10`), and `mcpConfigHash`. `resolveCliSessionReuse()` at `src/agents/cli-session.ts:114` compares the stored binding hashes against the current runtime values and returns an `invalidatedReason` of `"auth-profile"`, `"system-prompt"`, or `"mcp"` when a mismatch is detected, causing the session to be discarded rather than reused. Previously, session reuse after a model change or profile switch could persist a stale session that had been created under different auth or MCP credentials, a subtle but exploitable context-confusion channel. The binding store lives in `cliSessionBindings` on `SessionEntry` (line 176 of types.ts).

#### Classify "Failed to extract accountId from token" as auth error for failover ([`eb328a85e`](https://github.com/openclaw/openclaw/commit/eb328a85e))

`classifyFailoverReason()` at `src/agents/pi-embedded-helpers/errors.ts:921` delegates auth-error detection to a pattern list in `src/agents/pi-embedded-helpers/failover-matches.ts`. This commit adds the pattern `/\bfailed to (?:extract|parse|validate|decode)\b.*\btoken\b/` to the `auth` pattern array (line 94), so that Codex-style errors like "Failed to extract accountId from token" are correctly classified as `"auth"` failover reasons. Before the fix, such errors fell through to `null` and the agent did not trigger the auth-error failover branch, causing it to remain stuck on a non-functional provider rather than failing over to an alternate. The fix ensures proper model rotation on token-level auth failures.

#### Skip cli backends in models auth warnings ([`2a0405385`](https://github.com/openclaw/openclaw/commit/2a0405385))

The models list auth-warning filter in `src/commands/models/list.status-command.ts:193` now calls `isCliProvider(provider, cfg)` (defined at `src/agents/model-selection.ts:83`) to exclude CLI-backed providers from the "missing auth" warnings surfaced to operators. CLI providers authenticate through the installed CLI binary, not through API keys, so a missing API-key entry is not an error condition. Before this fix, operators saw spurious auth-warning entries for every CLI-backed model, creating noise that could obscure genuine auth failures on non-CLI providers.

#### Add memory-core extension sources ([`3d0050c30`](https://github.com/openclaw/openclaw/commit/3d0050c30))

Adds `getMemoryCommandSecretTargetIds()` at `extensions/memory-core/src/cli.runtime.ts:51`, which returns the set of config key paths that contain sensitive memory search API keys (`agents.defaults.memorySearch.remote.apiKey` and `agents.list[].memorySearch.remote.apiKey`). The secret-resolution function `resolveCommandSecretRefsViaGateway` is then called at line 59 with these target IDs to resolve stored credentials at runtime via the gateway's secret-reference mechanism. This commit is the infrastructure step for moving secret resolution into the memory-core extension, separating the secret boundary from the CLI core.

#### Move memory tooling into memory-core extension ([`e955d574b`](https://github.com/openclaw/openclaw/commit/e955d574b)) [ESCALATION]

Removes `agents.defaults.memorySearch.remote.apiKey` and `agents.list[].memorySearch.remote.apiKey` from `src/cli/command-secret-targets.ts`. These entries previously gave the CLI core direct access to memory-search API key secret targets. Ownership is transferred to `extensions/memory-core/src/cli.runtime.ts` via `getMemoryCommandSecretTargetIds()` (introduced by `3d0050c30`). The architectural concern requiring escalation: if the memory-core extension is not loaded, no component will claim these secret targets and they will not be resolved at runtime. The change is a necessary step toward proper extension-owned secret management, but requires that the extension loading path is guaranteed before secret resolution occurs.

#### Add browser plugin runtime package ([`197510f69`](https://github.com/openclaw/openclaw/commit/197510f69)) [ESCALATION]

Re-exports security controls from the browser-support plugin SDK (`auth.ts`, `ssrf.ts`, `path-guards.ts`) into a new browser plugin runtime package. Also adds a gateway request handler with allowlist validation and persistent-profile mutation checks. The escalation flag reflects the scope of the security surface reorganisation: consolidating these controls into a re-export layer means that the security properties of the browser runtime now depend on the correct re-export wiring being maintained. Verifying that `ssrf.ts` and `path-guards.ts` are still applied at all call sites that previously used direct imports is the key audit follow-up.
