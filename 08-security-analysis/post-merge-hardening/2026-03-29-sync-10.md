#### Allow plugin channels in hooks mappings ([`89881379d`](https://github.com/openclaw/openclaw/commit/89881379d))

The `channel` field in `HookMappingConfig` (defined in `src/config/types.hooks.ts:29`) changed from a narrow `"last" | string` union to `"last" | (string & {})`. This open-ended type allows runtime plugin channel IDs to be used as hook channel references. Runtime validation against registered/configured channels still happens in the gateway hooks runtime. The doc comment at line 26-27 explicitly notes that validation against plugin channels occurs at runtime, not at schema level. This aligns config schema capability with plugin ecosystem expectations without weakening runtime enforcement.

#### Memory: add configurable FTS5 tokenizer for CJK text support (openclaw#56707) ([`3ce48aff6`](https://github.com/openclaw/openclaw/commit/3ce48aff6))

Adds a `fts.tokenizer` config field to the memory-core extension enabling selection between `unicode61` (default) and `trigram` tokenizers for FTS5 full-text search (type defined in `extensions/memory-core/src/memory/manager-search.ts`). The `ftsTokenizer` property is tracked in sync-ops metadata (`extensions/memory-core/src/memory/manager-sync-ops.ts`) to detect tokenizer changes and trigger reindexing when needed. CJK language users benefit from improved search precision using the trigram tokenizer, which handles non-space-delimited text. No security control changes; included as security-adjacent due to memory schema migration and SQL tokenizer configuration.

#### Eliminate timing side-channel in HMAC signature validation ([`7626d18c6`](https://github.com/openclaw/openclaw/commit/7626d18c6))

`validateLineSignature()` in `extensions/line/src/signature.ts` previously returned `false` early when buffer lengths differed, bypassing `crypto.timingSafeEqual()` and leaking timing information about signature length. The fix pads both buffers to `maxLen = Math.max(hashBuffer.length, signatureBuffer.length)` before comparison, then calls `timingSafeEqual` unconditionally (storing the result before the `&&` length check to avoid JavaScript's short-circuit evaluation). This ensures constant-time execution regardless of signature length mismatch, eliminating the timing side-channel.

#### Land LINE timing-safe signature validation (#55663) (thanks @gavyngong) ([`fcc9fd162`](https://github.com/openclaw/openclaw/commit/fcc9fd162))

Adds `extensions/line/src/signature.test.ts` with test coverage for the timing-safe HMAC fix from `7626d18c6`. Specifically verifies that `crypto.timingSafeEqual` is called even when signature length mismatches, confirming the short-circuit avoidance behavior is preserved.

#### Accept strict Bot Framework and Entra service tokens ([`dc382b09b`](https://github.com/openclaw/openclaw/commit/dc382b09b))

Hardens MS Teams inbound token validation in `extensions/msteams/src/sdk.ts`. `createBotFrameworkJwtValidator()` (line ~416) now validates both Bot Framework and Entra service token issuers strictly: `botFrameworkValidator` and `entraValidator` (both `JwtValidator` instances) run in sequence — only tokens matching known Bot Framework or Entra issuer URLs are accepted. Previously non-conforming or spoofed issuer tokens could pass validation. Policy-based inbound drops are now logged at info level, improving observability.

#### Encrypt thumbnails in E2EE rooms using thumbnail_file ([`81432d6b7`](https://github.com/openclaw/openclaw/commit/81432d6b7))

In `extensions/matrix/src/matrix/send/media.ts`, thumbnail metadata for large images in E2EE rooms now uses `thumbnail_file` (encrypted content with `EncryptedFile` object containing `url` + crypto keys) rather than `thumbnail_url` (plaintext MXC URL). Line 143 sets `imageInfo.thumbnail_file = result.file` for encrypted rooms. This closes a gap where the main media was encrypted but the thumbnail could be leaked in plaintext via `thumbnail_url` in E2EE Matrix rooms.

#### Preserve Teams Entra JWT fallback on legacy validator errors ([`3d69ad830`](https://github.com/openclaw/openclaw/commit/3d69ad830))

Fixes JWT validation fallback logic in `extensions/msteams/src/sdk.ts` to preserve the Entra JWT path when legacy validator errors occur. Previously, error handling in the validator chain could short-circuit the fallback sequence, rejecting valid Entra tokens. The fix ensures the `entraValidator` path is always attempted when `botFrameworkValidator` throws on legacy token formats.
