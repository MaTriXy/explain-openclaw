#### Allow required Google Fonts origins in Control UI CSP ([`d123ade0c`](https://github.com/openclaw/openclaw/commit/d123ade0c))

The Control UI Content Security Policy in `src/gateway/control-ui-csp.ts` was updated to allow Google Fonts origins required by the UI. The `style-src` directive now includes `https://fonts.googleapis.com` and `font-src` now includes `https://fonts.gstatic.com`. Previously these origins were blocked by the CSP, which would prevent font stylesheets and font files from loading. This is a deliberate security boundary relaxation to support legitimate UI functionality — the CSP continues to restrict other content types. No new attack vectors are introduced since Google Fonts delivers static assets over HTTPS from a well-known CDN.

#### Fix group policy enforcement gaps ([`b0a8909a7`](https://github.com/openclaw/openclaw/commit/b0a8909a7))

Fixes a significant policy enforcement gap in the Feishu bot: the `groupConfig.enabled` flag was being parsed from configuration but was never actually checked before processing messages. As a result, groups configured with `enabled: false` would still receive bot responses, bypassing the intended disable mechanism. This commit adds an early-exit guard in `extensions/feishu/src/bot.ts` — when `groupConfig?.enabled === false`, the handler logs the refusal and returns immediately before any policy evaluation or message processing. A secondary fix corrects a misleading log message that was showing the sender's `open_id` instead of the group ID and `groupPolicy` value, making audit logs more accurate.

#### Honor wildcard group config for reply policy ([`60bf56517`](https://github.com/openclaw/openclaw/commit/60bf56517))

The `resolveFeishuGroupConfig` function in `extensions/feishu/src/policy.ts` now correctly applies a wildcard group configuration (`groups["*"]`) as a fallback when no explicit group entry matches by exact or case-insensitive key lookup. Previously, if no exact group key was found, the function returned `undefined`, meaning the wildcard catch-all configuration was silently ignored. After this fix, `wildcard = groups["*"]` is captured at the start of the function and returned as the final fallback — explicit matches remain higher priority. This ensures administrators can define a default policy for all unspecified groups via the `"*"` key, which was the intended behavior based on the config schema but was not being honored.

#### Scope notification wakeups to session ([`f1bb26642`](https://github.com/openclaw/openclaw/commit/f1bb26642))

Previously, `requestHeartbeatNow` calls in the `notifications.changed` event handler in `src/gateway/server-node-events.ts` were triggering heartbeat wakes without passing a `sessionKey`, meaning the wake was not scoped to the specific session that received the notification. This commit passes the resolved `sessionKey` to `requestHeartbeatNow` so that heartbeat wakes are triggered only for the relevant session. Additionally, `DeviceNotificationListenerService` in the Android app (`apps/android/app/src/main/java/ai/openclaw/android/node/DeviceNotificationListenerService.kt`) now adds self-notification filtering: both `onNotificationPosted` and `onNotificationRemoved` skip emitting when the notification originates from the OpenClaw app itself (checked via `entry.packageName == packageName`), preventing the gateway from reacting to its own notification events.

#### Canonicalize notification wake session ([`a8bcad3db`](https://github.com/openclaw/openclaw/commit/a8bcad3db))

In the `notifications.changed` case of `src/gateway/server-node-events.ts`, the raw session key string received from the event payload is now passed through `loadSessionEntry()` to obtain the canonical key before it is used as a routing identifier. Previously, non-canonical session key variants (e.g., case variations or alternate formats) could be used directly as event routing identifiers, causing inconsistencies in session tracking and heartbeat delivery. After canonicalization via `loadSessionEntry(sessionKeyRaw)`, only the resolved `canonicalKey` is passed downstream — to the system event emission and to `requestHeartbeatNow` — ensuring session identity consistency throughout the notification wakeup path.

#### Enable Android notify + notification events ([`9d3ccf475`](https://github.com/openclaw/openclaw/commit/9d3ccf475))

This commit enables two new capabilities on Android: (1) adds `NODE_SYSTEM_NOTIFY_COMMAND` to the Android platform defaults in `src/gateway/node-command-policy.ts`, enabling the notify command for Android nodes; (2) implements a `notifications.changed` event handler in `src/gateway/server-node-events.ts` that processes Android notification events forwarded from the Android-side DeviceNotificationListenerService. The handler includes input validation via `compactNotificationEventText()`, which enforces a `MAX_NOTIFICATION_EVENT_TEXT_CHARS = 120` character limit with whitespace normalization, preventing excessively large or crafted notification payloads from being injected as system events. Both the `title` and `text` fields are individually truncated before being emitted. This is new attack surface — Android notifications can now trigger gateway events — but the 120-character hard cap and whitespace normalization limit the scope of any payload that could be injected.
