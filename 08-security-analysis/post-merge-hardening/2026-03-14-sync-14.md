#### Share pinned sandbox entry finalization ([`0db62fc6c`](https://github.com/openclaw/openclaw/commit/0db62fc6c5))

A new `finalizePinnedEntry` private method in `SandboxFsPathGuard` consolidates path-escape validation that was previously duplicated at two call sites. Both call sites checked for `..` traversal (`startsWith("..")`) and absolute path injection (`isAbsolute()`). Extracting to a shared helper eliminates the risk of future divergence where one call site might have a check removed or weakened independently.

This is a code quality fix with a meaningful security property: the two call sites that mutate pinned sandbox entries are now guaranteed to run the same invariant checks. No behavioral change — the same checks that existed before still run.

#### Restore control-ui auth bypass and classify connect failures ([`e5fe818a7`](https://github.com/openclaw/openclaw/commit/e5fe818a74))

`shouldSkipControlUiPairing` in `connect-policy.ts` previously received `sharedAuthOk: boolean` as its second parameter and used it as a bypass gate. This was incorrect — the bypass should only apply to operator-role connections with `dangerouslyDisableDeviceAuth` configured, not to connections that merely have shared auth. The fix changes the second parameter to `role: GatewayRole` and evaluates `role === "operator" && policy.allowBypass` instead.

`evaluateMissingDeviceIdentity` also gains an operator-role check on the `dangerouslyDisableDeviceAuth` path, ensuring the device-identity skip is only available to operator-role clients. The commit additionally classifies connect failures into distinct rejection kinds (`reject-control-ui-insecure-auth`, `reject-unauthorized`, `reject-device-required`) to improve observability and enable precise error responses.

#### macOS: respect exec-approvals.json settings in gateway prompter ([`25f458a90`](https://github.com/openclaw/openclaw/commit/25f458a907))

The macOS exec-approvals gateway prompter previously hardcoded `allow-once` when no UI prompt could be shown (e.g., the app is backgrounded or the gate cannot present). This commit introduces a `PresentationDecision` struct in `ExecApprovalsGatewayPrompter` with fields `shouldAsk`, `canPresent`, `security`, `askFallback`, and `allowlist`. When a prompt cannot be presented, the code now reads the `security` field from the exec-approvals policy — specifically `ExecSecurity.full` maps to `allowOnce`, otherwise the decision is `deny`.

This closes a silent policy-bypass: previously, exec requests that arrived when the UI was not presentable would be silently approved once. After this fix, the configured `ExecSecurity` policy governs the fallback decision, making the prompter consistent with operator-configured behavior.

#### Harden Discord guild allowlist resolution ([`10afde99c`](https://github.com/openclaw/openclaw/commit/10afde99c1))

`resolveDiscordGuildEntry` in `src/discord/monitor/allow-list.ts` previously required a non-null `guild` object (a Discord.js `Guild` instance) to perform allowlist lookup. In webhook and certain interaction contexts, `guild` can be null while the raw guild_id is still available in the Discord API event payload. The fix adds an explicit `guildId?: string | null` parameter to the function. The resolution order is now: use `params.guildId?.trim()` first, falling back to `guild?.id`. ID-based lookup proceeds even if `guild` is null; slug and wildcard lookups still require the `guild` object.

Before this fix, a Discord interaction arriving in a guild context where the bot had not cached the `Guild` object could fall through allowlist checks, potentially allowing commands that should be blocked. The fix ensures the guildId from the raw payload is sufficient for ID-based allowlist matching.
