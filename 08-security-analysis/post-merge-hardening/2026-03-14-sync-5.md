#### Delivery Deduplication Anti-Replay ([`f07033ed3`](https://github.com/openclaw/openclaw/commit/f07033ed3))

Cron-triggered outbound deliveries can now be deduplicated across invocations. `COMPLETED_DIRECT_CRON_DELIVERIES` is an in-memory `Map` keyed by delivery ID that retains completed results for 24 hours (60 seconds in test mode, capped at 2000 entries). `pruneCompletedDirectCronDeliveries()` evicts entries older than the TTL window. `cloneDeliveryResults()` deep-copies result objects before storing them to prevent mutation aliasing.

When a cron delivery fires and the ID already exists in the map, the prior result is returned without re-sending — preventing duplicate message delivery in cases of network retry, worker restart, or overlapping cron schedules. This closes a replay window in the direct cron dispatch path (`src/cron/isolated-agent/delivery-dispatch.ts`).

#### Telegram Media Transport Policy Threaded into SSRF ([`8023f4c70`](https://github.com/openclaw/openclaw/commit/8023f4c70))

Before this fix, Telegram media fetches always used a direct `Agent`, ignoring any configured transport policy. The proxy/transport configuration was never passed down to `createPinnedDispatcher()`, meaning Telegram media downloads could bypass proxy-scoped SSRF controls entirely.

The fix introduces `PinnedDispatcherPolicy` — a discriminated union type with three modes: `"direct"`, `"env-proxy"` (uses `EnvHttpProxyAgent`), and `"explicit-proxy"` (uses `ProxyAgent`). `createPinnedDispatcher()` now accepts an optional `policy` parameter and routes to the appropriate undici dispatcher. A new helper `withPinnedLookup()` merges the pinned DNS lookup into connect options regardless of proxy mode, ensuring SSRF-safe resolution is preserved in all transport paths.

The Telegram media fetch now threads its transport policy through to the SSRF dispatcher, aligning with Audit 2 Claim 4 (DNS rebinding SSRF via web-fetch).

#### Zalouser Mutable Group Allowlist Auditing ([`f36d8c09f`](https://github.com/openclaw/openclaw/commit/f36d8c09f))

Adds `isZalouserMutableGroupEntry()` to `src/security/mutable-allowlist-detectors.ts` — a new detector that flags Zalouser group allowlist entries that are based on mutable names rather than stable identifiers. A group entry is flagged as mutable if it is not a purely numeric ID (fails `/^\d+$/`) and not a stable prefixed ID (fails `/^g-\S+$/i`). Wildcards (`*`) and empty entries are excluded from flagging.

The new detector is integrated into `src/security/audit-channel.ts` and `src/commands/doctor-config-flow.ts` to warn operators during audits and doctor checks when their Zalouser group allowlist contains name-based entries that could match different groups over time. This directly addresses the vulnerability class described in GHSA-f5mf-3r52-r83w.

#### Windows Gateway Launch Timeout Hardening ([`32d8ec948`](https://github.com/openclaw/openclaw/commit/32d8ec948))

`execSchtasks()` in `src/daemon/schtasks-exec.ts` replaces the unbounded execFileUtf8 call with `runCommandWithTimeout` configured with a 15-second main timeout and a 5-second no-output timeout. If either timeout fires, the error code is normalized to 124 and a descriptive message is placed in stderr.

`shouldFallbackToStartupEntry()` expands the fallback trigger conditions: previously only "access is denied" triggered the startup-entry fallback; now it also fires on code 124, "schtasks timed out", and "schtasks produced no output". `launchFallbackTaskScript()` replaces the old synchronous `launchStartupEntry` with a detached `spawn(detached: true, stdio: "ignore")` so the launcher process is completely decoupled from the gateway process.

This hardens the Windows gateway startup path against unresponsive or permission-blocked Task Scheduler installations, preventing the gateway from hanging indefinitely during onboarding or restart.
