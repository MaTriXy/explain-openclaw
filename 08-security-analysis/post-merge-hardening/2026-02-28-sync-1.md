#### Unify context window handling across discovery, merge, and OpenAI-compat transport ([`f16ecd1da`](https://github.com/openclaw/openclaw/commit/f16ecd1da))

The Ollama OpenAI-compatible transport now enforces concurrency and context-window integrity through a defense-in-depth chain. `isOllamaCompatProvider()` classifies endpoints by combining URL hostname pattern matching, port 11434 detection, and path validation (`/` or `/v1`), preventing misidentification of non-Ollama OpenAI-compat endpoints. `queryOllamaContextWindow()` fetches per-model context windows from the Ollama `/api/show` API with a 3-second timeout, enforcing `OLLAMA_SHOW_CONCURRENCY=8` concurrent discovery requests and capping model enumeration at `OLLAMA_SHOW_MAX_MODELS=200` to prevent resource exhaustion via large model registries. The new `injectNumCtxForOpenAICompat` per-provider config flag gates whether the `num_ctx` parameter is injected into OpenAI-compat requests, defaulting to `true` for detected Ollama endpoints and allowing operators to opt out. Fetch errors in `queryOllamaContextWindow()` fall back gracefully (no injection), ensuring that a misbehaving or attacker-controlled Ollama endpoint cannot stall the agent. This closes Audit 2 Claim 4 by unifying SSRF-relevant hostname validation and timeout enforcement across the discovery, merge, and transport code paths.

#### Remove post-compaction audit injection message ([`70a4f25ab`](https://github.com/openclaw/openclaw/commit/70a4f25ab))

Removes `post-compaction-audit.ts`, which implemented a Layer 3 post-compaction audit injection that constituted a persistent prompt injection vector. The removed module called `enqueueSystemEvent` to insert synthetic user-role messages into the agent conversation immediately after context compaction, instructing agents to re-read `WORKFLOW_AUTO.md` — a file that does not exist in standard workspaces and could be created by an attacker. Additionally, the module leaked raw regex patterns in user-facing warning messages, allowing workspace files matching those patterns to be read by agents without user awareness. Layers 1 and 2 (compaction summary preservation and AGENTS.md workspace context refresh) remain intact. This removal closes five related issues (#27697, #26851, #20484, #22339, #25600) and eliminates a self-injection path that bypassed the agent's normal tool-call flow.

#### Honor tools.fs.workspaceOnly=false for host write/edit ([`172583972`](https://github.com/openclaw/openclaw/commit/172583972))

The `workspaceOnly` filesystem boundary flag is now consistently propagated to the host write and edit tool constructors. Previously, `createHostWorkspaceWriteTool()` and `createHostWorkspaceEditTool()` received only the workspace root path, causing the internal workspace boundary check inside each tool to be ignored when `tools.fs.workspaceOnly=false` was set by an operator. The fix passes `{ workspaceOnly }` as an options argument to both constructors so that operator-configured `false` values are honored at the inner enforcement layer, not just the outer wrapping guard. A 199-line integration test covers the `workspaceOnly=false` path access behavior. This closes Audit 1 Claim 6 by ensuring the operator's explicit filesystem boundary configuration is respected end-to-end rather than only at the outermost layer.

#### Normalize whitespace-padded tool call names before dispatch ([`6b317b1f1`](https://github.com/openclaw/openclaw/commit/6b317b1f1))

Adds `trimWhitespaceFromToolCallNamesInMessage()` to normalize whitespace-padded tool call names before dispatch, preventing models from bypassing tool policy enforcement by emitting names such as `" read "` that fail exact-match tool lookup. Normalization applies to both live streamed embedded-runner tool calls and transcript history replay, ensuring consistent tool availability checks regardless of whether an agent is executing a new request or replaying a prior session. Tool names are also normalized in the session transcript repair path (`session-transcript-repair.ts`) for completeness. This closes Audit 2 Claim 7 by hardening the dispatch layer against whitespace-based name manipulation that could allow disallowed tools to be invoked.

#### Bound unauthenticated webhook rate-limit state (openclaw#26050) thanks @bmendonca3 ([`f943c76cd`](https://github.com/openclaw/openclaw/commit/f943c76cd))

Bounds the in-memory Feishu webhook rate-limit state map using `FEISHU_WEBHOOK_RATE_LIMIT_MAX_TRACKED_KEYS=4096`. `trimWebhookRateLimitState()` prunes entries whose rate-limit window has elapsed; `maybePruneWebhookRateLimitState()` enforces the hard cap by evicting the oldest-keyed entries when the map exceeds the configured limit. Without this bound, a high-volume stream of unauthenticated Feishu webhook requests using rotating source keys could grow the state map without limit, leading to out-of-memory conditions on the gateway process. The cap and eviction logic run on each inbound webhook before rate-limit state is updated, bounding memory use regardless of attacker throughput.
