#### Telegram transport fallback validation hardening ([`e4825a0f9`](https://github.com/openclaw/openclaw/commit/e4825a0f93))

Refactors the Telegram API transport fallback chain to enforce stricter SSRF dispatcher policy validation. Introduces the `TelegramTransportAttempt` type to structure fallback attempts, adds validation in `createPinnedLookup()` that requires at least one pinned address (throws if the list is empty), and adds the `PinnedHostnameOverride` type to support dispatcher override hostnames. IPv4 fallback escalation now correctly routes to pinned Telegram IP dispatcher attempts (e.g., `149.154.167.220`). Tests validate fallback escalation from `EHOSTUNREACH`/`ETIMEDOUT` to pinned IPs and that `createPinnedDispatcher` accepts override hostnames. **Audit impact:** Defense-in-depth for DNS rebinding SSRF mitigation (Audit 2 Claim 4); strengthens internal pinning invariants without directly closing any audit gap.

#### Trusted-proxy Control UI scope clearing ([`ccf16cd88`](https://github.com/openclaw/openclaw/commit/ccf16cd889))

Fixes a gateway authorization boundary issue where device-less trusted-proxy Control UI sessions could retain self-declared operator scopes (admin, secrets access). The change modifies the condition guarding `clearUnboundScopes()` in the connection message handler: previously, when `isControlUi` was true and the connection decision was `"allow"`, scope clearing was skipped — allowing trusted-proxy-authenticated sessions without a bound device identity to retain any scopes they declared. The fix adds `|| trustedProxyAuthOk` to the bypass condition, so sessions validated via `isTrustedProxyControlUiOperatorAuth()` are also subject to scope clearing. **Audit impact:** No direct audit claim, but hardens the gateway trust model for trusted-proxy Control UI auth paths; prevents privilege escalation via self-declared admin/secrets scopes on device-less sessions.

#### ACP startup hardening and PATH isolation ([`ea15819ec`](https://github.com/openclaw/openclaw/commit/ea15819ecf))

Hardens ACP daemon startup and service PATH configuration. The core security change is in `src/daemon/service-env.ts`: `buildMinimalServicePath()` and `getMinimalServicePathPartsFromEnv()` now prepend the daemon Node bin directory to the service PATH, preventing PATH hijack vectors where a compromised PATH entry before the Node bin could intercept daemon process execution. The commit also introduces error-state session recovery for ACP sessions, moves ACP configured routing behind plugin seams via a new binding registry/consumer architecture, and refactors startup initialization. The plugin seam boundary reduces the gateway startup attack surface by ensuring configured routing decisions go through defined plugin interfaces rather than direct module calls. **Audit impact:** Defense-in-depth — daemon PATH hardening relates to Audit 2 Claim 8 (env variable injection) defenses; plugin seam isolation reduces ACP startup exposure.
