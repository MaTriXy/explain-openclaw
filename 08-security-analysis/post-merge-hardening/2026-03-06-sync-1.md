#### Gateway: add SecretRef support for gateway.auth.token with auth-mode guardrails ([`72cf9253f`](https://github.com/openclaw/openclaw/commit/72cf9253f))

This commit introduces structured SecretRef support for `gateway.auth.token`, enabling operators to source gateway authentication tokens from environment variables, files, or executables rather than storing them as plaintext in `openclaw.json`. A SecretRef is a typed reference object resolved at runtime via the secrets subsystem, preventing token exposure in config files, backups, or log output.

Three new auth-mode guardrails accompany the feature. `auth-mode-policy.ts` enforces that `gateway.auth.mode` must be explicitly configured when both `gateway.auth.token` and `gateway.auth.password` are set (including via SecretRef), preventing ambiguous authentication mode fallback that could allow an attacker to coerce the gateway into using a weaker auth path. `auth-install-policy.ts` ensures that daemon service installation only infers token mode when explicitly configured or when no password option exists, blocking token generation bypass during automated setup. The `GatewaySecretRefUnavailableError` class (defined in `src/gateway/credentials.ts`) and the `isGatewaySecretRefUnavailableError()` function implement fail-closed semantics: if `gateway.auth.token` is configured as a SecretRef that cannot be resolved, the gateway refuses to start rather than silently operating without authentication.

All token resolution paths—CLI commands, daemon lifecycle, QR dashboard, extension relay authentication, and onboarding workflows—are updated to use `resolveSecretInputRef()` and `resolveSecretRefValues()`, providing a unified, testable resolution pipeline across all gateway entry points.
