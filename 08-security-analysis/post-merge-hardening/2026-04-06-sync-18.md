#### `58d2b9dd46` — Runtime model contextTokens caps

`contextTokens?: number` field is added to per-model config entries (`src/config/types.models.ts:68`) and the zod schema (`src/config/zod-schema.core.ts:310`). `resolveContextWindowInfo()` (`src/agents/context-window-guard.ts:22`) is updated at line 40 to prefer `contextTokens` over the native `contextWindow` reported by the provider when both are present. This gives operators an explicit runtime cap on effective context window size per model, independent of what the provider exposes. A global cap path is also available at `agents.defaults.contextTokens` (read at line 51). The openai-codex provider applies a 272K default cap over the native 1.05M window. This closes a defense-in-depth gap: before this change, agents could consume up to the provider's full reported capacity with no operator override, which could lead to unintended resource usage or context bloat. No existing audit claims addressed; new operator control.

#### `300fb36879` — Atomic JSON writes with secure file permissions

`saveJsonFile()` (`src/infra/json-file.ts:108`) is rewritten to use an atomic write-then-rename pattern: the payload is written to a temp path (`${targetPath}.${randomUUID()}.tmp`) via `writeTempJsonFile()` (`:89`) which opens the file with `JSON_FILE_MODE` (`0o600`, line 5), calls `fsyncSync` before closing, then `renameJsonFileWithFallback()` atomically promotes the temp file to the final path. `trySetSecureMode()` (`:8`) applies `0o600` permissions on both the temp path and the final path as a belt-and-suspenders measure. Parent directories are created with `JSON_DIR_MODE` (`0o700`, line 6). This hardening applies to every JSON config/session/data file written by the application. Previously, `writeFileSync` was used directly with no explicit permissions, leaving files potentially world-readable on systems with permissive umask defaults. The atomic rename also prevents partial-write corruption if the process is interrupted mid-write. Addresses the spirit of Audit 1 Claim 5 (Insufficient file permission checks) in a new context; new hardening.
