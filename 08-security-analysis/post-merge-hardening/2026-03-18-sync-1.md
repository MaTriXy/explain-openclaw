#### Forward plugin subagent overrides ([`1399ca5fc`](https://github.com/openclaw/openclaw/commit/1399ca5fcb))

Plugin subagent model override fields are now validated before propagation. The new helper `containsControlCharacters()` scans override values for ASCII control characters (bytes 0x00–0x1F and 0x7F), blocking injection attempts that embed control sequences in model name strings. A companion constant `OVERRIDE_VALUE_MAX_LENGTH` (256 characters) caps override value length, preventing buffer-boundary edge cases.

Model override propagation is now gated by the `allowModelOverride` flag on the plugin configuration. Only plugins that explicitly opt in receive override forwarding, preventing an untrusted plugin from silently re-routing inference to an attacker-controlled model endpoint.

These changes close two injection vectors in Audit 2: control character injection via override fields (Claim 7) and unauthorized model override propagation (Claim 8).

#### Inject oauth credential fs stubs ([`a413da9cc`](https://github.com/openclaw/openclaw/commit/a413da9cca))

Google OAuth credential handling now exposes a `setOAuthCredentialsFsForTest()` seam backed by a `CredentialFs` interface. Previously the credential read/write path called the real filesystem directly, making it impossible to exercise permission checks and access control logic in unit tests.

This commit does not change production credential storage — credentials remain in plaintext on the filesystem (a known gap from Audit 1 Claim 1). However, it enables security test coverage of the file permission verification path, allowing future tests to assert that `0o600` enforcement is exercised before plaintext writes.

#### Fix ssh sandbox key cp ([`59cd98068`](https://github.com/openclaw/openclaw/commit/59cd98068f))

SSH sandbox key copying is hardened in two ways. First, `normalizeInlineSshMaterial()` now strips Windows-style line endings before key material is stored, ensuring that CRLF sequences in a copied key do not produce a structurally malformed entry that would still pass format validation but fail at authentication time. Second, `buildSshFailureMessage()` now provides a structured error response that exposes corruption details when key material is detected as malformed — rather than failing silently.

Both changes operate alongside the existing `0o600` permission enforcement on the stored key file, which prevents unauthorized reads. Together they close Audit 1 Claim 5: the key copy path now normalizes, validates, and reports errors rather than storing potentially broken material at the correct permissions and silently failing later.

#### Block JVM, Python, and .NET env injection vectors ([`f84a41dcb`](https://github.com/openclaw/openclaw/commit/f84a41dcb8))

The host execution environment blocklist in `src/infra/host-env-security-policy.json` gains five new entries: `JAVA_TOOL_OPTIONS`, `_JAVA_OPTIONS`, `JDK_JAVA_OPTIONS`, `PYTHONBREAKPOINT`, and `DOTNET_STARTUP_HOOKS`. Each is a runtime-level hook variable that the respective VM or interpreter unconditionally reads before executing user code.

An attacker with control over the sandbox environment could set `JAVA_TOOL_OPTIONS=-javaagent:/tmp/evil.jar` to instrument any JVM process launched inside the sandbox, or use `PYTHONBREAKPOINT=evil_module.trace` to redirect Python's breakpoint hook to arbitrary code. `DOTNET_STARTUP_HOOKS` allows arbitrary .NET assembly injection at CLR startup. Blocking all five closes a class of "interpreter-layer injection" attacks that the prior blocklist (focused on `LD_PRELOAD`, `NODE_OPTIONS`, etc.) did not cover.

The generated Swift policy file `HostEnvSecurityPolicy.generated.swift` is also updated to keep the macOS app's policy in sync.

This commit strengthens Legitimate Gap 1 (gateway-side env var blocklist) but does not close it — three languages are now blocked, but the gap formally tracks the broader hardening work started in PR #12. The gap remains open pending a comprehensive audit of all interpreter-layer injection vectors.

#### Defer DM cite expansion until after auth ([`ebee4e221`](https://github.com/openclaw/openclaw/commit/ebee4e2210))

Tlon DM cite (message reference) expansion was previously triggered by scry calls before the sender had been authenticated. A cite embedded in an inbound DM could reach the expansion path with the sender's identity unchecked, allowing an unauthenticated party to trigger content resolution from the ship's data store.

The fix introduces `resolveAuthorizedMessageText()` which gates cite resolution behind an `authorizedForCites` flag. Cite expansion now runs only after the sender identity has been verified. A new `security.test.ts` file provides a proof-of-concept test case that validates the authorization ordering — the test directly reproduces the pre-fix behavior and asserts it is prevented after the fix.

This is a new control with no direct counterpart in the prior audits.
