#### Add IP validation, fix upgrade handler for non-loopback bind ([`e883d0b55`](https://github.com/openclaw/openclaw/commit/e883d0b55))

The `relayBindHost` Zod schema field previously accepted any string. This commit tightens it to `z.union([z.string().ipv4(), z.string().ipv6()])`, ensuring only valid IPv4 or IPv6 addresses are accepted at config parse time.

The WebSocket upgrade handler in `ensureChromeExtensionRelayServer()` previously rejected all connections from non-loopback remote addresses unconditionally. When `relayBindHost` is set to a non-loopback address (e.g. `0.0.0.0` for WSL2 bridge networking), this gate incorrectly rejected legitimate clients. The fix changes the condition to `!isLoopbackAddress(remote) && isLoopbackHost(bindHost)` — non-loopback remote connections are only rejected when the relay is bound to a loopback address. The `ChromeExtensionRelayServer` type gains a `bindHost: string` field for downstream inspection.

#### Make browser relay bind address configurable ([`436ae8a07`](https://github.com/openclaw/openclaw/commit/436ae8a07))

Adds `browser.relayBindHost` as a new config field (default: `127.0.0.1`, preserving the existing loopback-only behavior). The field is parsed in `resolveBrowserConfig()` and stored in `ResolvedBrowserConfig`. It flows through `ensureExtensionRelayForProfiles()` in `server-lifecycle.ts`, which passes the value as `bindHost` to `ensureChromeExtensionRelayServer()`. This allows users in WSL2 or other bridged-network environments to bind the relay to `0.0.0.0` or a specific interface address while keeping the secure loopback default for all other deployments.
