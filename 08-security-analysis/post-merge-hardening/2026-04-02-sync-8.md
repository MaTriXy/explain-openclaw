#### Revert: block sensitive external bind sources ([`14a779ee8`](https://github.com/openclaw/openclaw/commit/14a779ee8))

This commit reverts `8db20c196`, which had added dynamic home-subdirectory blocking to the sandbox bind-mount denylist. The revert removes three categories of protection from `src/agents/sandbox/validate-sandbox-security.ts`:

1. **Dynamic home-path blocking** — the reverted commit had introduced `BLOCKED_HOME_SUBPATHS` (`.aws`, `.config`, `.kube`, `.openclaw`, `.ssh`) and a `getBlockedHostPaths()` function that computed the full runtime set (static paths + home subdirs + resolved OpenClaw state directory) at each call. The REVERT replaces all calls to `getBlockedHostPaths()` with the static `BLOCKED_HOST_PATHS` constant (`:18`), which now contains only system-level paths (`/etc`, `/proc`, `/sys`, `/dev`, `/root`, `/run`, and Docker socket variants).

2. **Additional static paths removed** — `/var/lib/docker` and `/var/log` have been removed from `BLOCKED_HOST_PATHS`.

3. **Runtime alias resolution removed** — the `BlockedHostPathAlias` type and its resolution logic (which normalized symlinks and `~` expansions) are gone.

**Security impact:** Sandbox containers can now receive bind mounts pointing to `~/.aws`, `~/.ssh`, `~/.kube`, `~/.openclaw`, `~/.config`, and log/Docker data directories. This expands the sandbox's potential access to host credentials and configuration. The revert was presumably made for compatibility reasons (PR #59016). This is a regression in defense-in-depth. No existing audit claim directly requires home-subdir blocking, but the original fix was introduced as defense-in-depth hardening.

#### Add Telegram error suppression controls ([`74b9f22a4`](https://github.com/openclaw/openclaw/commit/74b9f22a4))

Introduces per-scope Telegram error policy configuration via new `errorPolicy?: "always" | "once" | "silent"` and `errorCooldownMs?: number` fields on `TelegramAccountConfig`, `TelegramTopicConfig`, `TelegramGroupConfig`, and `TelegramDirectConfig` (all in `src/config/types.telegram.ts`). The implementation lives in a new `extensions/telegram/src/error-policy.ts` module, which tracks per-message error cooldown state and prunes expired entries. The `bot-message-dispatch.ts` integration gates outbound error messages through the policy resolver before sending.

**Security impact:** No attack-surface change. The feature prevents noisy error floods (e.g., repeated 429 rate-limit or ECONNRESET replies) that could reveal internal error details to channel participants when `errorPolicy: "always"` is in use. Operators can now gate this to `"once"` per cooldown window or `"silent"`.

#### Make chat history max chars configurable ([`1654c3a85`](https://github.com/openclaw/openclaw/commit/1654c3a85))

Adds `gateway.webchat.chatHistoryMaxChars` (type `GatewayWebchatConfig.chatHistoryMaxChars?: number` at `src/config/types.gateway.ts:395`) to cap per-field text size in `chat.history` WebSocket responses. The config is applied at `src/gateway/server-methods/chat.ts:1227` (`cfg.gateway?.webchat?.chatHistoryMaxChars`). Default is 12,000 characters; the schema enforces a maximum of 500,000 (integer, exclusive lower bound > 0).

**Security impact:** The bounded schema cap prevents operators from accidentally configuring an unbounded in-memory expansion vector via the `chat.history` payload. The default 12,000-char limit matches the previous hardcoded behavior; raising it requires an explicit operator opt-in.

#### Pass dangerouslyForceUnsafeInstall through archive and source ([`fbe3ca4d7`](https://github.com/openclaw/openclaw/commit/fbe3ca4d7))

Fixes two silent propagation gaps in `src/plugins/install.ts`: the `dangerouslyForceUnsafeInstall` flag is now threaded through `installPluginFromArchive` (`:676`) and through the `archiveInstallParams` passed from `installPluginFromNpmSpec` (`:843`). Before this fix, calling an archive or npm-spec install with `dangerouslyForceUnsafeInstall: true` would silently drop the flag once control reached the archive sub-path, meaning safety checks were not actually bypassed as requested — but more importantly, the flag could be inconsistently applied across install paths.

**Security impact:** Addresses **Audit 2 Claim 8** (unsafe install propagation). Operators who intentionally use `dangerouslyForceUnsafeInstall` for trusted internal plugins can now rely on it being consistently honored across archive and source install paths. The security-relevant direction of this fix is that audit gates are now consistently applied (the flag propagates so the caller's intent is honored). No new bypass capability is introduced.

#### Escalate to model fallback after rate-limit profile rotation cap ([`4fa11632b`](https://github.com/openclaw/openclaw/commit/4fa11632b))

Adds `auth.cooldowns.rateLimitedProfileRotations` (config field at `src/config/types.auth.ts:43`) and a new helper `resolveRateLimitProfileRotationLimit()` at `src/agents/pi-embedded-runner/run/helpers.ts:40`, mirroring the existing `resolveOverloadProfileRotationLimit()` at `:36`. When the rotation counter reaches the cap (default: 1), the runner throws a FailoverError (from `src/agents/model-fallback.ts`) to escalate to the next model in the fallback chain instead of cycling through the same profiles indefinitely.

**Security impact:** Prevents a silent DoS scenario where all auth profiles share the same per-model quota (e.g., all profiles are Anthropic Sonnet-only accounts). Without this fix, exhausting all profiles on a rate-limited model caused infinite rotation loops — effectively stalling the gateway. The fix ensures graceful degradation to the next configured model.

#### Use browser image for browser runtime matching ([`1ce410a7b`](https://github.com/openclaw/openclaw/commit/1ce410a7b))

The new `resolveConfiguredDockerRuntimeImage()` function (`:16`) uses the `configLabelKind` discriminant to route browser runtimes to `sandboxCfg.browser.image` (`:24`) rather than defaulting to `sandboxCfg.docker.image`. Previously, browser sandbox containers were launched with the standard Docker image instead of the browser-specific one, resulting in an incorrectly configured runtime environment.

**Security impact:** Defense-in-depth misconfiguration fix. Browser sandbox containers now run with the correct image (which has browser-specific security profiles). Using a generic sandbox image for browser workloads could undermine browser-specific security hardening applied in the browser image.

#### Require explicit allowlist for /bot-logs to prevent info disclosure ([`72af92ba4`](https://github.com/openclaw/openclaw/commit/72af92ba4))

Adds a new `hasExplicitCommandAllowlist()` guard at `extensions/qqbot/src/slash-commands.ts:111` that checks whether `accountConfig.allowFrom` is a non-empty array of non-wildcard entries. The `/bot-logs` command handler (`:537`) now calls this check at `:549` before exporting log files. If the allowlist is absent, empty, or contains a wildcard (`*`), the command returns an authorization error.

**Security impact:** Closes an information disclosure vector. The `/bot-logs` command exports recent gateway log files (up to 4 files, 1000 lines each) as attachments. In a permissive default setup where `allowFrom` is left unconfigured, this command was accessible to any QQ user who could message the bot. The fix ensures log export requires an explicit, non-wildcard allowlist entry — consistent with the access control model used elsewhere in the QQ bot channel.
