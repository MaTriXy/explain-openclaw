#### Add model request transport overrides ([`61f13173c`](https://github.com/openclaw/openclaw/commit/61f13173c))

Introduces a new `models.providers.*.request` configuration field that allows per-provider header and auth overrides for model-inference requests. The new `sanitizeConfiguredModelProviderRequest()` function (`src/agents/provider-request-config.ts:306`) enforces strict transport boundaries: proxy and TLS settings are explicitly rejected with a thrown error, restricting overrides to headers and auth only. A new type file `src/config/types.provider-request.ts` centralizes the config shape. Secret redaction covers the new auth token and custom header paths, and the provider registry is updated with full request-field coverage. This commit has no direct audit claim link — it adds a constrained escape hatch for inference-path customization while preventing proxy/TLS misconfiguration at the boundary.

#### Add dangerous private-network media opt-in ([`4bfa9260c`](https://github.com/openclaw/openclaw/commit/4bfa9260c))

Adds a `network.dangerouslyAllowPrivateNetwork` config field (`src/config/types.telegram.ts:61`) that, when set, passes `allowPrivateNetwork: true` into the SSRF policy constructed by `buildTelegramMediaSsrfPolicy()` (`extensions/telegram/src/bot/delivery.resolve-media.ts:21`). Default behavior is unchanged — private-network targets remain blocked. The flag is explicitly named to signal risk; config metadata (`extensions/telegram/src/config-ui-hints.ts:56`) surfaces a warning for operator-visible contexts. This addresses **Audit 2 Claim 4** (DNS rebinding SSRF via web-fetch): the claim identified that Telegram media downloads had no private-network block. The fix adds the block by default and provides a clearly gated opt-in for trusted fake-IP or transparent-proxy environments. The claim is now mitigated for default deployments; operators who enable the flag accept the residual risk.
