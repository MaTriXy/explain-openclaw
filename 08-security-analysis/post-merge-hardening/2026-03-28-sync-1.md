#### Gateway: require verified scope for chat provenance ([`4b9542716`](https://github.com/openclaw/openclaw/commit/4b9542716))

Replaces the identity-based `isAcpBridgeClient()` check with a scope-based `canInjectSystemProvenance()` gate (`src/gateway/server-methods/chat.ts:302`) for the `systemInputProvenance` and `systemProvenanceReceipt` fields on `chat.send`. Previously, only clients whose `client.id` matched the ACP bridge identity could set these fields; the new check reads `client.connect.scopes` and requires `ADMIN_SCOPE`. The gate at `chat.ts:1275` is updated to use `canInjectSystemProvenance(client)`. The error message is updated from "reserved for the ACP bridge" to "require admin scope". This is a defense-in-depth hardening: scope-based gates survive client identity renames and correctly express the intended privilege boundary (Audit 1 Claim 6).

#### Add dedicated TUI gateway client auth ([`2b96569e2`](https://github.com/openclaw/openclaw/commit/2b96569e2))

Registers a new client identity `GATEWAY_CLIENT_IDS.TUI = "openclaw-tui"` (`src/gateway/protocol/client-info.ts:4`) and introduces `isOperatorUiClient()` (`src/utils/message-channel.ts:45`) that returns `true` for both `CONTROL_UI` and `TUI` clients. Previously, the TUI shared implicit auth assumptions with the Control UI path, with no distinct registration. The new `isOperatorUiClient()` function gives the TUI a named identity, enabling separate auth handling in `src/tui/gateway-chat.ts` and `src/gateway/server/ws-connection/auth-messages.ts`. This commit is paired with `f1de00c107` (below) which moves origin-check enforcement to the browser-only `isBrowserOperatorUiClient()` path.

#### Cap concurrent pre-auth WebSocket upgrades ([`cb5f7e201`](https://github.com/openclaw/openclaw/commit/cb5f7e201))

Introduces per-IP rate limiting for unauthenticated WebSocket upgrade attempts. New module `src/gateway/server/preauth-connection-budget.ts` implements a per-IP counter (`DEFAULT_MAX_PREAUTH_CONNECTIONS_PER_IP = 32` at line 1; configurable via `OPENCLAW_MAX_PREAUTH_CONNECTIONS_PER_IP`). `createPreauthConnectionBudget()` (line 23) returns an object with `acquire(clientIp)` and `release(clientIp)` methods; the gateway calls `acquire` before the WebSocket handler attaches and `release` on auth completion or failure. Connections whose IP cannot be resolved use a shared fallback bucket (`UNKNOWN_CLIENT_IP_BUDGET_KEY`) rather than failing open. This mitigates resource-exhaustion attacks that flood the gateway with unauthenticated upgrade requests before any auth check can fire.

#### Agents: block protected gateway config writes ([`76411b2af`](https://github.com/openclaw/openclaw/commit/76411b2af))

Adds a denylist of sensitive config paths that agents are not permitted to mutate via the gateway tool, even with owner credentials. `PROTECTED_GATEWAY_CONFIG_PATHS = ["tools.exec.ask", "tools.exec.security"]` is declared at `src/agents/tools/gateway-tool.ts:22`. `assertGatewayConfigMutationAllowed()` (line 77) parses the proposed config patch, applies legacy migrations, and checks whether any proposed change touches a protected path; it throws if so. The guard is enforced at `gateway-tool.ts:260` (config.apply action) and `gateway-tool.ts:277` (config.patch action). This closes a gap where a malicious or prompt-injected agent with owner-scope credentials could silently lower `tools.exec.security` from `"allowlist"` to `"full"` or set `tools.exec.ask` to `"never"`, removing human-in-the-loop approval for shell command execution.

#### Require read scope for models http ([`2d80dbfeb`](https://github.com/openclaw/openclaw/commit/2d80dbfeb))

Enforces operator scope authorization on the `/v1/models` HTTP endpoint (`src/gateway/models-http.ts:93`). Previously, `handleOpenAiModelsHttpRequest()` checked gateway-level bearer auth (`authorizeGatewayBearerRequestOrReply()`) but did not verify operator scopes; any bearer-authenticated client could enumerate models regardless of scope. The fix adds `authorizeOperatorScopesForMethod("models.list", requestedScopes)` — returning HTTP 403 with `{ ok: false, error: { type: "forbidden" } }` when the requesting scope lacks the required permission. This tightens the principle of least privilege for the OpenAI-compatible models list surface.

#### Route ollama helpers through plugin sdk ([`932248107`](https://github.com/openclaw/openclaw/commit/932248107))

Creates `src/plugin-sdk/ollama.ts` as a stable public re-export seam for Ollama provider helpers (`isOllamaCompatProvider`, `resolveOllamaCompatNumCtxEnabled`, `shouldInjectOllamaCompatNumCtx`, `wrapOllamaCompatNumCtx`, and others). Updates `src/agents/pi-embedded-runner/run/attempt.ts` to import from `plugin-sdk/ollama.ts` instead of directly from `extensions/ollama/api.js`. This enforces the CLAUDE.md import boundary rule: core code (`src/agents/`) may not reach directly into extension packages (`extensions/ollama/`). The plugin SDK seam ensures any future changes to Ollama internals pass through a versioned contract surface rather than bypassing it.

#### Move provider runtime into extensions ([`64bf80d4d`](https://github.com/openclaw/openclaw/commit/64bf80d4d))

Large structural refactor relocating the Ollama provider runtime from core into the `extensions/ollama/` package. New extension-side files include `extensions/ollama/src/stream.ts` (streaming and message conversion, ~755 lines), `extensions/ollama/src/setup.ts` (model configuration, ~488 lines), `extensions/ollama/src/embedding-provider.ts` (~207 lines), `extensions/ollama/src/provider-models.ts` (~127 lines), and `extensions/ollama/src/defaults.ts`. The memory-host SDK's Ollama embedding helper is slimmed by ~128 lines as it delegates to the extension. Moving provider runtime code into the extension package reduces core's dependency on Ollama internals and brings the Ollama integration into alignment with the plugin SDK boundary model, limiting blast radius if the Ollama extension is compromised or misbehaves.

#### Adapt compaction request auth ([`39fae14c7`](https://github.com/openclaw/openclaw/commit/39fae14c7))

Removes a local duck-typing shim (resolveModelAuth, ~34 lines) from `src/agents/pi-extensions/compaction-safeguard.ts`, which previously adapted between three incompatible `modelRegistry` API shapes (`getApiKeyAndHeaders`, `getApiKey`, `getApiKeyForProvider`). The extension now calls `ctx.modelRegistry.getApiKeyAndHeaders(model)` directly (line 621). If auth resolution fails, the safeguard logs a warning and cancels compaction with a user-visible reason string (`setCompactionSafeguardCancelReason()`), rather than silently proceeding with empty credentials. This eliminates a duck-typing surface that could have masked API incompatibilities in the model registry and ensures compaction fails loudly when auth is unavailable instead of continuing with an empty API key.

#### Keep TUI out of browser origin checks ([`f1de00c10`](https://github.com/openclaw/openclaw/commit/f1de00c10))

Corrects the origin-check gate in `src/gateway/server/ws-connection/message-handler.ts:409` to use `isBrowserOperatorUiClient()` (`src/utils/message-channel.ts:50`) instead of `isOperatorUiClient()` (`src/utils/message-channel.ts:45`). `isBrowserOperatorUiClient()` returns `true` only for `CONTROL_UI`; `isOperatorUiClient()` returns `true` for both `CONTROL_UI` and `TUI`. Before this fix, TUI connections — which do not send an HTTP `Origin` header — were incorrectly subject to the browser origin enforcement path (`enforceOriginCheckForAnyClient || isBrowserOperatorUi || isWebchat` at `message-handler.ts:411`), causing spurious auth failures for TUI clients. The fix ensures origin-header enforcement fires only for actual browser-based WebSocket connections. This commit depends on `2b96569e2` (above) which registered the dedicated TUI client identity.
