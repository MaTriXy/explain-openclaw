#### Support custom session IDs and auto-bind to current session ([`e7d9648fb`](https://github.com/openclaw/openclaw/commit/e7d9648fb))

Cron job session targeting was extended to support two new `sessionTarget` values (defined in `src/cron/types-shared.ts`): `"current"` auto-binds the job to whichever session created it, while `"session:<id>"` binds to a named persistent session. Previously only main and ephemeral isolated sessions were available. The implementation validates custom session IDs against existing sessions and falls back to isolated mode in headless/CLI environments where no session context is available. Bound targets survive gateway reloads and restarts.

From an authorization standpoint, a cron job bound to a specific session inherits that session's token scope and approval state. A job that binds to a session with broad pre-granted approvals can execute with wider permissions than an `isolated` job would. Operators should gate the `sessionTarget` field accordingly.

#### Relax local backend self-pairing and harden launchd restarts ([`8db6fcca7`](https://github.com/openclaw/openclaw/commit/8db6fcca7))

`shouldSkipBackendSelfPairing()` in `src/gateway/server/ws-connection/handshake-auth-helpers.ts` was extended to bypass device pairing for device-token auth (`usesDeviceTokenAuth`) in addition to the existing shared-secret bypass. This prevents same-host gateway-to-CLI connections from being blocked by pairing requirements when the device-token auth method is active. The commit also adds launchd restart health detection with status gathering and printing, and adds a compat-baseline auth test to confirm the new bypass path is properly exercised. No audit claim applies — this is defense-in-depth tightening of the auth handshake.

#### Clear stale lockout state when user re-authenticates ([`e490f450f`](https://github.com/openclaw/openclaw/commit/e490f450f))

`openclaw models auth login` now clears stale `auth_permanent` lockout and `billing` disabled state for all profiles matching the target provider when the user deliberately re-authenticates. Prior to this fix, expired or revoked OAuth tokens could leave profiles in a permanently locked-out state with no recovery path short of manual file edits. The cleanup uses `loadAuthProfileStoreForRuntime` (imported at `src/commands/models/auth.ts`) for correct multi-agent profile resolution, and wraps the housekeeping in `try/catch` so corrupt store files cannot block re-authentication. This directly improves the recovery path for Audit 1 Claim 8 (stale token state).

#### Skip device pairing when auth.mode=none ([`9bffa3422`](https://github.com/openclaw/openclaw/commit/9bffa3422))

Fixed a bug in `src/gateway/server/ws-connection/message-handler.ts` where `gateway.auth.mode=none` did not bypass device pairing. When authentication is disabled, the resolved auth context method is `"none"`, which left `sharedAuthOk` false and caused all existing `skipPairing` conditions to fail — closing browser WebSocket connections with code 1008 "pairing required". The fix short-circuits the check: `skipPairing` is set true when `resolvedAuth.mode === "none"`, since device pairing is itself an authentication mechanism and is correctly bypassed when the operator explicitly disables auth.
