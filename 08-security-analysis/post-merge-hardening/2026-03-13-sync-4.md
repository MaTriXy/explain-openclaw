#### Reuse hook agent routing normalization ([`eece5867`](https://github.com/openclaw/openclaw/commit/eece5867))

`resolveAllowedAgentIds()` was moved from `hooks.ts` into `audit-extra.sync.ts` so the same normalization that enforces `hooks.allowedAgentIds` at runtime can be reused by the security audit scanner. This refactor is the prerequisite for the `904db270` fix that follows — consolidating the normalization into a single exported function prevents the runtime and audit paths from diverging.

#### Audit unrestricted hook agent routing ([`904db270`](https://github.com/openclaw/openclaw/commit/904db270))

Added `isHookAgentRoutingUnrestricted()` helper and wired it into `collectHooksHardeningFindings()` in `audit-extra.sync.ts`. When `hooks.allowedAgentIds` is not set, the `/security` audit scan now emits a `hooks.allowed_agent_ids_unrestricted` finding at `warn` severity, alerting operators that any agent identity can act as a hook agent. The finding is keyed by gateway exposure tier so severity scales with network accessibility.

#### Honor trusted proxy hook auth rate limits ([`4da617e1`](https://github.com/openclaw/openclaw/commit/4da617e1))

The hook endpoint authentication throttler previously tracked request counts against the raw TCP socket address. When the gateway sits behind a reverse proxy all requests shared the same remote IP, making rate limits ineffective. The fix threads `cfg.gateway.trustedProxies` through the hook auth middleware so the throttler operates on the resolved client IP from the `X-Forwarded-For` chain, correctly isolating each originating client even when routed through a trusted proxy.

#### Commands: require owner for /config and /debug ([`08aa57a3`](https://github.com/openclaw/openclaw/commit/08aa57a3))

Added `rejectNonOwnerCommand()` in `src/auto-reply/reply/command-gates.ts`. Both `handleConfigCommand()` and `handleDebugCommand()` in `src/auto-reply/reply/commands-config.ts` now call it as the first gate, returning an error to non-owner users before any config read or debug output is produced. This enforces Audit 2 Claim 5 (privileged command access must require owner identity) at the command dispatch layer rather than relying on per-handler checks.

#### Gateway/ws: clear unbound scopes for shared-token auth ([`5e389d5e`](https://github.com/openclaw/openclaw/commit/5e389d5e))

When a WebSocket client authenticates with a shared token (device-less operator auth) the message handler now calls `clearUnboundScopes()` to strip any client-declared scopes not bound to that token's device record. Without this, a shared-token operator could self-declare `operator.admin` or other elevated scopes and have them accepted. Tests confirm that both shared-token and shared-password connects have client-declared scopes stripped and API calls requiring those scopes return `"missing scope: operator.admin"`. (`GHSA-rqpp-rjj8-7wv8`)

#### Flatten MiniMax auth to 4 direct choices ([`55f47e5c`](https://github.com/openclaw/openclaw/commit/55f47e5c))

MiniMax onboarding previously used a nested flow where the CN vs Global region selection preceded the auth method choice. The rewrite presents four explicit paths — MiniMax Global OAuth, MiniMax Global API Key, MiniMax CN OAuth, MiniMax CN API Key — as a flat list. Flattening removes an ambiguous intermediate state where selecting a region before confirming the auth type could cause credentials to be submitted to the wrong regional endpoint.

#### Harden ambiguous approval operand binding ([`33ba3ce9`](https://github.com/openclaw/openclaw/commit/33ba3ce9))

`resolveMutableFileOperandSnapshotSync()` in the node-host runner now fails closed for interpreter invocations that cannot be resolved to one stable file. Inline loader patterns (e.g. `node --loader …`) and shell-payload scripts that remain ambiguous emit `SYSTEM_RUN_DENIED: approval cannot safely bind this interpreter/runtime command`. Post-POSIX value-taking flag parsing ensures the real script file is captured after flags like `-o`/`-c`. `pnpm exec`, `npm exec`, and `npx` script runners are unwrapped before the operand binding step. (`GHSA-57jw-9722-6rf2`, `GHSA-jvqh-rfmh-jh27`, `GHSA-x7pp-23xv-mmr4`, `GHSA-jc5j-vg4r-j5jx`)

#### Harden Docker CI builds ([`9f08af1f`](https://github.com/openclaw/openclaw/commit/9f08af1f))

CI Docker build jobs now pin `docker/setup-buildx-action@v3` to ensure BuildKit is the builder backend across all build matrix entries. BuildKit provides more predictable layer caching, build-secret isolation, and mount controls. The commit also unblocks config documentation generation that was stalled due to the build configuration.

#### Gateway: harden custom session-store discovery ([`46f0bfc5`](https://github.com/openclaw/openclaw/commit/46f0bfc5))

Session store path discovery now resolves symlinks and rejects any resolved path that falls outside the expected root directory. New helpers `resolveStorePath()` (in `src/config/sessions/paths.ts`), `resolveAgentSessionDirsFromAgentsDir()`, `resolveAgentSessionDirsFromAgentsDirSync()`, and `mapAgentSessionDirs()` (in `src/agents/session-dirs.ts`) centralize the path resolution logic. Workspace-controlled symlinks pointing outside the agent session root are silently skipped, preventing path traversal attacks where a compromised workspace could redirect session storage to arbitrary filesystem locations.

#### Zalo: rate limit webhook secret guesses before auth ([`f96ba87f`](https://github.com/openclaw/openclaw/commit/f96ba87f))

The Zalo webhook handler now applies a rate limit to requests carrying an invalid `x-bot-api-secret-token` before authentication succeeds. The rate limit is keyed by the resolved client IP, so a flood from one unauthorized client does not exhaust the rate limit bucket for authenticated traffic arriving from a different client IP. A separate test confirms that a valid `x-bot-api-secret-token` from an unaffected IP still succeeds while the invalid-secret attacker is blocked.

#### Plugins: require explicit trust for workspace-discovered plugins ([`3e28e10c`](https://github.com/openclaw/openclaw/commit/3e28e10c))

`resolveEnableState()` in the plugin resolver now returns `reason: "workspace plugin (disabled by default)"` for plugins that are only auto-discovered from the workspace path. Such plugins are disabled unless explicitly listed in `plugins.allow` or individually enabled in their plugin entry. This prevents cloned repositories from executing workspace plugin code without an operator making an explicit trust decision. (`GHSA-99qw-6mr3-36qr`)

#### Moonshot: respect explicit baseUrl for CN endpoint ([`0a8fa0e0`](https://github.com/openclaw/openclaw/commit/0a8fa0e0))

The implicit Moonshot provider resolver now honors an explicit `baseUrl` (e.g. `api.moonshot.cn`) when constructing the provider configuration. Previously, even when `baseUrl` was set, the implicit provider always defaulted to the `.ai` endpoint, causing CN API keys to fail with HTTP 401. The fix ensures `providers.moonshot.baseUrl` is preserved in the resolved provider so platform.moonshot.cn keys authenticate correctly.

#### Security: require Feishu webhook encrypt key ([`7844bc89`](https://github.com/openclaw/openclaw/commit/7844bc89))

Feishu webhook mode now enforces that both `verificationToken` and `encryptKey` are provided. The Feishu config schema validates both fields as required for `connectionMode: "webhook"`, covering both top-level and per-account configurations. `resolveSecretLike()` is called for `encryptKey` in webhook mode to support SecretRef resolution. Without `encryptKey`, unsigned forged events with a valid `verificationToken` were accepted; requiring both fields closes this gap. (`GHSA-g353-mgv3-8pcj`)

#### Normalize Unicode command obfuscation detection ([`99170e24`](https://github.com/openclaw/openclaw/commit/99170e24))

`detectCommandObfuscation()` now normalizes fullwidth characters (e.g. `／` → `/`) and strips invisible Unicode code points (soft hyphens, zero-width joiners, etc.) before running pattern matching. Previously, substituting look-alike Unicode characters allowed the check to be bypassed entirely. The normalization step runs before all pattern matchers so the Unicode-obfuscated and ASCII variants are detected with the same rules. (`GHSA-9r3v-37xh-2cf6`)

#### Tighten preauth WebSocket handshake limits ([`eff0d5a9`](https://github.com/openclaw/openclaw/commit/eff0d5a9))

Added `MAX_PREAUTH_PAYLOAD_BYTES = 64 * 1024` and `DEFAULT_HANDSHAKE_TIMEOUT_MS = 3_000` constants to bound unauthenticated WebSocket connection data. WebSocket connections that send more than 64 KB of data before completing authentication are closed with close code 1009 ("preauth payload too large"). The handshake timeout ensures connections that stall at the authentication step are torn down within 3 seconds. Together these limits prevent resource exhaustion from large or slow unauthenticated handshakes. (`GHSA-jv4g-m82p-2j93`, `GHSA-xwx2-ppv2-wx98`)

#### Security: preserve Feishu reaction chat type ([`3e730c03`](https://github.com/openclaw/openclaw/commit/3e730c03))

`normalizeFeishuChatType()` now preserves the chat type resolved from the reacted message context rather than falling back to the event's `chat_type` field when ambiguous. Reaction events that cannot be resolved to a definitive chat type (`p2p` or `group`) now fail closed with a skip-log entry. Without this fix, a synthetic reaction event with an empty `chat_type` could be treated as `p2p`, bypassing group authorization and mention gating checks. (`GHSA-m69h-jm2f-2pv8`)

#### Hardening: require LINE webhook signatures ([`48cbfdfa`](https://github.com/openclaw/openclaw/commit/48cbfdfa))

Both the Express and Node LINE webhook handlers now require a valid `X-Line-Signature` header for all incoming POST requests, including empty-event "verification" probes. Previously, unsigned verification probes received a `200 OK`, confirming webhook endpoint reachability to attackers without authentication. Now, missing or invalid signatures return an error before the body is read. Oversized payloads are also rejected before JSON parsing to prevent memory exhaustion from crafted large bodies.

#### Harden pnpm sticky cache on PRs ([`29b36f8e`](https://github.com/openclaw/openclaw/commit/29b36f8e))

CI pnpm store caching on pull requests now uses Blacksmith sticky disks (persistent runner-local storage) instead of `actions/cache` when `use-sticky-disk: true`. The cache key includes `hashFiles('pnpm-lock.yaml')` to ensure cache entries are invalidated on lockfile changes. This prevents a cache poisoning scenario where a malicious PR could write a tampered pnpm store entry into a shared GitHub Actions cache that persists to trusted runs.

#### Agents: adapt pi-ai oauth and payload hooks ([`2f037f09`](https://github.com/openclaw/openclaw/commit/2f037f09))

Updated the pi-ai agent adapter to use the `@mariozechner/pi-ai` library's new `options.onPayload` callback for intercepting model response payloads, and switched to `getOAuthApiKey()`/`getOAuthProviders()` from `@mariozechner/pi-ai/oauth` for credential resolution. The previous direct field access broke with an upstream library API change. The update keeps OAuth credential handling within the library's official boundary rather than reaching into internal structures.

#### Switch pairing setup codes to bootstrap tokens ([`bf89947a`](https://github.com/openclaw/openclaw/commit/bf89947a))

`/pair` and `openclaw qr` pairing flows now embed a short-lived bootstrap token in the pairing payload instead of the long-lived shared gateway credentials. On the mobile side, `loadGatewayBootstrapToken()` and `saveGatewayBootstrapToken()` manage the ephemeral token lifecycle via `gateway.bootstrapToken.<instanceId>` preference keys. This means a pairing QR code or setup code captured in transit or from a chat log can no longer be used to authenticate directly against the gateway — the bootstrap token is exchanged for real credentials only once and then discarded. (`GHSA-7h7g-x2px-94hj`)
