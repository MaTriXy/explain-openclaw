#### Repair Telegram allowlist DM migrations (#27936) (thanks @widingmarcus-cyber) ([`1d4320293`](https://github.com/openclaw/openclaw/commit/1d4320293))

The doctor config flow now detects and repairs Telegram DM channels where `dmPolicy="allowlist"` was configured but `allowFrom` was never populated. New helper `hasAllowFromEntries()` inspects the allow list array (accepting string or number elements), returning `true` only when at least one valid entry is present. `maybeRepairAllowlistPolicyAllowFrom()` wraps the full repair flow: it compares top-level and nested `allowFrom` lists across the Telegram account structure, applies inheritance if either is non-empty, and records the repair action for audit logging. Telegram configs that were silently permissive due to a missing `allowFrom` are now migrated to proper enforcement before the gateway starts.

#### Enforce dm allowFrom inheritance across account channels (#27936) (thanks @widingmarcus-cyber) ([`45d868685`](https://github.com/openclaw/openclaw/commit/45d868685))

This commit extends `requireAllowlistAllowFrom` validation to channel provider schemas beyond Telegram — specifically the providers-core schema (covering Signal, Slack, Discord, and other core channels) and the WhatsApp provider schema. For every channel type that supports `dmPolicy=allowlist`, the Zod refine check now verifies that a non-empty `allowFrom` array is present whenever the policy is declared. Previously, cross-channel configs could declare an allowlist policy at one level and silently omit the corresponding allow-from list at another, resulting in the policy having no practical effect.

#### Skip allowFrom validation at account level (inherits from parent) ([`0fdac3138`](https://github.com/openclaw/openclaw/commit/0fdac3138))

Account-level channel objects inherit their `allowFrom` from the parent config and do not independently hold an allow list. Redundant `requireAllowlistAllowFrom` validation calls that were firing at the account level were removed to prevent false-positive config rejection errors. This is a correctness fix — the enforcement intent is preserved by the parent-level validation added in the preceding commits, while the account-level duplicate check that would erroneously reject valid inherited configs is eliminated.

#### Reject dmPolicy="allowlist" with empty allowFrom across all channels ([`cbed0e065`](https://github.com/openclaw/openclaw/commit/cbed0e065))

New core validator `requireAllowlistAllowFrom` (in `src/config/zod-schema.core.ts`) is wired into every channel provider schema that exposes `dmPolicy`. When `dmPolicy="allowlist"` is parsed, the Zod refine check immediately rejects the config if `allowFrom` is absent or empty. Companion helper `detectEmptyAllowlistPolicy()` (in `src/commands/doctor-config-flow.ts`) performs the same check in the doctor flow, surfacing the misconfiguration as a warning before the gateway starts. Together these additions close a silent misconfiguration path: users who set `dmPolicy=allowlist` without populating `allowFrom` previously received a gateway that treated incoming DMs as unrestricted. A new test file (`src/config/config.allowlist-requires-allowfrom.test.ts`) covers the multi-channel validation matrix.

#### Harden temp dir perms for umask 0002 (landed from #27860 by @stakeswky) ([`b1bbf3fff`](https://github.com/openclaw/openclaw/commit/b1bbf3fff))

`resolvePreferredOpenClawTmpDir()` now actively repairs world-writable or group-writable temp directories caused by a `umask 0002` environment (common in certain Linux server setups). New inner function `tryRepairWritableBits()` detects when a candidate directory's mode has group-write or world-write bits set, then calls `chmodSync(path, 0o700)` to restrict the directory to owner-only access. If the chmod fails (for example, on a directory owned by another user), the function returns `false`, allowing the caller to fall back to an alternative path or fail safely. This directly strengthens **Audit 1 Claim 5** (insufficient file permission checks) by ensuring the OpenClaw temp directory is never left world-writable regardless of the process umask.

#### Align sync boundary realpath canonicalization ([`7dad7cc2c`](https://github.com/openclaw/openclaw/commit/7dad7cc2c))

`resolvePathViaExistingAncestorSync()` and `resolveSymlinkHopPathSync()` in `src/infra/boundary-path.ts` previously used `fs.realpathSync.native()` for path canonicalization. On macOS, `realpathSync.native` resolves firmlinks and uses the OS-native path cache, which can produce subtly different canonical forms compared to the cross-platform `realpathSync`. If the gateway's boundary check used the native form while plugin handler routing used the standard form, an attacker could craft a path that passed the boundary guard but reached a protected endpoint — the attack vector described in GHSA-8j2w-6fmm-m587. This commit aligns both path-resolution functions to use `fs.realpathSync()`, eliminating the canonicalization mismatch and patching **GHSA-8j2w-6fmm-m587** (/api/channels gateway-auth boundary bypass via path canonicalization mismatch).
