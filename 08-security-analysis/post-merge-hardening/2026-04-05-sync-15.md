#### Pass hydrated config to ack reactions to fix SecretRef resolution ([`b51214ec3`](https://github.com/openclaw/openclaw/commit/b51214ec3))

Fixes a credential resolution bug in `extensions/discord/src/monitor/message-handler.process.ts:202-215`. When Discord processes an incoming message it registers `setReaction` and `removeReaction` callbacks used to acknowledge messages (e.g., the ack thumbs-up emoji). These callbacks called `reactMessageDiscord` and `removeReactionDiscord` without passing the `cfg` or `accountId` parameters, so the functions fell back to reading raw config from disk. Raw config contains unresolved secret references (file paths to secret storage) rather than resolved bot tokens — causing the ack reaction to use an unresolved reference instead of the live Discord token.

The fix passes the already-resolved `cfg` context and `accountId` into both reaction callbacks. The reactions now use the in-memory hydrated config that has already resolved secrets to live tokens. Authored by @FunJim (thanks credit in `bf6bd7432`).

#### Harden discord ack auth and gate fallout ([`bf6bd7432`](https://github.com/openclaw/openclaw/commit/bf6bd7432))

Three-part auth hardening for Discord:

1. **`agentId` required in `DiscordSubagentSpawningEvent`** (`extensions/discord/src/subagent-hooks.ts:29`): Changes `agentId?: string` to `agentId: string`, removing the optional flag. Call sites now must supply an explicit agent ID when spawning a Discord subagent — the previous optionality allowed callers to omit the ID, resulting in the fallback `"subagent"` string being used for agent identification across all sessions.

2. **`targetKind` tightened to union type**: Changes `targetKind?: string` in `DiscordSubagentEndedEvent` to `targetKind?: ThreadBindingTargetKind`, replacing the open string type with the closed `ThreadBindingTargetKind` union. This eliminates the possibility of passing unrecognized target kind strings that the event handler cannot pattern-match.

3. **Approval-native typing tightened** (`extensions/telegram/src/approval-native.ts`): Companion type fix propagated from the Discord change.

Test coverage is added in `.../discord/src/monitor/message-handler.process.test.ts` for the updated auth path. CHANGELOG entry present. Thanks @FunJim.

#### Honor whatsapp heartbeat account allowFrom ([`7c738ad03`](https://github.com/openclaw/openclaw/commit/7c738ad03))

Fixes access control enforcement in the WhatsApp heartbeat delivery path. The heartbeat runner sends periodic status messages to a configured set of recipients; those recipients are determined by the `allowFrom` configuration for the channel account. Previously `extensions/whatsapp/src/heartbeat-recipients.ts` resolved `allowFrom` from the top-level channel config regardless of which account the heartbeat was running for. In multi-account setups, all heartbeats shared the top-level policy, meaning per-account `allowFrom` restrictions were not applied.

The fix updates `resolveWhatsAppHeartbeatRecipients()` to call `resolveWhatsAppAccount({ cfg, accountId })` and use the resolved `allowFrom` from the account-scoped config key (`channels.whatsapp.accounts.${accountId}.allowFrom`). `src/channels/plugins/whatsapp-heartbeat.ts` and `src/channels/plugins/types.adapters.ts` are updated to thread the `accountId` through the heartbeat pipeline. Test coverage added in `extensions/whatsapp/src/heartbeat-recipients.test.ts`.

#### Honor whatsapp setup dm policy accounts ([`d1883470e`](https://github.com/openclaw/openclaw/commit/d1883470e))

Fixes per-account DM policy resolution in the WhatsApp setup surface (`extensions/whatsapp/src/setup-surface.ts`). Previously the setup wizard read and wrote DM policy from the top-level `channels.whatsapp.dm.policy` config key for all accounts. In multi-account configurations, this caused per-account DM restrictions to be silently overwritten with the top-level policy during setup.

The fix refactors `setup-surface.ts` to resolve the account-specific config key pair (`channels.whatsapp.accounts.${accountId}.dm.policy` + `allowFrom`) when `accountId` is present and non-default, and delegates account-scoped policy through a shared setup wizard proxy helper. Test coverage added in `extensions/whatsapp/src/setup-surface.test.ts`.
