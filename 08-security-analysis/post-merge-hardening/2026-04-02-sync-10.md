#### Exec approvals: unify effective policy reporting and actions ([`ba735d015`](https://github.com/openclaw/openclaw/commit/ba735d015))

This commit adds three new functions to `src/infra/exec-approvals.ts` that centralize the mapping between policy state and what decision options are presented to users:

- **`resolveExecApprovalAllowedDecisions()` at `:669`** — returns the set of allowed decisions (`allow-once`, `allow-always`) based on the current `ask` mode. When `ask=always`, the `allow-always` option is suppressed — the gateway will not offer persistent approval to users when the policy requires confirmation on every execution.
- **`resolveExecApprovalRequestAllowedDecisions()` at `:685`** — variant for approval request contexts.
- **`isExecApprovalDecisionAllowed()` at `:696`** — validates that a given decision is within the allowed set, used to reject disallowed decisions server-side.

The commit also adds a `DEFAULT_EXEC_APPROVAL_ASK_FALLBACK: ExecSecurity = "deny"` constant, ensuring that when `ask` is unconfigured the policy defaults to deny rather than permissive.

**Security impact:** Closes a UI-layer policy bypass. Previously, a user could potentially be shown the `allow-always` decision option even when the operator had configured `ask=always` (requiring per-use confirmation). Selecting `allow-always` in that state would circumvent the intent of the operator's policy. The fix ensures the allowed decisions are filtered by policy at both the display and validation layers.

- **`src/plugins/hooks.before-install.test.ts`** (new file, 199 lines) — comprehensive test coverage for the restored hook flow, including cases where the hook blocks installs, where force-unsafe-install interacts with hook blocks, and where partner policies enforce additional gates.

**Security impact:** The `before_install` hook is a partner-policy enforcement point. Its removal had left a gap where third-party policies (registered via the hook system) could not inspect or block plugin installations. The reversion restores this gate, ensuring partner-level security scanning and access-control checks can run before any plugin is installed to the user's workspace.

#### BlueBubbles schema alignment ([`75ab5bce6`](https://github.com/openclaw/openclaw/commit/75ab5bce6))

Adds `enrichGroupParticipantsFromContacts` to the core BlueBubbles Zod schema (the strict validator used during config load). The field was already present in the extension's own config schema but was missing from the shared core validator, causing config validation to reject valid configurations with "Unrecognized key" errors at startup.

**Security impact:** Defense-in-depth fix. Schema mismatches between extension and core validators can create situations where security-relevant config fields are silently ignored or rejected inconsistently. Keeping the two schemas aligned ensures the intended config semantics are applied uniformly.

#### Scope session create aliases to requested agent ([`1c83e2eec`](https://github.com/openclaw/openclaw/commit/1c83e2eec))

Fixes a session alias scoping bug in `src/gateway/server-methods/sessions.ts`. When `sessions.create` is called with a bare alias like `"main"`, the alias is now passed through `toAgentStoreSessionKey()` at `src/routing/session-key.ts:53`, which namespaces it under the requesting agent's ID. The canonical sentinel keys `"global"` and `"unknown"` are preserved as-is (pass-through without scoping).

**Security impact:** Before this fix, a bare `"main"` alias from agent A could resolve to or collide with a `"main"` session belonging to agent B if both agents shared a gateway. This is a cross-agent alias leakage: one agent could read or overwrite another agent's session data by using a predictable bare key name. The fix prevents this by ensuring each agent's bare aliases are scoped to its own namespace.

#### Coerce numeric Discord IDs to strings ([`d076153fc`](https://github.com/openclaw/openclaw/commit/d076153fc))

Updates `DiscordIdSchema` at `src/config/zod-schema.providers-core.ts:47` to accept numeric Discord IDs at config load time and coerce safe integers to strings, while rejecting unsafe or precision-losing values with an actionable error. The schema uses a `.union([z.string(), z.number()])` input followed by a `.transform()` that validates `Number.isSafeInteger(value) && value >= 0` before coercing to string. Unsafe values (outside JavaScript's safe integer range) are rejected with `z.NEVER` and an explanatory message directing the user to wrap the value in quotes.

**Security impact:** Input validation improvement. Discord snowflake IDs are 64-bit integers; JavaScript's `number` type loses precision for IDs above `2^53 - 1`. Before this fix, large numeric IDs were silently coerced to imprecise floats, which could cause allowlist mismatches — a user whose ID exceeded safe integer range might be granted or denied access based on a corrupted ID value. The fail-closed behavior (reject unsafe integers) ensures precision loss is surfaced to the operator rather than silently producing incorrect access-control decisions. The `openclaw doctor` repair flow is also aligned with this behavior.

#### Plugin activation boundary enforcement ([`7771c69ca`](https://github.com/openclaw/openclaw/commit/7771c69ca)) — ESCALATED

This commit addresses two distinct security concerns:

**1. Plugin activation boundary enforcement in `src/plugins/loader.ts`**

Adds `matchesScopedPluginRequest()` at `:283` — a helper that checks whether a plugin loading request matches a specific scoped plugin reference (by plugin ID and optional channel/provider scope). Two callsites are updated:

- `:1021` — activation check: scoped loader requests now use `matchesScopedPluginRequest()` instead of a loose string comparison, preventing a channel-ID scope leakage where a scoped request for `plugin-A/channel-X` could accidentally activate `plugin-A/channel-Y`.
- `:1514` — ambient load guard: plugins that have not been explicitly activated cannot be ambient-loaded by scoped requests. This prevents bundled-but-inactive plugins from being pulled into a session scope without explicit operator enablement.

**2. ADC credential path unicode preservation**

Adds `src/plugin-sdk/anthropic-vertex-auth-presence.ts` — a new auth presence helper that checks for Google Cloud ADC credentials. The `GCLOUD_DEFAULT_ADC_PATH` environment variable is handled with unicode normalization, preserving non-ASCII characters in paths (e.g., `~/.config/gcloud/application_default_credentials.json` on systems with non-ASCII home directory paths). The helper `hasAnthropicVertexMetadataServerAdc()` checks for the GCP metadata server opt-in flag.

**3. WhatsApp auth presence helpers**

Expands `src/plugin-sdk/whatsapp-auth-presence.ts` with:
- `hasWebCredsSync(authDir)` — synchronous check for `creds.json` in a WhatsApp auth directory
- `addAccountAuthDirs()` with `resolveUserPath()` — resolves custom auth directory paths from config, including `~`-expansion and normalization

**Security impact:** The plugin activation boundary fix closes a scope-leakage vector where an attacker with control over a scoped plugin request could cause unactivated bundled plugins to load, potentially exposing plugin capabilities that the operator had intentionally left inactive. The auth presence helpers are additive defense-in-depth: they check for credential presence before attempting authentication, avoiding unnecessary credential-scanning in environments where credentials are absent.
