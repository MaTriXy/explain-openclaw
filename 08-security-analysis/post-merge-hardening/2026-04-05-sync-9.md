#### Require owner access for /allowlist writes ([`3cd9aac6b`](https://github.com/openclaw/openclaw/commit/3cd9aac6b))

The `/allowlist` command lets administrators control which commands non-owner users can invoke. Before this fix, the command had no owner gate at its entry point — any authenticated session could issue `/allowlist` write operations. The fix adds a call to `rejectNonOwnerCommand` (`src/auto-reply/reply/command-gates.ts:32`) at `src/auto-reply/reply/commands-allowlist.ts:391`, before the admin scope check, matching the same gate already used by `/config`, `/plugins`, and `/send`. A non-owner caller now receives an immediate rejection rather than proceeding into the allowlist mutation logic. The threat is privilege escalation: without this gate, an attacker with a non-owner session could expand the allowed command set to gain access to commands they would otherwise be blocked from using.

#### Allow configured bundled channels past allowlists ([`f7d24c1ed`](https://github.com/openclaw/openclaw/commit/f7d24c1ed))

This is the first of a three-commit chain that hardens the allowlist enforcement boundary for bundled channels. It introduces the `explicitlyConfiguredBundledChannel` flag in `resolvePluginActivationState` (`src/plugins/config-state.ts`): the flag is true when the plugin origin is `"bundled"`, `explicitSelection.explicitlyEnabled` is true, and `explicitSelection.reason` is `"channel enabled in config"`. The allowlist rejection guard at line 337 is updated to skip the block when `explicitlyConfiguredBundledChannel` is set — ensuring that bundled channels the admin has explicitly configured in their config are not accidentally locked out by an active plugin allowlist. The two follow-on commits (`cd08facd7a` and `bc137951e9`) harden and then root-cause-fix the check.

#### Keep auto-enabled channels behind allowlists ([`cd08facd7`](https://github.com/openclaw/openclaw/commit/cd08facd7))

The second commit in the chain. It replaces the `explicitlyConfiguredBundledChannel` string-based check introduced in `f7d24c1ed` with a call to `isBundledChannelEnabledByChannelConfig` (`src/plugins/config-state.ts:429`) in the same expression at `src/plugins/config-state.ts:287`. The concern with the two-condition string check is that `explicitSelection.reason === "channel enabled in config"` could be satisfied by auto-enabled bundled channels whose activation reason matches the string but whose entry point is not a deliberate admin configuration. Delegating to `isBundledChannelEnabledByChannelConfig` provides a stricter structural check against the actual config object. This commit also adds `sourceConfig` and `sourceRootConfig` parameters to `resolveEffectiveEnableState` (`src/plugins/config-state.ts:448`) to thread the activation-time config snapshot through the call stack. The root cause (the snapshot not being propagated at the call site) is addressed by the next commit.

#### Preserve allowlist guard for auto-enabled bundled channels (#60233) (thanks @dorukardahan) ([`bc137951e`](https://github.com/openclaw/openclaw/commit/bc137951e))

The root cause fix in the three-commit chain. The prior commits (`f7d24c1ed`, `cd08facd7a`) added the `explicitlyConfiguredBundledChannel` bypass and tightened its check, but the activation-time config snapshot (`activationSourceConfig`) was never threaded into `resolveGatewayStartupPluginIds` (`src/plugins/channel-plugin-ids.ts:76`). Without it, the `sourceRootConfig` parameter inside `resolvePluginActivationState` (`src/plugins/config-state.ts:272`) was always `undefined`, so `explicitSelection` (`src/plugins/config-state.ts:282`) was computed against the wrong config — making auto-enabled bundled channels appear "explicitly configured" and allowing them through the allowlist gate they should not pass.

This commit propagates `activationSourceConfig` into `resolveGatewayStartupPluginIds` (`src/plugins/channel-plugin-ids.ts:80`) and derives `sourcePluginsConfig` from it, then passes both `sourceConfig` and `sourceRootConfig` into each plugin activation check. On the gateway side, `activationSourceConfig` is now forwarded at `src/gateway/server-plugins.ts:426`. With the correct config snapshot in place, the explicit selection resolves against the actual admin config, and the `src/plugins/config-state.ts` check is reverted from the structural `isBundledChannelEnabledByChannelConfig` (`src/plugins/config-state.ts:429`) call back to the two-condition string check — which is now reliable because it operates on a properly scoped config.

#### Use owning npm prefix for global updates ([`eb4b6f702`](https://github.com/openclaw/openclaw/commit/eb4b6f702))

OpenClaw's self-update path resolves a global npm binary to run `npm install -g openclaw`. Before this fix, the binary was looked up via `PATH`, which can silently route the update command to a different npm installation than the one that owns the current global package — a supply chain risk if a user's PATH is manipulated or if multiple npm installations are present. This commit adds two new functions to `src/infra/update-global.ts`: `inferNpmPrefixFromPackageRoot` (line 194), which walks up the package root directory (`<prefix>/lib/node_modules/<pkg>`) to derive the owning prefix, and `resolvePreferredGlobalManagerCommand` (line 224), which uses that prefix to locate the npm binary at `<prefix>/bin/npm` (or `<prefix>/npm.cmd` on Windows). If the binary exists at the resolved path, it is used instead of the bare `npm` command from PATH. The `resolveGlobalRoot` and `resolveGlobalPackageRoot` functions are updated to accept and forward a `pkgRoot` parameter. This ensures that global update operations are anchored to the npm installation that originally installed OpenClaw, preventing update hijacking via PATH manipulation.
