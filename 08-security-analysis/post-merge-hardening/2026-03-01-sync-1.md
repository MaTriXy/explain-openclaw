#### Docker Browser No-Sandbox Rollout ([`139271ad5`](https://github.com/openclaw/openclaw/commit/139271ad5))

The sandbox browser container launch now passes `OPENCLAW_BROWSER_NO_SANDBOX=1` as a Docker environment variable. Chromium's built-in setuid/namespace sandbox requires Linux PID namespace creation privileges that Docker does not grant by default; without this flag, Chromium fails to start inside a Docker container. The container itself provides the isolation boundary, making `--no-sandbox` the correct choice in this context. The `SANDBOX_BROWSER_SECURITY_HASH_EPOCH` constant was bumped from `"2026-02-21-novnc-auth-default"` to `"2026-02-28-no-sandbox-env"`, which invalidates all existing sandbox browser containers and forces recreation with the updated environment.

#### Outside-Workspace Error Code Added to fs-safe ([`f5c2be191`](https://github.com/openclaw/openclaw/commit/f5c2be191), [`d6552998e`](https://github.com/openclaw/openclaw/commit/d6552998e), [`44220ef24`](https://github.com/openclaw/openclaw/commit/44220ef24))

The `SafeOpenErrorCode` type in `src/infra/fs-safe.ts` gained a new `"outside-workspace"` value. Previously, `openFileWithinRoot` and `writeFileWithinRoot` threw `SafeOpenError("invalid-path", "path escapes root")` for path traversal attempts — the same error code used for hardlinks, symlinks, and non-file targets. Consumers could not distinguish a workspace-boundary violation from other invalid-path conditions and often fell back to a generic "not found" message.

The new `"outside-workspace"` code is now thrown when a resolved path escapes the workspace root. Each consumer was updated accordingly: `pi-tools.read.ts` maps it to `EACCES`, `browser/paths.ts` returns `"File is outside {scopeLabel}"`, and `media/server.ts` returns HTTP 400 with a descriptive message.

The `toSaveMediaSourceError` function in `src/media/store.ts` (`d6552998e`) was updated separately to handle the new code in its switch statement. Four test files (`44220ef24`) verify the full error mapping chain — `pi-tools.read.host-edit-access.test.ts`, `browser/paths.test.ts`, `media/server.outside-workspace.test.ts`, and `media/store.outside-workspace.test.ts`.

This partially addresses Legitimate Gap 3 (workspace boundary enforcement) by giving callers a distinct signal for path-escape violations.

#### Android Voice Playback Token Rotation ([`1d7b76a90`](https://github.com/openclaw/openclaw/commit/1d7b76a90))

The `playbackGeneration` counter in `TalkModeManager` was changed from a `volatile Long` to an `AtomicLong`. The counter is incremented on each new assistant reply via `incrementAndGet()`, ensuring that in-flight audio rendering from a previous reply cannot continue after the generation advances. `volatile Long` reads and writes are individually atomic on the JVM, but the increment+compare sequence (used to gate audio playback continuation) is not. `AtomicLong.incrementAndGet()` makes the whole operation atomic, preventing a race condition where concurrent voice stream threads could observe the same generation and replay stale audio from a prior reply.

### Audit Cross-Reference

The `f5c2be191`, `d6552998e`, and `44220ef24` commits partially address **Legitimate Gap 3** (workspace boundary enforcement) from the Medium article audit. The workspace boundary check in `openFileWithinRoot` now throws a distinct `"outside-workspace"` error code instead of the generic `"invalid-path"`, making path-escape violations detectable by callers. The underlying `isPathInside()` guard was already present; this change improves error observability and consumer accuracy.

The `1d7b76a90` commit addresses a **token race condition** in Android voice playback (Audit 1 Claim 4 domain). The `AtomicLong` change ensures the generation counter increment+compare sequence is atomic, preventing stale-audio replay across concurrent voice stream threads.

The `139271ad5` commit updates sandbox browser behavior for Docker deployments. The `--no-sandbox` flag is the Docker-appropriate isolation choice (the container provides the isolation boundary) and does not represent a security regression. The hash epoch bump forces container recreation.

No audit claims from Issue #1796 or the Medium article are directly closed by this sync. The 3 legitimate gaps identified in previous audits remain open.
