#### Add async requireApproval to before_tool_call ([`6ade9c474`](https://github.com/openclaw/openclaw/commit/6ade9c474))

Extends the `before_tool_call` plugin hook result type with a `requireApproval` field (`src/plugins/hooks.ts`) that enables plugins to natively pause agent execution and wait for real human approval. When a plugin returns `requireApproval`, the agent runtime routes the request through three new gateway methods (`plugin.approval.request`, `plugin.approval.waitDecision`, `plugin.approval.resolve` in `src/gateway/server-methods/plugin-approval.ts`) with scope guards. The gateway broadcasts the approval request to connected channels (Telegram, Discord) and the user responds via the `/approve` command, which falls back from exec approval lookup to `plugin.approval.resolve` when no exec approval is pending.

Several security-critical correctness fixes are bundled in this commit:

- **Host-generated approval IDs:** The server always generates IDs via `randomUUID()` — the plugin-provided `id` field was removed from the type, gateway request, and protocol schema, preventing plugins from forging or predicting approval IDs.
- **Block takes precedence:** `block` is checked before `requireApproval` so a higher-priority plugin block cannot be overridden by a lower-priority approval return.
- **Abort-signal racing:** `waitDecision` races against the abort signal so users are not stuck waiting for the full approval timeout after cancelling a run.
- **Timeout-allow param bug fixed:** When `timeoutBehavior` is `"allow"`, the runtime now returns the merged hook params (not the original params), preventing security plugins' parameter rewrites from being silently discarded on timeout.
- **APPROVAL_NOT_FOUND error code:** The `/approve` command's exec-to-plugin fallback uses structured error code matching instead of fragile string comparison.

This commit directly addresses **Audit2-claim5** (external approval enforcement — plugins can now natively pause execution for mandatory human approval rather than relying on a cooperative soft block). It also strengthens **Audit1-claim5** by adding extensive test coverage for the approval forwarder path (`src/infra/plugin-approval-forwarder.test.ts`, 334 lines) and the gateway method authorization (`src/gateway/server-methods/plugin-approval.test.ts`, 431 lines).

#### Preserve embedded auth on HTTP fallback ([`5e8db468f`](https://github.com/openclaw/openclaw/commit/5e8db468f))

Fixes a credential propagation gap in `src/agents/openai-ws-stream.ts` and `src/agents/pi-embedded-runner/run/attempt.ts` where the bearer token or API key established for the WebSocket path was not forwarded when the runtime fell back from WebSocket to HTTP for communication with the OpenAI Realtime API. Without this fix, HTTP fallback connections silently used incorrect or absent credentials, creating an unreliable auth surface that could mask authentication failures. The fix propagates embedded auth into the HTTP fallback request path so auth behavior is consistent regardless of which transport is active.

#### Media: enforce file mode after writes ([`206c29514`](https://github.com/openclaw/openclaw/commit/206c29514))

After every media write (downloaded file or buffered stream), `src/media/store.ts` now passes `MEDIA_FILE_MODE = 0o644` (line 19) directly as the `mode` option to `createWriteStream()` (line 220) and `fs.writeFile()` (line 298), ensuring files are created with the correct permissions atomically. Previously, the file mode was left to whatever the process umask produced — which could be `0o666` or broader depending on the runtime environment. The atomic mode enforcement ensures that media files stored in `~/.openclaw/media/` are always owner-readable/writable, group and others read-only, regardless of system umask. This directly addresses **Audit1-claim5** (media file permissions gap).

#### Media: respect umask and clean failed downloads ([`bb932beea`](https://github.com/openclaw/openclaw/commit/bb932beea))

Paired with `206c29514`, this commit removes residual uses of overly broad mode flags (e.g. `0o666`) in the media store and ensures that failed downloads clean up their partially-written files. Previously, a failed download could leave a partial file on disk in an indeterminate state. Now the cleanup runs on failure so partial media artifacts do not accumulate in `~/.openclaw/media/`. The test in `src/media/store.redirect.test.ts` is updated to verify the cleanup path. Completes the **Audit1-claim5** media permission fix together with `206c29514`.

#### Fail over and sanitize Codex server_error payloads ([`b1c982bb2`](https://github.com/openclaw/openclaw/commit/b1c982bb2))

Addresses information disclosure risk from verbose Codex provider error responses. Adds `sanitizeUserFacingText()` in `src/agents/pi-embedded-helpers/errors.ts` to strip sensitive fields (API keys, internal paths, provider-specific error details) from error text before it is displayed to users or emitted to logs. A companion module `src/agents/pi-embedded-helpers/failover-matches.ts` adds pattern matching to detect Codex-specific transient `server_error` payloads and trigger model fallback rather than propagating the raw error. The TUI formatter (`src/tui/tui-formatters.ts`) is updated to use the sanitized form. The `src/shared/assistant-error-format.ts` barrel is updated to re-export the sanitized formatter. This reduces the risk of leaking provider credentials or internal infrastructure details through verbose error messages that reach end users or persistent logs.

#### Prefer non-user writeable paths ([`c40884d30`](https://github.com/openclaw/openclaw/commit/c40884d30))

Introduces `resolveSystemBin()` and the `SystemBinTrust` enum in the new file `src/infra/resolve-system-bin.ts` to resolve system binary paths (ffmpeg, Chrome) preferentially from non-user-writable locations. Previously, binary resolution on Windows used `PATH` scanning, which could be poisoned by user-writable directories earlier in `PATH` (a binary substitution / PATH hijacking vector). `src/infra/windows-install-roots.ts` (new, 263 lines) provides Windows-specific known-good install roots (Program Files, system32) for trusted binary lookup. The `SystemBinTrust` enum distinguishes `SYSTEM` trust (e.g., `/usr/bin`, Program Files) from `SystemBinTrust.USER` trust (user-writable directories). Chrome and ffmpeg callers in `extensions/browser/src/browser/chrome.executables.ts` and `src/media/ffmpeg-exec.ts` are updated to use `resolveSystemBin()` with the appropriate trust level. `src/infra/path-env.ts` gains new tests (96 lines added) and `src/daemon/runtime-paths.ts` is updated to use trusted path resolution. This addresses **Audit1-claim6** (path traversal / binary substitution via user-writable PATH entries).
