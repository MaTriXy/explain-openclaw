#### Revoke active sessions on token rotation ([`91f7a6b0f`](https://github.com/openclaw/openclaw/commit/91f7a6b0f))

When the gateway auth token is rotated via `device.token.rotate`, any clients authenticated with the old token would previously remain connected until they naturally disconnected. This commit adds a `queueMicrotask` call to `disconnectClientsForDevice` immediately after a successful rotate, forcing any active WebSocket sessions authenticated against the old device token to disconnect and re-authenticate. This closes a race window (related to GHSA-rfqg-qgf8-xr9x) where a stolen-then-rotated token would still grant access for the duration of an existing session.

#### Quiet scoped manifest id warnings ([`bbd495ed6`](https://github.com/openclaw/openclaw/commit/bbd495ed6))

Reduces noise from plugin installation validation by suppressing expected warnings for scoped npm-style plugin manifest IDs. No security boundary changes; included because the manifest validation code path runs during plugin load.

#### Unify channel approvals and restore routing/auth ([`9ff57ac47`](https://github.com/openclaw/openclaw/commit/9ff57ac47))

A large refactor (19 production files) that centralizes the exec approval routing and authentication logic across all channels. Previously, channel-specific approval handlers each had their own sender validation, creating risk of inconsistency. This change consolidates approval routing through a single path with unified sender authentication checks, preventing potential auth bypass in cases where channel-specific handlers might diverge from the core approval policy. This hardens the control plane against Audit 1 Claims 5-6 style scenarios where approval flow inconsistency could be exploited.

#### Gateway: open config files without shell interpolation ([`5cc0bc936`](https://github.com/openclaw/openclaw/commit/5cc0bc936))

Previously the gateway config file opener passed the file path as a shell-interpolated string, which could enable shell injection if the config path contained shell metacharacters. This commit switches to `execFile` with an argument array, which bypasses shell interpolation entirely. This addresses Audit 2 Claim 7 (shell injection via incomplete input sanitization) at the config file level.

#### Sandbox: sanitize SSH subprocess env ([`cfe144595`](https://github.com/openclaw/openclaw/commit/cfe144595))

SSH subprocess spawns inside the sandbox now inherit a sanitized environment that blocks sensitive host variables including `LD_PRELOAD`, `DYLD_*` dylib insertion vectors, and other credential/injection-enabling env vars. This is defense-in-depth hardening directly addressing Audit 2 Claim 8, where host environment variable injection through `LD_PRELOAD`/`DYLD_INSERT_LIBRARIES` could allow code execution in spawned subprocesses.

#### Canonicalize Telnyx replay request keys ([`ad7766605`](https://github.com/openclaw/openclaw/commit/ad7766605))

Telnyx webhook replay protection requires comparing a signing key derived from the timestamp and nonce. If the key canonicalization was inconsistent (e.g., accepting both standard Base64 and Base64URL encoding), a replay attack could forge a valid signature using a variant encoding. This commit canonicalizes the request key format before signature verification, ensuring replay protection applies uniformly. Related to GHSA-37v6-fxx8-xjmx.

#### Exec approvals: prevent interpreter allow-always persistence ([`1a75906a6`](https://github.com/openclaw/openclaw/commit/1a75906a6))

The exec approval allow-always list previously stored interpreter entries that could be reused via alternate interpreter implementations (e.g., `python` vs `python3`, or different shell variants). This commit hardens the allowlist logic to prevent such reuse exploits, ensuring that allow-always grants are specific enough that an attacker cannot craft a semantically-equivalent invocation that slips through the allowlist check.

#### Gateway: tighten HTTP tool invoke authorization ([`29cb1e3c7`](https://github.com/openclaw/openclaw/commit/29cb1e3c7))

The HTTP gateway tool invocation path now enforces owner-only gating at the gateway surface. Previously, a bearer token holder could potentially invoke owner-restricted tools via the HTTP interface by exploiting a gap between the HTTP authorization check and the internal tool policy. This commit closes the RBAC bypass path (Audit 2 Claim 5 scenario: privilege escalation through the HTTP gateway tool interface), ensuring that HTTP bearer token clients cannot invoke tools that require owner-level authentication.

#### Harden shell-side approval guardrails ([`dd9d0bdd8`](https://github.com/openclaw/openclaw/commit/dd9d0bdd8))

Multiple shell-side approval bypass paths are closed in this commit: (1) env wrapper injection — `env VAR=val cmd` patterns used to prepend environment variables before a command while keeping the command name looking approved; (2) nested shell wrapping — using `sh -c 'approved-cmd && malicious-cmd'` to chain unapproved commands after an approved one; (3) prefixed command patterns — commands with path prefixes or whitespace padding that fool string-match approval checks. The hardened guardrails normalize command representations before approval matching to defeat these evasion techniques.

#### Harden viewer proxy access ([`30a169032`](https://github.com/openclaw/openclaw/commit/30a169032))

Loopback requirement is maintained, but the address classification correctly uses the actual remote address rather than trusting forwarded headers from untrusted sources. Related to GHSA-3xv9-89fm-7h4r (diffs viewer proxy misclassification).

This ensures that callers using the OpenResponses HTTP endpoint cannot invoke owner-only tools, regardless of the auth token they present. #### Replace readFileSync with symlink-safe, root-confined skill file loader ([`b7b46ad18`](https://github.com/openclaw/openclaw/commit/b7b46ad18))

Skill file loading previously used `readFileSync` without path validation, allowing symlink-following and path traversal to escape the skill directory. The new loader resolves the path, verifies the resolved path is within the skill root, and rejects symlinks. This prevents a malicious skill package from including symlinks that point outside the skill directory to read arbitrary host files.

#### Infra: block auth env vars from workspace dotenv ([`7a5c5f33d`](https://github.com/openclaw/openclaw/commit/7a5c5f33d))

Workspace `.env` files can now no longer override auth-sensitive environment variables including `ANTHROPIC_API_KEY`, `OPENAI_API_KEY`, OAuth tokens, and gateway credentials. The filter uses both exact name matching and prefix/suffix heuristics to catch variants (`*_API_KEY`, `*_TOKEN`, `OPENCLAW_*`, etc.). This closes the dotenv-based host-env takeover path described in GHSA-3qpv-xf3v-mm45 and GHSA-qcj9-wwgw-6gm8, where a malicious workspace `.env` file could rewrite OpenClaw's runtime credentials or redirect package registries.

#### Harden identifier entropy and delay jitter ([`ae703ab0e`](https://github.com/openclaw/openclaw/commit/ae703ab0e))

Increases the entropy of session IDs and approval tokens, and adds jitter to retry delays. The 6-file, 6-test change targets the identifier generation paths to make them harder to predict or brute-force. Delay jitter prevents timing attacks that could exploit predictable retry intervals to synchronize attack attempts with authentication windows.

#### Agents: pin apply-patch workspace mutations ([`32a4a47d6`](https://github.com/openclaw/openclaw/commit/32a4a47d6))

The `apply-patch` tool now uses `openat()`-style directory-bound file operations for all write paths. Previously, patch targets were resolved relative to the workspace root using string comparison, which could be bypassed with `../` sequences or symlinks pointing outside the workspace. The new approach uses OS-level directory pinning so that all file operations are bound to the workspace directory at the syscall level, making directory traversal and symlink escapes structurally impossible.

#### Treat OpenAI HTTP ingress as non-owner ([`f011d0be2`](https://github.com/openclaw/openclaw/commit/f011d0be2))

The OpenAI HTTP gateway ingress (`src/gateway/openai-http.ts`) now sets `isOwner: false` on all connections, regardless of the auth token presented. External callers using the OpenAI-compatible HTTP endpoint cannot invoke owner-only tools (exec, file write, gateway config), even with a valid token. This enforces the principle of least privilege: the OpenAI-compatible surface is for chat completions, not privileged operations.

#### Allow setup-code bootstrap auth for operator pairing ([`2dced6b4a`](https://github.com/openclaw/openclaw/commit/2dced6b4a))

**Security-relevant auth model change.** Bootstrap tokens for operator pairing are now multi-use and persistent rather than single-use and ephemeral. The operator role scope is expanded from `[node]` to `[node, operator]`, and profile validation switches from strict exact-match to permissive allowlist checking. Tokens are kept alive until explicitly revoked rather than consumed on first use.

**Threat model impact:** The previous single-use model provided a tight security window — a stolen bootstrap token would be consumed by the legitimate pairing and become invalid. Multi-use persistent tokens have a wider exploitation window if intercepted. Mitigations: tokens are still explicitly revocable, operator scope is scoped, and the change enables pairing scenarios that previously required workarounds. Users should revoke bootstrap tokens promptly after pairing completes.

#### Harden session-status tool visibility guard for all callers ([`4d369a340`](https://github.com/openclaw/openclaw/commit/4d369a340))

The `session-status` tool visibility guard previously only applied in certain caller contexts, leaving a path where agents could query sessions outside their visibility scope via direct tool invocation. This commit extends the guard to all callers uniformly, ensuring agent session isolation is enforced regardless of how the tool is invoked.

#### Filter thread history by sender allowlist ([`5cca38084`](https://github.com/openclaw/openclaw/commit/5cca38084))

MS Teams thread history loaded as context for agent triggers now filters messages by the sender allowlist. Previously, all thread participants' messages were included in the context regardless of whether the sender was on the allowlist, enabling prompt injection attacks from non-allowlisted thread participants whose messages would appear in the agent's context as trusted input.

#### Synology-chat: add webhook in-flight guard ([`7e0866971`](https://github.com/openclaw/openclaw/commit/7e0866971))

Adds a per-account concurrency guard to the Synology Chat webhook ingestion path. When the number of in-flight webhook requests for a single account exceeds the configured limit, the endpoint responds with HTTP 429 (Too Many Requests) instead of processing the request. This prevents DoS attacks via webhook flooding that could exhaust resources on Synology Chat deployments.

#### Enforce embeddings HTTP write scope ([`3216df792`](https://github.com/openclaw/openclaw/commit/3216df792))

The gateway's HTTP embeddings endpoint now requires the `operator.write` scope. Previously, any authenticated HTTP client could call the embeddings API regardless of their declared scope. This prevents unauthorized or minimally-scoped clients from incurring model API costs or using the embeddings service beyond their authorization.

#### Harden phone-control command scope checks ([`847912f3e`](https://github.com/openclaw/openclaw/commit/847912f3e))

Phone control commands (dialing, hanging up, sending DTMF) now enforce `operator.write` scope checks consistently regardless of which channel the caller is using. Previously, scope enforcement could be bypassed by callers arriving from channels that did not enforce the scope check at the channel level. This closes a privilege escalation path for phone control mutations.

#### OpenShell: harden mirror sync boundaries ([`3b9dab0ec`](https://github.com/openclaw/openclaw/commit/3b9dab0ec))

The OpenShell mirror sync now validates that mirrored paths don't include symlinks that escape the sandbox boundary. Previously, sandbox files containing symlinks pointing to host filesystem paths outside the designated mirror root could expose host files when the mirror sync dereferenced them. The hardened sync resolves symlinks before mirroring and rejects paths that escape the mirror root.

#### Harden talk-voice config persistence scope checks ([`a4e447a16`](https://github.com/openclaw/openclaw/commit/a4e447a16))

Voice configuration writes (Talk Voice `/voice set` commands) now require `operator.admin` scope, enforced regardless of the caller's channel. This is part of the same `operator.admin` scope hardening applied to chat.send system provenance injection and Telegram target writeback in `8b88b927cb`. Without this check, a non-admin channel could modify voice configuration outside its authorization level.

#### Gateway: harden node event trust boundaries ([`a77928b10`](https://github.com/openclaw/openclaw/commit/a77928b10))

Node-originated tool calls (via `TOOL_ALLOW_BY_MESSAGE_PROVIDER` in `src/agents/pi-tools.ts`) are now restricted to a safe allowlist: `canvas`, `image`, `pdf`, `tts`, `web_fetch`, `web_search`. Dangerous tools including `exec`, `read`, `write`, `edit`, and messaging tools are no longer available to node message providers. Additionally, inbound node event fields (`key`, `packageName`, `title`, `text`) in `src/gateway/server-node-events.ts` are now passed through `sanitizeInboundSystemTags` before use, and notification events are marked with `trusted: false` to prevent node-originated content from masquerading as trusted system messages.

#### Restore default operator scopes for pure HTTP token auth ([`fe2eb185f`](https://github.com/openclaw/openclaw/commit/fe2eb185f))

After `f0af18672` removed bearer-declared scopes, HTTP token-authenticated clients received no default scopes, breaking backward compatibility. This commit restores `CLI_DEFAULT_OPERATOR_SCOPES` (including `operator.write`) for HTTP Bearer token requests that do not include an `x-openclaw-scopes` header. Clients that declare explicit scopes via the header still receive only those declared scopes. This closes the scope deprivation path (related to GHSA-6p8r-6m93-557f) while maintaining backward compatibility for existing HTTP token clients.

#### Clear unbound scopes for trusted-proxy auth ([`8b88b927c`](https://github.com/openclaw/openclaw/commit/8b88b927c))

The new `shouldClearUnboundScopesForMissingDeviceIdentity` function in `src/gateway/server/ws-connection/connect-policy.ts` clears unrestricted (unbound) scopes for trusted-proxy authenticated connections that lack device identity. In shared proxy setups, proxy auth can bypass device pairing requirements for some clients — but those clients should not receive unrestricted scope grants. This closes a privilege escalation gap where proxy-authenticated sessions without device binding could accumulate elevated scopes.

#### Enforce ACP attachment roots ([`566fb73d9`](https://github.com/openclaw/openclaw/commit/566fb73d9))

ACP (Agent Code Platform) attachment handling in `src/auto-reply/reply/dispatch-acp.ts` now validates that all attachment paths are confined to the configured attachment roots. Relative path sequences (`../`), `file://` URLs, and symlinks that escape the roots are rejected before processing. The attachment cache in `src/media-understanding/attachments.cache.ts` receives the same treatment. This closes the outPath validation gap previously listed as a legitimate open gap — path traversal attacks via ACP media attachments are now structurally blocked.

#### MS Teams: validate webhook auth before JSON parsing ([`3834d4709`](https://github.com/openclaw/openclaw/commit/3834d4709))

The MS Teams webhook handler now validates the Bot Framework JWT signature (via `JwtValidator`) before passing the request body to the JSON parser. Previously, the JSON body was parsed before auth validation, meaning unauthenticated requests would consume CPU and memory for body parsing. This reordering prevents DoS attacks via large malformed unauthenticated payloads and ensures parser-level vulnerabilities (e.g., JSON bomb) cannot be triggered by unauthenticated callers.

#### Block BROWSER, GIT_EDITOR, GIT_SEQUENCE_EDITOR from inherited host env ([`bc3b05dce`](https://github.com/openclaw/openclaw/commit/bc3b05dce))

`BROWSER`, `GIT_EDITOR`, and `GIT_SEQUENCE_EDITOR` are added to the exec environment blocklist. An attacker with access to the host environment could set `GIT_SEQUENCE_EDITOR=malicious-cmd` so that any `git rebase -i` invoked by the agent would execute the attacker's command instead of the rebase editor. Similarly, `BROWSER` controls which binary runs when git or other tools open URLs. Includes regression tests for `git rebase -i` and `git log` exploit scenarios. This extends the GHSA-g8xp-qx39-9jq9 remediation (incomplete host-env-security-policy).

#### Gate audio preflight transcription on sender authorization ([`c4fa8635d`](https://github.com/openclaw/openclaw/commit/c4fa8635d))

Telegram audio preflight transcription (the step that converts voice messages to text before routing to the agent) is now gated on sender authorization. Previously, an unauthorized Telegram user could send a voice message and trigger the transcription pipeline, consuming API resources and potentially leaking audio content. The authorization check is now applied before any transcription is initiated.

#### Discord: gate audio preflight on member access ([`ee52f6422`](https://github.com/openclaw/openclaw/commit/ee52f6422))

Discord audio preflight transcription is now gated on guild member access verification. Non-members (or users outside voice channel access) can no longer trigger audio transcription, closing an information leakage path where voice channel audio could be transcribed for non-member users. This aligns Discord audio gating with the Telegram audio gating hardening applied in `c4fa8635d`. Related to GHSA-cqgw-44wg-44rf (Discord voice manager bypass).
