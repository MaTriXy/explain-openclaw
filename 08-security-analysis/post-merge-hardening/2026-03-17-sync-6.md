#### Sandbox Path Guard Consolidation ([`503932919`](https://github.com/openclaw/openclaw/commit/503932919f))

The sandbox remote filesystem bridge consolidates path traversal guards into shared modules. Previously, `remote-fs-bridge.ts` contained local implementations of `isPathInside()`, `isPathInsideContainerRoot()`, and `normalizeContainerPath()`. These have been removed in favor of imports from `path-guards.js` (host path guard) and `path-utils.js` (container-specific normalization and containment check). Centralizing these functions ensures path traversal prevention logic is consistent across the codebase and cannot drift out of sync with shared implementations.

#### Feishu Webhook Timing-Safe Comparison ([`223ae42c7`](https://github.com/openclaw/openclaw/commit/223ae42c79))

`isFeishuWebhookSignatureValid()` previously compared HMAC-SHA256 signatures using JavaScript's `===` string equality, which is vulnerable to timing side-channel attacks. The fix introduces `timingSafeEqualString()`, a wrapper around Node.js's `crypto.timingSafeEqual()` that converts both strings to `Buffer` before comparison. This closes a timing oracle that could theoretically allow an attacker to probe the Feishu webhook secret character-by-character. Brings Feishu in line with the existing timing-safe pattern already applied to other webhook handlers.

#### WhatsApp Login Tool Surface Narrowed ([`38bc364ae`](https://github.com/openclaw/openclaw/commit/38bc364aed))

WhatsApp login functionality has been extracted from the general runtime into a dedicated `runtime-whatsapp-login-tool.ts` module. The new `createRuntimeWhatsAppLoginTool()` function sets `ownerOnly: true`, ensuring that only the channel owner can trigger WhatsApp QR-code pairing or wait for a scan to complete. This narrows the attack surface by separating authentication-related tooling from general WhatsApp runtime capabilities and enforcing an explicit privilege gate.

#### Plugin Command Alias Conflict Detection ([`68d2bd27c`](https://github.com/openclaw/openclaw/commit/68d2bd27c9))

`validatePluginCommandDefinition()` now validates all `nativeNames` aliases (default, telegram, discord) in addition to the primary command name. `registerPluginCommand()` extends conflict detection to check all invocation keys against both primary keys and aliases of already-registered commands via the new `listPluginInvocationKeys()` helper. Without this, a plugin could register an alias that shadows an existing command from another plugin — a potential command hijacking vector.

#### Bundled MCP Tools in Embedded Pi ([`06459ca0d`](https://github.com/openclaw/openclaw/commit/06459ca0df))

Adds support for bundling MCP (Model Context Protocol) tools directly into the embedded Pi agent execution context. MCP tools are now available within the Pi agent's sandbox boundary without requiring a separate MCP server process. The change introduces `bundleMcpTools` configuration, runtime wiring in the Pi runner, and documentation updates. The security dimension is that MCP tools continue to operate within the existing Pi sandbox isolation; the trust boundary remains the Pi execution context rather than the host.

#### SecretRef Scoping and Doctor Path Hardening ([`da34f81ce`](https://github.com/openclaw/openclaw/commit/da34f81ce2))

Scopes `SecretRef` resolution to specific message contexts, preventing secrets from being resolved in unintended code paths. New modules `message-secret-scope.ts`, `command-secret-targets.ts`, and `command-secret-gateway.ts` define explicit resolution scopes for CLI message commands and gateway-backed commands. Doctor and status code paths (`doctor-config-flow.ts`, `status-all.ts`) are hardened to prevent inadvertent secret exposure during diagnostic operations. Comprehensive test coverage added (11 test files including a dedicated coverage test `command-secret-resolution.coverage.test.ts`).

#### Web Search Runtime Capability ([`4bba2888e`](https://github.com/openclaw/openclaw/commit/4bba2888e7))

Introduces a `webSearch` runtime capability to the plugin system. Plugins must explicitly declare this capability to receive web search tool access at runtime. The capability is enforced at the plugin runtime boundary, following the existing capability model (alongside `media`, `voice`, etc.). This ensures web search tooling is not available to plugins that have not explicitly requested and been granted the capability.
