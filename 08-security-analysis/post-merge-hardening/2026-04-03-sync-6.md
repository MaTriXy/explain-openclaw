#### Separate Gemini OAuth state from PKCE verifier ([`a26f4d0f3`](https://github.com/openclaw/openclaw/commit/a26f4d0f3e))

Fixed OAuth state separation in the Gemini browser sign-in flow. Previously the auth flow reused the PKCE code verifier value as the OAuth state parameter — a structural violation that defeated CSRF state separation. The fix introduces `generateOAuthState()` to create an independent 256-bit random hex state value, separate from the PKCE verifier. `buildAuthUrl()` now takes distinct `challenge` and `state` parameters. The insecure fallback in `parseCallbackInput()` that would accept verifier-derived values as state on error paths is removed. Callbacks must now present the exact state value generated at auth URL creation time; any mismatch throws `"OAuth state mismatch - please try again"`. This closes the auth bypass vector that existed when a CSRF attacker could supply the known verifier value as state.

**Addresses Audit 1 Claim 2** (missing CSRF protection in OAuth flow) and **Audit 1 Claim 8** (insufficient auth token validation). Thanks @eleqtrizit.

#### Align env key normalization in approval binding path ([`7eb094a00`](https://github.com/openclaw/openclaw/commit/7eb094a00d))

Fixed environment variable key normalization inconsistency between the approval request path and the execution path on Windows. On Windows, env var keys with special characters (e.g., `ProgramFiles(x86)`) were normalized differently in the approval binding phase versus the execution phase — causing the approval hash to not match at exec time, silently passing the binding check without a valid hash. The fix exports `normalizeHostOverrideEnvVarKey()` from `src/infra/host-env-security.ts` and switches `system-run-approval-binding` to use it instead of the generic `normalizeEnvVarKey()`. Adds a fail-closed validation in `matchSystemRunApprovalEnvHash()` (`src/infra/system-run-approval-binding.ts:160`): if execution env keys are present but the approval binding has no hash, the approval is rejected rather than silently passing. This prevents attackers from injecting unapproved environment overrides on Windows hosts with special-character env keys. Thanks @pgondhi987.

**Addresses Audit 1 Claim 5** (environment variable security and validation).

#### Close fail-open bypass in exec script preflight ([`8aceaf5d0`](https://github.com/openclaw/openclaw/commit/8aceaf5d0f))

Fixed a fail-open bypass in the Python/Node script preflight validation. The prior implementation used simplified regex to detect `python`/`node` invocations; complex invocations (e.g., wrapping `env` with option flags like `--chdir`, `--unset`, or multi-argument argv) could reach the execution path without triggering preflight checks. The fix adds a complete token parser for `env(1)` invocations via `PREFLIGHT_ENV_OPTIONS_WITH_VALUES` (a set of options that consume a following argument), `isShellEnvAssignmentToken()`, and `isEnvExecutableToken()` to correctly identify the actual interpreter after env arguments are stripped. Complex interpreter invocations that cannot be statically resolved now fail-closed with an explicit error requiring a direct `python <file>.py` or `node <file>.js` command form. The commit adds 798 insertions to `src/agents/bash-tools.exec.ts` and 706 to the preflight test suite. Thanks @pgondhi987 and @drobison.

**New security control.** Exec script preflight is now fail-closed for complex interpreter invocations.

#### Block SSRF redirect bypass via real-time route interception ([`462b4020b`](https://github.com/openclaw/openclaw/commit/462b4020bc))

Fixed a SSRF bypass in the browser tool's navigation guard. The prior implementation checked destination URLs post-hoc after `page.goto()` completed — meaning a redirect chain that started at a public URL and pivoted to an internal IP could reach the internal host before the check fired. The fix installs a Playwright `page.route()` handler before `page.goto()` so navigations to private or internal IPs are intercepted and aborted mid-redirect instead of after the request reaches the internal host. Blocked targets are permanently marked per-page so subsequent tool calls that attempt the same internal URL are immediately rejected without re-evaluating the route. This closes the redirect-based SSRF bypass on the Playwright browser tool path. Thanks @pgondhi987.

**New security control.** SSRF protection is now enforced at the routing layer, not post-navigation.

#### Evaluate shell wrapper inline commands against allowlist ([`8d81e76f2`](https://github.com/openclaw/openclaw/commit/8d81e76f23))

Fixed an allowlist bypass when a skill constructs a compound command via a shell wrapper (e.g., `sh -c "cat SKILL.md && gog-wrapper calendar events"`). The prior implementation compared `/bin/sh` against the allowlist instead of the actual target binaries, causing the entire compound command to be silently rejected or incorrectly allowed based on the shell wrapper alone. The fix adds recursive inline command evaluation in `src/infra/exec-approvals-allowlist.ts`: chain operators (`&&`, `||`, `;`) in the `-c` payload are detected, each sub-command is parsed independently via `analyzeShellCommand()`, and every sub-command is evaluated separately against the allowlist with accurate `segmentSatisfiedBy` tracking. Recursion is limited to depth 3 to prevent abuse, and is skipped on Windows (no POSIX shell semantics). Thanks @biao.

**New security control.** Shell wrapper inline commands are now individually evaluated against the exec allowlist. Closes #57377.
