#### Keep service-managed children attached (#38463, thanks @spirittechie) ([`e20f44509`](https://github.com/openclaw/openclaw/commit/e20f44509))

When OpenClaw runs as a system service (systemd on Linux, launchd on macOS), child processes previously used `detached: true` on all non-Windows platforms. Detached children could outlive the gateway, leaving orphaned worker processes when the service manager restarted the gateway unit. New helper `isServiceManagedRuntime()` checks for the `OPENCLAW_SERVICE_MARKER` environment variable, which service-mode launch helpers inject at startup. `createChildAdapter()` now sets `useDetached = process.platform !== "win32" && !isServiceManagedRuntime()`: in service-managed mode children stay attached, so systemd/launchd controls the full process group on restart or stop — no orphaned workers remain after a service restart.

#### Harden service-mode stale process cleanup (#38463, thanks @spirittechie) ([`cc7e61612`](https://github.com/openclaw/openclaw/commit/cc7e61612))

When the gateway restarts it kills any stale process holding the gateway port before binding. Previously the cleanup returned immediately after SIGKILL, leaving a race window where a rapid service restart could hit `EADDRINUSE` if the kernel had not yet released the port. `parsePidsFromLsofOutput()` now runs in a polling loop after SIGKILL: it re-queries lsof every `PORT_FREE_POLL_INTERVAL_MS` (50 ms) until lsof reports no PIDs holding the port, or until `PORT_FREE_TIMEOUT_MS` (2000 ms) elapses. Only then does cleanup return, ensuring the port is actually free before the new gateway instance tries to bind.

#### Avoid freezing Windows PATH in task scripts (#39139, thanks @Narcooo) ([`b9dd6e99b`](https://github.com/openclaw/openclaw/commit/b9dd6e99b))

On Windows, `buildCommonServiceEnvironment()` previously set `PATH` to `SharedServiceEnvironmentFields.minimalPath` — a snapshot of the PATH captured at install time. Scheduled Task launches would run with this frozen PATH, ignoring any post-install tool updates (new Node versions, updated binaries, etc.). `SharedServiceEnvironmentFields.minimalPath` is now typed as `string | undefined`. `buildCommonServiceEnvironment()` only sets `PATH` when `minimalPath` is defined; for Windows Scheduled Tasks, `buildSharedServiceEnvironment()` passes `minimalPath: undefined`, so the task inherits the current live system PATH instead of a stale install-time snapshot.

#### Degrade gracefully on missing env vars (#39050, thanks @akz142857) ([`764971235`](https://github.com/openclaw/openclaw/commit/764971235))

Config loading previously threw or produced an empty string when an `${ENV_VAR}` reference had no matching environment variable — causing startup failures when optional credentials were absent. Two new types are added to `src/config/env-substitution.ts`: `EnvSubstitutionWarning` (carries the missing var name and config path) and `SubstituteOptions` (holds an optional `onMissing?` callback). `substituteString()` gains an `opts?: SubstituteOptions` parameter; when `onMissing` is provided, missing variables fire the callback and preserve the original `${VAR}` placeholder rather than throwing. Config loading now passes `onMissing` to collect warnings, which are emitted at `warn` level, allowing the system to start with partial configuration instead of refusing to boot.

#### Share discord allowlist resolver scaffolding ([`dfe8cd028`](https://github.com/openclaw/openclaw/commit/dfe8cd028))

`src/discord/resolve-channels.ts` and `src/discord/resolve-users.ts` previously duplicated the allowlist resolution logic (guild lookup, empty-list handling, case-insensitive matching). Both files now import from the new shared `src/discord/resolve-allowlist-common.ts` module. Centralizing the allowlist resolver reduces duplication and ensures consistent behavior across channel and user allowlist resolution in Discord.

#### Centralize gateway auth env credential readers ([`a91731a83`](https://github.com/openclaw/openclaw/commit/a91731a83))

Gateway credential reading (Discord token, Slack token, Telegram token, and others) was previously scattered across 8 files, each independently reading env vars and implementing its own check for unresolved `${VAR}` placeholder strings. This commit introduces `ResolvedGatewayCredentials`, `GatewaySecretRefUnavailableError`, and `trimCredentialToUndefined()` in `src/gateway/credentials.ts`. `trimCredentialToUndefined()` rejects unresolved env var placeholders — values that still look like `${OPENCLAW_GATEWAY_TOKEN}` — by returning `undefined` rather than passing the placeholder string as a literal credential. All 8 scattered callers are refactored to use the centralized reader, making the rejection behavior consistent across all auth channels.

#### Share onboarding account id resolution prelude ([`f0b05869f`](https://github.com/openclaw/openclaw/commit/f0b05869f))

Onboarding flows for 17 channel providers each duplicated the account ID resolution prologue: reading the configured account, resolving defaults, and checking for a missing ID. Refactored into a shared prelude consumed by all 17 onboarding entry points, consolidating secret-input handling across extensions and keeping each onboarding command thin.

#### Resolve status SecretRefs with provider-safe env checks ([`330579ef9`](https://github.com/openclaw/openclaw/commit/330579ef9))

`inspectTokenValue()` in `src/telegram/account-inspect.ts` now uses `coerceSecretRef()` to normalize the token value and new helper `canResolveEnvSecretRefInReadOnlyPath()` to check whether the configured secret provider supports read-only env resolution. The helper calls `resolveDefaultSecretProviderAlias()` to identify the active provider and returns `true` only when it is the `env` provider. If the provider cannot resolve the ref, inspection returns unavailable rather than attempting the read — preventing unresolved `${VAR}` placeholder strings from surfacing in status output or logs.
