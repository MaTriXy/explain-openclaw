#### Centralize node identity resolution ([`27519cf06`](https://github.com/openclaw/openclaw/commit/27519cf06))

Node identity resolution hardening. New helpers `listEffectivePairedDeviceRoles()` and `hasEffectivePairedDeviceRole()` centralize the logic for determining which roles a node holds based on its current pairing and token state. Previously, node matching logic was scattered across callers with inconsistent handling of partially-paired or token-rotated devices. The refactoring ensures all node role checks go through a single authoritative path, reducing the risk of stale or incorrect identity decisions in approval and routing contexts.

#### Pin runtime snapshot and drop ttl cache ([`168ab94ee`](https://github.com/openclaw/openclaw/commit/168ab94ee))

Config runtime snapshot pinning. The previous TTL-based config cache could expire mid-session, causing the gateway to re-read and re-validate config at unpredictable times — a potential source of transient policy inconsistencies. This commit replaces the TTL cache with persistent runtime snapshot pinning: the first successful config load is retained via `resetConfigRuntimeState()` and held until explicitly cleared. This makes the effective config policy stable for the duration of a session, preventing edge cases where an in-flight config reload could briefly alter security-relevant settings like exec approval thresholds or provider permissions.

#### Align same-chat approval routing ([`3ec000b99`](https://github.com/openclaw/openclaw/commit/3ec000b99))

Same-chat approval routing unified. The helper `hasApprovalTurnSourceRoute()` replaces a set of platform-specific routing checks that had diverged across Telegram, Discord, and other channels. Inconsistent per-platform checks created risk of approval-bypass edge cases where one channel's routing logic differed from another's in corner cases. The refactoring centralizes the check, ensuring consistent same-chat approval behavior regardless of messaging platform.

#### Disambiguate legacy mac node identities ([`f16c176a4`](https://github.com/openclaw/openclaw/commit/f16c176a4))

Legacy node identity disambiguation. Added `clientId` and `clientMode` tracking fields to `NodeSession` to distinguish between legacy macOS node identities that previously mapped to the same identity under certain pairing configurations. Without this disambiguation, a legacy mac node could be misidentified during approval routing, potentially allowing approval requests to be routed to the wrong device. The fix ensures each node session carries the context needed to resolve ambiguous identity scenarios correctly.

#### Make same-chat approvals work across channels ([`574d3c521`](https://github.com/openclaw/openclaw/commit/574d3c521))

Same-chat approval routing extended to all deliverable chat channels. The `hasApprovalTurnSourceRoute()` check introduced in `3ec000b995` was expanded to cover additional channel types where approvals need to be matched to their originating chat context. Channels that support outbound delivery but weren't previously covered could fail to route approvals back correctly, silently falling through to an incorrect approver or failing the approval entirely. This closes a routing gap affecting cross-channel approval flows.

The new centralized module is the single source of truth for approval authorization, making it harder to introduce channel-specific bypasses and easier to audit the approval gate.

#### Cache bundle MCP runtime per session ([`6477d783e`](https://github.com/openclaw/openclaw/commit/6477d783e))

Per-session MCP runtime caching implemented. The underlying caching mechanism for MCP bundle runtimes is restructured to key by session, ensuring that each session gets an isolated MCP runtime instance. This is the structural change that enables the fix in `f5f8ba6d35` — without per-session keying, any hardening of individual cache entries is insufficient because the cache sharing itself is the vulnerability. Per-session isolation prevents one session's MCP credentials from being visible to or reusable by another session.

#### Add SSE transport support for remote MCP servers ([`d89bfed5c`](https://github.com/openclaw/openclaw/commit/d89bfed5c))

The new `mcp-sse.ts` module introduces `EventSource`-based connection handling for remote MCP endpoints. This is a significant new attack surface: remote MCP connections introduce URL-based credential handling paths (bearer tokens, API keys in headers or query parameters) that the subsequent hardening commits (`2c6eb127d9`, `4e03d899b3`, `62d0e12155`, `32b7c00f90`) address. The feature enables connecting to hosted MCP services but requires careful credential management to avoid leakage.

#### Address review feedback — warn on dropped headers ([`bf8303370`](https://github.com/openclaw/openclaw/commit/bf8303370))

Explicit warning added in `mcp-sse.ts` when authentication headers are dropped during SSE transport connection (for example, during URL origin mismatch). Previously, dropped auth headers were silent — the connection would proceed without authentication and there would be no observable signal that the auth credential was discarded. Making the failure observable (via warning log) prevents operators from unknowingly running unauthenticated MCP connections while believing authentication is active.

#### Redact sensitive query params in invalid URL error reasons ([`2c6eb127d`](https://github.com/openclaw/openclaw/commit/2c6eb127d))

Information disclosure fix in MCP SSE error handling. When URL parsing fails, error messages previously included the raw invalid URL — which could contain embedded credentials in query parameters (`api_key`, `apikey`, `password`, `credential`, `secret`). This commit redacts those parameters before the URL appears in any error reason string, preventing credentials from leaking into logs or error output when a misconfigured MCP server URL is rejected. Addresses Audit 1 Claim 1 (plaintext OAuth token storage in logs).

#### Handle Headers instances in SSE fetch and redact invalid URLs ([`4e03d899b`](https://github.com/openclaw/openclaw/commit/4e03d899b))

MCP SSE credential redaction extended to `Headers` instance handling in SSE fetch operations. When SSE fetch receives a `Headers` instance rather than a plain object, the previous code path did not apply the same credential redaction that plain-object paths went through. This meant that auth headers passed via `Headers` instances could appear in error output without redaction. The fix ensures `Headers` instances are normalized and redacted consistently with other auth header formats. Addresses Audit 1 Claim 1.

#### User headers override SDK defaults & expand redaction list ([`62d0e1215`](https://github.com/openclaw/openclaw/commit/62d0e1215))

MCP credential redaction list expanded and user-header precedence fixed. The redaction list now covers a broader set of sensitive parameter names beyond the initial set. Separately, user-provided auth headers now correctly override SDK-injected default headers — a bug where SDK defaults took precedence over user headers meant that operators configuring custom auth could be silently using SDK-default (possibly empty) credentials instead. Both changes strengthen the secrets management layer for MCP authentication. Addresses Audit 1 Claim 1.

#### Apply SSE auth headers to initial GET, redact URL credentials, warn on malformed headers ([`32b7c00f9`](https://github.com/openclaw/openclaw/commit/32b7c00f9))

SSE authentication hardening: three gaps closed. (1) Auth headers were not applied to the initial SSE `GET` request — only subsequent messages carried auth, leaving the handshake unauthenticated. (2) URL credentials in error messages were not being redacted. (3) Malformed headers were silently discarded without warning. Taken together, these fixes close the credential handling gaps across the full SSE authentication flow, ensuring the connection is authenticated from the first request, credentials don't leak through error paths, and header configuration problems are surfaced to operators. Addresses Audit 1 Claim 1.

#### [codex] Harden clawhub plugin publishing and install ([`e13392404`](https://github.com/openclaw/openclaw/commit/e13392404))

Clawhub plugin publishing and installation hardened with cryptographic integrity verification. A new `plugin-package-contract` package performs SHA256 integrity checks on plugin packages before installation, and a new `temp-download.ts` module manages secure temporary file lifecycle during downloads (ensuring downloads are validated before being moved into place). Exec process hardening in `src/process/exec.ts` adds additional guardrails to plugin execution paths. Together, these changes prevent tampered or malicious plugin packages from being silently installed — directly addressing Audit 2 Claim 5 (self-approving agent, no RBAC on plugin installs) by adding a cryptographic verification layer that cannot be bypassed by agent self-approval.
